# cookbook/ — 도메인별 상세 진단 가이드 (Tier-2)

## 파일 구조

```
brain/cookbook/
├── README.md                  # 이 파일 — 구조 규칙
├── {domain-id}.md             # 도메인별 상세 가이드
│   ├── 도메인 컨텍스트        # "왜 이런 스펙인가" — 배경 지식
│   ├── 데이터 접근            # SQL 템플릿, Operation API
│   └── 과거 사례              # 이슈별 요약 + 판정(스펙/버그)
```

## 콘텐츠 경계

| 내용 | 목적지 | 이유 |
|------|--------|------|
| "뭘 해라" (진단 절차, 확인 순서) | `COOKBOOK.md` Tier-1 진단 체크리스트 | 이슈 접수 시 빠르게 스캔 |
| "왜 이렇게 되어있나" (법적 근거, 비즈니스 배경, 설계 의도) | `cookbook/{domain}.md` 도메인 컨텍스트 | 코드 변경과 무관한 배경 지식 |
| "이전에 이런 일이 있었다" (이슈 요약, 판정) | `cookbook/{domain}.md` 과거 사례 | 도메인별 상세 사례 축적 |
| SQL 템플릿, Operation API | `cookbook/{domain}.md` 데이터 접근 | 조사 도구 레퍼런스 |
| 한 줄 용어 매핑 (사용자 표현 → 시스템 용어) | `domain-map.ttl` `g:*` 블록 | 라우팅 시 자동 매칭 |
| 스펙 자체 (코드 동작 상세) | 각 서브모듈 repo | 단일 소스 원칙 — 이 repo에 복사하면 부패 |

## 도메인 컨텍스트 작성 규칙

### 목적

"왜"에 집중한다. 스펙 복사가 아니라, 그 스펙이 존재하는 배경과 맥락을 기록한다.

- 법적 근거: 근로기준법, 소득세법 등 법령이 스펙에 직접 영향을 미치는 경우
- 비즈니스 배경: 고객사 운영 패턴, HR 업계 관행이 설계에 반영된 경우
- 자주 혼동되는 것들: 온콜 조사 중 실제로 혼동했던 개념/용어
- 구현 특이사항: 코드를 읽어도 의도가 드러나지 않는 설계 결정

### 분량

도메인당 50-100줄. 핵심만 기록하고, 상세는 아카이브 노트나 서브모듈 코드 링크로 대체한다.

### 구조

```markdown
## 도메인 컨텍스트

### 이 도메인이 하는 일
### 핵심 개념
### 주요 흐름
### 비즈니스 규칙
### 자주 혼동되는 것들
### 구현 특이사항

---
```

`---` 구분선 아래에 기존 "데이터 접근", "과거 사례" 섹션이 이어진다.

### 갱신 시점

- `ops-learn`의 S8 시그널에 의해 갱신된다
- 주 갱신 대상: "구현 특이사항", "자주 혼동되는 것들"
- 새 이슈 조사에서 "왜 이런 스펙인지"가 밝혀질 때마다 추가
