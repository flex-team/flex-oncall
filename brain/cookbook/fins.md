# 비용관리 (Expense Management / Fins) — 상세 진단 가이드

> COOKBOOK.md Tier-1에서 참조되는 상세 SQL 템플릿과 과거 사례 모음

## 도메인 컨텍스트

### 이 도메인이 하는 일

법인카드 관리, 경비 정산, 세금계산서 연동(국세청/홈택스), 비용정책 관리, 분개 처리를 담당한다. `flex-fins-backend` 에서 관리하며, 외부 금융 데이터 소스(카드사, 국세청 등)와의 연동이 핵심이다.

### 핵심 개념

- **증빙(`spending`)**: 카드 결제 증빙(자동 수집)과 수동 추가 증빙(ETC)으로 나뉜다. 카드 증빙은 `transactedTime`이 있지만, 수동 증빙은 null이다.
- **데이터 동기화(`data-synchronization`)**: 카드사/국세청 등 외부 소스에서 데이터를 수집하는 과정. 금융사마다 수집 가능 기간과 API가 다르다.
- **비용정책(`policy`)**: 시간 정책(RANGE), 금액 정책 등 회사별로 설정하는 지출 규칙. 정책 생성 시점 이전 증빙에는 위반 판정이 적용되지 않는다.
- **스크래핑**: 공동인증서(crosscert) 기반으로 홈택스/카드사 데이터를 수집하는 방식.

### 주요 흐름

1. **카드 데이터 수집**: 카드사 API → 승인/매입 데이터 수집 → 가맹점 정보 매칭 → spending 생성
2. **세금계산서 연동**: 국세청 API → 최근 12개월까지 수집 → 어드민쉘 수동 동기화 가능
3. **비용정책 평가**: 증빙 생성/변경 시 활성 정책 평가 → 위반 여부 판정

### 비즈니스 규칙

- **수집 기간 제한**: 국세청 세금계산서는 최근 12개월까지. 카드 데이터는 승인/매입 API별로 조회 가능 기간이 상이하여, 특정 기간 이전은 별도 코드 작업이 필요하다.
- **부분 취소 처리**: 카드사마다 부분 취소 데이터 전달 방식이 다르다. 하나카드는 취소금액 필드가 별도로 있어야 하는데 미사용하는 버그가 있었다(CI-4071).
- **증빙 담당자(assignee) resolve 시 퇴사자 처리**: 영수증 생성 시점에 담당자를 resolve할 때, 해당 시점 기준 휴직/퇴사자는 담당자에서 제외한다. 단, 영수증 발생 당시 재직 중이었던 구성원이 이후 퇴사한 경우에는 담당자에 남되 "퇴직"으로 표기한다. 거래일자 시점이 아닌 **현재 시점** 기준으로 표기한다. [CI-4375]

### 자주 혼동되는 것들

- **수동 증빙 vs 카드 증빙의 정책 평가**: 수동 증빙은 `transactedTime=null`이라 시간 정책(RANGE) 평가에서 무조건 FAIL. 카드 증빙은 `transactedTime`이 존재하여 정상 평가. 수동 증빙의 시간 정책 위반은 버그(EP팀 수정 예정).
- **연동 상태 vs 수집 범위**: 연동이 완료되어 있어도 수집 가능 기간을 초과하면 과거 데이터를 가져올 수 없다. 연동 상태와 수집 범위는 별개 개념.

### 구현 특이사항

- **어드민쉘 수동 동기화**: 수집 범위 내의 과거 데이터는 어드민쉘에서 수동으로 동기화 실행 가능. 범위 밖은 코드 수정이 필요하므로 담당 개발자에게 요청.
- **독립 서비스**: `flex-fins-backend`는 별도 서비스 + 별도 모듈 구조. 다른 flex 백엔드와 API 경계가 분리되어 있다.
- **Vespa 인덱스 (`fins_spending_entire_v1`)**: 영수증 > 제출 화면의 데이터 소스. 전자결재 상태 변경 시 impact → fins 구간에서 Vespa 인덱스가 갱신되어야 하며, 실패 시 화면 표시 불일치 발생. 수동 재동기화 API: `POST /api/operation/v3/impact/electronic-approval/customers/{customerId}/documents/{documentId}/publish` [CI-4332]
- **`spending_evidence_electronic_approval_document` 테이블**: 전자결재 문서 상태를 DB에 저장하는 테이블. 전자결재 반려 이벤트 Kafka 수신 실패 시 `status = IN_PROGRESS` 잔존 → 영수증 유효성 검증 실패("진행중인 문서가 있어" 오류) [CI-4312]
- **Bullseye(`fins_spending_entire_v1`) 영수증 조회 구조**: 지출결의서 수정 팝업에서 FE가 `size` 파라미터로 최대 조회 건수를 지정한다(현재 1000). Bullseye는 MatrixQL 기반으로 `continuation` 토큰을 지원하지 않아 단일 `limit` 조회만 가능 — 비용관리 출시 초기 결정. 영수증 건수가 limit을 초과하면 이전 영수증이 잘려 보이지 않는다 [CI-4334]

## 과거 사례

- **국세청 세금계산서 소급 연동**: 최근 12개월까지 수집 가능. 어드민쉘 수동 동기화로 즉시 처리. 카드 데이터는 승인/매입 API별 조회 기간이 상이하여 특정 기간 이전은 별도 코드 작업 필요 — **운영 대응** [CI-4179]
- **카드 부분 취소가 전체 취소로 표시**: 하나카드 부분취소 시 취소금액 필드 미사용 버그. hotfix + 영수증 재생성 — **버그** [CI-4071]
- **수동 증빙 시간 정책 위반 오표시**: 수동 추가 증빙(ETC spending)의 `transactedTime=null`이 RANGE 평가에서 FAIL 처리되어 위반으로 표시. 카드 증빙은 영향 없음. EP팀 수정 예정 — **버그** [CI-4229]
- **지출결의 반려 후 영수증 > 제출 화면 진행중 표시**: Vespa 인덱스(`fins_spending_entire_v1`)에 전자결재 반려 상태가 미반영(impact→fins 동기화 실패). operation API로 수동 동기화 완료 — **운영 대응** [CI-4332]
- **지출결의 반려 후 재작성 시 영수증 소실**: `spending_evidence_electronic_approval_document.status = IN_PROGRESS` 잔존으로 "진행중인 문서가 있어" 오류 발생. 동일 고객사(무하유) 반복 발생, Kafka 이벤트 소비 실패 근본 원인 — **운영 대응** [CI-4312]
- **지출결의 수정 팝업 이전 영수증 미표시**: 특정 날짜 이전 영수증이 지출결의 수정 팝업에서 보이지 않음. FE size=1000 limit 초과(해당 사용자 영수증 500건+). FE PR #1986으로 size 1000→ 핫픽스 배포. Bullseye continuation 미지원으로 운영 대응 불가 — **버그(FE)** [CI-4334]
- **카드 사용 내역 엑셀 다운로드 시 퇴사자 정보 노출**: 카드 담당자 조직 설정 후 증빙 엑셀 다운로드 시 퇴사자 포함. Bullseye 조회 경로가 부서 ID 기반 필터만 사용, `excludeResignedUser` 미적용. 스펙상 영수증 생성 시점 퇴사자는 제외되어야 함 — **버그** [CI-4375]
- **법인카드 해외결제 청구 금액 불일치**: 승인→매입 2단계 수집 구조의 시차. 승인 데이터(매시간)는 임시 환율, 매입 데이터(하루 1회)는 확정 환율. 매입 전 조회 시 승인금액 표시. Notion 스크래핑 정책에도 명시된 스펙 — **스펙** [CI-4401]

## SQL 템플릿

### 영수증 건수 조회 (사용자별)

```sql
-- 지출결의 수정 팝업 영수증 건수 초과 여부 확인
-- Bullseye limit(1000) 초과 시 이전 영수증 잘림
SELECT
  COUNT(*) AS total_count,
  SUM(CASE WHEN created_at >= DATE_SUB(NOW(), INTERVAL 30 DAY) THEN 1 ELSE 0 END) AS recent_30d_count
FROM flex_fins.spending s
WHERE s.customer_id = ?
  AND s.member_id = ?
  AND s.deleted_at IS NULL;
```

> `total_count`가 1000 초과이면 F4 확정.
