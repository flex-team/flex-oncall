### 감사로그 미기록 원인

감사로그에서 조회되지 않는 것은 **전자계약 서식 삭제에 대한 감사 로깅이 구현되어 있지 않기 때문**이다 (스펙)[^5].

- `customer_digicon_template` 테이블에 `deleted_by`, `created_by`, `updated_by` 컬럼 없음[^5]
- 삭제 시 이벤트 발행(Kafka, ApplicationEvent) 없음[^7]
- 삭제 시 별도 audit 테이블 기록 없음[^7]

### 스펙 vs 버그 판별

- **감사로그 미기록**: 스펙 (미구현) — 전자계약 도메인에 감사 로깅 자체가 설계되지 않음
- **서식 삭제 자체**: 문의자 본인이 수행한 정상 동작. 단, 고객이 "몇번이나 삭제되서 계속 만들고있다"고 표현한 점에서 **의도치 않은 삭제일 가능성**(UX 문제)이 있음

## 해결

### CS 답변
고객에게 삭제 수행자가 본인(yelee@nobland.com) 계정임을 안내한다.

- access log 기준으로 2026-03-19 13:36~16:36 KST 사이에 14건의 서식 삭제 API 호출이 확인됨
- 모든 호출이 동일 계정(yelee@nobland.com), 동일 IP(222.107.107.120)에서 발생
- 서식은 이미 어드민쉘로 복구 완료

### 추가 확인 필요 사항
고객이 "몇번이나 삭제되서 계속 만들고있다"고 말한 점에서, **의도치 않은 삭제**인지 확인이 필요하다:
1. 고객이 실제로 삭제를 의도했는지 (다른 서식을 삭제하려다 잘못 삭제?)
2. UI에서 삭제 확인 다이얼로그가 있는지 (UX 개선 여지)
3. 서식 갯수 제한이 있는지 — 고객이 직접 질문함[^4]

## 다음에 같은 문의가 오면
1. **access log 검색**: OpenSearch `flex-app.be-access-*` 인덱스에서 `json.ipath: "digicon/templates"` AND `json.method: "DELETE"` AND `json.authentication.customerId: {고객ID}` 로 검색
2. **삭제자 확인**: `json.authentication.email`, `json.authentication.userId` 필드로 삭제 수행자 특정
3. **감사로그 없음 안내**: 전자계약 서식 삭제는 감사로그에 기록되지 않음 (미구현). access log가 유일한 추적 수단
4. **복구 필요 시**: 어드민쉘 > 전자계약 > 삭제된 템플릿 복구 화면에서 복구 가능
