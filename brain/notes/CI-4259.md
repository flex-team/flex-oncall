## 원인 분석

**calendar-backend가 all-day event를 UTC 자정으로 저장하면서, KST 등 UTC+ 타임존 조회 시 전날 이벤트가 당일 결과에 포함되는 구조적 버그**

### 근본 원인 (코드 4곳에서 확인)

1. **all-day event는 UTC 자정으로 저장**[^21]:
   ```kotlin
   // CalendarEvent2.kt:31-33
   val startAt: LocalDateTime, // DATE인 경우 UTC로 자정
   val endAt: LocalDateTime,   // DATE인 경우 UTC로 자정
   ```

2. **attendee 인덱스 컬럼에 동일 값 저장**[^22]:
   ```kotlin
   // CalendarEventJdbcMapper.kt:66-67
   indexFrom = event.startAt,
   indexTo = event.endAt,
   ```

3. **조회 시 KST → UTC 변환 후 비교**[^23]:
   ```kotlin
   // CalendarEventQueryUseCase.kt:89-90
   startAfter = zonedDateTimeRange.from.withZoneSameInstant(ZoneOffset.UTC).toLocalDateTime(),
   endBefore = zonedDateTimeRange.to.withZoneSameInstant(ZoneOffset.UTC).toLocalDateTime(),
   ```

4. **SQL overlap 비교**[^24]:
   ```sql
   -- CalendarEventAttendeeJdbcRepository.kt:75-76
   AND index_to > :startDateTime
   AND index_from < :endDateTime
   ```

### 버그 메커니즘

3/30 all-day 휴가가 3/31 KST 조회에 포함되는 과정:

```
3/30 all-day event DB 저장값:
  index_from = 2026-03-30T00:00 (UTC 자정)
  index_to   = 2026-03-31T00:00 (UTC 자정)

3/31 KST 조회 파라미터 (CalendarEventQueryUseCase에서 UTC 변환):
  startDateTime = 2026-03-30T15:00 (3/31 00:00 KST → UTC)
  endDateTime   = 2026-03-31T15:00 (4/1 00:00 KST → UTC)

SQL 비교:
  index_to(3/31T00:00) > startDateTime(3/30T15:00) → TRUE
  index_from(3/30T00:00) < endDateTime(3/31T15:00) → TRUE
  → 전날 이벤트가 당일 결과에 포함됨
```

> 💡 **핵심**: all-day event의 날짜 범위(UTC 자정~자정)와 KST 쿼리 범위(UTC 15:00~15:00) 사이에 **9시간 overlap 구간**(15:00~00:00 UTC)이 발생. UTC+ 타임존에서는 항상 전날 all-day event가 당일 조회에 포함됨.

### 스펙 vs 버그 판별: **버그**

- all-day event는 특정 "날짜"를 나타내므로 타임존과 무관하게 **날짜 기준으로 비교**해야 함
- UTC datetime 기반 overlap 비교는 all-day event에 적합하지 않은 로직

## 해결안

### 근본 해결

calendar-backend의 event 조회 시 **all-day event(dateType=DATE)는 날짜 기준으로 비교**하도록 수정:

```sql
-- 현재 (datetime 기반, 타임존 문제)
AND index_to > :startDateTime AND index_from < :endDateTime

-- 수정안: dateType=DATE인 경우 날짜 비교로 분기
AND (
  (date_type = 'DATE' AND start_date >= :queryStartDate AND start_date < :queryEndDate)
  OR
  (date_type != 'DATE' AND index_to > :startDateTime AND index_from < :endDateTime)
)
```

또는 all-day event 저장 시 UTC 자정이 아닌 **요청 타임존의 자정**으로 저장하는 방식도 가능.

### 즉시 대응

현재 구조에서는 데이터 패치 불가 (조회 로직 문제이므로). 코드 수정이 필요.
