# CI-4259: 모바일앱 일정에서 휴가 이벤트 2건 중복 표시

> **상태**: 원인 특정 — 2026-04-02

## 증상

- **문제 정의**: 당일 휴가 1건을 사용한 구성원의 모바일앱 일정에 휴가 이벤트가 2건 표시됨
- **회사**: 오로라월드(주) (Customer ID: 56765)
- **요청자**: 유지원 (CS팀 접수) / 고객사 문의자: 김정원 (jungwon.kim@auroraworld.com)[^1]
- **대상자**: 이준호 (junholee@auroraworld.com, userId: 229504) 및 다른 구성원 복수[^2]
- **영향 범위**: 오로라월드(주) 내 복수 구성원 (구성원 문의가 계속 들어오는 상황)[^3]. `calibrateDateRange()` 는 전체 고객사에 적용되므로 잠재적으로 **전체 고객사**에 영향
- **문제 시점**: 2026-03-31
- 문의 내용:
  1. 당일 휴가를 사용하지 않은 것으로 보이는데 모바일앱 일정에 휴가 일정이 표시됨
  2. 확인 결과, 해당 구성원은 실제로 당일 휴가 1건을 사용했으나 일정 탭에 **2건**이 표시됨[^4]

- **결론**: 클라이언트 렌더링 문제가 아닌 **서버 응답 자체에서 이벤트가 중복 생성**되고 있는 상황

| 1 | 일반 이벤트 + 반복 이벤트 중복 | access log 확인 | ❌ 소거 — 2건 모두 일반 이벤트, 별개 eventId |
| 2 | 날짜 범위 매칭 시 타임존 변환 버그 | access log + DB 대조 | ❌ 소거 — 3/31 이벤트 자체가 DB에 없는 유령 |

### 조사 과정 (2026-04-02)

#### 1단계: Calendar API 응답 확인

Kibana access log 문서[^7]에서 calendar-api의 응답을 직접 확인:

> 📊 **calendar API 응답** — `GET /api/v2/calendar/calendars/{calendarId}/events`
>
> 요청 범위: `dateTimeMin=2026-03-31T00:00+09:00, dateTimeMaxExclusive=2026-04-01T00:00+09:00` (3/31 하루)
> calendar API가 동일 calendarId·동일 userIdHash에 대해 **2건의 TIME_OFF 이벤트**를 반환:

| # | eventId | startAt (date) | endAtExclusive (date) | 의미 |
|---|---------|---------------|----------------------|------|
| 1 | `01kmpbdpc75h9gygstnqntqx5e` | 2026-03-30 | 2026-03-31 | 3/30 하루 이벤트 |
| 2 | `01kmpbdpc75h9gygstnqntqx5h` | **2026-03-31** | 2026-04-01 | **3/31 하루 이벤트 — DB에 대응 기록 없음** |

두 eventId가 매우 유사(`...5e`, `...5h`)하여 동시에 생성된 것으로 판단됨

#### 2단계: DB 대조 — 실제 휴가 기록은 3/30 1건


→ 3개 테이블 모두 **3/30 1건만 존재**. 3/31 레코드 없음[^8]

#### 3단계: 코드 추적 — `calibrateDateRange()`

> 💡 **판단 근거**: DB에 3/30만 있는데 calendar에 3/30+3/31이 존재 → calendar event 생성 과정에서 3/31이 추가됨
> → calendar sync consumer의 `calibrateDateRange()` 에서 `issueDateTo.plusDays(1)` 확인[^9]
> → 이로 인해 sync 범위가 3/30→3/31로 확장됨

calendar sync 흐름:

```
1. 휴가 등록 (3/30) → Kafka EVENT_TIME_OFF_CREATED
2. FlexCalendarTimeOffEventConsumer → COMMAND_CALENDAR_SYNC 발행
   (issueDateFrom=3/30, issueDateTo=3/30)
3. FlexCalendarSyncEventConsumer.consumeMessage()
   → calibrateDateRange() → issueDateTo = 3/30.plusDays(1) = 3/31  ← 여기서 확장
4. FlexCalendarSyncEventConverter.convertPerCustomer()
   → filteredDayResults = dayResults.filter { it.date in 3/30..3/31 }  ← 3/31 포함
5. collectTimeOffTimeBlocks() → 3/30, 3/31 각각에서 TIME_OFF 블록 수집
6. calendar event 2건 생성
```

<summary>📋 코드 위치 상세</summary>

| 파일 | 라인 | 역할 |
|------|------|------|
| `flex-timetracking-backend` > `time-tracking/consumer/.../CalendarSyncEventExtensions.kt` | 8-16 | `calibrateDateRange()` — `issueDateTo.plusDays(1)` |
| `flex-timetracking-backend` > `time-tracking/consumer/.../FlexCalendarSyncEventConsumer.kt` | 46 | `calibrateDateRange()` 호출 지점 |
| `flex-timetracking-backend` > `time-tracking/service/.../FlexCalendarSyncEventConverter.kt` | 76 | `filteredDayResults` 필터 (`it.date in issueDateFrom..issueDateTo`) |
| `flex-timetracking-backend` > `time-tracking/service/.../FlexCalendarEventEnricher.kt` | - | `makeTimestampRange()` — issueDateTo+1일로 dayResults 조회 범위 추가 확장 |
| `flex-timetracking-backend` > `time-tracking/service/.../CalendarTimeBlockCollector.kt` | - | `collectTimeOffTimeBlocks()` — dayResult에서 TIME_OFF 블록 수집 |

</details>

### 미확정 간극

코드 경로는 확인했으나, **3/31의 dayResult에 실제로 TIME_OFF 블록이 존재하는지**는 직접 확인하지 못했다. `collectTimeOffTimeBlocks()` 는 dayResult의 timeBlocks 중 `category === TIME_OFF` 인 것만 수집하므로, 3/31에 TIME_OFF 블록이 없으면 event가 생성되지 않는다. 다만:
- calendar API에 3/31 이벤트가 실제로 존재하는 것은 사실[^7]
- `calibrateDateRange()` 외에 이 이벤트를 생성할 수 있는 다른 경로는 발견되지 않음
- [추론: work schedule lookup 서비스가 3/30 휴가의 영향을 3/31 dayResult에도 반영하는 것으로 추정. 귀속일이 다른 경우 등]
