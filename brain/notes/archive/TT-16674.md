## 현재까지 파악된 내용

### 권한 추가/변경 프로세스
권한 추가 시 AuthN/Z 플랫폼 팀과 협업하는 프로세스를 따른다[^2]:

1. **권한 컨셉 구성** (PM/PE) → AuthN/Z 팀과 계획 공유
2. **디자인 리뷰** (PD → 스쿼드 PM + 디자인 플랫폼 팀)
3. **Backend 개발** — PE가 권한 평가 코드 반영 → AuthN/Z 팀 코드 리뷰 → 권한 설정 데이터 마이그레이션
4. **Frontend** — 번역어 추가, UI 분기
5. **Prod 배포** — DEV → QA → Prod 순서, 마이그레이션은 AuthN/Z 팀 담당 (기본 익일, 최대 2~3일)

### 신규 API 전환 가이드 (권한 평가 방식)
`DomainAuthorityModel`을 상속받은 도메인 enum을 만들고, `AccessCheckApiService`를 사용하여 권한을 평가한다[^3]:

```kotlin
enum class WorkClockAuthority(
    override val objectKey: String,
    override val action: FlexObjectAction,
) : DomainAuthorityModel {
    WORK_CLOCK_REAL_TIME_RECORD_READ("work_clock_real_time_record", FlexObjectAction.READ),
}
```

```kotlin
accessCheckApiService.executeIfGrant(
    user = actor,
    authority = WorkClockAuthority.WORK_CLOCK_REAL_TIME_RECORD_READ,
    requestDomain = RequestDomainModel("work-clock", "getWorkClockRecords"),
) { ... }
```

### 분기 대상 API 분석
코드베이스 탐색 결과, 분기가 필요한 API는 work-clock 모듈에 한정된다[^4]:

| API | 현재 동작 | 변경 |
|-----|----------|------|
| `GET /api/v2/.../work-clock/users` | `WORK_SCHEDULE_READ` 권한으로 전체 기록 반환 | 새 권한 없으면 onGoing 기록만 필터 |
| `GET /api/v2/.../work-clock/users/{userIdHash}` | 동일 | 동일 |

분기 불필요 API:
- `GET .../current-status` → 이미 본인만 조회 가능[^5]
- work-schedule V3 API → 진행 중 위젯 기록 미포함[^6]
- tracking-search 대시보드 → 실시간 기록 미포함[^7]

### 분기 로직
```
WORK_SCHEDULE_READ 없음 → 403 (기존과 동일)
WORK_SCHEDULE_READ 있음 + 새 권한 없음 → onGoing == true 레코드팩만
WORK_SCHEDULE_READ 있음 + 새 권한 있음 → 전체 레코드팩
```

### onGoing 판정 기준
`UserWorkClockEventPackModel.isOngoing()` 함수 기준[^8]:
- 근무 시작부터 20시간(`CONTINUOUS_WORK_START_DURATION`) 이내
- 종료 기록이 없거나, 사용자가 직접 등록한 종료가 아닌 경우
