## 원인 분석

### 조사 과정

> 💡 **판단 근거**: Sentry traceId[^4] → access log에서 `UPER_400_011` 에러 확인[^1] → 에러 코드 매핑에서 `PERSONAL_EMAIL_INVALID` 확인[^2] → `UserPersonalValidatorImpl.kt:162`에서 `Rfc5322EmailAddressValidator.check()` 실패 확인[^3]

### 원인
- 구성원의 `personalEmail` 필드 값이 RFC 5322 이메일 형식 검증을 통과하지 못함[^2]
- personalEmail은 2026-03-18에 입력됨 (이전에는 null)[^7]. 입력 시 잘못된 형식으로 저장됨
- 대량변경 이력 없음[^8] — 직접 입력 시 형식 오류
- FE가 주소 변경 시 전체 개인정보를 번들로 전송 → 서버가 전체 필드를 검증 → 잘못된 personalEmail로 인해 주소 변경까지 차단[^3]

### 가설 목록

| # | 가설 | 상태 |
|---|------|------|
| 1 | `personalEmail`에 RFC 5322 위반 값이 저장되어 주소 변경 시 전체 폼 검증 실패 | ✅ 확정 — 고객 확인 완료[^9] |
| 2 | 대량변경에서 이메일 검증 우회하여 잘못된 값 입력 | ❌ 소거 — 동일 validator 사용[^5], 대량변경 이력 없음[^8] |

```kotlin
// UserPersonalValidatorImpl.kt:159-168
validatable.personalEmail
    ?.takeIf { it.isNotEmpty() }
    ?.also {
        if (!Rfc5322EmailAddressValidator.check(it)) {
            result.put(
                UserPersonalPermitColumn.PERSONAL_EMAIL,
                UserPersonalError.PERSONAL_EMAIL_INVALID,
            )
        }
    }
```

</details>

### 스펙 vs 버그 판별

**판단: 스펙** — 검증 로직은 정상 동작. 잘못된 형식의 personalEmail이 입력된 것이 원인.

## 해결
- CS에서 고객에게 개인 이메일 확인 요청[^9]
- 고객이 개인 이메일 문제 확인 후 수정[^9]
- 수정 후 주소 정상 변경 완료[^9]

## 발견한 스펙/제약

- `UserPersonalValidatorImpl`은 변경 대상 필드뿐 아니라 **전체 개인정보 필드를 검증**한다. `exchange()` 함수에서 변경하지 않는 필드는 기존 DB 값을 사용하여 validatable을 구성하므로, 기존 데이터에 잘못된 값이 있으면 다른 필드를 변경할 때도 차단된다[^5].

## 다음에 같은 문의가 오면

1. **먼저 확인**: access log에서 traceId로 에러 코드 확인 → `UPER_400_011`이면 personalEmail 검증 실패
2. **원인 판별**: 해당 구성원의 personalEmail 값을 확인 (FE 프로필 화면에서 "개인 이메일" 필드)
   - 이메일 형식이 올바르지 않으면 → 입력 오류
   - 이메일 형식이 올바르면 → 다른 UPER_400_* 에러 코드 확인
3. **조치**: 고객에게 개인 이메일을 올바른 형식으로 수정하도록 안내. 수정 후 원래 작업 재시도.
