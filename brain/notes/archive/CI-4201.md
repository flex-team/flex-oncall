# CI-4201: 구성원이 없는 조직이 종료 처리되지 않음

> **상태**: 해결 완료 — 2026-03-24
> **판정**: 스펙 (의도된 제약)

## 증상
- **문제 정의**: 예약발령으로 구성원이 모두 이동한 조직을 종료 처리할 수 없음 — "소속된 구성원에게 예약된 조직발령이 있다" 메시지 표시
- **회사**: 이삼오구 (Customer ID: 224251)[^1]
- **요청자**: ohyj@2359.co.kr (CS팀 @sieun 접수)[^1]
- **대상**: 해외영업지원팀 (조직)[^1]
- **영향 범위**: 해당 조직 (구성원이 모두 타조직으로 이동 완료된 상태)
- **문제 시점**: 4/13일자 예약발령 이후[^1]
- 문의 내용:
  1. 해외영업지원팀 구성원들이 북미사업팀 및 신설조직(해외영업팀, 해외사업지원팀)으로 조직이동 발령됨[^1]
  2. 발령 이후 해외영업지원팀에 구성원이 없는 상태인데 종료 처리가 불가[^1]
  3. 종료 시도 시 "소속된 구성원에게 예약된 조직발령이 있다"는 메시지가 표시됨[^1]
  4. 재현 조건: 예약된 조직발령이 있는 상태에서 구성원이 모두 다른 조직으로 이동된 조직을 종료 처리하려는 경우[^1]

---

## 원인 분석

**`validateWaitingPersonnelAppointmentForDepartmentUser`가 구성원의 미래 position 변경을 부서 구분 없이 체크하여, 다른 조직으로 "나가는" 예약발령도 블로킹하는 것은 의도된 보수적 제약이다.**

> 💡 **판단 근거**: 코드 분석만으로는 부서 필터 누락처럼 보이지만, CI-3748에서 동일 동작이 스펙으로 확인[^9]되었고, CORE-2013에서 역방향 제약(종료 예정 조직 → 발령 차단)도 설계[^10]되었으며, CORE-4623에서 이 제약의 UX 개선(자동 떼어내기 발령)이 별도로 진행[^11]된 점을 종합하면 의도된 설계로 판단된다.

### 스펙 vs 버그 판별

**스펙 (의도된 제약)**: "예약발령으로 구성원이 모두 이동 예정이어도, 발령이 실제 실행되기 전까지는 해당 조직의 기간을 변경할 수 없다"

- CI-3748: 동일 증상 → "예약 발령이 있으면 막는 것은 의도된 제약"으로 정리[^9]
- CORE-2013: 역방향도 설계됨 — 종료 예정 조직이면 해당 조직으로의 발령도 차단[^10]
- CORE-4623: 기존 제약이 불편 → "조직 종료 시 자동으로 떼어내기 발령" 개선 작업[^11]. 제약 자체를 버그로 뒤집은 것이 아니라 보완

### 검증 흐름 (3/24 기준)

```
해외영업지원팀 구성원들의 time series:
  - Segment A: 해외영업지원팀, begin=과거, end=4/12
  - Segment B: 북미사업팀/해외영업팀/해외사업지원팀, begin=4/13, end=MAX

validateWaitingPersonnelAppointmentForDepartmentUser 실행:
  ① getAllByDepartmentIdentities(해외영업지원팀, now, MAX)
     → Segment A 매칭 (3/24~4/12 구간 겹침) → 구성원 목록 반환
  ② getAllFrom(구성원들, tomorrow=3/25)
     → Segment B 반환 (타 부서의 4/13 발령) — 부서 필터 없음
  ③ filter { beginDateTime >= tomorrow }
     → 4/13 >= 3/25 → true
  ④ → DepartmentUserUsedForAppointmentReservationException (DEPA_400_017)
```

### validator 비교

|-----------|--------|----------|----------|
| `validateWaitingPersonnelAppointmentForDepartment` (57~82행) | `getAll(departmentIdentity, tomorrow, MAX)` | ✅ 있음 (해당 부서만) | ✅ |

### 5 Whys

1. 왜 조직 종료가 안 되는가? → `DEPA_400_017` 에러 발생[^3]
2. 왜 에러가 발생하는가? → `validateWaitingPersonnelAppointmentForDepartmentUser`에서 미래 발령 감지[^4]
3. 왜 미래 발령이 감지되는가? → `getAllFrom(users, tomorrow)`이 타 부서 발령도 반환[^5]
4. 왜 타 부서 발령을 반환하는가? → `findAllByUserIdInFrom`이 userId 기준으로만 조회[^6]
5. **근본 원인**: validator가 "나가는 발령"과 "들어오는 발령"을 구분하지 않는 로직 결함[^4]

### `DepartmentPersonnelAppointmentDetailValidator` 전체 코드

```kotlin
// 84~109행 — 버그가 있는 메서드
private fun validateWaitingPersonnelAppointmentForDepartmentUser(
    validatable: DepartmentValidatable,
): CoreValidatorResult<DepartmentPermitColumn, FlexException> {
    val result = CoreValidatorResult<DepartmentPermitColumn, FlexException>()

    // ① 현재~미래 시점에 이 조직에 소속된 구성원 조회
    val users = userPositionLookUpService.getAllByDepartmentIdentities(
        listOf(validatable.departmentIdentity!!), clock.instant(), TimeSeriesConstants.MAX
    )

    val zoneId = customerScopedTimeProvider.zoneId(validatable.customerIdentity)
    val tomorrow = clock.instant().atZone(zoneId).plusDays(1L).truncatedTo(ChronoUnit.DAYS).toInstant()

    // ② 해당 구성원들의 "모든 부서" 미래 position 조회 — 부서 필터 없음!
    if (users.isNotEmpty() &&
        userPositionLookUpService.getAllFrom(
                userIdentities = users,
                fromSearchTime = tomorrow,
            )
            .filter { it.beginDateTime >= tomorrow }
            .isNotEmpty()  // ③ 타 부서 발령도 여기서 걸림
    ) {
        result.put(
            if (hasBeginDateTimeChanged(validatable)) DepartmentPermitColumn.BEGIN_DATE_TIME
            else DepartmentPermitColumn.END_DATE_TIME,
            DepartmentUserUsedForAppointmentReservationException(),
        )
    }

    return result
}
```

## 해결안 / 조사 방향

### 고객 안내 (스펙 기반)

현재 동작은 의도된 제약이므로, 아래 중 하나로 안내:

**옵션 A**: 4/13 예약발령이 실행된 후 종료 처리
- 4/13 이후 구성원이 실제로 이동 완료되면 조직 종료 가능
- 가장 안전한 방법

**옵션 B**: 예약발령을 먼저 취소 → 조직 종료 → 발령 재생성
- 쿡북 F2(발령+조직변경 데드락 해소)와 유사한 워크플로우
- 단점: 발령 재생성 필요

**옵션 C**: Operation API/DB로 종료일 직접 설정 (운영 개입)
- validator를 우회하여 조직의 endDateTime을 직접 변경
- 단점: 운영 개입 필요, 데이터 정합성 수동 확인 필요

### 이성환 질문에 대한 답변

> "4월 14일로 하면 되는데 4월 13일로 하면 안되는 건가"

어떤 날짜로 하든 안 됩니다. validator가 체크하는 것은 **"구성원에게 미래 발령이 있는가"** 이지 **"종료일이 발령일과 겹치는가"**가 아닙니다. 구성원의 미래 position이 하나라도 존재하면 조직 기간 변경 자체를 차단합니다[^4].

> "예약발령이 있으면 무조건 안되는건가"

네, 해당 조직 소속 구성원에게 예약발령이 하나라도 있으면 조직 시작일/종료일 변경이 차단됩니다[^4]. CORE-4623에서 이 제약을 개선하는 작업이 진행됨[^11].

1. "조직 종료가 안 된다" + "소속된 구성원에게 예약된 조직발령이 있다" 메시지 → **스펙**

4. 급하면: 예약발령 취소 → 종료 → 발령 재생성 (옵션 B) 또는 운영 개입 (옵션 C)
