## 현재까지 파악된 내용

### 1. API 불일치 확인

박준홍 확인 — 동일 조건(2026-03-01~03-31, 근무규칙 22개, 부서 23개)에서 세 API의 응답이 다르다[^3]:

| API | 응답 | 비고 |
|-----|------|------|
| `POST /dashboard-summary` | `insufficientWorkUserCount: 1` ✅ | 카운트 |
| `POST /dashboards/work-schedule/InsufficientWork/users` | `userIdHashes: ["vYzRPMXD8Z"]` ✅ | 유저 ID 반환 |
| `POST /dashboards/insufficient-work` | `{"hasNext":false,"list":[]}` ❌ | 상세 리스트 빈 값 |

### 2. 근본 원인 — 퇴직자 필터링 기준 차이

재언 확인 — **노운 이슈**. 대시보드 카드 숫자와 실제 데이터 조회의 퇴직자 처리 기준이 다르다[^4]:

| 조회 유형 | 퇴직자 스킵 기준 | 이 케이스에서 |
|-----------|----------------|-------------|
| **카드 숫자** (summary) | 조회기간 **전체**가 퇴직 이후이면 스킵 | 3/1~3/31 중 3/1~3/16은 재직 → **스킵 안 됨** → 1명 카운트 |
| **데이터 리스트** (detail) | **특정 문서**가 퇴직 이후이면 스킵 | 3/30, 3/31 문서가 퇴직(3/17) 이후 → **스킵됨** → 0명 |

### 3. ES 데이터 검증

userId 618943의 ES workSchedule 문서 상태[^6]:

| 기간 | lastModified | lackOfWorkingMinutes | 상태 |
|------|-------------|---------------------|------|
| 3/1~3/22 | 03-18 14:11 | 0 | ✅ 퇴사 이벤트로 재동기화됨 |
| **3/23~3/29** | **없음** | - | ⚠️ 문서 부재 |
| **3/30~4/9** | **02-01 21:32** | **480** | ❌ 스테일 — 퇴사 전 미리 생성된 미래 문서 |

> 💡 **판단 근거**: 3/30, 3/31 문서의 `lastModified`가 **2026-02-01** — 퇴사(3/17) 2달 전에 생성된 미래 근무일정이 클린업되지 않았다[^6]
> → ES user 문서에서 `resignation.resignationDate: "2026-03-16T15:00:00Z"` (KST 3/17) 확인[^8]
> → DB `view_user` 에서 0건 반환 — 활성 유저 뷰에 없음 = 퇴사 확정[^9]
> → 3/18에 퇴사 이벤트로 3/1~3/22 문서가 재동기화(lack=0)되었으나, 3/30+ 문서는 클린업 누락[^6]

### 4. Summary API가 이 유저를 카운트하는 이유

Summary API의 ES 집계에서 3월 daily workSchedule 중 `violationResult.lackOfWorkingMinutes > 0`인 문서가 userId 618943에 2건(3/30, 3/31) 존재한다[^7]. 이 유저의 퇴직일(3/17)이 조회 기간(3/1~3/31) 시작 이후이므로 "조회기간 전체가 퇴직"에 해당하지 않아 카운트에서 제외되지 않는다[^4].

## 발견한 스펙/제약

- 대시보드 카드 숫자(summary)와 상세 데이터(detail) 간 퇴직자 필터링 기준이 다르다[^4]:
  - summary: 조회기간 전체가 퇴직 이후여야 스킵

- 퇴사 이벤트 처리 시 미래 workSchedule 문서가 완전히 클린업되지 않을 수 있다[^6]
