# CI-4397: 퇴사자에게 머물러 있는 문서 3건 강제 승인 요청

삭제된 구성원이 승인 라인에 잔존하여 워크플로우 문서 3건의 승인이 차단된 운영 처리 요청[^1]. "퇴직한 승인자 교체" 기능은 퇴사 이벤트 기반이므로 삭제된 구성원은 조회 불가[^2].

### 긴급도
일반 — 금전적/법적 영향 없음. 당일 내 처리 적절.

## 원인 분석

**verdict**: `operational`

`yangyang0613@braincommerce.com` (user_id: 584593)이 시스템에서 hard-delete되어 승인 라인 step 0에서 PENDING으로 잔존, 문서 3건의 승인 흐름을 차단[^5]. CI-4228과 동일한 패턴.

### 조사 과정

> 💡 COOKBOOK F1 플로우를 따라 조사 진행[^6]
> → `workflow_task` 조회: 3건 모두 IN_PROGRESS 확인[^5]
> → `approval_process` 조회: ONGOING 상태, deleted_at=9090-09-09 센티널[^7]
> → `approval_line_actor` + `resolved_user` 조회: step 0에서 user 584593이 PENDING[^8]
> → `view_user` 조회: user 584593 존재하지 않음 (hard-delete)[^9]
> → 원인 확정: 삭제된 구성원이 승인 라인에 잔존

## 해결

1. `bulk-approve-for-user` API로 user 584593(삭제된 구성원) step 0 강제 승인 — succeeded 3건[^10]
2. `bulk-approve-for-user` API로 user 557355(syoon, 활성) step 1 강제 승인 — succeeded 4건[^10]
3. 3건 각각 `sync-with-approval` 호출하여 workflow_task 상태 DONE으로 동기화[^11]
4. DB 검증: 3건 모두 workflow_task.status = DONE 확인[^12]

> 557355의 bulk-approve에서 의도하지 않은 4번째 문서(`20060168bbc04bc3a54c9b2664e4d075`)도 함께 승인됨. `bulk-approve-for-user`는 해당 사용자의 모든 PENDING 건을 처리하므로 예상 동작[^13].

## 다음에 같은 문의가 오면

1. **먼저 확인**: "퇴직한 승인자 교체"에서 구성원이 조회되지 않으면 → `view_user`에서 해당 구성원의 상태 확인. 존재하지 않으면 hard-delete된 구성원
2. **원인 판별**: hard-delete된 구성원이면 퇴사 이벤트 미발행 → `replacement-targets` 미등록 → 교체 기능 사용 불가
3. **조치**: 고객에게 강제 승인 동의 확인 후
   - `bulk-approve-for-user` API 호출 (categories: 대상 문서 유형에 맞게)
   - 워크플로우 문서이면 `sync-with-approval`로 workflow_task 동기화 필수
