## 증상

- **문제 정의**: 스와치그룹코리아 세콤 초기 연동 ODBC 설정 중 PostgreSQL 연결 오류 발생[^1]
- **회사**: 스와치그룹코리아(주) (Customer ID: 124876)
- **요청자**: CS팀 김한나[^1]
- **대상자**: 고객사 HR 담당자 hyungnam.jin@kr.swatchgroup.com[^1]
- **영향 범위**: 초기 연동 설정 단계 — 아직 연동 시작 전
- **문제 시점**: 2026-03-23
- 문의 내용:
  1. ODBC test 중 PostgreSQL 연결 오류 발생[^1]
  2. 에러: `connection to server at 'flex-secom.flex.team' (52.78.150.221), port 5432 failed: server closed the connection unexpectedly`[^2]
  3. 방화벽에 도메인 등록은 안내 완료[^1]
  4. Intercom 첨부에서 "연결설정에서 에스원 세콤 항목자체가 보이지 않는다"는 추가 정보 확인[^3]

## 원인 분석

`odbc_connection_limit = 0`으로 설정되어 PostgreSQL ROLE의 `CONNECTION LIMIT 0`이 적용, 모든 ODBC 연결이 차단됨[^12][^13].

### 가설 목록

| # | 가설 | 확인 방법 | 상태 |
|---|------|----------|------|
| 1 | Provider 계정 미생성/미활성화 | `v2_customer_external_provider` 조회[^6] | ❌ 소거 — 계정 존재, active=true[^12] |
| 2 | **odbc_connection_limit=0으로 PostgreSQL 연결 차단** | DB 조회 + 코드 분석[^12][^13] | ✅ 확정 |
| 3 | SSL/ODBC 설정 불일치 | openssl s_client 테스트[^15] | ❌ 소거 — SSL 핸드셰이크 정상 |

### 조사 과정

> 💡 **판단 근거**: DB 조회에서 `odbc_connection_limit = 0` 발견[^12]
> → 코드 분석: 이 값이 PostgreSQL `ALTER USER ... CONNECTION LIMIT 0`으로 반영됨[^13]
> → PostgreSQL에서 `CONNECTION LIMIT 0`은 해당 ROLE로의 새 연결을 완전히 차단
> → 세콤 기본값은 2인데[^14] 0으로 설정된 것이 비정상
> → Wireshark 캡처에서 TCP 핸드셰이크 성공 후 PGSQL startup에 서버 무응답 → RST 패턴이 이 원인과 정확히 일치[^16]

<details>
<summary>📋 코드 분석 — connection limit 적용 흐름</summary>

1. `v2_customer_external_provider.odbc_connection_limit` (MySQL) 값을 읽어서
2. `flex-work-event-transmitter-backend`의 `CustomerExternalProviderOdbcUserRepository.changeConnectionCount()`가[^13]
3. PostgreSQL에 `ALTER USER secom_br04jpgkek CONNECTION LIMIT 0` 실행
4. 이후 해당 ROLE로의 모든 새 연결이 거부됨

기본값 결정 코드: `CustomerExternalProviderConstants.kt:32-39`[^14]
- SECOM: 2, CAPS: 2, TELECOP: 3 (x serverCount)

JPA Entity 기본값은 0이지만(`CustomerExternalProviderJpaEntity.kt:80`), 등록 시 `getDefaultConnectionLimit()`으로 올바른 값이 설정되어야 함. 0이 들어간 경로는 추가 확인 필요.

</details>

<summary>📋 Wireshark 캡처 분석 — TCP 성공 후 PGSQL 무응답 패턴</summary>

고객사에서 캡처한 Wireshark 패킷[^16]:

| # | 시간 | 방향 | 프로토콜 | 내용 | 의미 |
|---|------|------|---------|------|------|
| 1047 | 3.70s | 서버→클라 | TCP | **RST, ACK** (빨강) | 이전 연결을 서버가 강제 종료 |
| 1856 | 6.01s | 클라→서버 | TCP | **SYN** | 새 TCP 연결 시도 |
| 1858 | 6.01s | 서버→클라 | TCP | **ACK** | TCP 핸드셰이크 완료 — TCP 연결 성공 |
| 1859 | 6.01s | 클라→서버 | **PGSQL** | `> ?` (녹색) | PostgreSQL startup 메시지 전송 |
| 1893~3267 | 6.2~15.5s | 클라→서버 | TCP | **TCP Retransmission** x7 (검정/진한 빨강) | 서버 응답 없음, 재전송 반복 |
| 4520 | 25.17s | 서버→클라 | TCP | **RST, ACK** (빨강) | 서버가 연결 강제 종료 |

**Wireshark 색상 의미:**
- **빨간색**: RST(Reset) — 비정상 연결 종료
- **진한 빨간/검정**: TCP Retransmission — 응답 미수신으로 재전송
- **녹색**: PGSQL 프로토콜 — PostgreSQL 애플리케이션 레벨 메시지

**해석**: TCP 3-way handshake 성공(방화벽 정상) → PostgreSQL startup 메시지에 서버 무응답 → 19초간 재전송 후 RST. `CONNECTION LIMIT 0`일 때 PostgreSQL이 세션 거부하면서 응답 없이 끊는 동작과 정확히 일치.

</details>

### 근본 원인 추적 (5 Whys)

1. 왜 ODBC 연결이 실패하나? → PostgreSQL이 연결을 거부함
2. 왜 거부하나? → 해당 ROLE의 `CONNECTION LIMIT = 0`
3. 왜 0인가? → `v2_customer_external_provider.odbc_connection_limit = 0`이 PostgreSQL에 동기화됨[^12]
4. 왜 0으로 저장되었나? → (추정) 계정 생성 시 기본값 설정 로직이 정상 실행되지 않았거나, 이후 수동/API로 0으로 변경됨
5. 왜 기본값 설정이 누락될 수 있나? → JPA Entity 기본값이 0이므로, `getDefaultConnectionLimit()` 호출이 누락되면 0이 그대로 저장됨[^14] (추가 확인 불가 — 생성 시점 로그 미보존)

### 스펙 vs 버그 판별

**설정 오류 (Configuration Issue)**. `CONNECTION LIMIT 0`은 의도된 스펙이 아님. 세콤 기본값은 2[^14]. 계정 생성(`2026-02-25`) 시점에 기본값이 올바르게 설정되지 않은 것으로 보임.

1. **먼저 확인**: `v2_customer_external_provider`에서 해당 customer_id의 `odbc_connection_limit` 조회
