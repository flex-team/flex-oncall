### 정책 조건 구조

시간 정책은 RANGE 연산자를 사용하며, `SpendingPolicyCommandConverter.kt:941-946`에서 조건을 생성한다[^10]:
```kotlin
rangeValue = PolicyRange(
    start = it.timeFrom,    // 21:30
    end = it.timeTo,        // 06:00
    startExclusive = false,
    endExclusive = false,
)
```

`Policies.kt:830`에서 overnight 범위(end < start)를 감지하고 날짜를 보정하는 로직이 존재한다[^5]. 그러나 이 로직에 진입하기 전에 null 체크(826행)에서 false로 반환되므로, overnight 처리 로직이 실행되지 않는다.

### 00:00이 전달되었다면

만약 null 대신 00:00(LocalTime)이 전달되었다면:
- 범위: start=21:30, end=06:00 (overnight)
- 00:00 <= end(06:00) → evaluateDatePassed = true
- evaluateDateTime = 다음날 00:00
- startDateTime(오늘 21:30) <= evaluateDateTime(다음날 00:00) → true
- endDateTime(다음날 06:00) >= evaluateDateTime(다음날 00:00) → true
- **결과: PASS (비위반)** — 정상 동작

### Slack에서 언급된 시간 vs 실제 시간

Slack QNA에서 김한나가 "오전 09:30 ~ 다음 날 오전 06:30"이라고 언급했으나, access log의 실제 정책 데이터는 **21:30 ~ 06:00**이다. Slack의 "09:30"은 "21:30"(오후 9시 30분)의 오기이거나, 다른 정책을 참조한 것으로 보인다.

## 해결안 / 조사 방향

### 즉시 대응 (표면 원인 해결)

**옵션 A**: `SpendingPolicyCommandConverter.kt:539`에서 null일 때 00:00으로 대체
```kotlin
evaluationValue = transactedTime ?: LocalTime.MIDNIGHT,
```
- 장점: 변경 범위 최소, 기존 `toEvaluationBasedAt()`의 의도와 일치
- 단점: 수동 증빙의 시간이 00:00이라는 가정을 강제

**옵션 B**: `SpendingPolicies.kt:2084`에서 null 대신 00:00 전달
```kotlin
transactedTime = LocalTime.MIDNIGHT,
```
- 장점: null 원점을 제거
- 단점: 다른 곳에서 transactedTime=null을 의미 있게 사용하는 경우 영향

### 근본 해결 (근본 원인 해결)

**옵션 C**: 수동 증빙의 시간 정책 평가를 건너뛰는 로직 추가
- 장점: "시간 정보가 없는 증빙에 시간 정책을 적용하지 않는다"는 명확한 비즈니스 규칙
- 단점: 구현 범위 넓음, PM 확인 필요

> EP 담당(윤진한)이 "다음 주 자동화 관리 배포"에서 해결 예정이라 언급. 구체적 수정 방향은 EP팀 확인 필요.
