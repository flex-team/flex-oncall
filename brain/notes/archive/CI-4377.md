## 원인 분석

### 조사 과정
> 💡 CI-4150과 동일 패턴임을 인지[^3]하고, DB 경로로 조사 시작
> → `flex_authorization.flex_grant_subject`[^7]에서 customer_id=215787 최고관리자 멤버 조회[^11]
> → 현재 최고관리자 4명: 867895, 868347, 887810, 942114 — **대상자 920385 미포함**[^11]
> → 대상자의 전체 grant 소속 조회 — **0건**. 퇴사 처리 시 모든 grant에서 물리삭제[^8] 완료[^12]
> → CI-4309(raccoon audit API) 접근법 검토 — Envers[^6] 기반이므로 권한 테이블 미포함, **적용 불가**[^5]
> → DB + Envers 경로에서는 이력 추적 불가로 판단
>
> ⚠️ **그러나 윤성복님이 OpenSearch 감사로그에서 직접 검색하여 실제 권한 변경 이력을 확인**[^14]
> → DB(Envers)에는 없지만, **OpenSearch에는 별도 경로로 권한 변경 이벤트가 기록**되어 있음[^15]
> → 결론: **DB 물리삭제와 무관하게 OpenSearch 감사로그로 이력 추적 가능** (보존 기간 내)

### 원인
- `flex_grant_subject`는 물리삭제 방식이라 DB에서 과거 이력 추적 불가[^12]
- Envers(제품 감사로그)는 권한 테이블을 기록 대상으로 포함하지 않음[^5]
- **하지만 OpenSearch 감사로그에는 권한 변경 이벤트가 기록됨** — DB/Envers와 별개 경로[^15]

## 해결

- CS 담당자(유지원)에게 퇴사 시점(2026년 2월경)과 입사 시점을 확인 중[^17]

## 발견한 스펙/제약
- DB(`flex_grant_subject`) 물리삭제 + Envers 미기록은 기존에 알려진 제약 (CI-4150)
- **OpenSearch 감사로그에는 권한 변경이 기록됨** — CI-4150 노트에서는 이 경로를 "마지막 수단"으로만 언급했으나, 실제로는 **유효한 조사 경로**[^15]

## 다음에 같은 문의가 오면
1. **먼저 확인**: `flex_grant_subject`에서 현재 grant 멤버십 확인 (현재 포함 여부)
   ```sql
   SELECT gs.subject_id, gs.created_by, gs.created_at
   FROM flex_grant_subject gs
   JOIN flex_grant g ON g.id = gs.grant_id
   WHERE gs.customer_id = ? AND g.title_key = 'authority.administrator_title';
   ```
2. **과거 이력 필요 시**: **OpenSearch 감사로그에서 직접 검색** — DB 물리삭제와 무관하게 권한 변경 이벤트가 기록되어 있음 (보존 기간 내)
3. **증빙 자료 제공**: 로그가 확인되면 해당 내용을 이메일로 발송 가능
4. raccoon audit API(CI-4309)는 Envers 기반이므로 권한 변경 이력에는 **사용 불가**
