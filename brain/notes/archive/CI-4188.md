### 원인
- 후발 추가된 reviewer(류혜주)의 `evaluation_reviewer` 레코드에서 `user_form_ids`가 빈 배열(`[]`)[^5]
- 일괄 설정(finalize) 이후 추가된 reviewer는 UserForm 초기화를 메시지 큐에 의존하는데, 해당 메시지가 정상 처리되지 않은 것으로 추정[^20]
- form이 없으면 admin 화면에서 "평가지 생성 중"으로 표시됨[^9]

### 근본 원인 (5 Whys)

1. **왜 "생성 중"이 표시되나?** → `evaluation_reviewer.user_form_ids`가 빈 배열(`[]`)이기 때문[^5]
2. **왜 user_form_ids가 비어있나?** → 해당 레코드는 평가 일괄 설정(finalize) 이후에 추가되었고, 후발 추가 시 UserForm 초기화 메시지 큐가 정상 동작하지 않음[^5][^20]
3. **왜 메시지 큐가 처리되지 않았나?** → 정확한 원인 미파악. 김보라가 "메시지 큐가 안 들어간 것으로 추정"이라고 언급[^20]
4. **왜 fallback이 없나?** → `EvaluationUserFormInitializeService`가 lazy initialization 방식을 fallback으로 제공하지만, reviewer가 직접 접근해야 트리거됨[^6]
5. **왜 admin이 보는 화면에서 문제가 되나?** → admin의 `list-received-evaluations` API는 reviewer의 form 상태를 보여주며, form이 없으면 "생성 중"으로 표시[^9]

## 해결
- 김보라가 raccoon Operation API의 `initialize-user-form`을 호출하여 UserForm을 수동 초기화[^20]
- 2026-03-23 08:34 UTC (17:34 KST)에 form_user_form 레코드 생성 확인[^13]
- 2026-03-23 08:42 UTC에 김현욱이 정상 노출 확인[^21]

## 발견한 스펙/제약
- 일괄 설정(finalize) 시: 모든 reviewer의 UserForm이 batch로 즉시 생성됨[^7]
- 후발 추가 시: UserForm 초기화가 메시지 큐에 의존. 큐 처리 실패 시 lazy initialization(reviewer 직접 접근)이 fallback[^6]
- 이 비대칭 설계로 인해 후발 추가 reviewer의 admin 화면 표시가 불완전할 수 있음

## 용어 발견
| 사용자 표현 | 시스템 용어 | 비고 |
|------------|------------|------|
| 평가지 생성 중 | evaluation_reviewer.user_form_ids = [] | form 미초기화 상태 |
| 평가지가 안 보여요 | UserForm lazy initialization 미트리거 | 후발 추가 reviewer에서 발생 가능 |

## 다음에 같은 문의가 오면

1. **먼저 확인**: `evaluation_reviewer` 테이블에서 해당 reviewee-reviewer 조합의 `user_form_ids`가 빈 배열(`[]`)인지 확인
   ```sql
   SELECT id, reviewee, reviewer, step_type, user_form_ids, created_at
   FROM evaluation_reviewer
   WHERE customer_id = ? AND evaluation_id = ?
   AND user_form_ids = '[]';
   ```
2. **원인 판별**: 빈 배열이면 후발 추가된 reviewer일 가능성 높음. `created_at`을 일괄 생성 레코드와 비교하여 후발 추가 여부 확인
3. **조치**: raccoon Operation API `initialize-user-form` 호출하여 수동 초기화. reviewer가 해당 평가를 열면 lazy init으로도 자동 해소되나, admin 화면 즉시 정상화가 필요하면 Operation API 사용
