# CI-4247: 원천세 신고 생성 시 2020년도 선택 불가

> **상태**: 해결 완료 (Not a Bug — 의도된 동작으로 판정) — 2026-03-30

## 증상

- **문제 정의**: 원천세 신고 생성 시 귀속연도로 2020년을 선택할 수 없음
- **회사**: 스케쳐스코리아 (Customer ID[^8]: 211308)
- **요청자**: CS팀 김민경 (minkyung@flex.team)
- **대상자**: 스케쳐스코리아 담당자 (2025년 도입 고객사[^9])
- **영향 범위**: 도입 이전 연도(2020~2024) 귀속 원천세 신고 건 전체
- **문제 시점**: 2026-03-30 (현재)
- 문의 내용:
  1. 2020년 12월 귀속, 2026년 3월 지급 건으로 원천세 신고를 생성하려 하나 2020년도 선택 불가
  2. 소득등록은 완료된 상태
  3. 2020~2024년도 귀속 원천세 신고서 작성이 가능한지 확인 요청
  4. 고객사가 오전 중 신고 필요 — 긴급도 높음[^1]

## 원인 분석

FE 귀속연월 picker 컴포넌트가 "작년 1월"부터만 선택을 허용하며, 2024년 이전 연도는 모든 월이 disabled 처리됨[^2]. BE에는 귀속연도 절대 범위 제한이 없음[^3].

### 가설 목록

| # | 가설 | 확인 방법 | 상태 |
|---|------|----------|------|
| 1 | FE 귀속연월 picker가 "작년 1월"부터만 선택 허용 | dev 환경 재현 | **확정** — 2025년 1월이 선택 가능 최소 경계[^4] |
| 2 | BE에 귀속연도 범위 검증 로직이 있음 | BE 코드 분석 | **소거** — `WithholdingStatusReportHeaderProps.checkIfValid()`에 절대 연도 제한 없음[^3] |

### 조사 과정

> **판단 근거**: 권재호님의 FE 코드 조사[^2] → BE `SettlementResultService` 코드 확인[^5] → dev 환경에서 실제 재현[^4]
> → BE API(`GET /api/v2/payroll/withholding-tax/settlements`)는 `YearMonth` 파라미터를 자유롭게 수용[^5]
> → FE 귀속연월 picker에서 2024년 이하 모든 월 버튼이 disabled 상태 확인[^4]
> → 결론: **순수 FE 제한**

<details>
<summary>BE 코드 분석 상세</summary>

**신고서 생성 API**: `POST /api/v2/payroll/withholding-tax/national/status-reports`[^6]
- 요청 DTO: `belongedYearMonth: YearMonth`, `paidYearMonth: YearMonth` — 타입 제한만 있고 범위 제한 없음

**정산 결과 조회 API**: `GET /api/v2/payroll/withholding-tax/settlements`[^5]
- 서비스 주석: "조회 범위는 최대한 넉넉하게 잡습니다. 정확한 셀렉트는 FE에서 합니다."
- `WithholdingTaxHeaderUtils.getBelongedDateRangeFor()`로 날짜 범위를 계산하지만 절대 연도 하한 없음

**검증 로직** (`WithholdingStatusReportHeaderProps.checkIfValid()`)[^3]:
- 반기별 신고의 귀속월(1월/7월)과 지급월 관계만 검증
- 절대 연도 범위 제한 없음

</details>

### 원인

지방소득세 모달의 연도 제한 컴포넌트(`FormField_귀속지급_연월.tsx`)를 원천세 모달에서 재사용(PR #183)하면서, "1년 미만의 것을 작업할 일은 거의 0에 가깝다"는 지방소득세 전용 판단이 원천세에도 그대로 적용됨[^2].

### 스펙 vs 버그 판별

**판정: FE 버그** — 지방소득세 컴포넌트 재사용 시 원천세의 요구사항(도입 이전 연도 귀속 신고)을 반영하지 않은 것.
- 피그마 디자인에서 연도 제한이 의도된 것으로 보이나[^7], 이는 지방소득세 기준 설계로 추정 [추론: 원천세 전용 디자인이 별도로 존재하지 않음]
- 세금팀(@Slack Group S03RRBARCP4)에 원천세에서의 의도 확인 요청 상태[^7]

## 발견한 스펙/제약

- 원천세 신고서 BE API는 `YearMonth` 타입만 검증하고 절대 연도 범위 제한이 없음 — FE에서 "정확한 셀렉트"를 수행하는 구조[^5]
- 반기별 신고(HALF)에서만 귀속월(1월/7월)과 지급월 관계 검증이 존재[^3]

## 해결안 / 조사 방향

1. **즉시 대응**: FE 원천세 귀속연월 picker의 연도 하한을 제거/확대. 지방소득세와 원천세의 min year를 컴포넌트 props로 분리.
