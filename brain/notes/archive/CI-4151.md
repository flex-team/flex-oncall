## 원인 분석

**76번 코드(휴복직보험료)의 `settledHealthInsurance`가 중도정산 건강보험 계산에서 2중 합산되는 버그.**

### 가설 목록

| # | 가설 | 확인 방법 | 상태 |
|---|------|----------|------|
| 1 | `RetireeYearEndHealthInsuranceCalculatorImpl`에서 76번 코드의 `settledHealthInsurance`가 `pastHealthInsuranceAmount`(leaveOfAbsence 경로)와 `currentHealthInsuranceAmount`(installment 경로) 양쪽에서 합산됨 | 코드 분석 + 데이터 검증 | ✅ 확정 (코드 분석 기반) |
| 2 | UI 표시 레벨에서 동일 금액이 중복 노출 | PayStub 팩토리 코드 확인 | ❌ 소거 — PayStub 팩토리는 base/leaveOfAbsence/yearEnd를 별도 행으로 분리하여 표시하며 중복 없음[^8] |
| 3 | `informedHealthInsurance` 합산 시 76번 정산보험료가 미차감되어 기본 보험료에 포함 | `informedHealthInsuranceTotalPayment` 로직 확인 | ❌ 소거 — `isSeparated=true`이므로 정산보험료를 정상 차감[^5] |

### 핵심 근거

76번 코드는 `HealthInsuranceSettlementReasonCode` enum에서 동시에 두 분류에 속한다[^4]:
- `LEAVE_OF_ABSENCE_REASON_CODES` → `isLeaveOfAbsence = true`
- `SETTLEMENT_REASON_CODES` (= YEAR_END + LEAVE_OF_ABSENCE) → `isInstallment = true`

이로 인해 동일한 `settledHealthInsurance` 값이 두 가지 computed property로 노출된다[^5]:
- `leaveOfAbsenceHealthInsurancePayment = settledHealthInsurance` (isLeaveOfAbsence)
- `healthInsuranceInstallmentAmount = settledHealthInsurance` (isInstallment)

`RetireeYearEndHealthInsuranceCalculatorImpl.calculate()`에서 이 두 값이 각각 다른 합산 경로에 포함되어 동일 금액이 2회 합산된다[^6]. 기존 통합 테스트는 74번 코드(정산분할고지보험료)만 커버하며, 74번은 `isLeaveOfAbsence = false`라서 이 중복이 발생하지 않는다[^7].

<details>
<summary>2중 합산 코드 흐름</summary>

```
pastHealthInsuranceAmount   += deductions.leaveOfAbsenceHealthInsurance   (= settledHealthInsurance)
currentHealthInsuranceAmount += recipients.healthInsuranceInstallmentAmount (= settledHealthInsurance)
→ 동일 금액 2회 합산
```

</details>

<summary>📋 조사 과정 상세</summary>

#### 5 Whys 체인

1. 왜 건강보험료가 2배인가? → `RetireeYearEndHealthInsuranceCalculatorImpl`에서 `pastHealthInsuranceAmount`과 `currentHealthInsuranceAmount` 양쪽에 76번 코드의 `settledHealthInsurance`가 포함됨[^6]
2. 왜 양쪽에 포함되는가? → 76번 코드가 `isInstallment = true`이면서 `isLeaveOfAbsence = true`이기 때문[^4]
3. 왜 두 분류에 동시에 속하는가? → `SETTLEMENT_REASON_CODES = YEAR_END_REASON_CODES + LEAVE_OF_ABSENCE_REASON_CODES`로 정의되어 있고, `isInstallment`이 이 합집합을 체크하기 때문[^4]
4. 왜 calculator가 이를 구분하지 못하는가? → calculator의 `pastHealthInsuranceAmount`는 base deduction result의 `leaveOfAbsenceHealthInsurance`를 합산하고, `currentHealthInsuranceAmount`는 recipient의 `healthInsuranceInstallmentAmount`를 합산하는데, 두 경로가 겹치는 경우를 고려하지 않음[^6]
5. 왜 calculator가 dual-path 구조를 갖게 되었는가? → PR #8500[^13]에서 `healthInsurancePaymentHistories` 단일 소스를 `yearEndDeductions` + `deductions.leaveOfAbsence` dual-path로 리팩토링하면서 도입됨
6. 왜 기존에 발견되지 않았는가? → 기존 테스트가 74번 코드(isLeaveOfAbsence=false)만 커버하여 76번 코드의 중복 경로가 테스트되지 않음[^7]. 또한 PR #8500 merge 시점(2026-02-27)에는 76번이 키인 대상이 아니어서 실제 값이 0이었음

#### calculator 호출 흐름

`RetireeYearEndInputService.applySocialInsurance()` → `TaxableEarningCalculationService.calculateByInsuranceRecipient()` → `RetireeYearEndHealthInsuranceCalculatorImpl.calculate()` 순으로 호출됨.

`calculateByInsuranceRecipient()`에 전달되는 `dto.payeeResults`는 해당 연도의 모든 정산 결과를 포함하며, 현재 정산의 base deduction result도 포함됨(추정)[^9].

#### 소거된 가설 상세

**가설 2 (UI 중복 표시)**: `PayStubDeductionDataFactory`는 건강보험을 "건강보험"(base), "건강보험 휴복직"(leaveOfAbsence), "건강보험 연말정산"(yearEnd)으로 별도 행에 표시. 각 행의 값은 `healthInsurancePayment = informed - settled`, `leaveOfAbsenceHealthInsurancePayment = settled`, `yearEndHealthInsurancePayment = yearEndHI`로 서로 다른 필드에서 가져오므로 UI 레벨 중복 없음.

**가설 3 (informedHealthInsurance 미차감)**: `informedHealthInsuranceTotalPayment = informedHealthInsurance - settledHealthInsurance` (76번은 `isSeparated=true`이므로 차감 적용됨). base deduction의 `healthInsurance = informedHealthInsuranceTotalPayment`이므로 정상 차감됨.

</details>

### 버그 도입 — PR #8500 (2026-02-27 merge)

PR #8500[^13]이 `RetireeYearEndHealthInsuranceCalculatorImpl`을 리팩토링하면서 dual-path 구조를 도입한 것이 근본 원인이다.

**기존 코드** (PR #8500 이전): `healthInsurancePaymentHistories`라는 단일 소스에서 합산[^13]
```
pastHealthInsuranceAmount = paymentHistories.sumOf {
    it.yearEndHealthInsurance + it.healthInsuranceInstallmentAmount + ...
}
```
→ 단일 소스이므로 code 76이 양쪽 분류에 속해도 1회만 합산됨.

**변경 후 코드** (PR #8500): 두 개의 독립 경로로 분리[^6]
```
pastHealthInsuranceAmount  = yearEndDeductions.sumOf { healthInsurance }
                           + deductions.sumOf { leaveOfAbsenceHealthInsurance }  ← 경로 1
currentHealthInsuranceAmount = recipients.sumOf { healthInsuranceInstallmentAmount }  ← 경로 2
```
→ code 76은 경로 1(`leaveOfAbsenceHealthInsurance`)과 경로 2(`healthInsuranceInstallmentAmount`) 양쪽에서 동일한 `settledHealthInsurance`가 합산됨.

이 시점에는 code 76이 키인 대상이 아니어서 `healthInsuranceInstallmentAmount = 0`이라 버그가 잠복했다.

### 트리거 — PR #8548 (2026-03-05 merge)

76번 코드를 키인 관리 대상에 추가한 PR[^11]이 트리거. 이 PR은 `HealthInsuranceInstallmentKeyInCandidateFactory`에 76번 코드를 추가했을 뿐, calculator 로직은 변경하지 않았다. PR #8500의 잠복 버그가 **76번 코드에 실제 installment 값이 들어오면서 표면화**되었다.

### 75번 코드 확인 결과

75번 코드(휴복직+정산분할고지)는 `YEAR_END_REASON_CODES`에만 포함(`isLeaveOfAbsence = false`)[^4]. 따라서 76번과 같은 leaveOfAbsence + installment 중복은 발생하지 않는다.

### 데이터 검증 제약

DB 컬럼(`settled_health_insurance`, `year_end_health_insurance` 등)이 암호화(varchar)되어 있어 직접 SQL로 값 비교 불가[^12]. 데이터 검증은 어드민 화면 또는 API 호출로 진행해야 한다.

### 스펙 vs 버그 판별

**버그**. 동일 `settledHealthInsurance`가 서로 다른 합산 경로에서 중복 카운트되는 것은 의도된 동작이 아님.
- 기존 테스트(74번 코드)에서는 발생하지 않는 경로
- `pastHealthInsuranceAmount` 주석에 "과거 정산"이라고 적혀 있으나 실제로는 현재 정산도 포함
- PR #8548이 calculator를 수정하지 않고 76번을 키인 대상에 추가하면서 잠재 버그가 활성화됨[^11]

### 불확실 포인트

fix가 올바른지 확신할 수 없는 이유:

1. **`deductions` 파라미터에 현재 정산이 포함되는지 미확정**: `pastHealthInsuranceAmount`에서 `deductions.leaveOfAbsenceHealthInsurance`를 합산하는데, 이 deductions에 현재 정산의 base deduction result가 포함되는지 코드 분석만으로는 확정하지 못함 (추정만 있음)[^9]. 만약 포함되지 않는다면 fix 전제가 달라진다.
2. **15번, 25번 코드에 대한 영향**: 15번(휴복직분할고지반환), 25번(휴복직분할고지)도 `isLeaveOfAbsence=true`이면서 `isInstallment=true`. fix가 이 코드들에도 동일하게 적용되는데, 이 코드들의 실제 운영 데이터에서의 동작을 검증하지 못함.
3. **분납 서비스와의 정합성**: `HealthInsurancePaymentInstallmentRepayService`에서도 `healthInsuranceInstallmentAmount`를 합산하여 분납 내역을 생성함[^15]. calculator 수정만으로는 분납 서비스의 동작과 정합성이 맞는지 확인 필요.
4. **실제 데이터 미검증**: DB 컬럼 암호화로 SQL 직접 조회 불가[^12]. 스케쳐스 유주연의 실제 정산 데이터를 확인하지 못함.
