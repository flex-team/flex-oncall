# CI-4260: 사업장 담당자 계정 급여정산 실행 시 알 수 없는 오류

## 증상
- **문제 정의**: 사업장 담당자 계정으로 급여정산 실행 시 인가(authorization) 타임아웃으로 500 에러 발생, 정산이 대상자 0명 상태로 stuck
- **회사**: 주식회사 이도 (Customer ID: 212445)
- **요청자**: 정인지 (inji@flex.team)[^1]
- **대상자**: sangyeon+yido1@flex.team (userId: 834904)[^1]
- **영향 범위**: 사업장 담당자 계정으로 정산 실행하는 모든 대규모 회사에서 재현 가능. Sentry 기준 30명 이상 영향[^2]
- **문제 시점**: 2026-03-31 10:18 KST (최초 에러), Sentry 이슈는 2026-03-12부터 반복 발생[^2]
- 문의 내용:
  1. 사업장 담당자 계정으로 급여정산 실행 시 "알 수 없는 오류" 발생[^1]
  2. 재실행 시 사업장 대상자 인원만 정상 실행되지만, 이전 실패 건이 대상자 0명으로 리스트에 남음[^1]
  3. 이전 이도 사업장 담당자 교육 때도 동일 현상 발생[^3]

## 원인 분석

### 조사 과정
> 💡 윤성복이 "인가 타임아웃" 가능성 지적[^6]
> → 담당자 계정 관할 205명이지만, 회사 전체 구성원 약 2,000명[^7]
> → 인가 로직이 "2천명 중 작업할 수 있는 인원을 찾는" 구조로 전체 구성원 조회[^8]
> → `resolveAccessibleUsers` callTimeout 6초에서 타임아웃 발생[^15]
> → settlement 생성(TX#1)은 커밋되었으나 payee 초기화(TX#2, TX 밖)에서 실패 → 대상자 0명 정산이 stuck[^12]

### 원인
- **직접 원인**: flex-permission `resolveAccessibleUsers` 호출 시 callTimeout 6초 초과[^6][^15]. 구성원 2,000명 회사에서 인가 조회가 6초 이상 소요
- **정합성 문제**: settlement 생성과 payee 초기화가 의도적으로 별도 트랜잭션[^13]. payee 초기화 실패 시 보상 로직 부재 → settlement만 IN_PROGRESS로 남음[^12]

## 해결
- **authorization 타임아웃 조정**: flex-permission-backend v3.58.3 배포 — callTimeout 6초→10초[^16]
- **payroll hotfix 배포**: [flex-payroll-backend#8714](https://github.com/flex-team/flex-payroll-backend/pull/8714) — 인가 라이브러리 버전업 + 정합성 이슈 개선[^17][^22]
- **데이터 보정**: 깨진 정산 3건(87549, 87560, 87561) CANCELED 처리[^25]
- **후속 티켓**: 트랜잭션 all-or-nothing 개선은 EPBE-293으로 팔로업[^26]. settlement_result_id = -1 정합성 개선은 EPBE-246으로 진행[^27]

## 발견한 스펙/제약
- settlement 생성(TX)과 payee 초기화(TX 밖)의 의도적 분리: payee 초기화에서 8개+ 외부 API를 병렬 호출하므로 하나의 TX로 감싸면 DB 커넥션 장기 점유[^13]
- 인가 로직은 "사업장 담당자의 관할 인원"이 아닌 "회사 전체 구성원"을 기준으로 조회[^8]

## 다음에 같은 문의가 오면

1. **먼저 확인**: Sentry에서 `FLEX-PROD-PAYROLL-API-67Q` 또는 `FlexRemoteUnknownStateException: timeout` 검색. 해당 고객사의 전체 구성원 수 확인 (1,000명+ 이면 의심)
2. **원인 판별**:
   - 타임아웃 에러 + 대상자 0명 정산 존재 → 인가 타임아웃으로 인한 정합성 깨짐
   - `Key -1 is missing in the map` 에러 → 깨진 정산의 결과 조회 시 발생 (Sentry 6CM)[^28]
3. **조치**:
   - 깨진 정산 조회: `SELECT s.id, s.settlement_status, COUNT(p.id) FROM work_income_settlement s LEFT JOIN work_income_settlement_payee p ON p.settlement_id = s.id AND p.customer_id = {cid} WHERE s.customer_id = {cid} AND s.created_at >= '{date}' GROUP BY s.id`
   - payee 0명 + IN_PROGRESS 건 → `UPDATE work_income_settlement SET settlement_status = 'CANCELED', version = version + 1 WHERE id IN (...) AND settlement_status = 'IN_PROGRESS'`
