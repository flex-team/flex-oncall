## 해결

### 즉시 대응 (완료 — 2026-04-07)

`POST /proxy/calendar/api/operation/v2/calendar/customers/113613/events/cleansing` 호출로 구글 캘린더에서 2건의 이벤트를 직접 삭제 처리함 → HTTP 200.[^3]

```json
{
  "oauthTokenId": 1,
  "calendarConnectionId": 50246,
  "calendarConnectionHistoryId": null,
  "googleEventIds": [
    "019d6540380b1c6893152b28e3e88820",
    "019d6540380b1c6893152b28e3e88823"
  ]
}
```

- `calendarConnectionId=50246`: `flex.v2_external_calendar_connection`의 AVO Time Off 그룹 캘린더 연결 (user_id=495969)[^3]
- `googleEventIds`: `flex_calendar.google_calendar_event_sync.google_calendar_event_id` 값[^3]
- `oauthTokenId`는 API 내부에서 실제로 사용되지 않음 — `findGroupBy`가 customerId+userId로 토큰을 직접 조회[^3]

### 근본 해결 (미완)

취소 승인 플로우에서 `calendar_event.deleted_at` 업데이트가 왜 누락됐는지 코드 분석 필요 → `ops-fix-issue CI-4350` 실행 권장

**코드 체인 조사 결과 (2026-04-07):**

취소 승인 완료 → calendar 삭제까지의 정상 경로는 다음과 같이 구현되어 있음[^4]:

```
취소 승인 완료 (TimeTrackingApprovalEvent APPROVE)
  → FlexCalendarApprovalEventConsumer
  → FlexCalendarEventAdapter.handleTimeOff()
      ├─ calculatedUserTimeOffRegisterEventLookUpService (등록 이벤트 조회)
      └─ userTimeOffCancelEventLookUpService.getCancelEventBy() (취소 이벤트 조회)
  → sendToExternalProviders() → sendToFlexCalendar()
  → FlexCalendarSyncServiceImpl.sync() → CalendarSyncCommand Kafka 발행
  → FlexCalendarSyncOperationCommandConsumer
  → FlexCalendarEventSyncAdapter.executeBatchRequest()
  → FlexCalendarSyncEventConverter:
      - makeDeletionCommands(): TimeTrackingFlexCalendarEventMap 기존 매핑 → 삭제 대상
      - makeCreationCommands(): 현재 유효한 work schedule 블록 → 생성 대상
        (취소 승인 완료된 이벤트는 work schedule 조회 시 필터링되어 제외됨)
  → executeDeleteCommandOnCalendarBackend() → calendar_event.deleted_at 업데이트
```

실제 버그 지점 특정 미완. 가능성 있는 지점: `getCancelEventBy()`의 `targetEventId` 매핑 불일치, 또는 `CalendarSyncCommand` 발행 후 `TimeTrackingFlexCalendarEventMap` 매핑 미조회[추론: 코드 체인은 연결되어 있으나 실제 실행 여부 미확인]

## 다음에 같은 문의가 오면

> 트리거: "취소 승인 완료됐는데 구글 캘린더에 일정이 남아있어요" + Metabase에서 "미연동 없음"으로 표시

1. **먼저 확인**: `flex_calendar.calendar_event`에서 해당 이벤트의 `deleted_at`, `status` 조회
   - `deleted_at=NULL` + `status=CONFIRMED` → 취소 승인 플로우 버그 (이 케이스)
   - `deleted_at` 채워짐 → 구글 API 호출 실패 케이스 → `google_calendar_event_delete_sync` 확인

2. **google_calendar_event_id 조회**: `google_calendar_event_sync` WHERE `event_id IN (...)` → `google_calendar_event_id` 추출

3. **cleansing API 호출**: `POST /proxy/calendar/api/operation/v2/calendar/customers/{customerId}/events/cleansing`
   - `calendarConnectionId`: `flex.v2_external_calendar_connection` WHERE `customer_id=? AND calendar_type='GROUP'`
   - `googleEventIds`: ②에서 조회한 ID 목록
   - `oauthTokenId`: 임의값(1) — 내부에서 사용 안 함

4. **근본 해결**: 취소 승인 플로우에서 `calendar_event.deleted_at` 업데이트 로직 누락 버그 — 별도 수정 필요
