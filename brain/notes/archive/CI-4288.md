## 원인 분석

### 원인 확정 (L1)

API 인사발령 시 `displayOrder`가 null이면 0으로 fallback되어 모든 직무의 정렬 순서가 동일해지고, 구성원 목록 정렬 함수에 tiebreaker가 없어 대표 직무 표시가 비결정적으로 동작한다[^5][^8].

### 핵심 근거 (L2)

1. DB 조회 결과: 2026-04-01 발령 대상 50건 모두 `display_order = 0` — 복수 직무임에도 순서 구분 없음[^7]
2. API 발령 코드: `displayOrder ?: 0` fallback — 프론트가 null을 보내면 전부 0[^6]
3. 엑셀 발령 코드: `displayOrder = index` — 리스트 인덱스 기반으로 올바르게 설정 (동일 기능의 구현 불일치)[^5]
4. 구성원 목록 정렬 함수(`UserInfoLookUpServiceImpl.kt:348-349`)에 `userPositionId` tiebreaker 없음 — `displayOrder`가 동일하면 정렬 비결정적[^8]

### 가설 목록

| # | 가설 | 확인 방법 | 상태 |
|---|------|----------|------|
| 1 | API 발령 시 displayOrder null → 0 fallback으로 모든 직무 동일 순서 | DB 조회: 50건 모두 display_order=0 확인 | ✅ 확정 |
| 2 | UserInfoLookUpServiceImpl 정렬에 tiebreaker 없음 → 비결정적 표시 | 코드 확인: L348-349에 userPositionId tiebreaker 누락 | ✅ 확정 (보조 원인) |

4. 엑셀 발령 코드와 비교: `UserPersonnelAppointmentExcelServiceImpl.kt:915,937`에서는 `displayOrder = index` 사용 → 구현 불일치 확인[^5]

### 근본 원인 추적 (5 Whys)

1. 왜 주조직 표현이 사람마다 다른가? → 복수 직무의 정렬 결과가 사람마다 다름
2. 왜 정렬이 다른가? → `displayOrder`가 모두 0이라 tiebreaker에 의존, tiebreaker가 없거나 DB 순서 의존
3. 왜 `displayOrder`가 모두 0인가? → API 발령 시 프론트가 `displayOrder`를 null로 보내면 서버에서 `?: 0` fallback[^6]
4. 왜 서버에서 인덱스 기반 fallback이 없는가? → 엑셀 발령(`index` 사용)과 API 발령(`?: 0`)의 구현 불일치 **(근본 원인)**[^5][^6]

### 스펙 vs 버그: **버그**

동일 기능(인사발령)에서 경로에 따라 `displayOrder` 설정 방식이 다름:
- 엑셀 발령: `displayOrder = index` (올바른 구현)[^5]
- API 발령: `displayOrder = it.displayOrder ?: 0` (순서 유실)[^6]

## 다음에 같은 문의가 오면

1. **먼저 확인**: 해당 구성원의 `user_position_time_series_segment` 데이터에서 `display_order` 값 확인
   ```sql
   SELECT seg.id, seg.user_id, seg.is_primary, seg.display_order, seg.job_title_id
   FROM flex.user_position_time_series_segment seg
   WHERE seg.customer_id = ? AND seg.user_id = ?
     AND seg.deleted_date_time IS NULL
     AND seg.end_date_time > NOW() AND seg.begin_date_time <= NOW()
   ORDER BY seg.display_order, seg.id;
   ```
2. **원인 판별**:
   - 복수 직무의 `display_order`가 모두 동일(0)이면 → API 인사발령 displayOrder fallback 버그
   - `display_order`가 정상적으로 구분되어 있는데도 표시가 다르면 → 정렬 함수 tiebreaker 문제 또는 다른 원인
3. **조치**:
   - `display_order` 동일 → 사용자가 의도한 순서대로 `display_order` UPDATE 패치
   - 근본 수정 PR이 머지되었는지 확인 후, 미머지라면 재발령 또는 수동 패치
