# CI-4166: 계열사 전환 시 로그인이 풀리는 현상 확인 요청

> **상태**: SPEC (의도된 동작) | **담당**: @juwon | **생성일**: 2026-03-19

- **Linear**: [CI-4166](https://linear.app/flexteam/issue/CI-4166/계열사-전환-시-로그인이-풀리는-현상-확인-요청)
- **Slack**: [스레드](https://flex-cv82520.slack.com/archives/CRU35U9FC/p1773904535153399)

## 증상

- 계열사 전환 시 로그인이 풀리고 로그인창으로 이동
- 회사: 뉴넥스(219495, 주조직) / 하이버(222773, 겸직). 양방향 전환 모두 발생
- 대상자: yunwg@brandi.co.kr, seojm@brandi.co.kr (Google OAuth2 로그인)
- 빈도: 하루에 한번 정도. 자동 로그아웃 설정은 양쪽 모두 7일

## 원인 분석

**SSO(OAuth2/SAML2/OIDC) PC웹 로그인 시 workspace refresh token을 발급하지 않는 설계로 인해, workspace access token(12h) 만료 후 계열사 전환 불가**

### 토큰 구조와 수명 불일치

| 토큰 | 수명 | 갱신 | 역할 |
|------|------|------|------|
| workspace access token | **12시간** | refresh token 필요 | 계열사 전환(`/tokens/customer-user/exchange`) |
| workspace refresh token | 7일 | — | workspace access token 갱신 |
| customer-user token | 7일 | 자체 갱신 | 단일 회사 내 API 호출 |

### 분기 로직 (`flex-authentication`)

`LoginAuthorizationServiceImpl.kt:107`:
```kotlin
val workspaceToken = if (deviceInfo.mobile || authentications.hasRefreshable()) {
    tokenGrantService.grantAccessToken(grantRequest)       // access + refresh
} else {
    tokenGrantService.grantAccessTokenOnly(grantRequest)   // access만, refresh = null
}
```

`AuthorizationStrategy.kt`의 `notRefreshableMethods`:
- `CUSTOM_OAUTH2`, `CUSTOM_SAML2`, `OIDC`, `V2_DELEGATION`

→ **PC 웹 + OAuth2 = refresh token 미발급**. 모바일은 PLA-2261(2023-06)에서 예외 추가하여 발급.

### 결과

1. 사용자가 Google OAuth2로 로그인 → workspace access token(12h)만 발급, refresh token 없음
2. 12시간 이내: 단일 회사 사용(customer-user token 7일) 정상 + 계열사 전환 정상
3. 12시간 경과: 단일 회사 사용 여전히 정상 (customer-user token 유효) + **계열사 전환 시 401** (workspace access token 만료, 갱신 불가)
4. 재로그인하면 새 workspace access token 발급 → 12시간 동안 다시 정상

### 로그 증거 (2026-03-20 yunwg)


## 결론

**SPEC (의도된 동작)**. SSO 로그인은 외부 IdP 세션에 의존하므로, 웹에서는 IdP 재인증이 보안상 올바르다는 설계 의도[^seongbok][^minhyeok].

- 성복: "현재 확인된 것으로는 일단 의도된 동작", "sso는 refresh token을 내려주지 않는게 기존 코드"[^seongbok]
- 민혁: "SSO 로그인 후 12시간이 지나고부터 계열사 전환시에 로그인이 풀리는 것이 스펙"[^minhyeok]
- 개선안으로 재로그인 배너 표시 정도만 논의 중. EPBE에 공유 예정(관빈)[^gwanbin]

## 해결

고객사는 계열사 생성한지 약 2달. 기존에 안 되던 것이 아니라 처음부터 이 동작이었을 가능성 높음[^juwon].
고객에게 "SSO 로그인 특성상 12시간 후 계열사 전환 시 재로그인이 필요하다"고 안내 (김한나님 진행)[^hanna].

## 다음에 같은 문의가 오면

1. **인증 방식 확인**: SSO(OAuth2/SAML2/OIDC)인지 비밀번호 로그인인지
2. **SSO + PC 웹이면**: workspace access token 12시간 만료로 인한 정상 동작 안내
3. **비밀번호 로그인인데 발생하면**: 다른 원인 조사 필요 (refresh token이 발급되므로)

## 시나리오

```gherkin
# language: ko
기능: SSO 로그인 후 계열사 전환 시 workspace access token 만료

  배경:
    주어진 유저가 뉴넥스(주회사)와 하이버(겸직회사)에 소속되어 있다
    그리고 유저가 Google OAuth2로 PC 웹에서 로그인했다

  시나리오: 12시간 이내 계열사 전환 성공
    주어진 로그인한 지 6시간이 경과했다
    만약 유저가 뉴넥스에서 하이버로 계열사 전환을 시도한다
    그러면 workspace access token이 유효하므로 exchange API가 200을 반환한다
    그리고 유저는 하이버 화면으로 전환된다

  시나리오: 12시간 경과 후 계열사 전환 실패
    주어진 로그인한 지 13시간이 경과했다
    만약 유저가 뉴넥스에서 하이버로 계열사 전환을 시도한다
    그러면 workspace access token이 만료되었고 refresh token이 없으므로 갱신 불가
    그리고 exchange API가 401 AUTH_401_000을 반환한다
    그리고 유저는 로그인 화면으로 이동한다

  시나리오: 단일 회사 사용은 12시간 경과 후에도 정상
    주어진 로그인한 지 13시간이 경과했다
    만약 유저가 뉴넥스 내에서 API를 호출한다
    그러면 customer-user token(7일)이 유효하므로 정상 동작한다
```
