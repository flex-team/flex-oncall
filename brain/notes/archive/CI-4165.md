# CI-4165: SECOM 외부 출퇴근 이벤트 중복 등록으로 진행중 블럭 다건 발생

> **상태**: 해결 완료

## 증상

- 사용자 668756 (`pppp@chronostudio.net`, 고객사 외부연동 ID: 253)
- 3월 18일 SECOM에서 WORK_START 이벤트가 **7건** 수신 (터미널 6, 7, 8, 9에서 분산)
- 그 중 **5건이 모두 work_clock_event로 등록**되어 위젯에 진행중 블럭이 여러 건 표시
- onGoing이 하나만 true여야 하는데 여러 개가 true로 반환됨

### 등록된 START 이벤트

| work_clock_event ID | 출근 시간 (KST) | terminal_id |
|---|---|---|
| 125661917 | 10:09 | 9 |

| 125667921 | 14:07 | 9 |

### 추가 사실

- SECOM WORK_STOP 2건은 `DISABLED_CONFIRM_SKIP`으로 등록 거부됨
- 각 START마다 PLAN_BY_AUTO로 REST_START/REST_STOP/STOP이 자동 생성 → 총 17건의 이벤트
- 외부 provider 이벤트(`v2_user_external_provider_event`)에는 10건 저장 (START 7 + STOP 2 + ENTER 1)

## 원인 분석

3개의 방어선이 각각 다른 이유로 뚫림.

### 방어선 1: Unique Key 중복 검증 — 통과 (정상)

서로 다른 SECOM 터미널에서 서로 다른 시간에 전송한 별개의 이벤트. Unique key = `providerId#userId#eventType#eventTime(millis)` 구조상 당연히 통과. 이건 버그가 아닌 정상 동작.

### 방어선 2: `isDraftEventRegistrationAllowed` — 이벤트 타입 미구분

`UserWorkClockDraftEventRegisterServiceImpl.kt:99-104`:

```kotlin
val hasWorkClockEventEarlierThanRequested = userWorkClockRecordPackOnRegistrationDate.recordPacks.any {
    it.workClockStartEvent.targetTime <= registerEvent.eventTime
}
return hasWorkClockEventEarlierThanRequested
```

**"기존 START보다 늦은 시간의 이벤트면 허용"** 하는 로직. STOP/REST 등 후속 이벤트를 위한 설계이지만, **이벤트 타입(START/STOP)을 구분하지 않아** 새로운 START 이벤트도 무조건 통과.

실제 데이터로 검증:
- 10:09 START → `recordPackOnRegistrationDate == null` → `true` (첫 이벤트)
- 12:44 START → 기존 START(10:09) <= 12:44 → `true` → **통과** (기존보다 늦으니까)
- 14:07 START → 기존 START <= 14:07 → `true` → **통과**
- 09:41 START → 기존 START(10:09) <= 09:41 → `false` → **차단** (기존보다 빠르니까)
- 09:50 START → 기존 START(10:09) <= 09:50 → `false` → **차단**

이것이 **7건 중 5건 통과, 2건 차단**의 이유.

### 방어선 3: `WorkClockContinuousRegisterValidator` (dry-run) — Kafka 동시성 race condition

코드상으로는 확실히 차단되어야 하는 구조:

```kotlin
// WorkClockContinuousRegisterValidator.kt:30-42
registerContext.workClockStatus.onGoingEventPack?.let {
    listOf(SimpleWorkClockRegisterValidationResult(
        validation = WORK_CLOCK_START_CONTINUOUS_NOT_ALLOWED,  // level = ERROR, earlyReturn = true
    ))
} ?: emptyList()
```

`WORK_CLOCK_START_CONTINUOUS_NOT_ALLOWED`는 `level = ERROR` (default), `earlyReturn = true` → `UserWorkClockUpdateDryRunServiceImpl:127-134`에서 `UserWorkClockRegisterFailResult` 반환 → `UserWorkClockStartByDraftEventServiceImpl:170-174`에서 `출근_dryRun에_성공했을_경우 = false` → wet-run 미실행이 정상 경로.

**하지만 DB 데이터가 반증:**
- 5건 모두 `v2_user_work_clock_draft_event.fail_message = NULL` → dry-run 통과했음
- 5건 모두 `registered_user_work_clock_event_id` 세팅됨 → wet-run 실행되어 work_clock_event 생성됨

**Race condition 근거:**

1. `registerWorkClockEvent()`은 `@Transactional(propagation = Propagation.REQUIRES_NEW)` → 독립 트랜잭션
2. 외부 이벤트 7건이 **25초 내에** 일괄 수신 (created_at: 17:06:21~17:06:46 KST)
3. 터미널 4대(6,7,8,9) → **서로 다른 Kafka 파티션 가능** → 다른 consumer 스레드에서 동시 처리
4. MySQL REPEATABLE READ 격리 수준에서 동시 시작된 트랜잭션은 **서로의 미커밋 데이터 미가시**

```
Thread A (terminal 9, 10:09):  TX_A start → checkWorkClockStatus() → 기존 이벤트 없음 → dry-run 통과 → wet-run → TX_A commit
Thread B (terminal 6, 12:44):  TX_B start → checkWorkClockStatus() → TX_A 미커밋 → 기존 이벤트 없음 → dry-run 통과 → wet-run → TX_B commit
Thread C (terminal 9, 14:07):  TX_C start → 동일 패턴 → ...
```

### 부수 발견: `isOngoing()` dead code 버그

```kotlin
// UserWorkClockEventPackModel.kt:69-83
return conditionA && (
    stopEvent?.recordType !== RECORD ||           // B
    (stopEvent?.recordType !== RECORD &&           // B (중복)
        (stopEvent?.targetTime ?: 0) >= requestTime)  // C (dead code)
)
```

`B || (B && C)` ≡ `B`. **시간 비교(C)가 dead code**. STOP 시간 경과 여부와 무관하게 PLAN_BY_AUTO STOP이면 항상 ongoing=true. 이번 이슈의 직접 원인은 아니지만 잠재적 버그.

### 이벤트 흐름

```
SECOM 터미널 4대 → Kafka (파티션 분산) → 다중 consumer 스레드
  → AbstractExternalEventRegisterService.registerEvent()  [@Transactional]
    → registerEventIfNotExist()          // unique key → 통과 (시간 다름)
    → registerWorkClockEvent()           [@Transactional(REQUIRES_NEW)]
      → isDraftEventRegistrationAllowed()  // 기존 START보다 늦으면 → true (타입 미구분)
      → registerDraftEvent()             // draft 저장
      → dryRunToRegisterStart()          // checkWorkClockStatus() → 동시 TX라 기존 이벤트 못 봄
        → WorkClockContinuousRegisterValidator → onGoingEventPack == null → emptyList (통과)
      → wet-run → work_clock_event 생성
      → TX commit
```

## 해결

- 조사 중

### 수정 방향 (검토 필요)

1. **`isDraftEventRegistrationAllowed()`**: 새 이벤트가 START 타입일 때, 같은 날 진행중 블럭이 있으면 차단
2. **동시성 방어**: 같은 유저+같은 날 외부 출퇴근 이벤트 처리 시 분산 락 또는 DB unique constraint 추가
3. **`isOngoing()` dead code 수정**: `B || C` 로 변경하여 STOP 시간 경과 시 ongoing=false
