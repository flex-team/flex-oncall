# CI-3979: 세콤 퇴근 타각 시 근무 기록 시간 조정 원인 확인

## 증상
- **회사**: 도그지어 (Customer ID: 18005)[^1]
- **문의자**: jm.park@dogsear.co.kr[^1]
- **대상 구성원**: ds.lim@dogsear.co.kr (user_id: 924127)[^1][^5]
- 문의 내용:
  1. 1/29 10:00에 근무 시작하여 1/30 00:34:29에 세콤[^4]으로 퇴근 타각[^1]
  2. 실제 기록은 **00:00으로 조정**되어 34분 29초가 잘림 (정상 기대값: 00:34:29)[^1]
  3. 근무 유형상 제한 설정이나 최대 근로시간 초과 등은 따로 없다고 함[^1]
  4. 세콤 타각으로 종료된 건이라 문의자가 직접 확인이 어려워 이슈로 요청[^1]

> ⚠️ 일자를 넘기는 야간 근무(1/29 10:00 ~ 1/30 00:34:29) 상황에서 세콤 퇴근 타각이 자정(00:00)으로 잘린 것이 핵심 증상이다.

---

## 원인 분석

### 원인

> 💡 **1/30에 대상 구성원(ds.lim)에게 종일휴가가 등록되어 있었기 때문에**, `WorkClockTimeAdjuster.adjustWorkClockStopTime()`의 휴가 겹침 체크(line 112~122)에서 퇴근 시간이 휴가 시작 시간인 **1/30 00:00:00 KST로 조정**된 것이 원인이다[^18].

세콤 태깅 시간(00:32:29)은 dryrun(사전 검증 단계 — 실제 저장 없이 조정 결과만 계산)까지 정상 전달되었으나, wetrun(실제 저장 단계 — 조정 결과를 DB에 반영) 단계에서 종일휴가에 의해 00:00:00으로 변경됨[^19].

<details>
<summary>📋 adjustWorkClockStopTime() 휴가 체크 코드</summary>

```kotlin
// WorkClockTimeAdjuster.kt:112-122
val overlappedTimeOffUse = timeOffUses
    .flatMap { it.times }
    .firstOrNull { eventTime.toEpochMilli() in it.timeOffUseRange }
val 휴가_진행중일_경우 = overlappedTimeOffUse != null
if (휴가_진행중일_경우) {
    return@run WorkClockTargetTimestamp(
        onTime = false,
        targetTimestamp = overlappedTimeOffUse!!.timeOffUseRange.first, // 휴가 시작 시간 = 00:00:00
        originalTimestamp = eventTime.toEpochMilli(),
    )
}
```

</details>

### 조사 과정

> 💡 **판단 근거**: 세콤 퇴근 00:32:29 → 위젯 STOP 00:00:00 조정 확인[^5]
> → dryrun에는 00:32:29 전달, wetrun에서 00:00:00 변경[^19]
> → `adjustWorkClockStopTime()` 내에서 변경 발생[^20]
> → 설정 기반 4가지 조정 경로 모두 소거[^8][^15]
> → **1/30 종일휴가 존재 확인 → 휴가 겹침 체크(line 112~122)에서 조정됨**[^18]

<details>
<summary>📋 과거 잘못된 판단 — "휴가 없음" 가정</summary>

초기 조사에서는 문의자가 "제한 설정이나 최대 근로시간 초과 등은 없다"고 했기 때문에[^1], 휴가가 없다고 가정하고 설정 기반 조정 경로를 먼저 조사했다. 이후 실제 DB에서 1/30 종일휴가가 등록되어 있음을 확인하여 원인이 확정되었다[^18].

</details>

## 해결
- **원인**: 대상 구성원(ds.lim)의 1/30 종일휴가가 등록되어 있어, 자정을 넘긴 퇴근 타각(00:32:29)이 휴가 시작 시간(00:00:00)으로 조정됨[^18]
- **판정**: 스펙에 의한 **정상 동작**. 00시 초과 시간을 인정하려면 1/30 휴가를 조정해야 함[^21]
- **개선 포인트**[^22]:
  - consumer 로그 보기 위해 과거 로그 복구 시 consumer가 아닌 api 인덱스를 살려야 함
  - 조정 로그 보강 필요

## 다음에 같은 문의가 오면

1. **먼저 확인**: 퇴근 시간이 자정(00:00:00)으로 잘린 경우, **대상 구성원의 다음날 휴가 등록 여부**를 먼저 확인
   - Metabase 대시보드: `https://metabase.dp.grapeisfruit.com/dashboard/256?customer_id={customer_id}`
2. **원인 판별**:
   - 다음날 종일휴가가 있으면 → 스펙에 의한 정상 동작. `adjustWorkClockStopTime()` 휴가 체크(line 112~122)에서 퇴근 시간을 휴가 시작 시간으로 조정함
   - 휴가가 없으면 → 다른 원인 조사 필요 (설정 기반 조정 경로 확인: 가설 1~4 참고)
3. **안내**: "자정을 넘긴 퇴근 시간이 다음날 종일휴가와 겹치면 휴가 시작 시간(00:00)으로 조정됩니다. 00시 이후 시간을 인정하려면 해당 휴가를 조정해주세요."
