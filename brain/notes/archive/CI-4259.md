## 원인 분석

**calendar-backend가 all-day event를 UTC 자정으로 저장하면서, UTC+ 타임존 조회 시 전날 이벤트가 당일 결과에 포함되는 구조적 버그**

### 근본 원인 (코드 4곳에서 확인)

1. **all-day event는 UTC 자정으로 저장**[^21]:
   ```kotlin
   // CalendarEvent2.kt:31-33
   val startAt: LocalDateTime, // DATE인 경우 UTC로 자정
   val endAt: LocalDateTime,   // DATE인 경우 UTC로 자정
   ```

2. **attendee 인덱스 컬럼에 동일 값 저장**[^22]:
   ```kotlin
   // CalendarEventJdbcMapper.kt:66-67
   indexFrom = event.startAt,
   indexTo = event.endAt,
   ```

3. **조회 시 UTC 변환 후 비교**[^23]:
   ```kotlin
   // CalendarEventQueryUseCase.kt:89-90
   startAfter = zonedDateTimeRange.from.withZoneSameInstant(ZoneOffset.UTC).toLocalDateTime(),
   endBefore = zonedDateTimeRange.to.withZoneSameInstant(ZoneOffset.UTC).toLocalDateTime(),
   ```

4. **SQL overlap 비교**[^24]:
   ```sql
   -- CalendarEventAttendeeJdbcRepository.kt:75-76
   AND index_to > :startDateTime
   AND index_from < :endDateTime
   ```

### 버그 메커니즘

3/30 all-day 휴가가 3/31 KST 조회에 포함되는 과정:

```
3/30 all-day event DB 저장값:
  index_from = 2026-03-30T00:00 (UTC 자정)
  index_to   = 2026-03-31T00:00 (UTC 자정)

3/31 KST 조회 → UTC 변환:
  startDateTime = 2026-03-30T15:00 (3/31 00:00 KST → UTC)
  endDateTime   = 2026-03-31T15:00 (4/1 00:00 KST → UTC)

SQL 비교:
  index_to(3/31T00:00) > startDateTime(3/30T15:00) → TRUE
  index_from(3/30T00:00) < endDateTime(3/31T15:00) → TRUE
  → 전날 이벤트가 당일 결과에 포함됨
```

> 💡 **핵심**: all-day event의 날짜 범위(UTC 00:00~00:00)와 KST 쿼리 범위(UTC 15:00~15:00) 사이에 **9시간 overlap 구간**이 발생. all-day event는 "날짜" 개념이므로 타임존과 무관하게 날짜 기준으로 비교해야 하는데, datetime overlap으로 비교하는 것이 근본 원인.

### 4차: 이벤트 생성 시점 추적 — 전환점

ULID 디코딩으로 **calendar event가 다른 시점/다른 calendar에서 생성**된 것을 발견:

| calendarEventId | 생성 시각 (ULID) | calendarId |
|-----------------|-----------------|------------|
| `01kmfank7w...` (첫 sync) | 3/24 16:06 | `01h3h20qxy...` (group calendar) |
| `01kmpbdpc7...5e` (현재) | 3/27 09:34 | `01jgk2dhtg...` (다른 calendar) |

이 시점에서 **"유령 이벤트"가 아니라 "다른 calendar의 정상 이벤트"**일 수 있다는 의심이 생겼다.

### 5차: 사용자 식별 오류 발견 — 핵심 전환점

access log의 이벤트에 포함된 `userIdHash: xBEYJjYZ8g`를 디코딩[^25] → **userId 781345 (손정욱)** — 이준호(229504)가 아니었다.

> 💡 이택규가 확인한 Kibana 로그[^7]는 이준호가 아닌 **손정욱의 calendar**를 조회한 것이었다. 이준호의 "유령 이벤트"를 추적하고 있었는데, 실제로 보고 있던 데이터는 손정욱의 것.

### 6차: 손정욱 기준 재분석 — 문제 재정의

손정욱 DB 조회[^20] → 3/27, 3/30, 3/31, 4/1 연차 확인. **3/30과 3/31 모두 정당한 휴가.**

이벤트 2건은 "유령"이 아니라 정상 데이터. 문제는 **3/31을 조회할 때 3/30 이벤트가 같이 반환**되는 것.

```

  ✓ "왜 전날의 all-day 이벤트가 당일 조회에 포함되는가" (올바른 질문)
```

### 7차: calendar-backend 코드 확인 — 원인 확정

`flex-calendar-backend` 서브모듈을 추가하여 조회 로직을 직접 확인:

1. `CalendarEventQueryUseCase.kt:89-90`[^23] — 쿼리 파라미터를 UTC로 변환
2. `CalendarEventAttendeeJdbcRepository.kt:75-76`[^24] — `index_to > :startDateTime AND index_from < :endDateTime` overlap 비교
3. `CalendarEvent2.kt:31`[^21] — all-day event는 "DATE인 경우 UTC로 자정" 저장

이 3개 조합으로 UTC+ 타임존에서 전날 all-day event가 당일 결과에 포함되는 메커니즘을 확정.

### 8차: "언제부터 버그였는가" — PR #866 검증

PR #866[^26] (2026-03-17)이 `.toLocalDateTime()` → `.withZoneSameInstant(UTC).toLocalDateTime()` 변경을 도입. 이것이 원인인지 검증하기 위해 **3/16(배포 전) access log**[^27]를 확인.

결과: **호출자가 보내는 타임존 포맷에 따라 달랐다.**

같은 순간(KST 3/31 자정)을 두 가지로 표현할 수 있다:

```
형태 A: 2026-03-31T00:00+09:00   (KST 포맷)
형태 B: 2026-03-30T15:00Z         (UTC 포맷)
```

PR #866 이전의 `.toLocalDateTime()` 은 타임존을 **그냥 벗겨버린다**:

| 입력 포맷 | `.toLocalDateTime()` 결과 | 3/30 event의 index_to(`3/31T00:00`)와 비교 | 전날 포함? |
|----------|--------------------------|-------------------------------------------|-----------|
| A (KST `+09:00`) | `03-31T00:00` | `03-31T00:00 > 03-31T00:00` → FALSE | ✅ 정상 (우연[^28]) |
| B (UTC `Z`) | `03-30T15:00` | `03-31T00:00 > 03-30T15:00` → TRUE | ❌ 버그 |

**같은 시각인데 포맷에 따라 결과가 달랐다.** PR #866 이후에는 둘 다 UTC로 변환되어 둘 다 버그.

3/16 access log에서 실제로 두 포맷이 혼재[^27]:
- GET 엔드포인트: `+09:00` (KST) → PR #866 이전에는 정상
- POST 엔드포인트: `T15:00Z` (UTC) → **PR #866 이전에도 이미 버그**

## 해결

### 즉시 대응: PR #866 리버트 (calendar 팀 주도)

한지우가 PR #866[^26] 리버트 후 배포를 진행 예정[^29]. PR #866의 원래 맥락은 "9시 이전의 일정이 표시되지 않는 이슈"[^30] 수정이었으나, all-day event에 대한 고려 없이 일괄 UTC 변환을 적용하여 이 버그를 유발.

> ⚠️ 리버트 시 PR #866이 수정했던 원래 이슈(9시 이전 일정 미표시)가 다시 발생할 수 있음. 리버트 후 두 이슈를 동시에 해결하는 방법을 검토할 예정.

### calendar 팀 합의 방향

전성윤이 제시한 수정 원칙[^31][^32]:
- `start_date_type = DATE` → UTC에서 time을 제거한 **날짜(DATE)**로만 비교
- `start_date_type = DATETIME` → 타임존에 따른 **date + time** 비교

이 원칙은 기존 노트의 "권장: Application-level post-filter" 해결안과 방향이 일치한다.

### 참고: 이전 분석에서 제시한 해결안

#### Application-level post-filter (안전, 변경 최소)

`CalendarEventQueryUseCase`에서 repository 조회 결과를 **후처리**하여 DATE 이벤트의 false positive를 제거:

```kotlin
// CalendarEventQueryUseCase.kt — repository 조회 후 추가
val queryStartDate = zonedDateTimeRange.from.toLocalDate()
val queryEndDate = zonedDateTimeRange.to.toLocalDate()

events.filter { event ->
    if (event.dateType == DateType.DATE) {
        // all-day 이벤트는 날짜 기준으로 필터 (타임존 무관)
        val eventStartDate = event.startAt.toLocalDate()
        eventStartDate >= queryStartDate && eventStartDate < queryEndDate
    } else {
        true // DATETIME은 SQL에서 이미 정확히 필터됨
    }
}
```

#### SQL 분기 (성능 우선)

attendee 테이블에 `date_type` 컬럼 추가 또는 event 테이블 JOIN. 변경 범위 큼.
