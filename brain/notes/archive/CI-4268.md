## 원인 분석

**draft에 여러날 휴가 `timeOffDeletion`이 잔존하여, 게시 시 `MultiDayTimeOffCancellationValidator`가 ERROR를 반환하는 것으로 추정** [추론: 코드 흐름 분석 + CI-3892 패턴 일치]

### 조사 과정

> 💡 **코드 분석 — 게시 검증 체인 확인**
> → 게시 시 dry-run API가 5개 validator를 실행[^3]:
> `OverlappedScheduleRegisterValidator`, `WeeklyPaidHolidayAndWeeklyUnPaidHolidayOverlapValidator`, `TimeOffAndWeeklyHolidayAndOverlapValidator`, `ShiftScheduleRegisterPermissionValidator`, **`MultiDayTimeOffCancellationValidator`**
> → 이 중 `MultiDayTimeOffCancellationValidator`가 `timeOffDeletions.filter { it.targetDates.size > 1 }` 로 여러날 휴가 삭제를 감지하여 ERROR 반환[^4]

> 💡 **코드 분석 — timeOffDeletions 확장 메커니즘**
> → `convertToRegisterModel`[^5]에서 draft의 `timeOffDeletion`(eventId만 저장)을 `publishedShifts`의 전체 날짜로 확장:
> `targetDatesByTimeOffEventId[eventId]` → 여러날 휴가의 모든 날짜가 `targetDates`에 포함됨
> → 즉, 하루만 삭제 대상이어도 해당 eventId의 전체 날짜가 삭제 대상으로 확장됨

> 💡 **코드 주석 발견 — 과거 인지된 문제**
> → `UserShiftScheduleV3Converter.kt` 에 주석[^6]: "단일 휴가에 대해서만 삭제 가능하게 스펙 변경됨. **하지만 과거에 여러날 휴가에 대한 삭제 draft가 존재할 순 있으며** 시간이 지나면 점차 없어질 것으로 예상"
> → 즉, 여러날 휴가의 `timeOffDeletion` draft가 잔존할 수 있는 것은 이미 인지된 상황

<summary>📋 추정 메커니즘 (코드 기반 추론)</summary>

```
1. 사용자가 어느 시점에 4/25~29 여러날 휴가 날짜 중 하나에서 스케줄 편집
   → draft DB에 해당 날짜의 timeOffDeletion=[eventId] 저장
2. 게시 시 convertToRegisterModel 실행
   → publishedShifts에서 eventId의 전체 targetDates 조회 → {4/25, 4/26, 4/27, 4/28, 4/29}
   → TimeOffEventWithDateModel.of(eventId, {4/25~4/29}) 생성
3. MultiDayTimeOffCancellationValidator 실행
   → timeOffDeletions.filter { it.targetDates.size > 1 } → 5일 > 1 → 매칭
   → WorkShiftRegisterValidationType.TIME_OFF_MULTI_DAY_CANCELLATION_NOT_ALLOWED (ERROR)
4. FE가 dry-run 응답의 ERROR를 "확인 필요"로 표시
```

</details>

  - 관련 이슈: [TT-15807](https://linear.app/flexteam/issue/TT-15807/교대-근무-drywet-run에서-여러날-휴가-검증-추가) — 교대 근무 dry/wet-run에서 여러날 휴가 검증 추가
