## 원인 분석

**FE가 activities 조회 시 `sort=ASC&size=1`로 요청하여 action history만 반환되고 댓글이 누락되는 현상. BE 응답은 정상.**

### 가설 검증

| # | 가설 | 확인 방법 | 상태 |
|---|------|----------|------|
| 1 | BE의 targetUid 매핑 오류로 댓글이 다른 승인 건에 연결 | 코드 분석 | ❌ 소거 — UUID 기반 고유키, 구조적 결함 없음[^4] |
| 2 | FE에서 승인 건 전환 시 targetUid를 잘못 참조 (캐시/상태 관리) | access log 응답 분석 | ❌ 소거 — BE 응답에 cross-contamination 없음[^6] |
| 3 | FE가 sort=ASC&size=1로 요청하여 댓글 누락 | access log 요청 파라미터 + 응답 대조 | ✅ 확정[^7] |

### 핵심 근거

1. access log 분석 결과, 동일 targetUid(7a844451...)에 대해 FE가 2개의 병렬 요청을 보냄[^7][^8]:
   - `sort=ASC&size=1` → action history 1건만 반환, 댓글 0건 (total=3이지만 **size=1이라 누락**)[^7]
   - `sort=DESC&size=3` → 최근 활동 3건 반환, 댓글 포함[^8]
2. 6개의 서로 다른 targetUid에서 댓글은 작성된 건(7a844451...)에서만 반환됨 — BE cross-contamination 없음[^6]
3. 관리자(myg)가 동일 댓글을 19초 간격으로 2번 POST → BE에 중복 저장됨 (중복 방지 로직 없음)[^9]

> 💡 **판단 근거**: BE 코드 분석에서 targetUid가 UUID 기반 고유키임을 확인[^4]
> → access log에서 다른 targetUid 응답에 댓글 0건 확인[^6]
> → FE 요청 파라미터 `sort=ASC&size=1`에서 댓글 누락 확인[^7]
> → 이지우님(FE)이 동일 발견 보고[^10]

1. **BE 코드 분석**: `flex-core-backend`의 `CoreUserDataApprovalChangeApiController.kt:847-849`에서 targetUid를 `UUID.randomUUID().toString().replace("-", "")`로 생성하는 것을 확인[^4]. 각 승인 건마다 고유한 UUID가 할당되므로 구조적으로 다른 건에 댓글이 연결될 수 없음.

2. **Thread 시스템 확인**: `flex-flow-backend`의 `ThreadTopicIdentifier.kt`에서 key=targetUid, type=USER_DATA로 복합키를 구성[^5]. targetUid가 고유하므로 다른 건과 교차될 가능성 없음.

3. **access log 분석 (카임 PM0v4gdpEX, 2026-03-24 09:54~10:00 KST)**: 6개 targetUid의 activities 응답을 대조한 결과, 댓글은 7a844451...에서만 반환됨[^6]. 이로써 BE cross-contamination 가설 소거.

4. **FE 요청 패턴 분석**: traceId `e2ac1c170b45a85eb513400260f2e5b8` 건에서 `sort=ASC&size=1` 요청이 total=3임에도 1건만 반환하는 것을 확인[^7]. 동일 targetUid에 대한 `sort=DESC&size=3` 요청은 댓글을 포함하여 정상 반환[^8].

5. **중복 POST 확인**: 2026-03-24 09:57:09, 09:57:28에 동일 내용 댓글이 2회 POST되어 각각 200 응답을 받음[^9]. BE에 idempotency 로직이 없어 중복 저장됨.

6. **FE 확인**: 이지우님이 Linear 코멘트에서 FE가 `sort=ASC&size=1`로 보내고 있음을 확인 보고[^10].

## 해결안

**스펙 vs 버그**: 버그 (FE)

### 즉시 대응 (표면 원인)

- FE에서 activities 조회 시 `size` 파라미터를 충분히 크게 설정하거나, 댓글 포함 여부를 고려한 페이지네이션 로직 수정
- 이지우님이 FE에서 size 결정 로직 조사 중[^10]

### 근본 해결

- FE의 activities 조회 패턴 재설계: `sort=ASC&size=1`은 header용 첫 활동 조회 목적이라면, 댓글 렌더링에는 `sort=DESC` 응답만 사용해야 함

### BE 부가 이슈

- 동일 내용 댓글 중복 POST 방지 로직 없음 (idempotency 미적용)[^9]
- 우선순위 낮음 — FE 버그 수정이 근본 원인 해결
