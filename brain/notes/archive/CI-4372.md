# CI-4372: 위젯 출퇴근 후 근무 그래프 미표시 — canCreateWorkRecordWithWorkClock 설정 off

> **상태**: 해결 완료 (스펙대로 동작 확인) — 2026-04-09

## 증상
- **문제 정의**: 위젯으로 출퇴근(STOP)했으나 근무 그래프/캘린더에 해당 일자 근무가 표시되지 않음
- **회사**: 신우디엔시 (Customer ID: 105238)
- **요청자**: 이주화 (CS팀)[^1]
- **대상자**: kkm6466@naver.com (user_id: 874362)[^2]
- **영향 범위**: 해당 회사에서 `canCreateWorkRecordWithWorkClock = false` 설정 기간 동안 위젯으로 출퇴근한 구성원
- **문제 시점**: 2026-04-08 (어제 근무)
- 문의 내용:
  1. 4/8 근무 종료/확정 후 근무 그래프가 조회되지 않음[^1]
  2. 유사 문의 1건 추가 인입[^3]

## 현재까지 파악된 내용

### 결론: 의도대로 동작한 상황 (Spec)

> 💡 안희종 정리[^11]:
> - 퇴근 시점에 근무 유형이 **미반영** 설정이었음
> - 미반영이니까 출퇴근 기록은 **인정 근무로 반영되지 않음**
> - 원래도 인정 근무가 없었음 → 결과적으로 인정 근무는 빈 채로 남음
> - **출퇴근 기록(위젯)은 남아있음**
> - 하지만 유저는 실시간 기록 조회 **권한이 없어서** 종결된 시점 이후로 못 봄
> - 결과적으로 아무것도 없는 것처럼 보이는데 **의도된대로 동작한 상황**
> - 관리자가 들어가면 실시간 기록은 보일 것

---

### FE 확인 결과 (박준홍)
- work-clock STOP API(`POST /api/v2/time-tracking/work-clock/users/{userIdHash}/stop`) 호출 시 HTTP 200 응답[^4]
- 응답의 `targetDayWorkSchedule.workRecords: []` (빈 배열) — 근무 기록 미생성[^4]
- 이후 `GET .../work-schedules` 에서도 4/8 `timeBlocks: []`[^4]
- 4/6, 4/7은 정상 (WORK 08:00~17:00, eventStatus: RECORD)[^4]
- **FE 코드(SummaryGraph, CalendarPage)는 데이터가 있으면 정상 렌더링 → FE 이슈 아님**[^4]

### BE 조사 과정

> 💡 **조사 흐름**:
> → 유저 조회 → 4/8 근무 이벤트 확인 → 근무 블럭 비어있음 발견
> → 위젯 이벤트는 정상(17:03 STOP) → 근무 유형 설정 확인
> → `canCreateWorkRecordWithWorkClock` 설정이 false였음 확인[^12]
> → 관리자가 오늘(4/9) 설정 변경한 이력 발견 → **의도된 동작 확인**

#### DB 조사 쿼리

**유저 조회**[^2]:
```sql
select customer_id, id as user_id, email
from view_user
where email = trim('kkm6466@naver.com');
-- customer_id: 105238, user_id: 874362
```

**4/8 근무 이벤트 — 블럭이 비어있음**[^5]:
```sql
select * from v2_user_work_record_event
where user_id = 874362 and date_from >= '2026-04-07';

-- event_id 64040800: 4/8 근무 이벤트 존재, 타입이 REGISTER (REGISTER_BY_WORK_CLOCK 아님)
select * from v2_user_work_record_event_block
where user_work_record_event_id = 64040800;
-- 결과: 블럭 비어있음
```

**오늘(4/9) 재등록 — 설정 복원 후 정상**[^5]:
```sql
select * from v2_user_work_record_event_block
where user_work_record_event_id = 64091197;
-- 오늘 다시 등록 → 정상 블럭 생성
```

**위젯 이벤트는 정상 기록됨**[^5]:
```sql
select * from v2_user_work_clock_event
where user_id = 874362 and date_from >= '2026-04-08';
-- 17:03:23에 STOP 정상 기록
```

**핵심 설정 — canCreateWorkRecordWithWorkClock**[^5]:
```sql
-- 근무 유형 조회
select * from v2_user_work_rule where user_id = 874362;
select * from v2_customer_work_rule where id = 200107;

-- 설정 확인
select * from v2_customer_work_record_rule where id = 200111;
-- 현재: can_create_work_record_with_work_clock = null (기본값 true)

-- 변경 이력
select * from v2_customer_work_record_rule_aud where id = 200111;
-- 오늘(4/9) 변경됨
```

#### Access Log 확인

진재언이 4/8 STOP 요청 시점의 access log를 확인[^12]:
- STOP API 요청 바디에 `canCreateWorkRecordWithWorkClock: false` 가 포함되어 있음
- **퇴근 당시 설정이 false였음을 access log로 확증**

#### 코드 분석

- `UserWorkClockStopServiceImpl.kt` L743: `canCreateWorkRecordWithWorkClock` 설정에 따른 분기 진입점[^8]
- L817-L821: 설정 false일 때 근무 블럭 생성 스킵하는 구체적 로직[^13]
- 설정이 false이면: work_record_event는 REGISTER 타입으로 생성되지만 work_record_event_block은 빈 채로 남음[^6]
- **null이면 true가 기본값**[^9]

#### 설정 변경 경위

- DB audit 결과 최고관리자(coolmkholic@naver.com)가 설정 변경[^7]
- dev 환경에서 아무것도 안 건드리고 저장하면 true로 전달됨 → **관리자가 의도적으로 false로 설정한 것**[^10]
- 진재언: "유저 의도라고 볼 수밖에.."[^14]
- 오늘(4/9) 다시 설정을 복원(null → 기본값 true)[^7]

#### REGISTER vs REGISTER_BY_WORK_CLOCK 타입 논의

- 김영준: "이것도 그냥 REGISTER_BY_WORK_CLOCK으로 했어야 했나.. DB에 STOP은 있는데 REGISTER_BY_WORK_CLOCK이 없어서 좀 이상했음"[^15]
- 진재언: "저 타입이 클락으로부터 등록된거니까, 그건 그대로 유지하는게 더 좋아보이네요 (부가정보는 별도컬럼에 저장?)"[^16]
- → 현재 타입 체계 유지, 추가 정보가 필요하면 별도 컬럼 활용 방향

## 영향 범위
- 신우디엔시(105238)에서 `canCreateWorkRecordWithWorkClock = false` 기간 동안 위젯 출퇴근한 구성원
- 유사 문의 1건 추가 인입으로 최소 2명 이상 영향[^3]
- 금전적 영향: 해당 기간 근무 기록이 인정 근무로 반영되지 않으므로 급여 계산에 영향 가능. 단, 관리자가 의도적으로 끈 설정이므로 시스템 결함은 아님.

## 발견한 스펙/제약
- `canCreateWorkRecordWithWorkClock = false`일 때 위젯 STOP은 200을 반환하지만 근무 블럭(인정 근무)을 생성하지 않음 — **의도된 스펙**[^6]
- 출퇴근 기록(work_clock_event) 자체는 남지만, 실시간 기록 조회 권한이 없는 유저는 확인 불가[^11]
- `canCreateWorkRecordWithWorkClock`이 null이면 true로 동작[^9]
- dev 환경에서 아무것도 안 건드리고 저장하면 true로 전달됨 → false는 명시적 의도[^10]
