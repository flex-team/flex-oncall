# CI-4195: 삭제된 진행 중 평가 복구

## 증상
- **문제 정의**: 고객사에서 결과 오픈 전 진행 중이던 평가를 전체 삭제한 후 복구 요청
- **회사**: 주식회사 애드쉴드 (Customer ID[^1]: 223656)
- **요청자**: CS팀 이도경[^2]
- **대상자**: 삭제를 수행한 관리자 (user_id: 939543)[^10]
- **영향 범위**: 해당 회사의 삭제된 평가 1건
- **문제 시점**: 2026-03-23 20:18 (삭제 시각)[^10]
- 문의 내용:
  1. 결과 오픈 전 진행 중이던 평가를 전체 삭제한 상태[^2]
  2. 원칙적으로 복구 불가 안내를 3차례 이상 전달했으나, 고객 중요도를 고려하여 복구 요청[^2]

## 원인 분석

### 조사 과정
> 💡 고객이 평가를 직접 삭제한 운영 건 — 시스템 버그가 아닌 사용자 조작에 의한 삭제
> → 이성환이 김보라에게 "평가 soft delete로 바뀐 상태죠?" 확인, 김보라 "네 맞습니다!" 응답[^3]
> → 이성환이 `flex_review.evaluation` 테이블에서 해당 회사의 삭제된 평가를 DB 조회하여 스크린샷 공유[^8]
> → 이도경이 고객으로부터 확인한 평가 제목 `Performance Snapshot Review`와 DB 결과를 대조[^11]
> → 김보라가 삭제 시점과 상태(BEFORE_START vs 진행 중)로 `.` 있는 건 제외, 대상 1건 확정[^12]
> → 김보라가 복구/백업 쿼리를 작성하고 이성환에게 결재 요청[^10][^16]

### 대상 평가 특정

삭제된 평가 후보 3건 중 진행 중(`IN_PROGRESS` 이상) 상태였던 것을 기준으로 특정[^11]:

| 평가명 | 상태 | 삭제 시점 | 대상 여부 |
|--------|------|----------|----------|
| `Performance Snapshot Review` | 진행 중 | 2026-03-23 20:18 | **대상** |
| `Performance Snapshot Review.` | BEFORE_START | 다른 시점 | 제외[^12] |
| `Performance Snapshot Review (Copy)` | 불명 | 불명 | 제외 — 고객이 "Copy는 아니다"라고 확인[^13] |

### 원인
- 고객사 관리자가 평가를 직접 삭제함 (user_id: 939543)[^10]
- 다른 평가를 삭제하려다 실수로 진행 중 평가까지 삭제한 것으로 추정[^14]

## 해결
- `evaluation` 테이블의 soft delete 필드를 NULL로 복구하는 DML 실행[^10]

```sql
-- 복구 쿼리
UPDATE flex_review.evaluation
SET deleted_at = null, deleted_user_id = null
WHERE id = '01kjvxpfvdd1jnngp5zhqa5s0m';

-- 롤백 쿼리
UPDATE flex_review.evaluation
SET deleted_at = '2026-03-23 20:18:47.522540', deleted_user_id = 939543
WHERE id = '01kjvxpfvdd1jnngp5zhqa5s0m';
```

- soft deleted 평가 복구 Operation API PR 생성: https://github.com/flex-team/flex-review-backend/pull/5181 [^15]
- 이성환 결재 완료 후 쿼리 실행[^16]
- 고객사에 복구 결과 확인 요청 (이도경 → 고객)[^17]

## 발견한 스펙/제약
- `evaluation` 테이블은 soft delete 방식 (`deleted_at`, `deleted_user_id` 컬럼)[^3]
- 복구 시 `deleted_at = NULL, deleted_user_id = NULL` 설정으로 복원 가능[^10]
- 향후 Operation API로 복구 지원 가능 (PR #5181)[^15]

## 다음에 같은 문의가 오면
1. **먼저 확인**: `flex_review.evaluation` 테이블에서 해당 회사의 `deleted_at IS NOT NULL` 레코드 조회 — 삭제된 평가 목록과 삭제 시점 확인
2. **원인 판별**: 고객이 삭제 시점/대상을 기억하는지 확인. 여러 건이면 평가 상태(BEFORE_START vs 진행 중)와 삭제 시점으로 대상 특정
3. **조치**:
   - Operation API PR #5181 머지 후: Operation API로 복구
   - Operation API 미배포 시: DML 직접 실행 (`deleted_at = NULL, deleted_user_id = NULL`), 결재 필요
