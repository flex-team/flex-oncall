# CI-4131: 초과근무수당 인정시간 입력 시 100의 자리 올림 계산

> **상태**: 해결 완료 (노이슈) — 2026-03-17

## 증상
- **회사명**: 주식회사 이도 (Customer ID: 212445)
- **환경**: PC 웹, PROD
- **온도감**: 중 (D+1~2)
- **라벨**: 급여/원천세/전표
- **리포터**: 안상연 (델리)
- 문의 내용:
  1. 초과근무수당 계산을 10의 자리에서 올림으로 설정해 둔 상태[^1]
  2. 정산 중 `지급` 탭 초과근무수당에서 인정시간을 입력하면, 결과 값이 100의 자리에서 올림되는 것처럼 계산됨[^1]
  3. 야간근무뿐 아니라 휴일근무에서도 동일 현상 발생[^2]

## 원인 분석

**노이슈** — 정산 생성 시 올림 설정이 스냅샷되어, 이후 설정 변경이 기존 진행 중인 정산에 반영되지 않는 정상 동작[^16].

고객은 기존에 **100의 자리 올림**을 사용하다가 **10의 자리 올림**으로 변경했으나, 변경 전 생성된 정산은 스냅샷된 이전 설정(100의 자리)을 유지. 신규 정산에서는 변경된 설정(10의 자리)이 정상 적용됨[^16].

참고: dev 조사에서 `decimalPoint` 공식 off-by-one 의심이 있었으나, prod 현상의 원인은 설정 스냅샷 동작이었음.

### 가설 목록

| # | 가설 | 확인 방법 | 상태 |
|---|------|----------|------|
| 1 | `decimalPoint` 공식이 급여 관례와 1단계 차이 | dev 재현 데이터 검증 | ✅ 확정 — "10의 단위" 설정 시 100의 배수 동작 확인[^11] |
| 2 | 수당 설정(TENS)과 계산 설정(TENTHS) 불일치 | DB 조회 | ✅ 확정 — 두 테이블 독립 관리, prod는 TENTHS[^9] |
| 3 | FE에서 금액 계산 시 잘못된 rounding 적용 | dev API 추적 | ❌ 소거 — BE가 계산하여 응답[^15] |

### 근본 원인: `decimalPoint` 공식 off-by-one

`flex-payroll-backend` > `commons/utils/number/.../RoundingIntegerDigitType.kt:15`[^5]:

```kotlin
val decimalPoint = exponent * (-1) - 1  // 현재 공식
```

| Enum | UI 라벨 | exponent | decimalPoint | setScale 결과 | 기대 결과 |
|------|---------|----------|-------------|--------------|----------|
| HUNDREDS | 100의 단위 | 2 | **-3** | 1000의 배수 | 100의 배수 |
| TENS | 10의 단위 | 1 | **-2** | **100의 배수** | **10의 배수** |
| ONES | 1의 단위 | 0 | -1 | 10의 배수 | 1의 배수(정수) |
| TENTHS | 소숫점 첫째 | -1 | 0 | 정수 | 정수 ✓ |

TENTHS만 정상이고, 나머지는 모두 1단계씩 큰 단위로 올림됨.

<details>
<summary>5 Whys — 근본 원인 추적</summary>

1. **왜** 113,732원이 113,800원으로 올림되는가? → `setScale(-2, UP)` 적용 (100의 배수)[^11]
2. **왜** setScale(-2)인가? → `TENS.decimalPoint = -2`[^5]
3. **왜** TENS의 decimalPoint가 -2인가? → 공식 `exponent * (-1) - 1 = 1 * (-1) - 1 = -2`[^5]
4. **왜** `-1`이 추가되었는가? → 한국 수학 관례("10의 자리에서 올림" = 10의 자리 숫자를 올림 → 100의 배수)를 따른 것으로 추정
5. **왜** 급여 관례와 다른가? → 급여에서 "10의 단위" = "10원 단위 올림" = 10의 배수를 의미하나, 수학에서 "10의 자리에서 올림" = 100의 배수. **UI 라벨은 "10의 단위"(급여 관례)**이지만 **코드는 수학 관례를 따름**

</details>

### prod 설정 불일치

prod DB에서 `payroll_legal_payment_setting` = `TENTHS`[^9]. 그러나 고객 제공 결과(90,100 / 42,600)는 **100의 배수**로, `TENS`(decimalPoint=-2)의 동작과 정확히 일치[^12]:
- 90,090 → setScale(-2, UP) → **90,100** ✓
- 42,546 → setScale(-2, UP) → **42,600** ✓

고객은 "10의 자리 올림 설정"이라 했으나, 법정지급항목 설정(현재 TENTHS)과 결과(100의 배수)가 불일치. 두 가지 가능성:
1. 고객이 **수당 관리 UI**에서만 설정 변경 → `allowance_global`(TENS)만 업데이트, `payroll_legal_payment_setting`(TENTHS) 미변경
2. 계산 당시 설정이 TENS였으나 이후 변경됨 (설정 변경 이력 미확인)

## 스펙 vs 버그 판별

**스펙(정상 동작)** — 정산 생성 시 올림 설정을 스냅샷하여 기존 정산의 일관성을 보장하는 설계. 설정 변경 후 신규 정산에서는 정상 반영됨[^16].

## 해결

**노이슈로 종결** — 고객에게 기존 정산은 스냅샷된 설정을 사용하며, 신규 정산부터 변경된 설정이 적용됨을 안내[^16].

<details>
<summary>참고: decimalPoint 공식 off-by-one 관찰 (별도 이슈)</summary>

dev 조사 과정에서 `RoundingIntegerDigitType.decimalPoint` 공식이 급여 관례와 1단계 어긋남을 발견[^5][^11]. 이 티켓의 원인은 아니었으나, "10의 단위" 선택 시 100의 배수로 동작하는 것은 별도 검토 대상일 수 있음.

| Enum | UI 라벨 | decimalPoint | setScale 결과 | 급여 관례 기대 |
|------|---------|-------------|--------------|--------------|
| TENS | 10의 단위 | -2 | 100의 배수 | 10의 배수 |
| ONES | 1의 단위 | -1 | 10의 배수 | 정수 |

</details>

## 다음에 같은 문의가 오면
1. **설정 변경 시점 확인**: 고객이 올림 설정을 최근 변경했는지 확인
2. **기존 정산 vs 신규 정산 구분**: 정산 생성 시 올림 설정이 스냅샷됨. 기존 진행 중 정산은 이전 설정을 유지하는 것이 정상
3. **신규 정산 생성 안내**: 변경된 설정을 적용하려면 새 정산을 생성하도록 안내
4. **설정 테이블 확인**: `payroll_legal_payment_setting` WHERE customer_id=? AND type='GROUP_EXCEEDED_WORK_EARNING' / `work_income_over_work_payment_calculation_basis`의 스냅샷 설정과 비교
