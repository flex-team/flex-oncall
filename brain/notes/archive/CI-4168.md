# CI-4168: 전자계약 서식 반복 삭제 — 삭제자는 문의자 본인

> **상태**: 조사 완료 — 2026-03-19

## 증상

- **문제 정의**: 전자계약 서식이 반복적으로 삭제되며, 삭제자를 특정할 수 없음 (감사로그 미조회)
- **회사**: 노브랜드 (Customer ID[^9]: 219834)
- **요청자**: CS팀 이주화 (고객 yelee@nobland.com 대리)
- **대상자**: yelee@nobland.com
- **영향 범위**: 해당 회사 전자계약 서식 — 전체 26건 중 14건 삭제 상태[^1]
- **문제 시점**: 2026-03-19, 13:36~16:36 KST에 집중 (일부 과거 삭제 건 존재)[^1]
- 문의 내용:
  1. 전자계약 서식 삭제자 확인 요청
  2. 양식 개수 제한 여부 확인

---

## 원인 분석

### L1 — 한 줄 요약

서식 삭제자는 문의자 본인(yelee@nobland.com, userId: 898461)이며, 웹 UI에서 직접 삭제 API를 호출한 것으로 확인됨[^2][^3][^4].

### L2 — 핵심 근거

1. **DB 조회**: `customer_digicon_template` 테이블에서 customer_id=219834의 삭제된 서식 14건 확인. 특히 14:10:25~26 KST에 5건이 동일 초에 삭제됨[^1]
2. **Access log 추적**: traceId `d89bbb6d1bb1d2b82715c54c73f01ef6`로 호출 체인 추적 → permission-api에서 `customers/219834/users/898461` 확인[^2][^3]
3. **사용자 특정**: `view_user` 테이블에서 email LIKE '%yelee%' AND customer_id=219834 → id=898461 매칭[^4]

### 가설 검증

| # | 가설 | 확인 방법 | 상태 |
|---|------|----------|------|
| 1 | 회사 내 사용자가 삭제 API 호출 | DB deletedAt + access log traceId 추적 | ✅ 확정 — 문의자 본인(userId 898461)[^2][^4] |
| 2 | 시스템/배치에 의한 자동 삭제 | 코드에서 template 삭제 트리거 검색 | ❌ 소거 — access log에서 웹 UI 요청 확인[^2] |
| 3 | 양식 개수 제한 초과로 자동 삭제 | 코드에서 count/limit 로직 검색 | ❌ 소거 — 개수 제한 로직 없음[^5] |

> 💡 **판단 근거**: 가설 2, 3 소거의 결정적 단서는 access log의 요청 헤더.
> User-Agent가 `Mozilla/5.0 (Windows NT 10.0; Win64; x64)`이고, referer가 `https://flex.team/digicon/request`였다[^2].
> 이는 웹 브라우저에서 flex 웹 UI를 통한 직접 요청임을 의미하며, 시스템/배치 호출이 아님을 확정한다.

### 조사 과정

1. **코드 탐색**: `flex-digicon-backend`에서 서식 삭제 API 위치 확인 — `DELETE /api/v2/digicon/templates/{hash}`, soft delete[^8] 방식 (deletedAt 갱신)[^5][^6]
2. **DB 조회**: customer_id=219834의 서식 26건 중 14건 삭제 상태. **14:10:25 UTC에 5건 동일 초 삭제** 패턴 발견[^1]
3. **Access log 조회**: `flex-app.be-access-2026.03.19` 인덱스에서 digicon-api DELETE 요청 중 219834 관련 건 필터링[^2]
4. **traceId 추적**: permission-api 호출에서 userId=898461 특정[^3] → view_user 테이블에서 yelee@nobland.com 확인[^4]

### 코드 위치

| 위치 | 설명 |
|------|------|
| `flex-digicon-backend` > `api/.../CustomerDigiconTemplateController.kt:66-74` | 삭제 API 엔드포인트[^5] |
| `flex-digicon-backend` > `service/.../CustomerDigiconTemplateServiceImpl.kt:64-71` | 삭제 로직 (soft delete: deletedAt 갱신)[^6] |
| `flex-digicon-backend` > `operation-api/.../DigiconOperationService.kt:28-66` | 복구 API (deletedAt = null)[^7] |

### 스펙 vs 버그

**스펙 (정상 동작)** — 삭제 API 자체는 정상 동작[^5]. 단, 감사로그(audit log)에 서식 삭제 이벤트가 기록되지 않아 삭제자 추적이 access log에 의존하는 점은 개선 필요[^10].

## 해결

### 즉시 대응 (CS 안내)

1. 서식 삭제자는 문의자 본인(yelee@nobland.com)임을 안내
2. 양식 개수 제한은 없음을 안내
3. 본인이 의도하지 않은 삭제라면, 같은 계정을 다른 사람이 사용하고 있는지 확인 필요

### 개선 사항 (코드 변경)

1. 서식 삭제 시 감사로그(audit log) 기록 추가 — 향후 삭제자 추적 용이
2. `deleted_by` 컬럼 추가 검토

---

## 다음에 같은 문의가 오면

1. Access log에서 `DELETE /api/v2/digicon/templates` + digicon-api 서비스로 검색
2. traceId로 호출 체인 추적 → permission-api 호출에서 userId 확인
3. `view_user` 테이블에서 userId → 이메일/이름 매핑
4. 삭제자가 문의자 본인이면 → 의도하지 않은 삭제인지, 계정 공유 여부 확인 안내
