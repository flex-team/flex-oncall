## 원인 분석

**Matrix(Vespa) 인덱스 동기화 실패 — 확정**

세금계산서 목록 화면은 DB를 직접 읽지 않고 Matrix(Vespa)[^5] 인덱스 `fins_spending_entire_v1` 을 읽는다. DB 변경이 성공해도 Matrix 갱신이 실패하면 화면에 반영되지 않는다.

> 💡 **판단 근거**: 이성환 "matrix이슈" 식별[^2] → 임형태 MatrixQL로 Vespa 데이터 확인: DB에 담당자 지정돼 있으나 화면 미반영[^12] → 18:30 전후 검색엔진 부하로 일부 문서 색인 실패 확인[^11] → 김성민 전체색인 실행으로 복구[^13]

### 데이터 흐름

```
사용자 액션 (담당자 지정/제출)
  → API 호출 → Kafka command produce → 즉시 "처리 완료" 안내
  → [비동기] Consumer → DB 변경 → Bullseye[^8] publish → Matrix 갱신
                                      ↑ 여기서 실패 시 화면 미반영
```

### 가설 목록

| # | 가설 | 확인 방법 | 상태 |
|---|------|----------|------|
| 1 | Matrix(Vespa) 갱신 실패 — DB에는 정상 저장됐으나 18:30 부하로 색인 실패 | 임형태 MatrixQL 확인 + 부하 시점 특정 | ✅ 확정[^11][^12] |
| 2 | Kafka command 자체 소비 실패 — DB 변경도 안 됨 | DB에 담당자 저장 확인됨 → 소거 | ❌ 소거 (DB에 데이터 있음)[^12] |

### 담당자 지정 API 흐름

**엔드포인트**: `POST /api/v2/fins/spending/spending-processes/assignees`[^9]

```
Controller.bulkUpdateSpendingAssignees()
  → SpendingService.acceptBulkUpdateSpendingAssignees()  ← Kafka command 발행 후 즉시 리턴
     → Kafka topic produce: SpendingEvidenceDemandBulkUpdateAssigneeCommandV1
        → (Consumer) SpendingCommandHandler.on(...)
           → SpendingService.bulkUpdateSpendingAssignees()  ← 실제 DB 변경
              → publishBullseye()  ← Matrix 갱신 시도
```

핵심: `acceptBulkUpdateSpendingAssignees`는 Kafka command produce만 하고 즉시 리턴. "처리 완료" 안내는 DB 변경 전에 뜬다.

### 세금계산서 제출 API 흐름

**엔드포인트**: `POST /api/v2/fins/spending/tax-invoices/spending-processes/{spendingProcessId}/evidences/submit`[^10]

제출은 동기 트랜잭션으로 DB 변경 후 Bullseye publish를 시도한다.

### Matrix 갱신 2-track 구조

**Track 1 (direct)**: 트랜잭션 커밋 후 coroutine으로 비동기 Vespa 업데이트[^6]
- **실패 시 재시도 없음**: `logger.warn(ignored) { "publish bullseye is failed. ignored." }`
- `businessOperation != null` 조건 충족 시에만 실행

**Track 2 (Kafka fallback)**: 이벤트 → Kafka consumer → 배치 Bullseye publish[^7]
- 예외 발생해도 `catch (ignored: Exception)` 후 continue
- 해당 spending의 Matrix 갱신만 실패하고 넘어감

## 해결

- **즉시 대응**: 김성민이 전체색인 실행 완료[^13] → 고객사 확인 대기[^15]
- **표면 원인**: 2026-04-08 18:30 전후 검색엔진(메트릭스) 부하로 일부 문서 색인 실패[^11]
- **근본 원인**: Bullseye publish 실패 시 재시도 메커니즘 부재. Track 1은 실패 무시[^6], Track 2도 예외를 삼키는 구조[^7]. outbox pattern 미적용.
- **성능 이슈 연관**: 임형태가 성능팀에 멘션, 별도 성능 이슈와 연관된 추가 이슈로 분류[^14]

## 다음에 같은 문의가 오면

1. **증상 확인**: 비용관리 화면에서 담당자/제출 상태가 반영 안 되는지 확인
2. **Matrix 이슈 판별**: MatrixQL로 Vespa 데이터를 직접 조회하여 DB에는 저장돼 있는데 화면에만 안 보이는지 확인
3. **조치**: 전체색인 실행 요청 (김성민 또는 비용관리팀)
   - 또는 Operation API: `POST /api/operation/v2/fins/spending/customers/{customerId}/synchronize`
