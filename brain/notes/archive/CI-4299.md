# CI-4299: DEV 승인 정책 관리 — 근무/휴가 승인 정책 미응답

> **상태**: 조사 중 — 2026-04-02

## 증상
- **문제 정의**: DEV 환경에서 설정 > 승인 정책 관리 화면의 근무/휴가 승인 정책이 응답에 내려오지 않음
- **영향 범위**: DEV 환경 전체 (prod 정상)[^1]
- **요청자**: 이지선[^1]
- **대상 customer**: customerId=1673, customerIdHash=`dzDgMA8gPw`[^5]
- **요청자 계정**: jiseon.lee+2@flex.team (userId=16118)[^5]
- 문의 내용:
  1. 승인 정책 관리 > 근무, 휴가 승인 정책이 API 응답에 포함되지 않음
  2. prod에서는 정상 동작 확인됨

## 원인 분석

### 조사 과정

> 💡 access log → traceId 추적 → 권한 체크 흐름 → 데이터 조회 흐름 순으로 추적

#### 1. Access log 확인
- API: `GET /api/v2/approval/customers/dzDgMA8gPw/policy-target-objects` → **200** (1373ms)[^5]
- 응답에 `APPROVAL_DOCUMENT`(17건), `USER_DATA`(5건), `EVALUATION`(2건) 카테고리만 존재[^5]
- **`WORK_RECORD`, `TIME_OFF` 카테고리 없음**[^5]

#### 2. TraceId 호출 체인 추적
traceId: `ede00e53ff06a557b430c512287ab793`[^5]

| 서비스 | 경로 | 상태 | 역할 |
|--------|------|------|------|
| permission-api | features check | 200 | 권한 확인 |
| core-api | USER_DATA 카테고리 | 200 | 사용자 데이터 정책 조회 |
| goal-api | objectives approval-settings | 200 | OKR 승인설정 |
| authorization-api | batch-check (x2) | 200 | 인가 확인 |
| **time-tracking-api** | — | **호출 없음** | WORK_RECORD/TIME_OFF 조회 |

#### 3. 권한 체크 흐름 분석
`doWithGrantedReadPermissions()` (ApprovalPolicyPermissionChecker.kt:92)[^6]:
1. `APPROVAL_POLICY`는 `isDefault=true` → defaultPermissionChecker로 먼저 체크[^7]
2. 이 체크가 true이면 **ALL 카테고리**가 `getPerCategories()`에 전달됨
3. → 권한 체크 단계에서 WORK_RECORD/TIME_OFF가 제외된 것이 아님

#### 4. 데이터 조회 흐름 분석 (핵심)
`getPerCategories()` (TargetObjectQueryService.kt:74)[^8]:
```
1. approvalPolicyReadUseCase.getAll() → approval_policy DB에서 정책 조회
   → customerId=1673에 WORK_RECORD 212건, TIME_OFF 328건 존재 (DB 확인)
2. policyTargetQueryService.getAllTargetInfo() → 각 도메인 서비스에서 target 정보 조회
   → WORK_RECORD/TIME_OFF는 timetracking 서비스 호출
   → .filter { !it.deleted } 로 삭제된 항목 제외
3. targetInfos.map { ... } → targetInfos 기준으로 응답 생성
   → targetInfos에 없는 카테고리는 응답에서 제외됨
```

**timetracking 호출 경로**: `DomainTargetObjectQueryService.getAllTargetInfo()` → `timeTrackingCustomerApprovalPolicyLookupService.getCustomerApprovalPolicyDetails()`[^9]
- 이 호출이 빈 리스트를 반환하거나, 반환된 항목이 모두 `deleted=true`이면 해당 카테고리가 응답에서 빠짐
- try-catch에서 예외 시 `emptyList()` 반환 + `logger.error` 기록 (TargetObjectInfoQueryService.kt:101)[^10]
- **에러 로그 없음** → 예외가 발생하지 않았음. 정상 처리 후 빈 결과 반환 추정

**timetracking 권한 조회 경로**: `TimeTrackingCustomerApprovalPolicySettingPermissionInternalApiClient`[^11]
- `GET /api/v2/internal/time-tracking/customers/{customerId}/approval-policies/{category}/settings.options`
- DEV access log에서 이 endpoint 호출이 **오늘 하루 전체 0건** → 이 internal API가 DEV에서 호출되지 않고 있음

### 확인된 DB 상태
- `flexdev_approval.approval_policy` (customerId=1673)[^12]:
  - `WORK_RECORD`: 212건
  - `TIME_OFF`: 328건
  - `APPROVAL_DOCUMENT`: 94건 (응답에 포함됨)
  - DB에는 데이터가 있으나 API 응답에 누락

- `flexdev.v2_customer_work_approval_rule` (customerId=1673)[^3]:
  - 7개 type 보유 (LEAF_* 2종 누락은 별도 이슈)

### 원인 (확정)
approval-api → timetracking 내부 API 호출이 DEV에서 전면 실패. **DEV 환경 전체 문제** (특정 customer 아님)[^13].

**검증 결과:**
- timetracking API 직접 호출(외부 URL) → WORK_RECORD 19건, TIME_OFF 31건 **정상 반환**[^14]
- 권한 API 직접 호출 → WORK_RECORD/TIME_OFF 모두 **GRANTED**[^15]
- 하지만 approval-api access log에서 오늘 WORK_RECORD를 포함한 응답 = **0건** (5건 호출 모두 누락)[^13]
- timetracking access log에서 `approval-policies/{category}` 패턴 호출 = **오늘 0건** (approval-api에서 timetracking으로의 내부 호출이 아예 도달하지 않음)[^16]
- approval-api ERROR 로그 = **0건** (TargetObjectInfoQueryService의 try-catch가 예외를 삼키고 emptyList() 반환)[^10]

**근본 원인 (확정)**: SB4 마이그레이션(`1de7a9d`, PR #1140)에서 `team.flex.timetracking:internal-api-client`가 4.0.0 호환 버전을 아직 출시하지 않아, timetracking AutoConfiguration을 제외하고 **fallback 빈을 등록**[^17][^18].

`ExternalServiceFallbackConfiguration.kt` 에서 세 가지 fallback 빈을 등록:

| fallback 빈 | 동작 | 영향 |
|-------------|------|------|
| `TimeTrackingCustomerApprovalPolicyLookupService` | **`emptyList()` 반환** | WORK_RECORD/TIME_OFF 데이터 조회 → 항상 빈 결과 → **응답에서 누락** |
| `TimeTrackingCustomerApprovalPolicySettingPermissionsLookupService` | 모든 권한 `GRANTED` | 권한 체크는 통과 (증상 은폐) |
| `TimeTrackingCustomerApprovalPolicyCommandService` | no-op (빈 동작) | 정책 생성/수정 시 timetracking에 알림 미전송 |

이로 인해:
- 실제 HTTP 호출이 발생하지 않음 → access log 없음, 에러 로그 없음
- APPROVAL_DOCUMENT, USER_DATA, EVALUATION은 다른 서비스에서 조회하므로 영향 없음

| 구분 | Prod | DEV |
|------|------|-----|
| Spring Boot | 3.5.10 | 4.0.4 |
| flex commons | 3.80.0 | 4.0.0 |
| flex v2 backend | 3.303.0 | 4.0.0 |
| 앱 버전 | 3.49.0 | 4.0.0-SNAPSHOT |
