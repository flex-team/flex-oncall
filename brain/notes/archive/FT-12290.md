# FT-12290: 캡스 연동 방화벽 해제 관련

## 증상

- 회사: 타이거치과 (customer_id: 200407)
- 환경: PC 웹
- 캡스(CAPS) 연동 시도 중 1.3 ODBD 데이터 원본(32비트) 프로그램에서 TEST 버튼 클릭 시 연결 실패
- 고객은 사내 방화벽이 막혀있어 해제 방법 안내를 요청

## 원인 분석

### 1차 문의 (2024-12-06): 패스워드 오입력

- 캡스 기사가 원격 지원 중 **패스워드를 잘못 입력**한 것이 원인
- DB 로그에도 패스워드 실패 로그가 남아있었음 (김영준 Enhance 확인)
- flex 측에서 해당 패스워드로 정상 로그인되는 것을 확인 → 고객 측 입력 오류 확정
- 캡스 기사가 통화 중 PW 재입력하여 해결

### 2차 문의 (2024-12-09): 연결 설정 완료 후 시작 시 정지로 돌아감

- 연결설정은 완료되었으나 "시작" 클릭 시 다시 "정지" 상태로 복귀
- 고객사 개발자가 외부 IP를 차단하고 있어 **방화벽 예외 설정을 위해 flex IP 주소를 요청**
- flex 정책: IP는 변경될 수 있으므로 제공하지 않음. 도메인 기반 예외처리를 안내
- 고객에게 CMD 명령어로 네트워크 연결 상태 확인 후 결과 전달을 요청

## 해결

### 1차: 패스워드 재입력으로 해결
### 2차: 고객사에 네트워크 진단 명령어 안내 후 방화벽 예외처리 유도

## 비고

### 핵심 교훈

1. **연결 실패 ≠ 방화벽 문제**: 연결 실패 시 가장 먼저 의심할 것은 **인증 정보(패스워드) 오입력**. 방화벽은 그 다음
2. **IP 제공 불가 정책**: flex 서버 IP는 변경될 수 있으므로 고객에게 직접 제공하지 않음. 도메인(`flex-caps.flex.team`) 기반 예외처리를 안내
3. **고객이 "방화벽 문제"라고 단정 지어도 검증 필요**: 오류 메시지만으로는 원인 특정이 어려움. DB 로그 확인이 필수

### 방화벽 문의 대응 가이드 (CMD 명령어)

고객에게 아래 명령어 실행 후 화면 캡처를 요청:

```bash
# Windows
nslookup flex-caps.flex.team        # DNS 확인 — amazonaws 나오면 정상
telnet flex-caps.flex.team 5432     # 포트 연결 — 아무 화면 안 나오면 연결 성공, "연결할 수 없다"면 차단
tracert flex-caps.flex.team         # 경로 추적
```

- `nslookup`: amazonaws 주소 나오면 DNS 정상
- `telnet`: "연결할 수 없다"만 안 나오면 통신 가능 (빈 화면 = 연결 성공)
- ODBC 연결 + 세콤/캡스 프로그램에서 flex 연결정보 입력 후 성공했다면 방화벽 문제는 거의 아님
