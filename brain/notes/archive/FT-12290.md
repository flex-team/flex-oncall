## 원인 분석

### 1차 문의 (2024-12-06): 패스워드 오입력

#### 조사 과정

> 💡 **판단 근거**: 고객이 "방화벽 문제"라고 주장했으나, 스크린샷 요청 후 확인한 결과 DB 로그에 패스워드 실패 로그가 남아있었음[^5]
> → 오류 화면 스크린샷 요청 → 오류 메시지 확인 → DB 로그에서 패스워드 실패 확인[^5]
> → 직접 해당 패스워드로 로그인 시도 → 성공[^5]
> → 고객 측(캡스 기사)의 PW 입력 오류로 확정

#### 원인

- 캡스 기사가 원격 지원 중 **패스워드를 잘못 입력**한 것이 원인[^5]
- 계정 만료로 재생성 후 새 PW를 올바르게 입력하지 않음[^4]

### 2차 문의 (2024-12-09): 방화벽 차단 (실제 방화벽 문제)

#### 조사 과정

> 💡 **상황**: 1차와 달리 이번에는 실제 방화벽 차단이 원인일 가능성 있음
> → 고객사 개발자가 "외부 IP를 차단 중, flex IP 주소 필요"라고 구체적으로 언급[^4]
> → 안희종: 과거 선례 확인 — IP 제공은 정책상 불가[^6]
> → 도메인(`flex-caps.flex.team`) 기반 예외처리 안내로 방향 전환[^6]
> → CMD 명령어로 실제 연결 여부 확인 후 방화벽 예외처리 가이드 전달[^7]

#### 원인

- 고객사 방화벽이 외부 IP를 일괄 차단하여 flex 서버에 접근 불가[^4]

## 해결

### 1차: PW 재입력으로 해결

- 캡스 기사가 통화 중 PW 재입력 → 정상 연결 확인[^4]

### 2차: 네트워크 진단 + 도메인 기반 방화벽 예외처리 안내

- flex IP 직접 제공 대신 도메인(`flex-caps.flex.team`)으로 예외처리하도록 안내[^6]
- 고객에게 CMD 명령어 실행 후 캡처 전달 요청[^7]

---

## 발견한 스펙/제약

- **flex IP 제공 불가 정책**: flex 서버 IP는 변경될 수 있으므로 고객에게 직접 제공하지 않음. 도메인 기반 예외처리를 안내해야 함[^6]
- **연결 실패 원인 다양성**: ODBC TEST 버튼 실패 = 방화벽 문제라는 공식 가이드가 있으나, 실제로는 PW 오입력 등 다른 원인도 동일 증상을 보임[^3][^5]

## 다음에 같은 문의가 오면

1. **먼저 확인**: "방화벽 문제"라고 주장하더라도 먼저 **인증 정보(패스워드) 오입력 여부 확인** — DB 로그에서 패스워드 실패 로그 검색
2. **원인 판별**:
   - DB 로그에 패스워드 실패가 있으면 → 고객에게 PW 재입력 요청
   - PW가 정상인데도 실패하면 → 방화벽 차단 의심, 아래 CMD 명령어로 네트워크 진단
3. **방화벽 차단 시 조치**:
   - **IP 제공 금지** (변경 가능성 있음)
   - 고객에게 아래 명령어 실행 후 캡처 전달 요청:
     ```bash
     # Windows (명령 프롬프트)
     nslookup flex-caps.flex.team        # DNS 확인 — amazonaws 나오면 정상
     telnet flex-caps.flex.team 5432     # 아무 화면 안 나오면 연결 성공, "연결할 수 없다"면 차단
     tracert flex-caps.flex.team         # 경로 추적
     ```
   - 결과 캡처 확인 후 고객사 IT 담당자에게 `flex-caps.flex.team` 도메인으로 방화벽 예외처리 요청 안내
