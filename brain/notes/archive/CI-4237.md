# CI-4237: 캡스 타각 이벤트 지연 전달로 수동 근무 시작과 충돌

> **compact**: 2026-03-31. git log로 원본 추적 가능.

## 증상
- **문제 정의**: 캡스 타각 이벤트가 12분 지연 전달되는 사이 사용자가 수동으로 근무를 시작하여, 이후 도착한 이벤트가 스킵됨
- **회사**: 클레로보틱스 주식회사 (Customer ID: 70136)
- **요청자**: hkjee@clerobotics.com[^1]
- **대상자**: hkjee@clerobotics.com (userId: 565537)[^2]
- **영향 범위**: 1명
- **문제 시점**: 2026-03-27 08:13 KST
- 문의 내용:
  1. 오전 8시 13분 캡스 지문 인식이 정상 진행되었으나 10분 이상 지나도 플렉스에 반영되지 않음
  2. 팀원 전체의 아침 캡스 출입기록이 플렉스로 정상 연동되지 않아 수동으로 근태 체크 진행

## 원인 분석

#### 조사 과정
> 💡 **판단 근거**: `v2_user_external_provider_event` 테이블에서 이벤트 수신 시각 확인[^3]
> → 08:13:54에 발생한 타각 이벤트가 캡스 → 플렉스로 전달된 시각은 08:25:56 (12분 지연)[^3]
> → `v2_user_work_clock_draft_event` 확인 결과 해당 이벤트 기준 start 레코드 없음[^4]
> → 컨슈머 로그 확인: 2026-03-27 08:25:33에 사용자가 직접 START 수행 → 08:25:56에 도착한 캡스 이벤트가 스킵됨[^4]

#### 원인
- 캡스 → 플렉스 이벤트 전달이 약 12분 지연됨[^3]
- 지연 대기 중 사용자가 수동으로 근무를 시작(08:25:33)하여, 이후 도착한 캡스 이벤트(08:25:56)가 이미 근무 중 상태로 간주되어 스킵됨[^4]
- 이벤트 지연은 당일 일시적인 현상이었으며 이후 정상화됨[^5]

## 해결
- 별도 조치 없이 자연 해소[^5]
- 지연이 해소된 이후에는 정상 연동 확인됨

## 발견한 스펙/제약
- 캡스 이벤트가 지연 전달되는 사이 사용자가 수동으로 START하면, 이후 도착하는 캡스 이벤트는 처리되지 않음[^4]

## 다음에 같은 문의가 오면

1. **먼저 확인**: `v2_user_external_provider_event` 테이블에서 해당 유저 + 날짜로 이벤트 수신 시각 확인 ([Metabase](https://metabase.dp.grapeisfruit.com/question/3565))
   - `created_at`(이벤트 발생) vs 실제 DB 저장 시각 차이 확인
2. **원인 판별**:
   - 이벤트 자체가 없음 → 캡스 → 플렉스 연동 설정 문제
   - 이벤트는 있으나 지연됨 → 캡스 서버 측 전송 지연 (일시적일 가능성 높음)
   - 이벤트는 왔으나 draft/근무가 생성 안 됨 → `v2_user_work_clock_draft_event` 확인, 수동 조작 여부 확인
3. **조치**:
   - 이벤트 지연: 이후 정상화 여부 모니터링 후 안내
   - 수동 START로 인한 스킵: 해당 draft를 직접 수정하거나 사용자에게 수동 입력 안내
