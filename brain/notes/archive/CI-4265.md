# CI-4265: 업그레이드 이전 정산 결과의 지급액 0원 표기

## 증상
- **문제 정의**: 지난 정산 내역에서 23년 11월 실지급액은 정상 표시되나, 정산 상세 진입 시 지급액이 0원으로 표시됨
- **회사**: 녹십자수의약품 주식회사 (Customer ID: 37424)
- **요청자**: CS팀 @김한나[^1]
- **대상자**: 문의자 youngeun.jo@gcvp.co.kr (대상 구성원 특정 안 됨)[^1]
- **영향 범위**: 해당 회사 2023-11~12 귀속 전체 + 기타 귀속월 일부 — **460명 미마이그레이션**[^3]
- **문제 시점**: 2023년 11월 귀속 정산 (현재 지속)
- 문의 내용:
  1. 급여정산 > 급여 > 지난 정산 내역 > 23년 11월: 실지급액 정상 표시[^1]
  2. 정산 상세 진입 시 지급액 0원[^1]
  3. 지급단계에서는 지급항목과 금액이 정상 표시[^1]
  4. 마이그레이션 영향 여부 확인 요청[^1]

## 원인 분석

### 조사 과정
> 💡 CI-4227에서 확인된 구→신 테이블 마이그레이션 누락 패턴과 동일[^2]
> → 2025-03-19 backfill[^8] 시 검증 로직 실패로 `work_income_payment_settlement_payee_result_item` 테이블 데이터 미생성[^3]
> → DB 조회로 customer_id=37424의 460건 미마이그레이션 확인[^3]
> → dryRun(traceId=ccf8d3d2a0b37e4298ce68011c77f586)으로 legacy fallback[^11] 동작 확인[^5]
> → SR-level 스킵 조건 분석으로 wetRun 안전성 확인[^6]

### 원인
- 2025-03-19 backfill 시 검증 로직 실패로 일부 귀속월의 `work_income_payment_settlement_payee_result_item` 테이블 데이터가 생성되지 않음[^3]
- CI-4227과 동일 원인이며, customer_id만 다름 (CI-4227: 139856, CI-4265: 37424)[^2]

## 영향 범위

| 항목 | 값 |
|------|-----|
| settlement result 수 | 234건[^3] |
| payee result 전체 | 8,142건[^3] |
| 마이그레이션 완료 | 7,682건[^3] |
| **미마이그레이션** | **460건**[^3] |
| 미마이그레이션 SR 수 | 21개[^3] |

문의 대상인 2023-11은 3개 SR에 **116명 전원 미마이그레이션**. 2023-12도 3개 SR에 **309명 미마이그레이션**[^3].

## 정산 상태 확인

- 귀속월: 2023-11, 2023-12 (문의 대상), 기타 2022-04~2026-02
- 정산 상태: COMPLETED — 과거 확정된 정산이며, 정산 자체는 정상 완료 상태
- 지급 결과 항목(payee result item)만 신 테이블에 누락된 상태

## 소급 적용 / 재정산

- 재정산 불필요 — 정산 자체는 정상. 구 테이블 blob → 신 테이블 row 마이그레이션만 수행
- Operation API(`migrate-ci4227`)를 통한 마이그레이션으로 해결[^4]

## 금전적 영향

- 실제 급여 금액 오류 없음 — 표시만 0원으로 보이는 문제
- 원천징수 신고 영향 없음

## 460건 미마이그레이션 payee 분류[^7]

| 카테고리 | 건수 | 날짜 범위 | 원인 | wetRun 결과 |
|---------|------|----------|------|------------|
| NORMAL_BLOB | 436 | 2022-04 ~ 2025-02 | 정상 legacy blob 존재 | **마이그레이션 완료** |
| TINY_BLOB | 6 | 2025-04 ~ 2025-05 | 40 bytes — 데이터 손상 | 마이그레이션 불가 |
| NO_RECORD | 18 | 2025-12 ~ 2026-02 | legacy 레코드 없음 | 마이그레이션 불가 |

## 해결

1. CI-4227 fix 코드(PR #8691[^12])가 이미 배포된 상태에서 Operation API로 마이그레이션 실행[^4]
2. dryRun 실행 (2026-04-06 11:41 KST) — 425건 migrate, 7,717건 skip, ERROR 0건[^4]
3. **wetRun 실행 (2026-04-06 12:06 KST)** — dryRun과 동일 결과로 실제 반영 완료[^13]
4. 고객 확인 완료 — 정상 반영 확인됨[^14]

|------|--------|--------|

WARN 패턴: 총액 불일치 4건 (SR 34891, 44576), 지급항목 구성 차이 2건 (TAX03/TAX05 등), 구 테이블 레코드 없음 ~20건[^13]

## 발견한 스펙/제약
- `migrateBySettlementResult`의 SR-level 스킵 조건: fold 결과가 기존 SR items와 일치하면 null 반환하여 로그에 미포함. 그러나 payee items는 SR 반환값과 무관하게 같은 트랜잭션 내에서 저장됨[^6]
- TINY_BLOB (40 bytes) — `AllowanceEarningDataEncryptedJsonConverter`가 파싱할 수 없는 비정상 blob. CI-4227 customer_id=156610에서도 동일 패턴[^7]

## 다음에 같은 문의가 오면

1. **먼저 확인**: `work_income_payment_settlement_payee_result_item` 테이블에 해당 payee result의 항목이 존재하는지 확인. 0건이면 마이그레이션 누락.
2. **원인 판별**: customer_id로 미마이그레이션 payee 수 조회. 1건이면 개별 건, 다수이면 backfill 누락 패턴(CI-4227 동일 원인).
3. **조치**: Operation API `migrate-ci4227/{customerId}?dryRun=true`로 먼저 확인 후, 문제 없으면 `dryRun=false`로 실행. WARN 로그(총액 불일치, 항목 구성 차이)는 legacy→신 데이터 구조 차이로 인한 것이며 무시 가능.
