## 원인 분석

**원인 확정 (2단계):**

1. **교체 실패의 근본 원인**: `ApprovalPolicy.update()` 시 `validateHasCustomerSuperAdmin()` 검증이 교체(replacement) 흐름에서도 실행되어, 정책 내 다른 actor가 `CUSTOMER_SUPER_ADMIN` 타입이면 `ApprovalPolicyNewSettingOfCustomerSuperAdminNotAllowed` 예외로 교체가 실패한다[^12].
2. **UI 노출의 원인**: 실패한 교체 건(`active=true`, FAILED)이 deactivate되지 않고, API가 이를 필터링 없이 반환한다[^5][^6]. 해당 회사 80건(퇴직자 4명), eileen의 TIME_OFF 7건이 UI에 노출[^10].

### 조사 과정

> 💡 **판단 근거**: access log 응답 분석[^1] → 7건 모두 `isDeleted: true`, `status: FAILED` 확인 → 코드 추적으로 API 필터링 로직 없음 확인[^5][^6] → DB 조회로 80건 stale 레코드 + 5회 교체 실패 이력 확인[^10][^11] → consumer 에러 로그에서 `ApprovalPolicyNewSettingOfCustomerSuperAdminNotAllowed` 예외 확인[^12] → 코드 분석으로 `ApprovalPolicy.update()` 검증이 교체 흐름에서도 실행됨을 확인[^13]

### 가설 목록

| # | 가설 | 확인 방법 | 상태 |
|---|------|----------|------|
| 1 | API에서 삭제된 정책/FAILED 건을 필터링하지 않음 | 코드 분석 | ✅ 확정 — 필터링 로직 없음[^5][^6] |
| 2 | 정책 삭제 이벤트 처리 실패로 replacement target이 deactivate되지 않음 | consumer 로그/DB 확인 | 🔍 추가 확인 필요 |
| 3 | 정책 삭제 이벤트 자체가 발행되지 않음 | Kafka consumer log 확인 | 🔍 추가 확인 필요 |
| 4 | 교체 로직에서 CUSTOMER_SUPER_ADMIN 검증이 의도치 않게 발동 | consumer 에러 로그 + 코드 분석 | ✅ 확정 — `ApprovalPolicy.update()` > `validateHasCustomerSuperAdmin()` 가 교체 시에도 실행되어 기존 정책의 CUSTOMER_SUPER_ADMIN actor에 걸림[^12][^13] |

```
Controller (getAllApprovalReplacementTargets)
  → ApprovalReplacementTargetQueryService.getApprovalReplacementTarget()
    → approvalReplacementTargetReadUseCase.getAll(customerId, requestAt)
      → approvalReplacementTargetRepository.findAllNeedToReplacement(customerId)
```

**핵심 SQL 쿼리** (`findAllActiveByCustomerIdAndNeedToReplacement`):

```sql
SELECT art.*
FROM approval_replacement_target art
LEFT JOIN approval_replacement_transaction_mapping artm
  ON art.id = artm.source_id AND artm.status = 'COMPLETED'
WHERE art.customer_id = :customerId
  AND artm.source_id IS NULL   -- COMPLETED 트랜잭션이 없는 건만
  AND art.active               -- active = true인 건만
```

이 쿼리는:
- `active = true` AND COMPLETED 트랜잭션이 없는 건을 모두 반환
- **FAILED 상태인 건도 반환됨** (COMPLETED가 아니므로)
- **삭제된 정책의 건도 반환됨** (정책 삭제 여부를 확인하지 않으므로)

**`isDeleted` 값 결정 로직** (`ApprovalReplacementServiceModelMapper`):

```kotlin
deleted = targetInfo?.deleted ?: true
```

- `policyTargetQueryService`에서 정책 정보를 조회
- 정책이 삭제되었거나 조회되지 않으면 `deleted = true`
- 하지만 **이 값으로 필터링하지 않고 그대로 응답에 포함**

**정책 삭제 이벤트 Consumer** (`ApprovalPolicyEventConsumer`):
- `DELETED` 이벤트 수신 시 `deleteAllByTarget()` 호출 — 정상 로직
- 하지만 전체가 `runCatching { ... }.onFailure { log.error(it) }` 로 감싸져 있어 **에러 시 재시도 없이 로그만 남김**

## 해결안 / 조사 방향

### 즉시 대응 (표면 원인 해결)

**Option A: DB에서 stale 레코드 deactivate**
- `approval_replacement_target` 테이블에서 해당 회사 80건(퇴직자 4명)의 `active` 를 `false` 로 변경
- 장점: 즉시 해결, 고객 확인 가능
- 단점: 다른 고객의 동일 문제는 해결되지 않음

### 근본 해결 (근본 원인 해결)

**Option B: 교체 흐름에서 CUSTOMER_SUPER_ADMIN 검증 우회**
- `ApprovalPolicyUseCase.replaceApprovalLineUser()`에서 `update()` 대신 검증을 건너뛰는 별도 메서드를 사용하거나, `validateHasCustomerSuperAdmin()`에 교체 컨텍스트 조건 추가
- 장점: 교체 실패의 근본 원인 해결, 기존 CUSTOMER_SUPER_ADMIN이 포함된 정책도 교체 가능
- 단점: 검증 우회로 인한 사이드이펙트 검토 필요

**Option C: API 레벨 필터링 추가**
- `DtoMapper` 또는 `ApprovalReplacementTargetQueryService`에서 `isDeleted=true` + `status=FAILED` + `relatedProcesses=[]` 인 건을 필터링
- 장점: 전체 고객에 적용, 방어적. 다른 원인의 stale 건도 자동 처리
- 단점: 근본적으로 교체 실패 문제는 해결되지 않음

**Option D: 이벤트 Consumer 에러 핸들링 개선**
- `ApprovalPolicyEventConsumer`의 에러 핸들링 — 에러 시 재시도 또는 DLQ 전송
- 장점: 정책 삭제 시 cleanup 누락 방지
- 단점: 이미 발생한 stale 데이터는 별도 마이그레이션 필요

**권장**: Option A (즉시 대응) + Option B (교체 검증 수정) + Option C (방어적 필터링) 병행
