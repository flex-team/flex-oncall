## 원인 분석

**evaluation_reviewee 삭제는 hard delete(물리 삭제)이다.** `evaluation` 자체의 soft delete(CI-4195)와 달리, 대상자 삭제 시 DB에서 완전히 제거되어 **애플리케이션 레벨 복구가 불가능**하다.

### 핵심 증거

1. `evaluation_reviewee` 테이블에 `deleted_at` 컬럼이 없음[^5] — soft delete 인프라 자체가 부재
2. `bulkDeleteReviewees()` 에서 Spring Data JDBC `delete(entity)` 호출 → `DELETE FROM` 실행[^6]
3. DB 조회 결과 해당 평가에 26명만 남아있고, 문성수는 reviewee/reviewer 양쪽 모두 없음[^3][^4]

### 가설 목록

| # | 가설 | 확인 방법 | 상태 |
|---|------|----------|------|
| 1 | soft delete여서 DML 복구 가능 | Entity/DDL에 deleted_at 컬럼 확인 | ❌ 소거 — deleted_at 없음[^5] |
| 2 | hard delete이므로 DB binlog/백업에서만 복구 가능 | 코드 분석 + DB 데이터 확인 | ✅ 확정[^6][^3] |

## 해결
- **결론**: 복구 불가 → 고객에게 지원 어렵다고 안내[^7]
- **사유**: evaluation_reviewee는 hard delete 구조이며, 연쇄 삭제된 데이터(reviewer, userForm, nomination 등) 복원이 복잡하고 시간이 오래 걸림[^8]
- 김보라가 이주화에게 고객 안내 요청 → 이주화 승인[^7]

## 다음에 같은 문의가 오면
1. **evaluation 삭제인지, evaluation_reviewee(대상자) 삭제인지 구분**
   - evaluation 삭제 → soft delete. `flex_review.evaluation` WHERE deleted_at IS NOT NULL로 확인 후 DML 복구 가능 (CI-4195 참고)
   - evaluation_reviewee(대상자) 삭제 → **hard delete. 애플리케이션 복구 불가**
2. 대상자 삭제인 경우, DB binlog 복구만 가능하나 6개 테이블 정합성 맞춰야 하므로 현실적으로 어려움
3. 고객에게 복구 불가 안내 + 대상자 재추가 및 평가 재작성 권고

## 발견한 스펙/제약
- `evaluation` (평가 세션): soft delete (`deleted_at`, `deleted_user_id`)
- `evaluation_reviewee` (평가 대상자): **hard delete** — soft delete 인프라 없음
- 대상자 삭제 시 `evaluation_reviewer`, `form_user_form`, `evaluation_reviewee_step_mapping`, `reviewee_evaluation_item`, `evaluation_reviewer_nomination` 모두 연쇄 hard delete
