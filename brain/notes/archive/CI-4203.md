## 원인 분석

**younghoon.jo@ohora.com은 리마인드 알림을 발송하지 않았다.** 오늘 글루가에서 `remind/pending-approval` API를 호출한 사람은 다른 2명이다[^1].

### 가설 목록

| # | 가설 | 확인 방법 | 상태 |
|---|------|----------|------|
| 1 | younghoon.jo가 근태 대시보드에서 리마인드 발송 | access log에서 userId=786603의 remind API 호출 확인 | ❌ 소거 — 해당 userId의 호출 없음[^1] |
| 2 | 같은 회사의 다른 사용자가 리마인드 발송 | access log에서 customerId=194114의 remind API 호출 전수 조회 | ✅ 확정[^1] |

### 조사 과정

> 💡 **판단 근거**: access log에서 `remind/pending-approval` API + customerId=194114 조건 전수 조회[^1]
> → younghoon.jo(userId=786603) 호출 건수 = 0
> → 실제 호출자는 jinhee.kim(12:06)과 jungil.um(13:23)
> → euna.go는 리마인드 대상 사용자의 **승인권자**로서 알림을 수신한 것으로 판단[^2]

<details>
<summary>📋 리마인드 알림 발송 흐름</summary>

1. 대시보드에서 "승인 대기중" 카드 클릭 → 리마인드 버튼 클릭
2. `POST /action/v2/time-tracking/customers/{hash}/remind/pending-approval` 호출
3. `PendingApprovalRemindServiceImpl`에서 대상 사용자의 미승인 항목 조회
4. 각 항목의 승인 프로세스에서 ONGOING 상태의 미승인 승인권자 추출
5. 승인권자에게 알림 발송 (Kafka → 알림 시스템)

</details>

## 해결안 / 조사 방향

이 이슈는 **스펙 확인 요청**으로, 코드 수정이 필요하지 않다.

- **답변**: younghoon.jo@ohora.com은 리마인드를 발송하지 않았음
- **실제 발송자**: jinhee.kim@ohora.com (12:06) 또는 jungil.um@ohora.com (13:23)
- 문의자가 진술한 "1시 10분"에 가까운 것은 13:23 호출(jungil.um)이나, 12:06 호출(jinhee.kim)의 알림이 지연 수신되었을 가능성도 있음

## 다음에 같은 문의가 오면

1. **회사 customerId 확인** → Linear 이슈에서 추출
2. **access log 조회**: `flex-app.be-access-{날짜}` 인덱스에서 `json.ipath: "remind/pending-approval"` + `json.authentication.customerId: {id}` 필터
3. 결과에서 `json.authentication.email`로 발송자 특정, `json.requestBody`의 `userIdHashes`로 대상 사용자 특정
4. 알림 수신자는 대상 사용자의 **승인 프로세스에서 ONGOING 상태인 미승인 승인권자**
