### 코드 분석: 두 화면의 퇴직자 판정 기준 비교

#### (그림 상) 정산 대상자 화면 — payee `employmentStatus`
`GET /api/v2/payroll/settlements/work-income/{settlementIdHash}/payees`[^6]

응답의 `employmentStatus`는 **DB에 저장된 스냅샷 값**을 그대로 반환한다[^7]. 정산 대상자 생성 시 flex-core의 `UserStatus`로부터 결정된다[^8].

```kotlin
// PayeeLookUpServiceImpl.kt:325-338
private fun UserStatusEvaluateDto?.getEmploymentStatus(): EmploymentStatus {
    if (this == null || UserStatus.COMPANY_JOIN_SCHEDULED in this.statuses) {
        return EmploymentStatus.UNKNOWN
    }
    if (UserStatus.LEAVE_OF_ABSENCE_ACTIVE in this.statuses) {
        return EmploymentStatus.ON_LEAVE
    }
    if (UserStatus.RESIGNATION_ACTIVE in this.statuses) {
        return EmploymentStatus.OFFBOARDED
    }
    return EmploymentStatus.IN_EMPLOYMENT
}
```

| UserStatus | employmentStatus |
|------------|-----------------|
| `RESIGNATION_ACTIVE` (퇴직 완료) | **OFFBOARDED** |
| `RESIGNATION_SCHEDULED` (퇴직 예정) | **IN_EMPLOYMENT** |
| `LEAVE_OF_ABSENCE_ACTIVE` | ON_LEAVE |
| 그 외 | IN_EMPLOYMENT |

> ⚠️ `RESIGNATION_SCHEDULED`(퇴직 예정)는 체크하지 않아 **IN_EMPLOYMENT**으로 분류된다[^8]. 또한 이 값은 정산 생성 시점에 한 번 세팅되고 이후 업데이트되지 않는다[^9].

#### (그림 하) 정산 목록 홈 — `payeeCountByEmploymentStatus`
`POST /action/v2/payroll/work-income/settlements/list-by-filter`[^10]

응답의 `meta.payeeCountByEmploymentStatus`는 DB에 저장된 `WorkIncomeSettlementMetaInfo` 엔티티의 값을 반환한다[^11].

```kotlin
// WorkIncomeSettlementMetaInfoService.kt:86-95
private fun countPayeesByEmploymentStatus(
    payees: Collection<WorkIncomeSettlementPayeeModel>,
    settlement: WorkIncomeSettlementModel,
): Map<WorkIncomeSettlementPayeeEmploymentStatus, Int> {
    val (retirees, employees) = payees.partition { it.checkOffboarded(settlement) }
    return mapOf(
        WorkIncomeSettlementPayeeEmploymentStatus.EMPLOYEE to employees.size,
        WorkIncomeSettlementPayeeEmploymentStatus.RETIREE to retirees.size,
    )
}
```

```kotlin
// WorkIncomeSettlementPayeeModel.kt:50-51
fun checkOffboarded(settlement: WorkIncomeSettlementDatesProps) =
    offboardedDate?.let { it <= settlement.baseWorkCalculationToDate } ?: false
```

| 조건 | 분류 결과 |
|------|----------|
| `offboardedDate` 존재 AND `<= baseWorkCalculationToDate` | **RETIREE** |
| `offboardedDate` 없음 (null) | **EMPLOYEE** |
| `offboardedDate > baseWorkCalculationToDate` | **EMPLOYEE** |

---

#### 일관성 분석

| 구분 | 정산 대상자 화면 (`employmentStatus`) | 정산 목록 홈 (`payeeCountByEmploymentStatus`) |
|------|--------------------------------------|----------------------------------------------|
| **판정 기준** | flex-core `UserStatus` enum | payee `offboardedDate` vs 정산기간 종료일 |
| **저장 방식** | DB 스냅샷 (정산 생성 시 1회) | DB 스냅샷 (메타정보 생성/갱신 시) |
| **갱신 시점** | 정산 대상자 생성 시 고정, 이후 미갱신[^9] | `createMetaInfo()` 또는 `refreshPayeeCounts()` 시[^12] |

> 💡 **판단 근거**: 두 화면은 **서로 다른 기준**으로 퇴직 여부를 판정한다[^8][^12].
> → 정산 대상자 화면: `UserStatus.RESIGNATION_ACTIVE` 여부 (퇴직 "완료"만 OFFBOARDED)
> → 정산 목록 홈: `offboardedDate <= baseWorkCalculationToDate` (퇴직일이 정산기간 내이면 RETIREE)
>
> 퇴사예정자(`RESIGNATION_SCHEDULED`, offboardedDate가 정산기간 내)의 경우:
> - 정산 대상자 화면: **IN_EMPLOYMENT** (퇴직 완료가 아니므로)
> - 정산 목록 홈: offboardedDate가 세팅되어 있으면 **RETIREE**, 아니면 **EMPLOYEE**
>
> 따라서 두 화면 간 불일치가 구조적으로 발생할 수 있다.
