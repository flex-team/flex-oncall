# CI-4142: 워크플로우 승인 요청 메일 알림 미수신 확인 요청

## 증상
- **문제 정의**: 워크플로우[^4] 2차 승인자가 승인 요청 메일 알림을 받지 못함
- **회사**: 에스제이월드 (Customer ID: 222420)
- **요청자**: 이주화 (CS팀), 원 문의자 ws.yeom@sjworld.com[^1]
- **대상자**: hdr0513@sjworld.com (user_id: 927524, 임원)[^1]
- **영향 범위**: 1명[^1]
- **문제 시점**: 2026-03-17 (워크플로우 문서 번호 2026-187)
- 문의 내용:
  1. 워크플로우 문서 번호 2026-187의 2차 승인자(hdr0513@sjworld.com)가 승인 요청 메일 알림을 받지 못함[^1]
  2. 기존에는 워크플로우 승인 요청 메일 알림을 정상적으로 수신하고 있었음[^1]
  3. 알림 설정은 켜져 있고, 스팸메일함에도 해당 메일이 없음[^1]
  4. 제품 내(인앱) 알림 수신 여부는 미확인 — 대상 구성원이 임원이라 추가 확인이 어려운 상황[^1]
  5. flex 측에서 메일 발송이 정상적으로 이루어졌는지 확인 요청[^1]

## 원인 분석

### 조사 과정

> **조사 경로**: notification_deliver DB 조회 → 인앱 알림 정상 확인[^5] → mail_send_history 0건 발견[^6] → BEI-151로 기록 중단임을 확인[^13] → 코드 분석으로 발송 흐름 파악[^11][^12] → Kafka produce 로그 확인[^14] → SES 이벤트 로그에서 Send+Delivery 확인[^18] → **flex 측 전체 정상, 수신자 측 문제 확정**

8개 가설을 순차적으로 소거하며 원인을 추적했다:

| # | 가설 | 상태 |
|---|------|------|
| 1 | `notification_deliver`에 발송 기록 자체가 없음 | ❌ 소거 — 인앱 알림 3건 존재[^5], 다른 수신자도 동일 타입 알림 정상[^9] |
| 2 | 알림 타입이 `FLEX_ELECTRONIC_APPROVAL_REQUEST_APPROVE`일 것 | ❌ 소거 — 실제 타입은 `FLEX_WORKFLOW_TASK_REQUEST_APPROVE`[^10] |
| 3 | `user_notification_type_setting`에서 EMAIL 채널 비활성화 | ❌ 소거 — 레코드 없음(기본값: 전체 활성)[^7] |
| 4 | `NotificationCondition.resolve(EMAIL)` 에서 `isSendable()` false | ❌ 소거 — Kafka produce 로그 존재 → checkCondition 통과[^14] |
| 5 | `sendEmail()` 내부에서 예외 발생 (Kafka 발행 실패) | ❌ 소거 — Kafka produce 성공 로그 확인[^14] |
| 6 | `mail_send_history`에 기록이 없어 메일 미발송 | ❌ 소거 — BEI-151로 recording 로직 자체가 제거됨[^13] |
| 7 | **메일은 발송되었으나 수신 서버에서 미전달** | ✅ **확정** — SES Delivery 확인, bounce/complaint 없음[^18] |
| 8 | 메일 consumer에서 처리 실패 (SES/Mailgun 발송 오류) | ❌ 소거 — SES Send+Delivery 이벤트 존재[^18]. consumer 비즈니스 로그는 구조적 한계로 미확인[^16], 접속 로그 0건[^15] |

### 원인
- flex 측 메일 발송은 전 구간 **정상**[^18]
- SES에서 sjworld.com 수신 서버로 **Delivery 완료** (bounce/complaint 없음)[^18]
- 수신자 측 메일 서버 내부 라우팅, 스팸 필터, 메일 클라이언트 설정 등의 문제로 추정

## 해결
- flex 측 정상 발송 확인 결과를 고객에게 안내하여 종료[^19]
- 고객사 IT 부서에 메일 서버 내부 로그 확인을 권고[^19]

## 다음에 같은 문의가 오면

1. **먼저 확인**: `notification_deliver` + `notification` 테이블 JOIN으로 인앱 알림 생성 여부 확인[^5]
   ```sql
   SELECT nd.id, n.notification_type, nd.created_at
   FROM flex_pavement.notification_deliver nd
   JOIN flex_pavement.notification n ON nd.notification_id = n.id
   WHERE nd.receiver_id = ? AND nd.created_at >= UNIX_TIMESTAMP('2026-03-01') * 1000
   ORDER BY nd.created_at DESC;
   ```
2. **원인 판별**:
   - 인앱 알림 없음 → 알림 생성 자체가 안 됨 (flow-backend 문제 또는 사용자 상태 문제)
   - 인앱 알림 있음 + 메일 없음 → 아래 순서로 확인:
     1. `user_notification_type_setting`에서 EMAIL 비활성화 여부[^7]
     2. `notification_topic_deliver`에서 토픽 구독 해제 여부[^8]
     3. OpenSearch `flex-app.be-consumer-*`에서 Kafka produce 로그 확인[^14]
     4. **admin-shell 이메일 발송 로그**로 Delivery/Bounce 확인[^21] (또는 SES 이벤트 OpenSearch `flex-prod-ses-feedback-*`에서 `MessageObject.mail.destination` 필터[^18])
3. **조치**:
   - SES Delivery 확인됨 → **flex 측 정상**. 고객에게 수신 서버/스팸 필터 확인 요청
   - SES Bounce 확인됨 → SES Suppression List 등록 여부 확인, AWS 티켓으로 해제 요청
   - Kafka produce 로그 없음 → `checkCondition` 또는 `sendEmail()` 실패. pavement-consumer 에러 로그 확인

> **주의**: `mail_send_history` 테이블은 BEI-151(2026-02-20)로 기록이 중단됨. 메일 발송 여부는 이 테이블이 아닌 SES 이벤트 OpenSearch로 확인해야 한다[^13].
