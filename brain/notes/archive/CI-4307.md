## 원인 분석

**버그 확정 (Known Issue).** `work_record_import=NONE` 회사에서 월 중도 퇴사자의 주휴수당 default item dateRange가 전체 정산 기간(03-31)으로 설정되어, base payment 범위(~03-17)와 불일치 → API 응답에서 item 누락.

### 가설 목록

| # | 가설 | 확인 방법 | 상태 |
|---|------|----------|------|
| 1 | recipient 미생성 (hourlyAllowanceTargetUserIds 미포함) | DB 조회 | ❌ 소거 — DB에 레코드 존재[^4] |
| 2 | payee 미등록 | DB 조회 | ❌ 소거 — payee 테이블에 존재[^3] |

| 4 | **base payment dateRange 불일치로 API 응답에서 item 필터링** | 코드+DB 대조 | ✅ 확정[^6][^7][^8] |

### 조사 과정

> 💡 **판단 근거**: DB에 recipient 존재 확인[^4] → access log에서 API 200 응답 확인, 7명 중 user 926921만 `items:[]`[^5] → 코드에서 `createReadData()` JOIN 로직 발견[^6] → base payment dateRange 불일치 확인[^7][^8]

4. **dateRange 불일치 확인**:

   - JOIN 조건 `basePayment.dateRange.contains(item.dateRange.to)` → 03-17 범위에 03-31 미포함 → null → 제외[^6]

</details>

### 5 Whys

1. 왜 주휴수당 대상자에 안 나오나? → API가 `items:[]`로 반환
2. 왜 items가 비어있나? → `createReadData()`의 basePayment JOIN에서 필터링[^6]
3. 왜 JOIN이 실패하나? → item.dateRange.to(**03-31**)가 basePayment.dateRange(**~03-17**)에 미포함
4. 왜 item의 to가 03-31인가? → default recipient 생성 시 전체 사용자 MAX dateRange 사용[^8]
5. **왜 개별 사용자 범위를 사용하지 않나?** → `createRecipients()`에서 전체 `regularPayPerHours`의 min/max로 산출 (버그)[^8]

## 발견한 스펙/제약

- 정산 템플릿의 `work_record_import` 설정이 정산에 상속됨[^9]
- `shouldFetchWorkScheduleResults()` = `workRecordImport === WorkRecordImportType.FLEX` — NONE이면 TT 주휴수당 데이터 미조회[^10]
- `work_record_import=NONE` 회사는 TT 주휴수당 데이터를 사용하지 않고, default recipient(0원)로 생성됨
- 주휴수당 API 응답의 `items`는 DB `statutory_leisure_payment_item`에서 직접 읽되, base payment과 `mapNotNull` JOIN을 거침 — base payment 없으면 item 제외
