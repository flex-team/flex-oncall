# TT-16783: 외부 연동(SECOM) 등록 시 다법인 workspace에서 customerKey 충돌 오류

> **상태**: 종료 — 2026-03-17 (테스트 계정, 실사용자 영향 없음)

## 증상
- **회사**: workspace 39602 (다법인 구조)[^1]
- **요청자**: customerId 222911 (신규 SECOM 등록 시도)
- **에러 발생 시각**: 2026-03-17 11:57:49, 11:58:59 KST
- Sentry에서 prod 오류 감지[^2]:
  1. SECOM 외부 연동 등록 시 `java.lang.IllegalStateException` 발생
  2. 에러 메시지: `하나의 계열사 안에 서로 다른 외부 연동 정보가 존재합니다.`
  3. 상세: `동일한 workspace(39602)내에 여러 외부 연동처가 존재합니다. (customerKeys=[secom_jy0r76pz1k, secom_k1ebayvkzy])`

## 원인 분석

### 에러 발생 위치

에러는 `registerNewProvider` 메서드에서 발생한다[^3].

```kotlin
// CustomerExternalProviderUpdateServiceImpl.kt:209-212
// customerKeys가 2개 이상이면 else 분기에서 에러 throw
```

동일 workspace 내 동일 providerType(SECOM)에 대해 `customerKeys`[^4]가 2개 이상 존재할 때, 어느 key를 사용할지 결정할 수 없어 예외를 발생시킨다.

### 코드 흐름

1. `registerNewProvider` 호출
2. `findCustomersInSameWorkspace`로 workspace 39602의 모든 customer 조회
3. `findAllByCustomerIdIn`으로 해당 customer들의 external provider 조회
4. `filter { it.providerType == SECOM }` → 2건
5. `map { it.accountId }.distinct()` → [secom_jy0r76pz1k, secom_k1ebayvkzy] (size=2)
6. `when (customerKeys.size)` → else 분기 → error

### DB 조사 결과

workspace 39602에 속한 외부 연동 provider 전체:

| id | customer_id | account_id | provider_type | 생성일 (KST) | 수정일 (KST) |
|---|---|---|---|---|---|
| 178 | 72326 | caps_jy0r76pz1k | CAPS | 2024-11-14 00:45:56 | 2024-11-14 00:45:56 |
| 188 | 72326 | secom_jy0r76pz1k | SECOM | 2024-11-20 22:33:57 | 2025-02-26 00:08:34 |
| 198 | 110972 | secom_k1ebayvkzy | SECOM | 2024-12-03 08:05:12 | 2024-12-03 08:05:12 |
| 278 | 110972 | caps_k1ebayvkzy | CAPS | 2025-03-05 02:43:56 | 2025-04-09 19:33:39 |
| 371 | 72326 | telecop_jy0r76pz1k | TELECOP | 2025-07-09 00:24:55 | 2025-07-17 02:09:55 |

- `jy0r76pz1k` = 72326의 customerId 해시, `k1ebayvkzy` = 110972의 customerId 해시
- **SECOM, CAPS 모두 동일한 문제 존재** — 같은 workspace 내에서 providerType별로 customerKey가 2개씩
- TELECOP은 72326만 등록되어 있어 현재는 문제 없음

workspace_customer_mapping 조회 결과:

| customer_id | workspace_id | 매핑 생성일 (KST) |
|---|---|---|
| 72326 | 39602 | 2022-08-11 00:11:06 |
| 110972 | 39602 | 2023-10-18 19:59:48 |

두 customer 모두 외부 연동 등록(2024-11~12월) **이전부터** 동일 workspace에 속해 있었다.

### customerKey 생성 로직

`generateCustomerKey`는 customerId[^5]를 해시하여 key를 생성한다[^3]. customer마다 고유한 key가 만들어지므로, 다법인 구조에서 각 customer가 별도로 등록되면 서로 다른 key가 생성된다.

### 정상 플로우 vs 현재 상태

정상적인 등록 흐름:
1. 첫 번째 customer가 **case 0** (신규 생성)으로 진입 → customerKey 생성
2. 이후 같은 workspace의 다른 customer가 **case 1** (기존 key 재사용)으로 진입 → 첫 번째에서 생성된 key를 공유

현재 workspace 39602의 상태:
- SECOM providerType에 대해 **서로 다른 accountId(customerKey)가 2개** 존재[^1]
  - `secom_jy0r76pz1k` (customer 72326)
  - `secom_k1ebayvkzy` (customer 110972)

### 근본 원인: 다법인 지원 코드 추가 시 기존 데이터 마이그레이션 누락

git 이력 확인 결과, 2024-12월 시점의 `register` 메서드에는 **다법인(workspace) 조회 로직이 없었다**[^6].

당시 코드(`5974b8a2a2` 커밋 기준):
- `getProvider(customerId, providerType)`로 **해당 customer만** 조회
- `provider == null`이면 바로 `generateCustomerKey` 호출하여 신규 생성
- workspace 내 다른 customer의 provider 존재 여부를 전혀 확인하지 않음

**시간순 재구성:**

| 시점 | 이벤트 |
|---|---|
| 2022-08 ~ 2023-10 | 72326, 110972 모두 workspace 39602에 매핑 |
| 2024-11-14 | 72326이 CAPS 등록 → `caps_jy0r76pz1k` (다법인 체크 없는 코드) |
| 2024-11-20 | 72326이 SECOM 등록 → `secom_jy0r76pz1k` (동일) |
| 2024-12-02 | 110972가 SECOM 등록 → `secom_k1ebayvkzy` (동일 — workspace 체크 없이 자기 해시로 생성) |
| 이후 | 다법인 지원 코드 추가 — `registerNewProvider`에 workspace 내 customerKey 통일 로직 도입 |
| 2025-03-04 | 110972가 CAPS 등록 → `caps_k1ebayvkzy` (이 시점 코드 버전 미확인) |
| 2026-03-17 | 222911이 SECOM 등록 시도 → 새 코드가 workspace 내 customerKey 2개 감지 → **에러** |

**근본 원인: 다법인 지원 코드가 추가될 때, 기존에 각자 생성된 customerKey 데이터의 마이그레이션(통일)이 누락됐다.**

## 해결 방향

- **단기**: workspace 39602의 customerKey를 하나로 통일하는 데이터 패치
  - SECOM: `secom_jy0r76pz1k` / `secom_k1ebayvkzy` 중 하나로 통일
  - CAPS: `caps_jy0r76pz1k` / `caps_k1ebayvkzy` 중 하나로 통일
  - 외부 연동 DB(PostgreSQL) 계정도 함께 정리 필요
- **장기**: 다법인 지원 이전에 등록된 다른 workspace에도 동일 문제가 있는지 전수 조사
  - 동일 workspace 내 동일 providerType에 서로 다른 accountId가 존재하는 케이스 스캔

## 결론

요청자 customerId 222911은 **테스트 계정**(주토피아 경찰서)이므로 실사용자 영향 없음. 이슈 종료.
