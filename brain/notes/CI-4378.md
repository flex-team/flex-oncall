# CI-4378: 미래 근무유형 예약 취소 시 자동근무기록 재등록 누락

## 증상

- **문제 정의**: 고정근무 유형에 자동근무기록이 활성화되어 있으나, 특정 구성원의 내근무에 근무가 전혀 표시되지 않음
- **회사**: 더파운더즈 (Customer ID: 60371)
- **요청자**: ti.ahn@thefounders.kr (CS 접수)
- **대상자**: hm.kim@thefounders.kr (user_id: 934203)
- **영향 범위**: 해당 구성원 1명 (동일 근무유형의 다른 구성원은 정상)
- **문제 시점**: 2026-04-01 이후 전체
- 문의 내용:
  1. 근무유형에서 자동 근무 기록이 활성화되어 있음[^1]
  2. 모든 구성원 동일한 유형인데 이 구성원만 내근무에 자동근무 기록 그래프가 안 보임[^1]
  3. 내근무 상단 "주기" 영역을 눌러도 자동근무기록 설정 UI가 없음[^2]

---

## 원인 분석

### 조사 과정

> 💡 **단서 1 — 근무유형 이벤트**: `v2_user_work_rule`에서 이 유저만 유일하게 CANCEL 이벤트가 존재[^3]

| id | 근무유형 | 기간 | 이벤트 | 시점 |
|---|---|---|---|---|
| 1417812 | 236571 ([기본][통상] 9:30~18:30) | 1970-01-01 ~ 9999-12-31 | REGISTER | 2026-01-19 |
| 1455910 | 236566 ([기본][통상] 9:00~18:00) | 2026-04-01 ~ 9999-12-31 | REGISTER | 2026-03-04 |
| **1466335** | **236566** | **2026-04-01 ~ 9999-12-31** | **CANCEL** | **2026-03-17** |

→ 03-04에 236566으로 04/01부터 변경 예약 → 03-17에 취소 → 236571로 fallback[^3]

> 💡 **단서 2 — work rule 해석은 정상**: `UserWorkRuleResultCalculator`의 필터 체인(ActiveFilter)이 CANCEL된 236566을 제거하고 236571이 04/01 이후를 커버[^4]. 시스템은 CANCEL 후 빈 기간이 생기지 않도록 검증도 수행[^5]

> 💡 **단서 3 — auto work plan 데이터가 핵심**: 내근무 스케줄은 OpenSearch가 아닌 DB(`v2_user_work_plan` + `v2_user_repetitive_work_plan`)에서 계산[^6]

`v2_user_work_plan` 조회 결과:

| id | 기간 | 이벤트 | 시점 | 설명 |
|---|---|---|---|---|
| 2742564 | 01-19 ~ 9999-12-31 | REGISTER | 01-19 | 원래 236571 기반 자동근무 |
| 2828808 | 04-01 ~ 9999-12-31 | RESET | 03-04 | 236566 등록 시 기존 계획 초기화 |
| 2828809 | 04-01 ~ 9999-12-31 | REGISTER | 03-04 | 236566 기반 자동근무 등록 |
| **2839951** | **04-01 ~ 9999-12-31** | **RESET** | **03-17** | **236566 CANCEL 시 초기화** |
| **2839952** | **1970-01-01 ~ 2026-01-18** | **REGISTER** | **03-17** | **잘못된 기간으로 재등록!** |

→ CANCEL 후 재등록(#5)이 `04-01 ~ 9999-12-31`을 커버해야 하는데, `1970-01-01 ~ 2026-01-18`로 들어감[^7]
→ 04/01 이후는 RESET(#4)만 남고 REGISTER가 없어서 **내근무가 비어 있음**

### 원인

**`handleAfterUserWorkRuleMappingRemove`의 auto work plan 재등록 로직 버그**[^8]

```kotlin
// UserWorkRuleAfterChangeService.kt:71-89
targetUserWorkRules
    .sortedBy { it.applyTimeStampFrom }
    .forEach { uwr ->
        userWorkPlanUpdateService.registerAutoWorkPlanByWorkRule(
            ...,
            DateRange(uwr.dateFrom, uwr.dateTo),  // ← 여기가 문제
            eventTimeStamp,
        )
    }
```

- `resetAutoWorkPlan`으로 04/01+ 구간을 지운 뒤, `registerAutoWorkPlanByWorkRule`에 work rule 전체 범위(1970~9999)를 전달
- 내부에서 기존 REGISTER(id=2742564, 01-19~9999)가 있다고 판단하여 **gap(1970-01-01~01-18)만** 채움
- RESET에 의해 무효화된 04/01+ 구간은 기존 REGISTER가 커버한다고 잘못 판단 → **재등록 누락**

### FE 자동근무 설정 UI 미노출 (추가 발견)

- FE 코드에서 고정근무(FIXED) 타입이면 `checkIsAutoWorkPlanEnabled = false`로 자동근무 설정 UI를 숨김[^9]
- 이는 **의도된 스펙** — 고정근무는 BE가 자동 생성하므로 유저 설정 불필요[^10]
- 하지만 BE 자동 생성이 실패하면 **UI에서 복구할 방법이 없는 상태**

## 해결

### 즉시 대응 (데이터 보정)

`v2_user_work_plan`에서 236566 관련 이벤트 4건 삭제 → 원본 REGISTER(id=2742564)가 04/01+ 커버

```sql
-- 롤백용 백업: 삭제 전 SELECT * FROM flex.v2_user_work_plan WHERE id IN (2828808, 2828809, 2839951, 2839952);
-- → INSERT문으로 변환해서 보관

-- 블럭 삭제 (REGISTER 2건의 repetitive work plan)
DELETE FROM flex.v2_user_repetitive_work_plan
WHERE user_work_plan_id IN (2828809, 2839952);

-- 이벤트 삭제
DELETE FROM flex.v2_user_work_plan
WHERE id IN (2828808, 2828809, 2839951, 2839952);

-- 결과: 원본 REGISTER(id=2742564, 01-19~9999)만 남음 → 04/01+ 정상 커버
```

### 코드 버그 수정 (요청 필요)

**파일**: `flex-timetracking-backend` > `work-rule/service/src/main/kotlin/team/flex/workrule/user/UserWorkRuleAfterChangeService.kt:71-89`

**문제**: `resetAutoWorkPlan`으로 지운 구간과 `registerAutoWorkPlanByWorkRule`로 채우는 구간이 불일치

**수정안**:
```kotlin
// AS-IS: work rule 전체 범위 → 내부에서 gap만 채움 → RESET 구간 누락
DateRange(uwr.dateFrom, uwr.dateTo)

// TO-BE: reset 시작일을 하한으로 사용 → RESET 구간부터 재등록 보장
val resetStartDate = max(eventDate, applyTimeStampFrom.toLocalDate(appliedZoneId))
DateRange(max(resetStartDate, uwr.dateFrom), uwr.dateTo)
```

**원인 상세**: `resetAutoWorkPlan`이 04/01+를 지운 뒤, 재등록 시 work rule 전체 범위(1970~9999)를 전달하면 `registerAutoWorkPlanByWorkRule` 내부에서 기존 REGISTER(01-19~9999)가 아직 유효하다고 판단하여 gap(1970~01-18)만 채움. RESET으로 무효화된 구간은 인지하지 못함.

**테스트 시나리오**: "미래 날짜로 근무유형 변경 예약 → 적용 전 취소 → 원래 근무유형의 자동근무기록이 취소 이후 날짜에 존재하는지 검증"

## 다음에 같은 문의가 오면

1. **먼저 확인**: `v2_user_work_rule WHERE user_id = {대상}` — CANCEL 이벤트 존재 여부
2. **원인 판별**:
   - CANCEL 있음 + `v2_user_work_plan`에 문제 기간의 RESET만 있고 REGISTER 없음 → **이 버그**
   - CANCEL 없음 → 다른 원인 (유저 수동 삭제, 근무유형 설정 등)
3. **조치**: `v2_user_work_plan`에서 해당 CANCEL 관련 이벤트(RESET + 잘못된 REGISTER) 삭제. 롤백 쿼리 보관 필수.
