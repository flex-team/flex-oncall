## 원인 분석

**두 계열사의 자동 로그아웃(세션 만료) 설정 시간 차이로 인한 정상 동작 (스펙)**[^7]

### 메커니즘

1. 뉴넥스와 하이버의 **자동 로그아웃 설정 시간**[^9]이 서로 다름
2. 세션 시간이 **긴 회사에서 짧은 회사로** 전환할 때, 짧은 회사 기준으로 세션이 이미 만료된 것으로 판단됨
3. exchange API(계열사 전환 시 대상 회사의 인증 토큰을 발급하는 API)가 대상 회사의 세션 유효성을 체크 → 만료 → `401 AUTH_401_000` 반환 → 로그인 풀림

> 💡 **판단 근거**: 윤성복님(인증 도메인 전문가)이 이전 동일 이슈 사례[^10]를 근거로 원인 특정[^7]. 김한나님(CS)이 자동 로그아웃 설정 UI를 통한 안내 진행 확인[^11].

### 스펙 vs 버그 판별: **스펙**

- 계열사별로 독립적인 세션 만료 설정을 가지는 것은 **의도된 설계**[^7]
- 전환 시 대상 회사의 세션 만료 기준을 적용하는 것도 **정상 동작**
- 이전에도 동일 증상으로 문의가 있었고 동일하게 "설정 맞추기"로 안내한 사례 존재[^10]

### 가설 목록

| # | 가설 | 확인 방법 | 상태 |
|---|------|----------|------|
| 1 | 계정 병합 후 인증 수단이 주회사 workspace에만 존재하여 전환 시 검증 실패 | Linear 코멘트에서 실제 원인 확인 | ❌ 소거 — 원인은 세션 시간 차이[^7] |
| 2 | `\|\|1@merged.user` suffix가 exchange API에서 무효 처리 | 〃 | ❌ 소거 |
| 3 | workspace access token의 workspaceUuid 매칭 실패 | 〃 | ❌ 소거 |
| 4 | **두 회사의 자동 로그아웃 설정 시간 차이** | 윤성복님 확인[^7] | ✅ 확정 |

<summary>📋 access log 분석 결과 (참고)</summary>

오늘(3/19) exchange API 응답 상태 분포[^5]:
- **200 성공**: 10,000건 이상
- **401 실패**: 211건 (전체 대비 약 2%)

401 실패 토큰 분석 (100건 샘플):
- 만료된 토큰: 15건 (15%)
- 유효한 토큰(만료 전): 85건 (85%) — workspace access token의 exp 기준. 대상 회사의 세션 만료 기준과는 별개.
- 49개 고유 workspace에서 발생

401 응답의 `elapsedTime`이 3-6ms로 매우 짧음 → 인증 레이어에서 즉시 거부. 성공 요청은 47-54ms.

</details>

## 다음에 같은 문의가 오면

1. **계열사 구조 확인**: 대상자가 어떤 회사들에 소속되어 있는지 확인 (Metabase `dashboard/212?고객사_id=`)
2. **자동 로그아웃 설정 비교**: 각 계열사의 자동 로그아웃 설정 시간이 동일한지 확인
3. **설정이 다르면**: "세션 시간이 긴 회사에서 짧은 회사로 전환 시 로그인이 풀리는 것은 정상 동작이며, 두 회사의 자동 로그아웃 설정을 맞추면 해결됩니다" 안내
4. **설정이 같은데도 발생하면**: 다른 원인 조사 필요 (인증 수단, 세션 상태 등)
