## 원인 분석

**승인 시스템(flex-approval-backend)에서는 2025-12-22에 APPROVED 처리 완료되었으나, core 시스템(flex-core-backend)의 `user_data_approval_process`가 ONGOING 상태로 잔류. 승인 이벤트가 core 컨슈머에 전달/처리되지 않아 `UserDataApprovalApprovedServiceImpl.approve()` 콜백이 미실행.**

> 💡 **판단 근거**: `approval_process.status = APPROVED`(5건)[^5] → `user_data_approval_process.status = ONGOING`(5건, 동일 target_uid)[^6]
> → 이벤트 재발행 후 OpenSearch 로그 확인[^10] → core-consumer가 이벤트를 수신하지만 **approve() 콜백 실행 중 예외 발생**
> → `UserDocumentFileLookUpServiceImpl.getAllByIdentities(:131)`에서 `CoreResourceNotFoundException` — 삭제 대상 파일이 존재하지 않음[^11]

<summary>코드 분석 — 승인 콜백 흐름</summary>

승인 완료 시 core 시스템의 처리 흐름[^8]:

1. `ApprovedApprovalNotificationFactory.onApprovedToRequester()` — 승인 이벤트 수신
2. → `userDataApprovalApprovedService.approve()` 호출
3. → `UserDataApprovalApprovedServiceImpl.approve()`:
   - user_data_approval_process status → APPROVED
   - 정책 타입별 실제 데이터 업데이트 (PERSONAL, FAMILY, WORK_EDU_EXPERIENCE, DOCUMENT, LICENSE)
   - 예: `userPersonalBundleService.update()`, `dependentFamilyApplicationService` 등

이 콜백이 실행되지 않으면:
- user_data_approval_process는 ONGOING 유지
- 프로필 데이터는 변경되지 않음
- 할일(todo)은 진행중 유지

</details>

## 상태 보정 시 주의

- **DB 직접 상태 변경(user_data_approval_process.status → APPROVED) 금지**: 이렇게 하면 상태만 바뀌고 실제 프로필 데이터 반영이 누락됨[^8]
- **`sync-with-approval` 부적합**: 이 API는 workflow_task용이며, user_data_approval_process에는 영향 없음
- 이벤트 재발행(`re-produce-messages`, `produce-approval-process-event`) 모두 실패 — Kafka로 이벤트는 발행되나 **어떤 서비스에서도 소비하지 않음**[^10]

### 남은 옵션

**옵션 A: `user_data_approval_diff` 데이터 보정 (즉시 대응)**
- 문제 건의 diff에서 `deleteFiles` 참조를 제거하면 approve() 콜백이 정상 실행될 수 있음
- 이후 `re-produce-messages`로 이벤트 재발행하여 처리 완료

**옵션 B: approve() 로직 방어 코드 추가 (근본 해결)**
- `UserDocumentFileServiceImpl.operate()`에서 없는 파일을 skip하도록 방어 처리
- 또는 `UserDocumentFileLookUpServiceImpl.getAllByIdentities()`에서 부분 실패를 허용

### 근본 해결

- `approve()` 콜백이 **멱등하지 않음** — 파일이 중간에 삭제되면 영구적으로 처리 불가 상태에 빠짐
- 승인 이벤트 전파 실패가 CI-4182, CI-4019, CI-4326으로 반복되는 패턴
- approve() 콜백의 방어 로직 강화 필요 (없는 리소스 skip, 부분 실패 허용)

## 발견한 스펙/제약

- USER_DATA 카테고리는 workflow_task를 사용하지 않음 — 할일 상태는 user_data_approval_process.status에서 직접 파생[^7]
- 프로필 변경 승인의 "할일" 상태와 "데이터 반영"이 모두 core 콜백(`approve()`)에 의존하므로, 콜백 실패 시 둘 다 실패함[^8]
