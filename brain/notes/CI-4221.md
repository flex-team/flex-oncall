## 원인 분석

사용자가 보고한 "오류"는 **두 가지 별개의 현상이 겹친 것**으로 확인되었다.

### 현상 1: dry-run 검증 실패 (400)

양일 모두 **첫 번째 업로드 시도의 dry-run이 400을 반환**[^1]했다. 이는 엑셀 데이터에 검증 오류가 있을 때의 정상 동작이다. 사용자가 파일을 수정하여 재시도하면 통과한다.

### 현상 2: 비정상적으로 느린 처리 시간

서버 access log 기준, 모든 실제 업로드(POST /upload)는 **200으로 정상 완료**되었다[^2]. 그러나 처리 시간이 비정상적으로 길었다:

| 파일 크기 | 이 회사 (cid=185358) | 다른 회사 (cid=175919) |
|----------|---------------------|---------------------|
| ~35KB | **216초** (3.6분) | — |
| ~49KB | — | **1초** |

> 💡 **판단 근거**: 비슷한 크기 파일에서 200배 이상의 속도 차이 → 회사별 데이터 규모(구성원 수, 업로드 대상 날짜 범위)에 따른 처리량 문제[^3]

### 원인 (확정)

1. **dry-run 400**: 엑셀 데이터의 검증 오류 (스펙에 의한 정상 동작). `UserSalaryContractNotFoundException` 등이 원인
2. **느린 업로드**: `UserWorkScheduleV2ExcelUploadMappingService`가 각 구성원/날짜별로 **내부 API를 순차 호출**(N+1 패턴)[^6]하여, 구성원 수 × 날짜 수에 비례하여 처리 시간 증가
3. **사용자 에러의 직접 원인**: 서버 처리 시간(216초)이 **CloudFront origin_read_timeout(180초)**[^11]을 초과 → CF가 **504 Gateway Timeout** 반환. 서버는 200으로 정상 완료했으나 유저에겐 에러 표시

### Timeout 체인 (확정)

```
axios (timeout 없음) → CloudFront (180초) → ALB (300초) → 서버 (216초 처리)
                           ↑ 여기서 504 반환
```

| 요청 | 처리 시간 | CF 180초 초과 | 유저 경험 |
|------|----------|:------------:|----------|
| #3 upload (35KB) | 216초 | ✅ | 504 에러 |
| #5 upload (35KB) | 217초 | ✅ | 504 에러 |
| #8 upload (25KB) | 211초 | ✅ | 504 에러 |
| #10 upload (16KB) | 27초 | ❌ | 정상 |
| #12 upload (21KB) | 102초 | ❌ | 정상 |

### 스펙 vs 버그

- **dry-run 400**: ✅ 스펙 (검증 실패 시 정상 동작)
- **느린 업로드**: ⚠️ 성능 이슈 (기능적 버그는 아니지만, 구성원/날짜가 많으면 타임아웃 가능성)

## 해결안 / 조사 방향

### 즉시 대응 (표면 원인 해결)
- **CS 안내**: dry-run 400은 엑셀 데이터 검증 실패이므로, 반환된 엑셀 파일의 오류 표시를 확인하여 데이터 수정 후 재업로드 안내
- **분할 업로드 안내**: 구성원/날짜가 많을 경우 파일을 분할하여 업로드하면 타임아웃 없이 처리 가능

### 근본 해결 (성능 개선 — 필요 시)
- 업로드 처리의 N+1 패턴 개선 (batch API 호출로 전환)
- 구성원/날짜 수에 따른 처리 시간 증가가 선형 이상이므로, 대량 업로드 시 비동기 처리(큐 기반) 전환 검토
- **Timeout 설정 확인 완료**: CloudFront origin_read_timeout=180초[^11], ALB idle_timeout=300초[^10], axios timeout=없음
- CloudFront 180초가 병목 → CF timeout을 늘리거나, 서버 처리를 180초 이내로 줄이거나, 비동기 전환 필요

   - status=500 → 서버 에러. app log에서 traceId로 추적
3. **조치**: 구성원/날짜가 많은 파일은 분할하여 업로드하도록 안내. 근본 해결은 비동기 전환 필요.
