## 증상

보상휴가[^1]를 부여한 뒤 회수(withdraw)했으나, 근무 화면의 자물쇠(🔒) 표기가 사라지지 않아 퇴근 기록을 수정할 수 없다[^2].

## 현재까지 파악된 내용

- 회사 전체에 보상휴가 일괄 부여가 활발하게 이루어지고 있음 (3/10~3/18 사이 100건+ 부여)[^3]
- 3/10 날짜에 대한 근무스케줄 잠금 이벤트: **LOCK_REGISTER 26건, LOCK_CANCEL 2건만 존재**[^4]
- LOCK_CANCEL된 2건은 소량 assign (3분, 15분)이고, 대량 assign(200분+)은 lock이 유지됨[^4]

## 원인 분석

**보상휴가 회수 시 `usedMinutes == 0`인 경우에만 잠금을 해제하는 의도적 설계에 의해, 일부라도 사용된 보상휴가의 잠금은 회수 후에도 유지된다.**

### 핵심 증거

1. **코드**: `UserCompensatoryTimeOffWithdrawService.doWithdraw()` 에서 `usedMinutes == 0` 필터링 후에만 `cancelLock` 호출[^5]
2. **코드 주석**: "보상휴가를 일부라도 사용한 상태에서 회수하는 경우 → lock을 해제하면 안된다. (보상휴가 부여의 원장인 초과근무기록이 변경되면 안됨)"[^5]
3. **DB**: 26건의 LOCK_REGISTER 중 2건만 LOCK_CANCEL — cancel된 2건은 모두 소량(3분, 15분) assign으로 usedMinutes가 0이었을 가능성이 높음[^4]

### 가설 목록

| # | 가설 | 확인 방법 | 상태 |
|---|------|----------|------|
| 1 | 회수 시 lock 해제 로직이 `usedMinutes == 0` 조건부로만 동작 | 코드 분석 | ✅ 확정 — 코드와 주석으로 확인[^5] |
| 2 | 해당 유저의 보상휴가가 "사용(use)" 상태여서 lock 해제 불가 | DB 조회 — withdrawal 테이블 확인 | ❌ 소거 — 회수 자체가 2건뿐, usedMinutes 이전 문제[^7] |
| 3 | 회수 자체가 일부만 완료됨 (전체 미완료) | withdrawal 레코드 존재 여부 확인 | ✅ 확정 — 3/10 대상 26건+ 중 2건만 withdrawal 존재[^7] |

### usedMinutes 계산 구조 (심층 분석)

> `usedMinutes > 0`이 되려면 `v2_user_custom_time_off_use` 에 실제 사용(USE) 레코드가 있어야 한다. 암묵적 사용은 불가능.[^8]

- 계산 경로: `UserCompensatoryTimeOffBucketDomainModel.usedMinutes` → `usedTimeOffMinutesMap` → `CompensatoryTimeOffUseMinutesInfoCalculator.calculate()` [^8]
- 입력: `targetUserCustomTimeOffBucket.totalUseHistories` (v2_user_custom_time_off_use 테이블)[^9]
- USE 레코드 0건 → `usedMinutes = 0` **무조건**
- 따라서 **회수만 정상 실행되면 lock은 반드시 해제됨** (사용 이력이 없는 한)

### Lock 매칭 구조

- 각 LOCK_REGISTER는 개별 LOCK_CANCEL 필요 (1:1 매칭, `referenceUserWorkScheduleLockEventId`로 연결)[^10]
- 같은 날짜에 여러 assign의 lock → **전부** cancel되어야 해당 날짜 잠금 해제
- 활성 lock 판별: `LOCK_REGISTER.id`가 어떤 `LOCK_CANCEL.referenceId`에도 없으면 active[^10]

### 스펙 vs 버그 판별

**판정: 스펙 (정상 동작) — 고객 측 운영 미완료**

- 고객이 "모두 회수했다"고 했지만, DB상 3/10 날짜 대상 보상휴가 26건+ 중 **2건만 회수** 완료[^7]
- 회수된 2건(3분, 15분 assign)은 lock이 정상적으로 해제됨 (LOCK_CANCEL 이벤트 존재)[^4]
- **나머지 24건+은 회수되지 않아 lock이 유지** — 코드는 정상 동작
- `usedMinutes == 0` 조건부 lock 해제 설계도 별개로 존재하지만, 이번 건에서는 회수 미완료가 직접 원인
- 회수된 2건 모두 usedMinutes==0으로 lock 정상 해제 → `usedMinutes` 조건 로직도 정상 동작 확인

## 해결안 / 조사 방향

### 즉시 대응 (권장)

**고객 안내: 해당 구성원의 보상휴가를 모두 회수하도록 안내**
- 현재 3/10 날짜에 대해 대부분의 보상휴가가 회수되지 않은 상태
- 해당 구성원(gw0914@irisbr.com)의 3/10 관련 보상휴가를 **전부** 회수하면 lock이 해제될 가능성 높음
- 단, 보상휴가를 이미 사용(use)한 이력이 있다면 회수 후에도 lock이 유지됨 (`usedMinutes > 0` 조건)
- 사용 이력이 있어서 lock이 풀리지 않는 경우 → 수동 lock 해제 필요

### 수동 lock 해제 (fallback)

보상휴가를 전부 회수했는데도 lock이 유지되는 경우:
- `v2_user_work_schedule_lock_event` 에 해당 유저의 LOCK_CANCEL 이벤트를 수동 삽입
- 장점: 즉시 해결 가능
- 단점: 초과근무기록 보호 장치가 해제됨

### 근본 개선 (검토 필요)

보상휴가를 사용한 상태에서 회수 + lock 해제를 허용할지 제품 차원의 결정이 필요:
- **근로자에게 유리** ✅: lock 해제하여 퇴근 기록 수정을 허용하면 근로자의 기록 정정 권리 보장
- **리스크** ⚠️: 초과근무기록이 변경되면 이미 사용된 보상휴가와 정합성이 깨질 수 있음
