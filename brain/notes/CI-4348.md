**문의 접수**: 2026-04-03 / **조사일**: 2026-04-07

## 관련 이력

2026-01-05 커밋(`1d7d4027`, [PR #456](https://github.com/flex-team/flex-openapi-backend/pull/456))에서 연관된 버그가 수정됐다:

```diff
- .mapValues { convertWorkClockToWorkTimeBlock(workFormMap, dayWorkClocks) }  // 전체 기간 팩을 모든 날짜에 할당
+ .mapValues { convertWorkClockToWorkTimeBlock(workFormMap, it.value) }        // 해당 날짜 팩만 처리
```

다만 CASE 2/3는 단일 날짜 조회(`2025-07-04/2025-07-04` 등)이므로 이 수정의 영향을 받지 않는다 — 단일 날짜 범위에서는 `dayWorkClocks` ≈ `it.value`로 동작이 동일하다. 이번 버그는 PR #456 이전에도 잠재적으로 존재했을 가능성이 있으며, "갑자기 들어온 문의"의 정확한 원인(API 신규 사용, 근무 수정 케이스 증가 등)은 미확인이다.

---

## 증상

퇴근 타각 없이 근무 수정으로 확정한 날짜에서 두 가지 이상 현상 발생:

1. **동일 시간대 근무 블록 중복** — 같은 `blockFrom`/`blockTo`를 가진 블록이 2개 반환
2. **`blockTo = null` 인 잔재 블록** — 휴게 블록이 종료 시간 없이 응답에 포함

고객이 보고한 케이스:
- CASE 1: `2025-04-02` 퇴근 미처리 후 근무 수정 — 데이터 미확인 (아래 참고)
- CASE 2: `2025-07-04` 과거 근무 이력 — **버그 확인됨**
- CASE 3: `2025-02-03` 과거 근무 이력 — **버그 확인됨**

## 원인 분석

### API 구조

OpenAPI는 TT로부터 두 소스를 병렬 조회하여 같은 날짜 버킷에 합산한다:

```
workBlocksMapByDate[date] ←── (1) connected-event-packs  (타각 이벤트 팩)
                           ←── (2) work-schedules         (확정 근무 데이터)
```

퇴근 타각 없이 근무 수정으로 확정한 경우, **두 소스 모두 같은 날짜 데이터를 가져 중복이 발생**한다.

### STOP=null 팩을 노출하는 것은 스펙

슬랙 스레드 확인 결과(윤성복, 2026-04-03 17:49):

> "종료시간이 없는건 확정안한 실시간 근무도 조회가능하기위해서 스펙이긴 합니다"

STOP=null 팩은 **현재 진행 중인 근무를 표현하기 위해 의도적으로 노출**하는 값이다.

### 버그 — 근무 확정 후에도 STOP=null 팩이 함께 노출됨

문제는 STOP=null 팩 자체가 아니라, **같은 날짜에 workSchedule(확정 데이터)이 존재하는 경우에도 STOP=null 팩을 같이 내보내는 것**이다.

```kotlin
// UserWorkScheduleWithWorkClockMappingService.kt:184
.filter { it.workClockStopEvent?.recordType != WorkClockRecordType.RECORD }
```

이 필터는 STOP=RECORD(정상 퇴근 타각)만 제외한다.  
`STOP=null` → `null?.recordType = null` → `null != RECORD = true` → **통과**

근무 수정으로 확정해도 원본 event pack의 STOP 이벤트는 추가되지 않으므로, 확정된 날짜임에도 STOP=null 팩이 workSchedule과 함께 이중으로 노출된다.

> **START.recordType으로 구분 불가**: 실시간 진행 중 근무와 근무 수정으로 확정된 케이스 모두 START.record_type = RECORD (오늘 v2_user_work_clock_event START 이벤트 40,150건 전부 RECORD 확인).

### 버그 2 — REST_STOP이 이전 블록을 닫지 않음

```kotlin
// UserWorkScheduleWithWorkClockMappingService.kt:207~213
if (event.event.eventType in listOf(STOP, REST_STOP, CANCEL)) {
    return@fold acc  // 이전 블록의 blockTo를 업데이트하지 않고 반환
}
```

REST_STOP 이후 더 이상 이벤트가 없으면 REST 블록의 blockTo가 null인 채로 남는다.

### 케이스별 확인 데이터

**CASE 2** — `2025-07-04` (traceId: `23427c61046dbd6d7f97a5a1c1b6bd68`)

| connected-event-packs → | REST [12:00, **null**] | 잔재 ❌ |

|------|------|------|

**CASE 1** — `2025-04-02` 데이터 미확인

이메일에 기재된 URL(`dates/2025-04-02/2025-04-03`)은 4월 전체에서 성공 응답 0건(모두 401). 실제 성공 요청은 월 단위 조회(`2025-04-01/2025-04-30`, traceId: `146bf66925e6408d6c826b88d62e3a0d`, 2026-04-02 16:03 KST)로, 해당 날짜 event pack은 STOP=RECORD라 중복 없음. 고객이 문의한 근무 수정 후 상태의 응답 데이터는 확인하지 못했으나 CASE 2/3와 동일 메커니즘이 적용됐을 것으로 추정.

## 수정 방향

**파일**: `flex-openapi-backend/openapi/api/src/main/kotlin/team/flex/openapi/timetracking/schedule/UserWorkScheduleWithWorkClockMappingService.kt`

**Option B (스펙 의도에 부합)** — 날짜별 workSchedule 존재 시 해당 날짜의 workClock 블록 스킵

스레드의 의도: "workSchedule(확정) 없으면 실시간 팩 노출, 있으면 확정 데이터만 노출"

- 실시간 진행 중(오늘): workSchedule 미존재 → workClock 유지 ✅
- 과거 확정 날짜 (CASE 2/3): workSchedule 존재 → workClock 스킵 ✅  
- blockTo=null 잔재도 자연히 제거됨 ✅
- 엣지케이스: 교대근무 등 당일 workSchedule이 미리 생성되는 경우 → 확인 필요

**Option C** — 합산 후 deduplication (workSchedule 우선, blockFrom 기준 중복 제거)
- blockTo=null 잔재 블록도 함께 정리 가능
- workSchedule 없는 날짜는 영향 없음 (가장 방어적)
- Option B보다 범위가 좁음 — 동일 blockFrom이 아닌 중복은 잡지 못함
