## 해결

Operation API를 사용하여 승인 정책 승인권자를 일괄 교체했다.[^6]

### 수행 내역

- **실행 시각**: 2026-04-09 19:58:19 KST[^6]
- **API**: `POST /action/operation/v2/approval/policy/customers/55747/replace-user-to-user`
- **대상**: 39개 정책 (APPROVAL_DOCUMENT 6건 + TIME_OFF 33건)[^7]
- **파라미터**: `beforeUserId: 808508`, `afterUserId: 897921`
- **검증**: API 실행 후 동일 쿼리로 재조회 → **0건** 확인[^8]
- **고객 확인**: 안상연님 "감사합니다!!" (2026-04-09 19:59)[^9]

### Operation API 상세

```
POST /action/operation/v2/approval/policy/customers/{customerId}/replace-user-to-user
```

- **승인 정책** 내 유저 1:1 교체 전용 API[^3]
- **Controller**: `ApprovalPolicyOperationController.kt` — **Service**: `ApprovalPolicyOperationUseCase.replaceAllUsers()`[^10]
- 로직: 각 정책의 기본 Step + 조건부 Step(요승설[^14]) + 참조자(referrers)에서 beforeUserId → afterUserId 교체 후 Kafka 이벤트 발행[^10]
- 롤백: 역방향(afterUserId → beforeUserId)으로 다시 호출[^11]
- 원래 SK케미칼 요청 처리용으로 만든 API (권민철님 확인)[^12]
- ⚠️ **승인 요청(진행 중 문서)은 이 API 대상이 아님** — 진행 중 문서는 "퇴사자 승인 변경" 별도 기능 사용 필요[^11]

#### Request Body 구조

```json
{
  "targetIdentities": [
    { "category": "<ApprovalPolicyCategory>", "key": "<정책 대상 키>" }
  ],
  "beforeUserId": 808508,
  "afterUserId": 897921
}
```

- `targetIdentities`: 교체 대상 정책 목록 — `approval_policy` 테이블의 `target_category`, `target_key` 에 매핑[^10]
- **ApprovalPolicyCategory enum**: `TEMPLATE`, `APPROVAL_DOCUMENT`, `WORK_RECORD`, `TIME_OFF`, `TIME_OFF_PROMOTION`, `USER_DATA`, `OBJECTIVE`, `EVALUATION`

#### targetIdentities 조회 쿼리

```sql
SELECT DISTINCT ap.target_category, ap.target_key
FROM flex_approval.approval_policy ap
JOIN flex_approval.approval_step astp ON astp.approval_policy_id = ap.id
JOIN flex_approval.approval_step_actor asa ON asa.approval_step_id = astp.id
WHERE ap.customer_id = ?
  AND asa.resolve_target_type = 'USER'
  AND asa.resolve_target_value = ?
  AND ap.deleted_at = '9090-09-09 09:09:09'
  AND astp.deleted_at = '9090-09-09 09:09:09';
```

## 다음에 같은 문의가 오면

1. **먼저 확인**: `view_user` 에서 기존/변경 대상 이메일로 user_id 조회 → 위 쿼리로 대상 정책 수 확인
2. **원인 판별**: 퇴사인지 단순 담당자 변경인지 확인
   - 퇴사 + 진행 중 문서 있음 → "퇴사자 승인 변경" 기능 별도 사용 필요 (이 API로는 승인 요청 교체 불가)
   - 단순 담당자 변경 또는 퇴사(승인 정책만) → Operation API 사용
3. **조치**: Operation API 호출
   - `POST /action/operation/v2/approval/policy/customers/{customerId}/replace-user-to-user`
   - DB 쿼리로 `targetIdentities` 조회 → request body 조립 → 실행
   - 실행 후 동일 쿼리로 0건 확인

## 비고

- `approval_step_actor` 에는 `resolve_target_type = 'USER'` 외에 `RELATIVE_DEPT_HEAD` 등 다른 타입도 존재. 이번 요청은 직접 지정(USER)된 승인권자만 변경하면 됨.
- 진행 중인 문서(`approval_line_actor`)의 승인권자는 이번 변경 대상에 포함되지 않음.
- **유사 이슈 패턴**: 승인권자 일괄 변경은 퇴사/계정 변경에 따라 반복적으로 인입됨 (CI-1602, CI-913, CI-4228, CI-2367 등). 이번 건은 퇴사가 아닌 단순 담당자 변경.
