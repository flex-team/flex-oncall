> **상태**: 조사 완료 — 2026-04-08

## 증상
- **문제 정의**: 워크플로우 문서에서 본인의 승인 차례임에도 승인 버튼이 보이지 않는다
- **회사**: 바이엘코리아 (Customer ID: 199890)
- **요청자**: youjung.choi.ext@bayer.com[^1]
- **대상자**: 최유정 (youjung.choi.ext@bayer.com, hashedId: 248NmJoBEe)[^1][^4]
- **영향 범위**: 해당 문서 1건 (문서번호: 2026-832) + CI-4362에서 확인된 시스템 전체 78건/일 sync failure[^12]
- **문제 시점**: 2026-04-08
- 문의 내용:
  1. 워크플로우 문서에서 본인 승인 차례가 되어 승인하려 하나 승인 버튼이 보이지 않음[^1]
  2. 1단계 승인권자가 승인한 것으로 상태 표시되나 활동 내역에 승인 내역이 보이지 않음[^2]

## 원인 분석

**1단계 승인 처리(act API) 시 타임아웃이 발생하여, impact-api에서 이벤트 동기화가 실패 → 최유정의 UserBox 미생성 → `myBoxInfo = null`**

### 가설 목록

| # | 가설 | 확인 방법 | 상태 |
|---|------|----------|------|
| 1 | Step 0 승인 이벤트 처리 실패 → UserBox 미갱신 → myBoxInfo null | `workflow_task.last_modified_date` 확인 | ✅ 확정 |
| 2 | 이벤트 미처리의 원인이 act API 타임아웃 | 임형태님 로그 분석으로 확인 | ✅ 확정[^13] |

### 조사 과정

> 💡 **판단 근거**: FE가 myBoxInfo null 확인[^4]
> → `myBoxInfo`는 `document.userBoxes.find { it.user.sequence == requester.userId }`로 생성 — UserBox가 없으면 null[^7]
> → UserBox는 `user_workflow_task_box` DB 테이블에 저장됨[^8]
> → UserBox 갱신은 `ApprovalProcessUpdatedEvent` 수신 → `repository.save()` → `patchAllUserBoxes()` 체인[^9]
> → **`workflow_task.last_modified_date = 2026-04-07 12:35:49`** — 문서 생성 시점과 동일, 이후 한 번도 갱신 안 됨[^10]
> → 이벤트가 impact-api에서 처리되지 않았음이 확인됨
> → 임형태님 추가 조사: act API에서 `FlexRemoteUnknownStateException: java.io.InterruptedIOException: timeout` 발생 확인[^13]
> → 48시간 내 동일 타임아웃 14건 발생[^14]

<details>
<summary>📋 myBoxInfo 생성 코드 분석</summary>

`flex-impact-backend` > `approval-document/api/.../ApprovalDocumentResponse.kt:16-38`:

```kotlin
data class ApprovalDocumentResponse(
    val myBoxInfo: MyBoxInfoDto?,  // nullable
) {
    constructor(message: ApprovalDocumentMessage, ...) : this(
        myBoxInfo = message.document.userBoxes
            .find { it.user.sequence == message.occurredBy.sequence }?.let {
                MyBoxInfoDto(it, message.document)
            },
    )
}
```

`userBoxes`에서 현재 사용자의 `sequence`와 일치하는 UserBox를 찾지 못하면 `null` 반환.

UserBox는 `user_workflow_task_box` DB 테이블에 저장되며, `ApprovalProcessUpdatedEventHandler` → `repository.save()` → `patchAllUserBoxes()` → `accessibleApprovers()` → `saveUserBoxesOf()` 체인으로 갱신됨.

`accessibleApprovers()` (`ApprovalDocument.kt:374-389`):
```kotlin
private fun accessibleApprovers() = approvalProcess.lines
    .filterNot { it.status === ApprovalLineStatus.PENDING }
    .flatMap { l ->
        l.actors.flatMap { a ->
            a.resolvedTarget.userIds.map {
                ApprovalDocumentUserBox.ofApprover(
                    user = ApprovalDocumentUser.of(userSequence = it, ...),
                    approvalRequired = l.status === ApprovalLineStatus.ONGOING &&
                        a.status === ApprovalLineActorStatus.PENDING,
                )
            }
        }
    }
```

PENDING이 아닌 라인(ONGOING, APPROVED)의 actor들에 대해 UserBox를 생성. step 1이 ONGOING이 되어야 최유정의 UserBox가 생성됨.

</details>

### 원인

- **표면 원인**: 최유정의 `user_workflow_task_box` 레코드가 존재하지 않아 API에서 `myBoxInfo = null` 반환[^7][^10]
- **근본 원인**: 이선영의 step 0 승인 act API 호출 시 **타임아웃** 발생(`FlexRemoteUnknownStateException: java.io.InterruptedIOException: timeout`)[^13]. approval-api에서는 승인이 커밋되었으나, impact-api로의 후속 동기화(이벤트 처리)가 타임아웃으로 실패 → `patchAllUserBoxes()` 미실행 → 최유정의 UserBox 미생성[^9][^10]
- 48시간 내 동일 타임아웃 14건 발생 — 구조적 문제[^14]
- CI-4362에서 동일 서비스의 sync failure가 78건/일로 발생하는 패턴과 일치[^12]

### 스펙 vs 버그

**판정: 버그 (이벤트 동기화 실패)**
- approval-api: step 0 APPROVED, step 1 ONGOING으로 정상 처리됨[^4]
- impact-api: 이벤트 미처리로 UserBox 미갱신 — 동기화 버그[^10]

## 영향 범위
- 이 문서 1건에 국한되지 않음. CI-4362 조사에서 **78건/일 sync failure** 확인[^12]
- impact-api의 `ApprovalProcessUpdatedEvent` 처리 실패는 구조적 문제로, 모든 고객사에서 발생 가능

## 현재 상태 확인

- `workflow_task.status = IN_PROGRESS`, `last_modified_date` 생성 이후 미갱신[^10]
- approval-api의 승인 프로세스 데이터는 정상 (step 0 APPROVED, step 1 ONGOING)[^4]
- 비정상 전이 아닌 이벤트 동기화 실패

## 차단 범위

- 최유정이 해당 문서(2026-832)를 승인할 수 없어 후속 프로세스 차단
- 우회: impact-api에 수동으로 이벤트 재처리를 트리거하면 해소 가능 [추론: CI-4182/CI-4019에서 sync-with-approval로 해결한 패턴 참고]

## 발견한 스펙/제약

- `myBoxInfo`는 nullable 필드. `userBoxes.find { it.user.sequence == requester.userId }`로 결정되며, UserBox가 없으면 null[^7]
- UserBox 갱신은 approval 이벤트 수신에 의존하는 비동기 구조 — 이벤트 유실 시 UI에 즉시 영향[^9]
- `accessibleApprovers()`는 PENDING이 아닌 라인의 actor만 UserBox로 생성 — step이 ONGOING으로 전환되어야 해당 step의 승인자 UserBox가 생성됨[^11]

## 다음에 같은 문의가 오면

1. **먼저 확인**: FE에서 `myBoxInfo`가 null인지 확인 (API 응답 `GET /api/v3/approval-document/approval-documents/{documentKey}`)
2. **원인 판별**:
   - `myBoxInfo = null` → `workflow_task.last_modified_date`가 생성 이후 갱신되었는지 DB 확인
   - 미갱신이면 → 승인 이벤트 동기화 실패 (이 이슈와 동일)
   - 갱신되었는데 null → UserBox 생성 로직 자체의 버그 (다른 원인)
3. **조치**: `sync-with-approval` API로 해당 문서의 상태 동기화 수동 트리거
