## 원인 분석

**버그** (확정) — 도메인 담당자(이윤선) 확인[^3]. 원인이 2가지 존재:

1. **FE가 `period.start`를 ~3-4초 미래 시간으로 전송** → BE가 `now < period.start` 판정 → `RESERVED`[^9]
2. **FE가 "지금 시작" 직후 동일 API를 empty body(`{}`)로 재호출**[^12] → period가 null로 리셋될 가능성

이성환: "서버에서는 아무것도 안보냈는데 받아주는 게 맞는지를 판단해서 픽스 필요"[^13]

### 가설 목록

| # | 가설 | 확인 방법 | 상태 |
|---|------|----------|------|
| 1 | FE가 `period.start`를 미래 시간으로 전송하여 BE가 RESERVED로 판정 | access log에서 requestBody의 period.start와 서버 시각 비교 | ✅ 확정[^9] |
| 2 | FE가 "지금 시작" 후 empty body로 period API 재호출 | access log 패턴 분석 | ✅ 확정[^12] |
| 3 | BE 상태 판정 로직 자체의 버그 | 코드 리뷰 | ❌ 소거 — 로직은 정확, 입력값이 문제[^6] |

### 핵심 증거

**1. 상태 결정 로직** — `now < period.start`이면 `RESERVED`[^6]:

<details>
<summary>📋 EvaluationStepCommandService.kt 상태 결정 코드</summary>

```kotlin
// EvaluationStepCommandService.kt:126-132
this.status = when {
    period == null -> EvaluationStepProgress.BEFORE_PROGRESS
    now < period.start -> EvaluationStepProgress.RESERVED      // ← 이 분기에 진입
    now in period -> EvaluationStepProgress.IN_PROGRESS
    now > period.endInclusive -> EvaluationStepProgress.FINISHED
    else -> this.status
}
```

`now`는 `clock.instant()`로 서버 시간을 사용. FE가 보낸 `period.start`가 `now`보다 미래이면 무조건 `RESERVED`.

</details>

**3. 자동 복구 메커니즘** — 크론 스케줄러가 매분 `RESERVED` → `IN_PROGRESS` 전환[^7]

### 5 Whys

1. 왜 "예약" 상태로 저장되었나? → `now < period.start` 조건에 진입[^6]
2. 왜 `now < period.start`인가? → FE가 보낸 `period.start`가 서버 시간보다 ~3초 미래[^9]
3. 왜 FE가 미래 시간을 보내는가? → (추정) 클라이언트 시계가 서버보다 3-4초 빠름, 또는 FE가 오프셋 추가
4. 왜 BE가 이 차이를 보정하지 않는가? → strict 비교, tolerance 없음[^6]
5. 왜 이후 자동 복구되는가? → 크론 스케줄러가 매분 전환[^7]

### 스펙 vs 버그 판별

> 💡 **판단 근거**: 도메인 담당자(이윤선) 확인[^3] → BE 로직은 올바르나 FE-BE 시간차 tolerance 부재[^6] → UX 결함 → **버그** (확정)

## 해결안/조사 방향

- **즉시 대응**: 크론 스케줄러가 자동 복구 완료.

### 근본 해결 (재발 방지)

이성환 의견: FE가 왜 이렇게 보내는지 파악 + BE가 empty body를 받아주는 게 맞는지 판단 필요[^13]

**FE팀 분석 결과 (이지훈, 김보라):**
- empty body(`{}`)는 "예약 취소" 버튼 클릭 시에만 전송됨 — "지금 시작" 후 자동 재호출이 아님[^14]
- 클라이언트 시간 기준으로 "예약"과 "지금 시작"을 판별하는 것은 무리[^15]
- 현재 API는 "예약"을 위해 `period.start`를 받는 설계 — "지금 시작"과 "예약"을 구분하는 API 변경이 필요[^16][^17]

**합의된 수정 방향: API 변경** (담당: 이성환, 기한: 2026-03-25)
- "예약"과 "지금 시작"을 구분할 수 있도록 API를 변경한다[^17]
- BE tolerance 적용(옵션 A)만으로는 근본 해결이 어려움 — 클라이언트 시간 의존 구조 자체가 문제
- 이성환: 근본원인을 제거하는 방향으로 진행 예정[^18]
- 이지훈: tolerance 방안(몇 초 빼서 대응)도 옵션으로 제안[^19]
- 이성환 임시 수정안: FE에서 "지금 시작" 시 당일 0시를 보내고, 다음 배포에서 API 분리[^20]
- 온도감: 낮음 — 내일 또는 차주 배포 가능[^21]

<details>
<summary>초기 검토 옵션 (참고용)</summary>

**옵션 A: BE tolerance 적용** — `period.start`가 `now`와 5~10초 이내면 `IN_PROGRESS` 판정. FE 수정 없이 가능.

**옵션 B: FE에서 서버 시간 기준 `period.start` 생성** — 서버 시간 조회 API 필요. 근본적 해결.

**옵션 C: "지금 시작" 전용 API 분리** — `POST .../start-now`로 서버에서 `period.start = now` 설정. 의도 명확. ← 현재 합의 방향과 유사

</details>
