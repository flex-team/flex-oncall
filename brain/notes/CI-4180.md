### 5 Whys

1. **왜 500 에러?** → `NoSuchElementException` 미처리[^8]
2. **왜 예외 발생?** → `userWorkRules.first { userId == 529380 }` — 해당 유저의 매핑 없음[^8]
3. **왜 매핑이 없는가?** → 마지막 이벤트가 CANCEL이고, 남은 267146은 `active=0`[^7][^9]
4. **왜 비활성 근무유형에 매핑이 남아있나?** → CI-4148 취소 버그로 267146 REGISTER 취소 불가 → PR #12027 배포 후 수동 복구 시 43명 중 4명 누락 → 267146 비활성화[^4]
5. **(근본 원인)** 두 가지:
   - **데이터**: CI-4148 취소 버그 후 수동 복구에서 4명 누락 (운영 실수)
   - **코드**: `validateBulk()`이 비활성 근무유형 매핑을 방어하지 않음 (`.first{}` → 500 에러)[^8]

### 코드 수정 옵션

**옵션 A (권장, 즉시)**: `.first{}` → `.firstOrNull{}` + 핸들링 — validate/register에서 비활성 근무유형 매핑 케이스를 처리
- 장점: 에러 방지, 비교적 간단한 수정
- 단점: 비활성 매핑 데이터 자체는 남아있음

**옵션 B (근본)**: 근무유형 비활성화 시 매핑 정리 — 비활성화 API에서 해당 근무유형에 매핑된 유저의 이벤트를 자동 CANCEL
- 장점: 근본 원인 해결, 데이터 정합성 보장
- 단점: 영향 범위가 넓어 신중한 설계 필요

담당자(김영준)가 별도 버그 수정 계획 중[^19]. `validateBulk()` / `registerBulk()` 의 `.first {}` → `.firstOrNull {}` 변경 + 비즈니스 예외 반환 (400)

## 발견한 스펙/제약
- 근무유형 삭제 스펙: 유저 맵핑이 하나라도 있으면 삭제 불가[^18]. 단, 맵핑이 없는 상태에서 삭제 후 유저 맵핑을 취소하면 삭제된 근무유형이 복원되는 엣지 케이스가 존재한다[^17].
- `getActiveByCustomerAndIdentities()` 는 `active=true`인 근무유형만 반환 — 유저 매핑 해석 시 비활성 근무유형 조회 불가
- LookUp 알고리즘은 REGISTER/CANCEL 이벤트만으로 유효 매핑을 결정하며, 근무유형의 active 상태는 고려하지 않음

## 다음에 같은 문의가 오면

1. **500 에러 + "근무유형이 없는 상태" 확인**: `v2_user_work_rule`에서 해당 유저의 마지막 이벤트가 `CANCEL`인지 확인
2. **CANCEL이면**: 유효한 REGISTER가 없는 상태 → 데이터 복구 필요
3. **복구**: 비활성(`active=0`) 근무유형의 CANCEL 이벤트를 INSERT하여 유효하지 않은 매핑 정리[^16]
4. **이러한 유저가 더 있을 수 있으므로**, 동일 고객사 전수 조사 → 529380 패턴(마지막 이벤트 CANCEL + 비활성 근무유형 잔존) 확인[^13]
