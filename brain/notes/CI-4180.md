## 원인 분석

**버그 확정**: `UserWorkRuleBulkUpdateService.validateBulk()`에서 현재 매핑된 근무유형이 없는 유저에 대해 `.first {}` 호출 시 `NoSuchElementException` 발생. 예외 처리가 없어 500 에러로 전파된다.[^8]

**핵심 증거:**
1. DB: 해당 유저의 마지막 이벤트가 `CANCEL`이라 3/23 이후 유효한 매핑 없음[^7]
2. 코드: `userWorkRules.first { userId == ... }` — 빈 결과에서 `.first {}` 호출 시 무조건 예외[^8]
3. 267146(10시-7시_20260317)은 `active=0`(비활성화)이라 `getActiveByCustomerAndIdentities`에서도 조회 불가[^9]

### 5 Whys

1. **왜 500 에러?** → `NoSuchElementException` 미처리[^8]
2. **왜 예외 발생?** → `userWorkRules.first { userId == 529380 }` — 해당 유저의 매핑 없음[^8]
3. **왜 매핑이 없는가?** → 마지막 이벤트가 CANCEL이고, 남은 267146은 `active=0`[^7][^9]
4. **왜 CANCEL 되었는가?** → CI-4148 이슈 대응 중 예약 변경 과정에서 발생한 것으로 추정[^4]
5. **(근본 원인)** `validateBulk`에서 유저의 현재 매핑이 없는 케이스에 대한 방어 처리 부재[^8]

### 스펙 vs 버그 판별

**버그** ✅ — 2건 복합

| 구분 | 내용 |
|------|------|
| **버그 1 (표면)** | `validateBulk`/`registerBulk`에서 `.first {}` → `NoSuchElementException` 미처리. `.firstOrNull {}`로 변경 후 비즈니스 예외 반환 필요 |
| **버그 2 (근본, 추정)** | 예약 취소 후 유효 매핑이 사라지는 데이터 상태 자체가 비정상. CI-4148 버그의 후속 영향 |

### 버그 재현 시나리오 (김영준 분석)[^17]

> 해당 버그를 만든 유저 액션 타임라인:

1. 2026-03-19 20:55:03 — **근무 유형**(CustomerWorkRule) 267147 삭제[^17]
2. 2026-03-19 20:55:25 — **유저 근무 유형**(UserWorkRule) 1466400 취소[^17]

> 근무 유형 삭제 스펙: 유저 맵핑이 하나라도 있으면 삭제 불가[^18]. 그런데 이 케이스는 맵핑이 없는 상태에서 근무유형을 삭제한 후, 유저 맵핑을 취소하면서 `active=true`인 (이미 삭제된) 근무유형이 남아있게 되는 상황[^17].

### 코드 수정 (예정)
- `validateBulk()` / `registerBulk()` 의 `.first {}` → `.firstOrNull {}` 변경 + 비즈니스 예외 반환 (400)
- CI-4148 수정 PR(#12027)[^12]과 별도 PR로 진행 가능
- 담당자(김영준)가 별도 버그 수정 계획 중[^19]

## 발견한 스펙/제약
- 근무유형 삭제 스펙: 유저 맵핑이 하나라도 있으면 삭제 불가[^18]. 단, 맵핑이 없는 상태에서 삭제 후 유저 맵핑을 취소하면 삭제된 근무유형이 복원되는 엣지 케이스가 존재한다[^17].

## 다음에 같은 문의가 오면

1. **500 에러 + "근무유형이 없는 상태" 확인**: `v2_user_work_rule`에서 해당 유저의 마지막 이벤트가 `CANCEL`인지 확인
2. **CANCEL이면**: 유효한 REGISTER가 없는 상태 → 데이터 복구 필요
3. **복구**: 비활성(`active=0`) 근무유형의 CANCEL 이벤트를 INSERT하여 유효하지 않은 매핑 정리[^16]
4. **이러한 유저가 더 있을 수 있으므로**, 동일 고객사 전수 조사 → 529380 패턴(마지막 이벤트 CANCEL + 비활성 근무유형 잔존) 확인[^13]
