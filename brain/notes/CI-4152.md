# CI-4152: 잘못 발송·서명된 완료 계약서 삭제 가능 여부 확인

> **상태**: 조사 완료 / 관망 중 — 2026-03-18

## 증상
- **문제 정의**: 잘못 발송·서명 완료된 전자계약서[^11]의 삭제 가능 여부 확인
- **회사**: (주)카카오페이증권 (Customer ID[^12]: 35284)
- **요청자**: CS팀 dokyung.lee
- **대상자**: 특정 구성원 (이름 미특정)
- **영향 범위**: 1명 (해당 구성원)
- **문제 시점**: 2026-03-18
- 문의 내용:
  1. 고객사에서 계약서를 잘못 발송하여 구성원에게 연봉이 잘못 노출됨[^1]
  2. 해당 구성원이 이미 계약서에 서명을 완료한 상태[^1]
  3. 완료된 계약서를 삭제한 선례가 있는지, 삭제가 가능한지 확인 필요[^1]
  4. 매우 긴급한 상황으로 빠른 확인 요청[^1]

## 원인 분석

**SUCCEED(서명 완료) 상태 계약서를 삭제·취소하는 코드 경로는 존재하지 않는다.**

- 전자계약(Digicon) 도메인, 라벨도 `전자계약(Digicon)`[^1]
- 과거 전자계약 삭제 선례는 COOKBOOK.md, INDEX.md에 없음[^2]
- CI-4107은 양식 렌더링 이슈로 직접 연관 없음[^2]

### 조사 과정

> 💡 **판단 근거**:
> 1. `DigiconProgressStatus.cancelable()` = `this === IN_PROGRESS` — SUCCEED 상태에서는 취소 불가[^3]
> 2. `DigiconCandidateSetProgressStatus.deletable()` = `setOf(IN_REVIEW).contains(this)` — 세트 삭제도 IN_REVIEW만 가능[^4]
> 3. Operation API에 계약서 삭제/취소 엔드포인트 없음[^5] — `restore-deleted-templates`(템플릿 복원)과 `refresh-candidate-set-status`(상태 갱신)만 존재
> 4. "회수", "recall", "withdraw", "revoke" 기능 전무[^6]
> → 서명 완료 계약서를 코드 레벨에서 삭제/취소할 수 있는 경로가 없다

### 가설 목록

| # | 가설 | 확인 방법 | 상태 |
|---|------|----------|------|
| 1 | 기존 코드에 SUCCEED 계약서 삭제/취소 경로 존재 | 코드 전수 조사 (cancel, delete, 회수, recall, withdraw, revoke) | ❌ 소거 — 코드에 없음 |
| 2 | DB 직접 조작으로 삭제한 과거 선례 존재 | COOKBOOK, INDEX, 과거 노트 검색 | ❌ 소거 — 선례 없음 |
| 3 | DB 직접 조작으로 삭제가 기술적으로 가능 | 엔티티 관계 분석 | ✅ 확정 — 아래 상세 |

### 스펙 vs 버그 판별

**스펙이다.** 서명 완료된 계약서는 법적 효력이 있는 문서이므로, 삭제 불가는 의도된 설계다[^3][^4]. 과거 flex-digicon-backend PE(임준규)에 따르면, 이미 서명한 계약서를 삭제하는 것은 법적 효력이 없으며 재계약이 정석적인 해결법이다[^13].

<details>
<summary>📋 DB 직접 삭제 시 영향 범위 (엔티티 관계도)</summary>

SUCCEED 상태 계약서를 DB에서 직접 삭제할 경우, 아래 테이블을 계단식으로 처리해야 한다:

```
digicon (progress_status = 'SUCCEED')
├── digicon_party_history (digicon_id FK) — 서명 이력 [삭제 필요]
├── digicon_party (digicon_id FK) — 계약 당사자 [삭제 필요]
├── file_key — S3 저장 PDF [별도 삭제 필요 (추정)]
└── digicon_candidate_set (역참조) — 세트 상태 갱신 필요할 수 있음
    └── digicon_candidate_unit — 유닛 상태 갱신 필요할 수 있음
```

**삭제 순서** (FK 의존성):
1. `DELETE FROM digicon_party_history WHERE digicon_id = ?`
2. `DELETE FROM digicon_party WHERE digicon_id = ?`
3. `DELETE FROM digicon WHERE id = ?`
4. digicon_candidate_set/unit의 progress_status 재계산 필요 여부 확인
5. S3 파일 삭제 (file_key 기반)

**주요 테이블 구조:**

| 테이블 | 핵심 컬럼 | 비고 |
|--------|----------|------|
| `digicon` | id, customer_id, user_id, contractor_user_id, progress_status, file_key | 메인 계약 테이블[^7] |
| `digicon_party` | digicon_id, entity_id, entity_type, current_status | 계약 당사자[^8] |
| `digicon_party_history` | digicon_id, digicon_party_id, status, execution_place_ip_address, created_date_time | 서명 이력 (IP 주소 포함)[^9] |

**Soft delete 미지원**: digicon, digicon_party, digicon_party_history 테이블에는 `deleted_at` 컬럼이 없다. 템플릿(`customer_digicon_template`)과 placeholder만 soft delete 지원[^10].

## 해결안 / 조사 방향

### 옵션 1: DB 직접 삭제 (즉시 대응)

digicon 관련 3개 테이블에서 해당 계약 레코드를 직접 DELETE한다.

- **장점**: 즉시 처리 가능, 구성원에게 더 이상 노출되지 않음
- **단점**: 서명 이력(IP, 시점) 등 법적 증거가 완전히 삭제됨. 복구 불가. S3 파일도 별도 처리 필요
- **위험도**: 🔴 — 데이터 영구 삭제, 감사 추적 불가

### 옵션 2: 상태값만 변경 (SUCCEED → CANCELED)

digicon 테이블의 `progress_status`를 `CANCELED`로 UPDATE하고, digicon_party의 `current_status`도 `CONTRACTOR_CANCELED`로 변경한다.

- **장점**: 데이터 보존, 감사 추적 가능, 복구 용이
- **단점**: UI에서 "취소됨"으로 여전히 보일 수 있음 (구성원 계약 목록에서 보이는지 확인 필요)
- **위험도**: 🟡 — 데이터는 보존되나 UI 동작 확인 필요

### 옵션 3: 고객사에게 새 계약서 재발송 안내 (운영 조치)

잘못된 계약서는 유지하고, 올바른 내용으로 새 계약서를 발송하도록 안내한다.

- **장점**: 시스템 개입 없음, 리스크 0
- **단점**: 구성원이 잘못된 연봉을 이미 확인한 상태라 고객사 입장에서 불충분할 수 있음

### 권장

**옵션 3 (재계약 안내)**가 정석이다[^13]. DB 조작(옵션 1, 2)은 서명 완료된 계약서의 법적 효력에 영향을 주지 못하므로, 올바른 내용으로 새 계약서를 재발송하도록 고객사에 안내하는 것이 적절하다.
