# CI-4244: 워크플로우 문서 강제 승인 요청

## 증상
- **문제 정의**: 승인 라인에 경영지원팀(조직)이 설정된 워크플로우 문서에 대해, 기존 팀원 전원 퇴사로 승인 진행 불가 — 강제 승인 요청
- **회사**: 세잎프닥 (Customer ID[^1]: 161808)
- **요청자**: wldus7312@safedoc.kr (CS 인터콤 문의)[^2]
- **대상자**: junhs@safedoc.io (현재 경영지원팀 구성원, 입사 전 문서 승인 불가)
- **영향 범위**: 워크플로우 문서 6건 — 2026-67, 2026-66, 2026-46, 2026-43, 2026-23, 2026-10
- **문제 시점**: 2026-03-27
- 문의 내용:
  1. 결재라인이 [경영지원팀]으로 설정된 문서 존재
  2. 기존 경영지원팀 구성원 전원 퇴사하여 승인 불가
  3. 현재 경영지원팀 소속 junhs@safedoc.io는 입사 전 올라온 문서 승인 불가
  4. **조직이 승인라인에 있을 경우 UI의 "퇴직자 승인자 교체" 기능도 사용 불가**[^2]

## 원인 분석

CI-4228 케이스와 동일 패턴이나, **조직(팀)이 승인라인에 있다는 차이점** 존재.

### 가설 목록

| # | 가설 | 확인 방법 | 상태 |
|---|------|----------|------|
| 1 | 조직 승인라인의 퇴직자는 `approval_replacement_target`에 등록되지 않아 교체 기능 불가 | CI-4228 패턴 적용 — 동일 근본 원인으로 추정 | `✅ 확정` [추론: CI-4228 분석 결과 재적용] |
| 2 | `bulk-approve-for-user` API로 퇴직한 팀원 userId 기준 강제 승인 가능 | API 호출 + 결과 확인 | ⏳ 조사 중 (userId 특정 필요) |

### 조사 과정

> 💡 **판단 근거**: COOKBOOK F1 플로우 히트[^3] → CI-4228 아카이브 노트 확인[^4]
> → CI-4228과 동일: 퇴직자 교체 기능은 퇴사 이벤트 기반이며, **조직이 승인라인인 경우 해당 기능 UI에서 차단됨**[^2]
> → 강제 승인(Operation API) 접근으로 전환

**CI-4228과의 차이점:**
- CI-4228: 개인 계정 삭제(deleted) → userId 명확
- CI-4244: 조직(경영지원팀)이 승인라인 → **퇴직한 팀원의 userId 특정 필요**
- 대상이 워크플로우 문서(문서번호: 2026-XX) → 카테고리가 `TIME_OFF`/`WORK_RECORD`가 아닐 수 있음

## 해결안

### 즉시 대응 — Operation API 강제 승인 (CI-4228 F1 플로우 적용)

**전제 조건 (아직 미확인):**
1. 경영지원팀에서 퇴사한 구성원들의 **userId** 확인 필요
2. 워크플로우 문서의 **카테고리** 확인 (TIME_OFF/WORK_RECORD 외 다른 카테고리 가능성)

**처리 절차 (CI-4228 패턴):**

```
① 퇴직자 승인자 교체 API 시도 (실패 예상 — 조직 케이스는 퇴사 이벤트 미발행)
   POST /proxy/approval/api/operation/v2/approval/replacement-targets/migrate
   Body: { "customerIds": [161808], "resignationDateRange": { "start": "...", "endInclusive": "..." } }

② bulk-approve-for-user API로 강제 승인
   POST /proxy/approval/api/operation/v2/approval/process/customers/161808/users/{resignedUserId}/bulk-approve-for-user
   Body: { "categories": ["{해당 카테고리}"] }
```

**대안**: 문서번호 기반 워크플로우 Operation API 사용 (CI-3769 사례[^5])
- Notion 워크플로우 이슈 대응 런북 참조 필요[^6]
