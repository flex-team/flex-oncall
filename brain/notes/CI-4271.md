## 현재까지 파악된 내용

### 겸직 유저 매핑 조회

> 💡 **조회 접근법**: `view_user.primary_user_id` 로 역추적하면 겸직 등록 방식에 따라 누락이 발생할 수 있다[^2].
> → `member_user_mapping` 테이블을 통해 동일 `member_id` 로 묶인 유저를 조회하는 것이 정확하다[^3].

**정확한 조회 쿼리:**
```sql
SELECT customer_id, id AS user_id, email
FROM view_user
WHERE id IN (
    SELECT user_id
    FROM member_user_mapping
    WHERE member_id = (
        SELECT member_id
        FROM member_user_mapping
        WHERE user_id = (
            SELECT id FROM view_user WHERE email = trim('{대상 이메일}')
        )
    )
);
```

**`member_user_mapping` 기반 개별 조회 결과** (최종 확정):

> ⚠️ IN 절로 3명 한꺼번에 조회하면 member_id가 뒤섞여 잘못된 매핑이 나온다[^4]. 반드시 1명씩 개별 조회할 것.

| 이메일 | 이스트소프트(10682) user_id | 이스트시큐리티(28621) user_id | 비고 |
|---|---|---|---|
| leezho@estsoft.com | 848113 | 848804 | ✅ 겸직 매핑 확인[^5] |
| hmjeong@estsoft.com | 802049 | 805412 | ✅ 겸직 매핑 확인[^5] |
| jina@estsoft.com | 117753 | **없음** | ⚠️ 이스트시큐리티에 겸직 미등록[^6] |

> ⚠️ jina@estsoft.com(117753)은 이스트시큐리티(28621) 매핑이 존재하지 않는다[^6]. 겸직 등록 자체가 되어 있지 않으므로 주법인 변경 불가. 고객에게 겸직 등록 선행 필요를 안내해야 한다.

### 처리 방법

Operation API로 주법인 변경 처리[^7]:

```
PUT /action/v2/operation/core/users/primaryUser
```

**요청 파라미터:**
```json
{
  "changes": [
    { "sourceUserId": "{현재 주법인 user_id}", "targetUserId": "{새 주법인이 될 user_id}" }
  ]
}
```

**전환 조건:**[^7]
1. 양쪽 유저(source, target) 모두 퇴직 예정이 아니어야 함
2. 양쪽 유저 모두 온보딩 완료 상태여야 함

**변경 후 영향:**[^7]
- 이메일/인증이 새 주법인(이스트시큐리티) 기준으로 변경됨 → 고객 안내 필요

## 발견한 스펙/제약

- **겸직 유저 조회 시 `member_user_mapping` 사용 필수**: `view_user.primary_user_id` 만으로는 겸직 매핑을 완전하게 찾을 수 없다. `member_user_mapping.member_id` 로 동일 멤버에 묶인 모든 유저를 조회해야 정확하다[^3][^6].
- **Operation API 내부 처리**: `PrimaryUserAppointmentServiceImpl` 에서 email swapping + `primaryUserId` 갱신 + `member_user_mapping.isPrimary` 플래그 변경을 한 트랜잭션으로 수행[^8]
