# CI-4236: 워크플로우 다운로드 완료 안내 메일 수백통 중복 발신

## 증상
- **문제 정의**: 워크플로우 문서 일괄 다운로드 완료 후, 다운로드 완료 안내 메일이 분단위로 수백통 중복 발신됨
- **회사**: 번개장터 (Customer ID: 63328)
- **요청자**: earnest.nam@bunjang.co.kr[^1]
- **대상자**: 동일 (문의자 = 다운로드 수행자로 추정)[^1]
- **영향 범위**: 1명 (해당 다운로드 건)[^1]
- **문제 시점**: 2026-03-27 (오늘)[^1]
- 문의 내용:
  1. 워크플로우 문서 일괄 다운로드를 수행함[^1]
  2. 압축파일 안의 파일은 약 1만건이며, 1건의 압축파일로 정상 수신됨[^1]
  3. 그러나 다운로드 완료 안내 메일이 분단위로 계속 수신 중이며, 현재 수백통을 받은 상태[^1]
  4. 다운로드 자체는 완료되었으나 메일 알림이 멈추지 않음[^1]

## 원인 분석

**Kafka consumer 리밸런스로 인한 file merge 메시지 무한 재처리 → 매 시도마다 다운로드 완료 알림 발송**[^2]

### 가설 목록

| # | 가설 | 확인 방법 | 상태 |
|---|------|----------|------|
| 1 | 1만건 파일 merge가 Kafka consumer poll timeout(`max.poll.interval.ms`[^10]) 초과 → 리밸런스 → offset 미커밋 → 무한 재처리 → 매번 알림 발송 | Slack 실시간 모니터링 (윤성복) + topic lag 확인 | ✅ 확정 |

### 조사 과정

> 💡 **판단 근거**: Slack 스레드에서 윤성복님이 실시간으로 확인한 증거[^2] → 코드 흐름 확인[^3][^4][^5] → 메커니즘 확정

1. Slack에서 "file storage merge요청이 무한재시도되고있는것같네요"[^2] → 알림 타입 `FLEX_FILE_STORAGE_DOWNLOAD_PREPARATION_COMPLETED` 확인[^6]
2. "해당 토픽 lag이 계속늘고있습니다"[^2] → Kafka consumer 문제 확인
3. "너무 큰 요청이라 오래걸려서 컨수머에서 제외되고 커밋안되고 다른곳에서 재시도하고를 반복"[^2] → 원인 확정

### 근본 원인 (5 Whys)

1. 왜 메일이 수백통 발송되었나? → file merge 완료 시 발송되는 알림이 반복 트리거됨
2. 왜 반복 트리거되었나? → Kafka `command.flex.file-storage.merge.v1` 메시지가 무한 재처리됨[^3]
3. 왜 무한 재처리되었나? → consumer가 `max.poll.interval.ms`를 초과하여 그룹에서 제외, offset 미커밋[^2]
4. 왜 처리 시간이 초과했나? → 1만건 파일을 zip으로 병합하는 작업이 Kafka consumer의 poll 간격을 초과[^2]
5. 왜 대량 요청을 방어하지 못했나? → file merge 요청에 파일 수 제한이나 처리 시간 기반 방어 로직이 부재 [추론: 코드에 제한 로직 미발견]

### 스펙 vs 버그

**버그** — 대량 파일 merge 시 Kafka consumer 리밸런스로 인한 무한 재처리는 의도된 동작이 아님. consumer의 처리 시간 제한을 고려한 방어 로직이 필요.

## 해결

### 실제 근본 원인 (임형태 확인)

`render` → `impact` → `file` 순서로 요청이 전달되는 구조에서, 2단계(impact→file) 처리 시간(~7초)이 1단계(render→impact) 타임아웃을 초과. render가 작업 미완료로 인식하여 **1분 주기로 재시도** → file merge 요청이 827건으로 폭증[^11]

- `flex_flow.render_job` 에서 번개장터 요청 확인: 19,032건 + 12,781건 문서 다운로드[^11]
- 03/27 오전 10시에 impact→file merge 처리가 정상 완료되어 재시도 종료[^11]
- 근본 해결: renderer 타임아웃 개선 → [EPBE-230](https://linear.app/flexteam/issue/EPBE-230) (임형태 → Enterprise Product BE)[^12]

### 대응 경과

1. **윤성복**: `max.poll.interval.ms` 10분으로 증가 → consumer 안정화[^2]
2. **구경일**: file-storage consumer CPU 2배 증설[^13]
3. **이성환**: TODO 상태 잔여 351건 → DONE으로 UPDATE → consumer skip 유도[^14]

## 해결안 / 조사 방향 (초기 분석)

### 즉시 대응 (표면 원인 해결)

1. **Kafka consumer group의 해당 메시지 offset 수동 커밋** — 무한 재처리 중단
   - file-storage 팀/담당자가 수행
   - 장점: 즉시 반복 발송 중단
   - 단점: 근본 해결 아님

2. **topic lag이 계속 증가 중이라면 consumer 일시 중지 후 offset skip**
   - 긴급 대응 시 사용

### 근본 해결 (근본 원인 해결)

1. **file merge 요청에 파일 수 제한 추가**
   - 예: 최대 N건 초과 시 분할 처리 또는 거부
   - 장점: 대량 요청 자체를 방어
   - 단점: 사용자 경험에 영향 (큰 요청 불가)

2. **file-storage consumer의 `max.poll.interval.ms` 조정**
   - 대량 파일 처리 시간을 수용할 수 있는 값으로 증가
   - 장점: 간단한 설정 변경
   - 단점: 다른 문제를 마스킹할 수 있음 (근본적이지 않음)

3. **merge 처리를 Kafka consumer와 분리 (비동기 처리)**
   - consumer는 merge 요청만 접수하고 즉시 commit → 실제 merge는 별도 스레드/워커에서 처리
   - 장점: consumer poll timeout과 처리 시간이 독립적
   - 단점: 구현 복잡도 증가

4. **idempotency 추가**
   - merge request ID 기반으로 이미 처리된 요청은 skip
   - 장점: 재처리 시에도 중복 알림 방지
   - 단점: 추가 상태 관리 필요

### 추천

즉시 대응으로 **1번 (offset 수동 커밋)** 수행 후, 근본 해결로 **3번 (비동기 분리)** 또는 **4번 (idempotency)** 적용을 권장.

## 다음에 같은 문의가 오면

1. **Kafka consumer group 상태 확인**: prod Kafka UI에서 `prod-flex-file-storage-api-file-storage-merge-command-handler` 그룹의 state/lag 확인
2. **DB에서 동일 요청 반복 확인**: `SELECT customer_id, status, merged_file_name, COUNT(*) FROM flex.v2_file_merge WHERE customer_id = ? GROUP BY 1,2,3` — 동일 파일명 수백건이면 render 타임아웃 재시도
3. **즉시 대응**: `max.poll.interval.ms` 증가 + CPU 증설 + TODO→DONE UPDATE
4. **근본 원인은 render→impact 타임아웃** — EPBE-230이 해결되면 재발 방지

## 발견한 스펙/제약

- 알림 타입 `FLEX_FILE_STORAGE_DOWNLOAD_PREPARATION_COMPLETED`는 여러 백엔드(yearend, payroll, fins, timetracking)에서 공통으로 사용하는 파일 다운로드 완료 알림[^4]
- file merge 명령은 `command.flex.file-storage.merge.v1` Kafka 토픽을 통해 file-storage 서비스로 전달됨[^3]
- file-storage 서비스는 이 oncall repo의 서브모듈에 포함되어 있지 않음
