# CI-4312: 지출결의서 반려 후 재작성 시 영수증이 사라지는 문제

> **상태**: 진행 중 — 2026-04-03

## 증상

- **문제 정의**: 지출결의서(문서번호 2026-1193) 반려 후 재작성할 때, 해당 영수증에 "진행중인 문서가 있어" 메시지가 뜨면서 영수증이 사라지는 문제
- **회사**: 주식회사 무하유 (Customer ID: 69518)
- **요청자**: hjjeong@muhayu.com[^1]
- **대상자**: eunjin@muhayu.com[^1]
- **영향 범위**: 해당 고객의 전자결재-fins 간 상태 동기화가 전반적으로 깨진 상태 — 최근 30건 전부 IN_PROGRESS[^5]
- **문제 시점**: 문서 생성일 2026-03-31[^5]
- **문서번호**: 2026-1193 (`document_id: ac68b35b-1031-4852-ba6e-c6d0e3a469df`)[^6]
- 문의 내용:
  1. 지출결의서 반려 후 재작성 시 영수증이 사라짐[^1]
  2. "진행중인 문서가 있어" 라는 메시지가 표시됨[^1]
  3. CS 확인: 영수증 제출 상태는 맞으나, 재작성된 문서에 첨부된 영수증에 "진행 중인 문서가 있음"으로 탈락[^7]

## 현재까지 파악된 내용

- 유지원(CS)이 Deli에서 확인: 영수증 제출 상태는 정상(SUBMITTED)이나, "지출결의 진행중"으로 표시됨[^7]
- 임형태: "일시적으로 메트릭스 조회 상황에서 오류가 뜨면 그럴 수 있는데 반복되는 현상일까요?" + "해당 영수증이 첨부 가능한 상태인지도 확인 필요 — 제출 상태여야 함"[^6]
- 임형태: "로그 보고 핑 드리겠습니다"[^6] — 조사 진행 중

## 원인 분석

**CI-1078과 동일 원인 — DB로 확정.** `spending_evidence_electronic_approval_document` 테이블의 status가 전자결재 실제 상태(REJECTED)와 동기화되지 않고 IN_PROGRESS로 잔존하여, 재작성 시 영수증 유효성 검증에서 탈락.

### 가설 목록

| # | 가설 | 확인 방법 | 상태 |
|---|------|----------|------|
| 1 | `spending_evidence_electronic_approval_document.status`가 IN_PROGRESS 잔존 | DB 조회 | ✅ **DB로 확정**[^5] |
| 2 | Bullseye 일시적 오류 (임형태 제기) | DB vs Bullseye 비교 | ❌ 소거 — DB 자체가 불일치[^5] |
| 3 | 영수증 state 문제 (임형태 제기) | DB 조회 | ❌ 소거 — 모두 SUBMITTED 정상[^5] |

> 💡 **판단 근거**: CI-1078 Slack 스레드(2025-04-15)에서 윤진한이 동일 메커니즘 확인[^8]

> → 해당 고객 최근 30건 전부 IN_PROGRESS — 동기화 전반적 실패[^5]

**전자결재 상태 변경 이벤트 흐름**:

`flex-fins-backend` > `integration/electronic-approval/kafka-consumer/.../ElectronicApprovalDocumentEventHandler.kt`에서 REJECTED 이벤트를 수신하면 `replaceAll()`로 상태를 갱신하는 구조이나, **이벤트 수신 실패 시 복구 메커니즘이 없다.**

**영수증 유효성 검증**: 재작성 시 각 영수증이 "다른 진행중 문서에 사용 중"인지 검증. `spending_evidence_electronic_approval_document.status = IN_PROGRESS`인 레코드가 있으면 해당 영수증은 탈락.

### 근본 원인

1. **표면 원인**: `spending_evidence_electronic_approval_document.status`가 IN_PROGRESS로 잔존[^5]
2. **근본 원인**: 전자결재 반려 이벤트의 Kafka 수신 실패 시 복구 메커니즘 부재[^4]. 이벤트가 한 번 누락되면 상태 불일치가 영구적으로 지속된다. [추론: CI-1078에서도 재현 불가였고, 해당 고객 최근 30건이 전부 IN_PROGRESS이므로 이 고객에 대한 Kafka 동기화 자체가 동작하지 않고 있을 가능성]

### 코드 위치

| `flex-fins-backend` > `integration/electronic-approval/kafka-consumer/.../ElectronicApprovalDocumentEventHandler.kt` | 전자결재 상태 이벤트 수신 |

## 발견한 스펙/제약

- 전자결재 문서 상태와 fins 쪽 `spending_evidence_electronic_approval_document` 상태의 동기화는 **Kafka 이벤트 기반 단방향**이며, 실패 시 재시도/복구 메커니즘이 없다[^4]
- `electronic-approval` 모듈에는 operation API가 없어 상태 보정을 위한 수동 동기화 API가 존재하지 않는다[^4]
- CI-1078 당시 "개별 문서 동기화 오퍼레이션" 필요성이 제기되었으나 미구현 상태 (FINS-8594)[^8]
- 재작성 시 영수증 유효성 검증: 제출 상태 + 다른 진행중 문서에 사용되지 않아야 함[^8]

## 해결안 / 조사 방향

### 즉시 대응 (표면 원인 해결)

**데이터 보정** — CI-1078과 동일 방법. 임형태가 조사 진행 중[^6].

1. `spending_evidence_electronic_approval_document` 테이블의 status를 전자결재 실제 상태와 일치시킴
2. Bullseye 재동기화: `POST /api/operation/v2/fins/spending/customers/69518/synchronize`

### 근본 해결 (재발 방지)

| # | 방안 | 장점 | 단점 |
|---|------|------|------|
| 1 | **전자결재 문서 동기화 operation API 신규 구현** (FINS-8594) | 수동 보정 가능, 운영 대응 시간 단축 | 개발 공수 |
| 2 | **Kafka consumer 실패 시 DLQ + 재처리 로직 추가** | 근본적 해결 | 인프라 변경 |
| 3 | **주기적 정합성 체크 배치** | 이벤트 누락 사후 감지 | 실시간 대응 불가 |

### 추가 확인 필요

- 해당 고객 최근 30건 전부 IN_PROGRESS — 이 고객의 Kafka 동기화가 전반적으로 실패하고 있는지 확인 필요
