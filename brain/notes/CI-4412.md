### 성격
인사발령 히스토리 리스트(`HistoryList`)에 메모 수정 전/후 내용이 모두 표시되는 UI 현상.

코드 분석 결과:
- FE `HistoryList`: `getSetsRemote`로 모든 PersonnelAppointmentSet 목록 조회 → 각 Set마다 `HistoryListItem` 렌더링, 각 Set의 `memo` 표시
- BE `patchSet()`: 기존 PersonnelAppointment 레코드를 UPDATE — 새 Set 생성 없음
- "메모 수정하기"(`MemoUpdateModal`): `PATCH /sets/{id}` 호출 → 기존 Set memo UPDATE만 — 이것만으로는 새 Set 미생성
- "발령 수정하기"(`EditDrawer`): 케이스별 분기
  - 메모/적용일만 변경 → `updateSet (PATCH)` (기존 Set 업데이트)
  - 인사정보 부분 변경 → `updateDetail (PATCH)` + `updateSet (PATCH)` (기존 Set 업데이트)
  - `isCanceled=true` 케이스 → `createSet (POST)` **새 Set 생성** → HistoryList에 이전 Set + 새 Set 모두 표시

**핵심 가설** [추론: EditDrawer useUpdate.ts 코드 분석]: "발령 수정하기" 중 취소된 발령을 재등록하는 케이스(`isCanceled=true`)에서 새 Set이 생성되고, HistoryList에 이전 Set(old memo)과 새 Set(new memo)이 함께 표시되는 것이 "수정 전후 메모 모두 표기" 현상의 원인. 이 구조가 의도된 히스토리 추적 설계인지 확인 필요.

### 영향 범위

## 원인 분석

**verdict**: `spec`

"재발령(Recreate)" 기능의 의도된 동작이다. 취소된 발령을 재등록할 때 기존 Set을 수정하는 것이 아니라 새로운 Set을 생성하는 것이 설계 의도이며, HistoryList는 모든 Set(취소된 이전 Set + 신규 Set)을 목록으로 표시한다.

### 가설 목록

| # | 가설 | 확인 방법 | 상태 |
|---|------|----------|------|
| 1 | `isCanceled=true` 케이스의 createSet 호출이 의도된 재발령 설계 (spec) | FE 코드 주석 + UI 레이블 확인 | ✅ 확정 |
| 2 | createSet 호출이 버그 — 메모 수정 시 불필요하게 새 Set 생성 | MemoUpdateModal 코드 확인 | ❌ 소거 — "메모 수정하기"는 PATCH만 호출, createSet 미사용 |

### 핵심 근거

1. **FE 코드 명시적 주석**: `useUpdate.ts` 라인 265에 `// NOTE: 전체 취소된 발령은 새로운 Set으로 재발령` — 설계 의도가 코드에 문서화되어 있음[^5]
2. **UI 진입점 분리**: `isCanceled=true`는 HistoryList의 **"재발령(Recreate)" 버튼** 클릭 시에만 설정됨. "메모 수정하기" 버튼과 별도 경로[^5]
3. **"메모 수정하기"는 새 Set 미생성**: `MemoUpdateModal`은 `PATCH /sets/{id}`만 호출 — 메모만 수정하는 경우에는 기존 Set이 UPDATE됨[^6]

### 조사 과정

> 💡 "메모 수정 시 전후 내용 모두 표기" 현상 → FE HistoryList가 모든 Set을 렌더링하는 구조 확인 → "어떤 경우에 새 Set이 생성되는가" 추적 → `useUpdate.ts`에서 `isCanceled=true` 분기 발견 → 해당 분기의 진입점(`onClickRecreate`)과 코드 주석 확인 → 의도된 재발령 기능으로 확정

### 고객 안내 내용

"발령 취소 후 재등록 시 이전 발령 이력과 새 발령이 모두 목록에 표시되는 것은 의도된 동작입니다. 이는 인사발령 변경 이력을 추적하기 위한 설계로, 취소된 발령 기록도 감사 목적으로 보존됩니다. 메모를 단순 수정('메모 수정하기')할 때는 이 현상이 발생하지 않습니다."

## 발견한 스펙/제약
- **재발령 시 새 Set 생성**: "발령 수정하기" 중 취소된 발령을 재등록("재발령" 버튼)하면 기존 Set을 수정하지 않고 신규 Set을 생성. HistoryList는 전체 Set 목록을 표시하므로 이전/신규 두 Set이 모두 표시됨. 의도된 감사 추적 구조.
- **"메모 수정하기"와 "재발령"은 별도 경로**: "메모 수정하기"(`MemoUpdateModal`)는 PATCH만 호출 → 새 Set 미생성. "재발령"(`isCanceled=true`)만 POST → 새 Set 생성.
