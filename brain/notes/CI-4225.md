## 원인 분석

### L1 원인 확정

육아휴직(`paymentRatio=0`) + 지급항목 `allowance_on_leave_rule=DAILY_BASE` 조합으로 자동산정 시 0원 계산됨. 스펙에 의한 정상 동작.

### 가설 테이블

| # | 가설 | 확인 방법 | 상태 |
|---|------|----------|------|
| 1 | 위탁보육수당이 변동급(`allowanceGroupKey=NULL`)으로 등록되어 자동산정 시 무조건 0원 | `customizable_allowance` 테이블의 `contract_allowance_group_key` 확인 | 소거 — 3개 Q02 항목 모두 NOT NULL (고정급)[^3] |
| 2 | 고정급이지만 DAILY_BASE 규칙에서 육아휴직 `paymentRatio=0`이라 0원 | `leave_of_absences` JSON의 `paymentRatio` + `allowance_on_leave_rule` 확인 | **확정**[^4][^5] |

### 핵심 근거

> **판단 근거**:
> 1. `allowance_global` 테이블에서 Q02 3개 항목 모두 `allowance_on_leave_rule = DAILY_BASE`, `variability = FIXED_PAY` 확인[^4]
> 2. `work_income_settlement_payee` (settlement_id=87186, user_id=890916)에서 `leave_of_absences` JSON 확인 — `paymentRatio: 0`, `reason: PARENTING`[^5]
> 3. `MonthlyAllowanceRecipientCalculator.kt:84-89` — DAILY_BASE 규칙에서 `recipientAmount x days x paymentRatio(0) / calendarDays = 0원`[^2]

### 조사 과정

<details>
<summary>상세 조사 과정</summary>

1. `AllowanceRecipientCalculationCoordinator.kt:60-64` — `allowanceGroupKey`가 null이면 변동급으로 분류하여 0원 처리. 가설 1 수립.[^1]
2. `customizable_allowance` 테이블 조회 — `contract_allowance_group_key`가 3개 모두 NOT NULL. 가설 1 소거.[^3]
3. `MonthlyAllowanceRecipientCalculator.kt:22-27` — ON_LEAVE remark일 때 `allowanceOnLeaveRule` 적용. DAILY_BASE에서 `paymentRatio` 곱셈.[^2]
4. `work_income_settlement_payee` 조회 — `leave_of_absences` JSON에서 `paymentRatio=0` 확인. 가설 2 확정.[^5]

</details>

### 5 Whys

1. 왜 보육수당 자동산정 시 0원인가? — DAILY_BASE 규칙에서 `paymentRatio`를 곱하는데 값이 0
2. 왜 `paymentRatio`가 0인가? — 대상자가 육아휴직(PARENTING)이고, 육아휴직 급여지급률은 0이 정상
3. 왜 DAILY_BASE 규칙이 적용되는가? — 해당 지급항목의 `allowance_on_leave_rule` 기본값이 DAILY_BASE

### 스펙/버그 판별

**스펙** — DAILY_BASE 규칙은 휴직 급여지급률을 반영하도록 의도적으로 설계됨. 육아휴직 급여지급률 0도 인사 설정에 의한 정상값. 고객의 니즈("복지 차원 지급")는 해당 지급항목의 설정 변경으로 해결 가능.

## 해결안 / 조사 방향

### 스펙 판정 — 설정 변경으로 해결 가능

**방법**: 위탁보육수당 지급항목의 `allowance_on_leave_rule`을 `DAILY_BASE` → `FULL`로 변경

- `FULL` 규칙: `금액 × 일수 × probationRate / 정산일수` — **paymentRatio를 곱하지 않음**
- 따라서 휴직 중에도 전액 지급됨

**변경 방법**: 고객사 관리자가 직접 변경 가능[^7]
1. **지급항목 마스터 설정**: `POST /api/v2/payroll/allowances/{allowanceIdHash}` — 급여 > 지급항목 설정 화면
2. **정산 내 개별 변경**: `PUT /api/v2/payroll/settlements/work-income/{settlementIdHash}/payments/allowances/item/{allowanceIdHash}` — 정산 내 지급항목 설정 (확정 전에만 가능)

> 고객사에 "해당 지급항목의 **휴직월 지급 방법**을 **전액지급(FULL)**으로 변경"하도록 안내하면 됨

### 참고: Q02 항목 3개 존재

이 회사에는 Q02 비과세 코드의 지급항목이 3개 등록됨[^4]:

| allowance_global_id | code | grant_method | 생성일 |
|---------------------|------|-------------|--------|
| 1029146 | A005 | FIXED (일괄) | 2025-11-12 |
| 1030366 | A023 | PER_PERSON (개인별) | 2025-11-18 |
| 1047508 | A037 | PER_PERSON (개인별) | 2026-02-09 |

3개 모두 `allowance_on_leave_rule = DAILY_BASE`. 대상자에게 해당하는 항목만 `FULL`로 변경 필요.

## 발견한 스펙/제약
- `DAILY_BASE` 규칙: `금액 x 일수 x 수습률 x 휴직급여비율 / 기간일수` — 육아휴직(비율 0)에는 항상 0원[^2]
- `FULL` 규칙: `금액 x 일수 x 수습률 / 정산일수` — 휴직 비율 미적용[^2]
- `NONE` 규칙: 무조건 0원[^2]
- Q02(보육수당) 2026년 이후: 비과세 한도 = 기준금액 x 자녀 수

## 다음에 같은 문의가 오면

1. **확인**: 해당 지급항목의 `allowance_on_leave_rule` 값 확인 (`allowance_global` 테이블)
2. **판별**: `DAILY_BASE`이고 휴직자 `paymentRatio=0`이면 → 이 이슈와 동일
3. **조치**: 고객사에 "해당 지급항목의 **휴직월 지급 방법**을 **전액지급**으로 변경" 안내
