### 핵심 스펙 정리

- **"관리자만 보기(`is_only_visible_for_admin`)"의 정확한 의미**: **양식 목록 페이지에서 비관리자에게 노출하지 않음** 뿐이다. 양식 작성 자체를 비관리자에게 차단하는 기능이 아니다.
- **공유 링크 작성 흐름은 의도된 패스**: 관리자가 `/workflow/archive/*?workflow-action=create&workflow-template-key=...` 형식의 URL을 복사해 비관리자에게 전달하면, 비관리자는 해당 링크로 양식을 작성 가능. 이것이 제품 요구사항.
- **CI-4416 원 문의의 재해석**: 올릿 CS팀은 "권한자만 보기"가 작성 권한까지 포괄한다고 오해한 것으로 보임. 실제로는 **목록 가시성만 제한**하는 기능이며, 링크 공유 작성은 허용되는 정상 플로우다.

### 성격

### 원인 (확정)

**결재 문서 draft 생성 API(`POST /api/v3/approval-document/approval-documents/draft`)가 양식의 `is_only_visible_for_admin` 필드를 체크하지 않는다.** 가시성 필터는 **양식 목록 조회(`filterReadPermitted`)에만** 적용되고, **templateKey로 양식을 직접 사용해 draft를 만드는 경로**에는 체크가 없다. 그래서 UI 목록에 더 이상 보이지 않아도, FE 캐시·즐겨찾기·과거 화면 등 **목록 조회 필터를 거치지 않는 진입점**을 통해 templateKey만 확보되면, 권한 없는 사용자가 기안서를 작성·상신할 수 있다.

이번 CI-4416은 이 구조적 결함이 **실제 발화**한 케이스: sy.choi가 수 년간 써온 "명함 신청" 양식이 2026-03-23에 `is_only_visible_for_admin = TRUE`로 변경됐으나[^13], FE에는 여전히 접근 경로가 남아 있었고, sy.choi가 3주 후(2026-04-13) 이전처럼 상신하자 Draft API가 막지 못했다[^14].

### 핵심 증거 (L2)

1. **가시성 필터 존재 지점**: `flex-impact-backend` > `approval-document-template/service/.../ApprovalDocumentTemplatePermissionResolveService.kt:74-96` — `filterPermitted()` 내부 `.filter { hasAdminPermission || !it.onlyVisibleForAdmin }`[^7]. `hasAdminPermission`은 `WORKFLOW_TEMPLATE_READ/UPDATE/DELETE` 권한 중 하나라도 있으면 `true`.
2. **가시성 체크 누락 지점**: `flex-impact-backend` > `approval-document/service/.../ApprovalDocumentDraftService.kt:44-68` `ApprovalDocumentDraftServiceImpl.draft()` — `findReferenceByIdentifier(identifier)`로 템플릿만 조회하고, `onlyVisibleForAdmin` 검사 없이 그대로 `draft`를 `save`한다[^8].
3. **우회 경로 입구**: `flex-impact-backend` > `approval-document/api/.../ApprovalDocumentDraftController.kt:50-72` `POST /api/v3/approval-document/approval-documents/draft`는 `ApprovalDocumentCreateRequest.assignedBy: ApprovalDocumentTodoRequest?`를 받아 Todo 기반 작성 요청을 처리한다[^9]. `assignedBy` 없이 `document.templateKey`만으로도 draft 생성 가능.

## 발견한 스펙/제약

> ⚠️ 재평가 후 정정됨. 아래 항목은 **스펙상 의도된 설계**로 확인됨 (2026-04-13 추가 사용자 피드백).

- **"관리자만 보기(`is_only_visible_for_admin`)"의 범위**: **양식 목록 페이지(`filterReadPermitted`)에서만 가시성 제한**. 개별 양식을 templateKey로 직접 사용하는 draft/submit 경로는 의도적으로 제한하지 않음[^7][^8][^23].
- **공유 링크 작성 플로우**: 관리자가 URL(`?workflow-action=create&workflow-template-key=<key>`)을 복사해 비관리자에게 공유 → 비관리자가 링크 통해 작성. 이것은 **의도된 제품 요구사항**. 대표 사례: 퇴사자에게 퇴사 워플 작성 요청[^23].
- **`hasAdminPermission` 판정 기준**: `WORKFLOW_TEMPLATE_READ` / `WORKFLOW_TEMPLATE_UPDATE` / `WORKFLOW_TEMPLATE_DELETE` 권한 중 하나라도 보유하면 관리자로 간주. 이 판정은 **목록 가시성 필터에서만** 의미를 가지며, 작성 권한 판정에는 쓰이지 않는다[^7].
- **`assignedBy` (Todo 경로) vs 공유 링크 (URL 경로)** — 두 개가 구조적으로 별개의 "비관리자에게 작성 요청" 채널:
  - `assignedBy`: Todo 기반 위임, `workflow_task.requester_id` 에 기록됨[^9][^14]
  - 공유 링크: URL query param 기반, DB에 별도 흔적 남지 않음[^22]. 즉 "누가 링크를 보냈는지" 시스템 차원에서 추적 불가
