### 근본 원인 (5 Whys)

1. **왜 급여명세서 지급금액이 0원?** → `work_income_payment_settlement_payee_result_item` 테이블에 지급 항목 행이 없음[^7]
2. **왜 신 item 테이블에 행이 없는가?** → 2025-03-19 backfill 시 2023-11 데이터 35건이 누락됨[^8][^9]
3. **왜 backfill에서 누락?** → **chicken-and-egg 문제 확정**[^15][^16]. JPA Entity가 신 테이블 기반으로 리팩토링된 상태에서 backfill이 `paymentSettlementResult.sortedResult.items`를 "레거시 소스"로 사용하지만, 이 값이 실제로는 신 테이블(빈 데이터)에서 역산됨 → `newPaymentItems = []` → `isEmpty()` 스킵. **재실행해도 동일하게 누락됨**
4. **왜 지급만 문제이고 공제는 정상?** → 공제는 `work_income_base_deduction_settlement_payee_result` 테이블에 개별 컬럼으로 저장되어 구→신 이관 불필요. 지급만 JSON blob → 개별 행 변환이 필요한 구조[^10]

## 해결안 / 조사 방향

### 즉시 대응 (표면 원인 해결)

**옵션 A: 해당 귀속월 엑셀 재업로드**
- 고객에게 같은 엑셀 파일로 2023-11 급여자료를 다시 이관하도록 안내
- 현재 코드는 신 item 테이블에 정상 기록하므로 해결 가능
- 장점: 고객 자가 해결 가능, 코드 변경 불필요
- 단점: 해당 고객 1사만 해결, 다른 고객사는 여전히 누락

**옵션 B: 수동 backfill 실행**
- 구 테이블 JSON blob에서 데이터를 추출하여 신 item 테이블에 INSERT
- 장점: 1회 실행으로 대량 수정 가능
- 단점: JSON blob이 암호화되어 있어 애플리케이션 레벨 처리 필요

### 근본 해결 (근본 원인 해결)

**옵션 C: backfill 코드 수정 후 재실행** (권장)
- 현재 backfill Operation API[^14]는 **재실행해도 동일하게 누락**됨 — chicken-and-egg 문제[^16]
- 수정 필요: `migratePayeeResult()`에서 `paymentSettlementResult.sortedResult.items` 대신 구 테이블(`work_income_payment_settlement_payee_result`)의 `allowance_settlement_result_data` JSON blob을 직접 읽도록 변경
- 수정 후 `POST /api/operation/v2/payroll/settlement-results/payment/item/migrate/{customerId}` 로 누락 고객 일괄 재실행
- 장점: 모든 영향 고객사 일괄 해결
- 단점: 코드 수정 필요, 구 JSON blob 파싱 로직 추가 또는 복원 필요

### 추가 확인 필요

- [ ] backfill 코드 수정 (구 JSON blob 직접 읽기) 후 재실행
- [ ] 누락된 전체 고객사/귀속월/인원 수 정확한 집계
- [ ] 원천세 신고에 실제 영향이 있는지 급여팀과 확인

## 발견한 스펙/제약

- 급여자료이관 시 지급 항목은 `allowance_global` 테이블의 `allowanceGlobalCode`와 매칭하여 저장. 매칭 실패 시 해당 항목은 조용히(silently) 무시됨 — 에러/경고 없음[^10]
- 지급 항목 저장 구조가 2025-02에 JSON blob → 개별 행 테이블로 변경됨. 공제는 처음부터 개별 컬럼 구조[^10]
