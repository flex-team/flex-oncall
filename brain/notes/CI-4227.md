### 근본 원인 (5 Whys)

1. **왜 급여명세서 지급금액이 0원?** → `work_income_payment_settlement_payee_result_item` 테이블에 지급 항목 행이 없음[^7]
2. **왜 신 item 테이블에 행이 없는가?** → 2025-03-19 backfill 시 2023-11 데이터 35건이 누락됨[^8][^9]
3. **왜 backfill에서 누락?** → ~~chicken-and-egg 문제(신 테이블이 비어있어 구 데이터를 읽지 못하고, 구 데이터가 없으면 신 테이블을 채울 수 없는 순환 의존)~~ **수정: 마이그레이션 실행 시점 검증 로직 실패 추정**[^17][^18][^19][^20]. 아래 "추가 조사" 참조
4. **왜 지급만 문제이고 공제는 정상?** → 공제는 `work_income_base_deduction_settlement_payee_result` 테이블에 개별 컬럼으로 저장되어 구→신 이관 불필요. 지급만 JSON blob → 개별 행 변환이 필요한 구조[^10]

### 원인 분석 수정 (2026-03-27 추가 조사)

#### 핵심 발견

1. **items 생성(01:32)이 v3.77.0 배포(13:40)보다 12시간 앞선다**[^19]. backfill 코드(PR #7282)는 v3.76.x에 포함되지 않았으므로, 사세영님은 **로컬 빌드를 prod DB에 직접 연결**하여 실행한 것으로 추정된다.

2. **로컬 빌드 시점이 Entity 리팩토링(03-13 20:25) 전이라면**, 구 Entity의 `paymentSettlementResult` getter가 JSON blob OneToOne에서 직접 데이터를 읽으므로 **chicken-and-egg 없이 정상 작동**한다. migration controller 작성자(사세영님)가 자신이 개발한 코드를 바로 실행했으므로, 리팩토링 전 빌드를 사용했을 가능성이 높다.

3. **이 가설의 근거**: chicken-and-egg가 전역적이면 단 한 건도 성공할 수 없다[^20]. 그러나 DB에서 25개월분 items가 정상 생성되었고, 2023-11만 누락되었다. 구 테이블 데이터(JSON blob)는 2023-10/11/12 간 구조적 차이가 없다[^21].

4. **2023-11 누락 원인**: Entity 리팩토링이 아닌, **migration controller 자체의 검증 로직 실패**로 추정. 03-11~03-12 커밋에서 추가된 검증: "실제 지급하는 수당만 item 생성"(필터링) + "항목/합계 비교"(과세총액 검증). 필터링 후 합계가 헤더 과세총액과 불일치하면 `IllegalStateException` → settlement_result 단위 트랜잭션 롤백[^15].

5. **참고**: customer_id=156610의 2023-11 JSON blob에 비정상적으로 작은 레코드 존재(min_len=40, 정상은 936+)[^22]. 데이터 손상 가능성.

#### 현재 코드의 chicken-and-egg

현재 코드(v3.77.0 이후)에서는 Entity 리팩토링이 적용되어 **migration controller를 재실행하면 chicken-and-egg가 발생**한다[^16][^20]. 이것이 PR #8691의 `LegacyPaymentDataReader` 추가 배경이다. 다만 이것은 **원래 누락의 원인이 아니라, 재실행을 막는 이차적 장벽**이다.

## 해결안 / 조사 방향

### 해결 방향 (2026-03-27 갱신)

**PR #8691** (`LegacyPaymentDataReader` 추가)이 chicken-and-egg 장벽을 해결하므로 재실행 경로를 확보한다.

**주의**: PR #8691은 legacy fallback 시 과세총액/지급총액 validation을 스킵한다. 이것은 원래 2023-11 누락의 원인(검증 로직 실패 추정)을 우회하는 효과가 있지만, **의도적 설계인지 확인 필요**.

### 남은 리스크

1. **원래 누락 원인 미확정**: 2023-11에서 검증 실패를 일으킨 구체적 데이터 패턴이 불명. validation 스킵으로 우회하면 해결되지만, 잘못된 데이터가 들어갈 가능성도 있다.
2. **customer_id=156610 데이터 손상**: 2023-11 JSON blob에 비정상 레코드(40 bytes) 존재[^22]. 이 레코드는 LegacyPaymentDataReader로 읽어도 정상 파싱이 안 될 수 있다.

### 추가 확인 필요

- [ ] PR #8691의 validation 스킵이 의도적인지, 2023-11 데이터에 안전한지 검증
- [ ] 누락된 전체 고객사/귀속월/인원 수 정확한 집계
- [ ] 원천세 신고에 실제 영향이 있는지 급여팀과 확인
