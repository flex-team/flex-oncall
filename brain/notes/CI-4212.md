- **문제 시점**: 2026년 3월 정기급여 중도정산[^1]

## 원인 분석

**결론**: 중도정산 recipient[^12]의 보험료는 생성 시점의 정산 데이터(보수총액)로 1회 계산·저장되며, 이후 이관 데이터가 추가되어도 자동 재계산되지 않는 **버그**. 워크어라운드 존재 — 리셋하면 정상값 복구됨[^6].

> ⚠️ **이한결 추가 발견**: 24일 19시에 키인(수동 입력) 이력이 존재[^14]. CS에 따르면 "23일 19시부터 잘못된 금액 → 키인으로 보정 시도 → 오늘 리셋" 순서[^15]. 이한결은 현재 남은 기록으로는 최초 잘못된 산출의 원인 파악이 거의 불가능하다고 판단[^14].

### 핵심 근거

1. **코드**: `RetireeYearEndSettlementSocialInsuranceRecipientFactory.createRecipient()`에서 보험료를 계산·저장[^9]. 이 메서드는 사용자 액션(리셋/최신정보반영/감면수정) 시에만 호출됨
2. **CS 확인**: "123 순서로 리셋하니 금액이 잘 반영되었습니다"[^6] — 리셋하면 정상값 복구됨
3. **DB**: 2026년 recipient db_created_at: 2026-03-23[^8]. "26년 일부 이관" 상태에서 생성되어 불완전한 보수총액 기반으로 보험료 계산·저장됨

## 해결안

- **즉시 대응**: 보험료 "리셋" 안내 (DELETE /premium API → recalculate)[^10]. 다른 이관 회사 동일 증상 확인 필요.
- **근본 해결**: 옵션 A(이벤트 기반 재계산) 또는 옵션 B(조회 시 dirty check) 중 선택 필요.

<details>
<summary>근본 해결 옵션 상세</summary>

- **옵션 A**: 정산 결과 변경 시 recipient 보험료 재계산 이벤트 추가
  - 장점: 근본 해결
  - 단점: key-in 값 초기화 위험, 성능 부하
- **옵션 B**: recipient 조회 시 dirty check → 정산 결과 변경 시 자동 재계산
  - 장점: 조회 시점에 항상 최신 데이터 보장
  - 단점: 조회 성능 저하

</details>
