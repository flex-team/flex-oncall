## `postAnnualTimeOffNextAssignTime` = 16일의 계산 근거

operation API 버킷 조회 결과를 코드 로직에 대입하여 검증[^12]:

### 구성원 연차 부여 구조

- **입사 추정**: 2025-08-26 (첫 MONTHLY 버킷 2025-09-26 기준)
- **소정근로시간**: 450분 (7시간 30분)
- **1년차**: 월별 부여 (MONTHLY) — 매월 26일에 1일(450분)씩, round 2~12
- **2년차~**: 연간 부여 (YEARLY) — 매년 1/1 기준

### 코드 계산 과정 (requestDate = 2026-04-01)

**① YEARLY 필터**: 전체 버킷에서 `timeRule.time === YEARLY` 만 추출 → MONTHLY 11개 제외

**② `getNextBuckets(2026-04-01)` — `validUsageFrom > 2026-04-01` 필터**:

| validUsageFrom | 타입 | assignValue | assigned | used | **remaining** |
|----------------|------|-------------|----------|------|--------------|
| 2026-01-01 | REGULAR 6일 | 6 | 2700 | 1230 | 1470 — ❌ 과거 버킷 |
| 2026-01-01 | ADDITIONAL 1일 | 1 | 450 | 0 | 450 — ❌ 과거 버킷 |
| **2027-01-01** | **REGULAR 15일** | **15** | **6750** | **0** | **6750** ← 선택 |
| **2027-01-01** | **ADDITIONAL 1일** | **1** | **450** | **0** | **450** ← 선택 |
| 2028-01-01~ | (이후 연도) | | | | — 가장 가까운 게 아님 |

> `getNextBuckets`는 `minBy { validUsageFrom }` 으로 **가장 가까운 미래**를 선택하고, 같은 `validUsageFrom + validUsageTo` 인 버킷을 모두 포함[^11]

**③ 합산**: 6750 + 450 = **7200분** ÷ 450(소정근로) = **16일** ✅

### 정리

`postAnnualTimeOffNextAssignTime = 16일` = **2027년 부여 예정 연차 (정기 15일 + 추가 1일)**

## 발견한 스펙/제약

- `postAnnualTimeOffNextAssignTime`: YEARLY 버킷만 필터 → 가장 가까운 미래 버킷의 `remainingTimeOffMinutes` 합산[^9]
- `getNextBuckets`: `validUsageFrom > requestDate` 인 것 중 `minBy { validUsageFrom }` 선택. 같은 기간(`validUsageFrom + validUsageTo`)인 버킷이 여러 개면 모두 포함[^11]
- 분→일 변환: `days = totalMinutes / 소정근로시간`, `hours = 나머지/60`, `minutes = 나머지%60`[^10]
- MONTHLY 버킷은 YEARLY 필터에서 제외되므로, 1년차 월별 부여 구성원의 경우 `postAnnualTimeOffNextAssignTime`이 2년차 이후 YEARLY 버킷을 가리킴
