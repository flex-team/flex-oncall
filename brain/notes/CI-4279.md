# CI-4279: 원천세 신고서 전월 미환급세액 미반영

> **상태**: 조사 중 — 2026-04-01

## 증상
- **문제 정의**: 2월 원천세 신고서의 차월이월 환급세액이 3월 원천세 신고서의 전월 미환급세액으로 자동 불러와지지 않음
- **회사**: (주)오피스디포코리아 (Customer ID: 187232)
- **요청자**: 김한나 (CS)[^1]
- **대상자**: 김수환 (suhwan.kim@officedepot.co.kr)[^2]
- **영향 범위**: 해당 회사 급여 담당자
- **문제 시점**: 2026-04-01 기준 진행 중
- 문의 내용:
  1. 2월 원천세 신고서(2월귀속 2월지급)의 **차월이월 환급세액**이 3월 원천세 신고서(2월귀속 3월지급 / 3월귀속 3월지급, 2건)의 **전월 미환급세액**으로 자동 반영되지 않음[^1]
  2. 현재 고객이 수동 키인으로 처리한 상태이며, 자동 반영되지 않는 원인 확인 요청[^2]

## 원인 분석

### 조사 과정

> 💡 **판단 근거**: 신고서 생성 로직(`WithholdingStatusReportModelFactoryImpl.buildReconciliationSummaryFrom()`)을 코드 분석[^3]
> → 직전 지급월(2026-02)의 CLOSED 신고서를 조회하여 `maxByOrNull { withholdingStatusReportId }`로 선택[^3]
> → DB에서 해당 회사의 2~3월 신고서 전체 조회[^4]
> → 2026-02 지급월에 CLOSED 신고서가 2건 존재: id 8690(2월귀속), id 8692(1월귀속)
> → id 8692가 ID가 더 크므로 선택됨 — 그런데 8692의 차월이월환급세액은 **0원**
> → 실제로 환급세액 97,738,270원을 가진 것은 id 8690

#### DB 데이터: 오피스디포코리아 2~3월 원천세 신고서

### 원인

`WithholdingStatusReportModelFactoryImpl.buildReconciliationSummaryFrom()`(라인 379~388)[^3]에서 직전 지급월의 CLOSED 신고서 중 **`withholdingStatusReportId`가 가장 큰 것**을 선택하는 로직이 문제다.

같은 지급월(2026-02)에 CLOSED 신고서가 2건 있을 때:
- id **8690** (2월귀속 2월지급): 차월이월환급세액 = **97,738,270원** ← 이것이 선택되어야 함
- id **8692** (1월귀속 2월지급): 차월이월환급세액 = **0원** ← ID가 더 커서 선택됨

코드의 의도는 "최신 귀속월 → 최신 생성"(PAY-9212)[^3]이지만, 실제로는 `maxByOrNull { withholdingStatusReportId }`가 **귀속월과 무관하게 ID만으로** 선택하므로, 1월귀속 신고서(8692)가 2월귀속 신고서(8690)보다 나중에 생성되어 잘못 선택된다.

```kotlin
// 현재 코드 (WithholdingStatusReportModelFactoryImpl.kt:383-387)
amountCarriedOverFromPrevMonth = content
    .filter { reportHeader.isHalfType() == it.header.isHalfType() }
    .maxByOrNull { it.withholdingStatusReportId }  // ID 기준 → 귀속월 무시
    ?.let { it.reconciliationSummary.amountToCarryOverToNextMonth() }
    ?: Money.ZERO,
```

> ⚠️ 주석에는 "최신 귀속월 → 최신 생성인 것"이라고 되어 있지만, 실제 코드는 `maxByOrNull { id }`로 귀속월을 고려하지 않는다. 주석과 코드가 불일치한다[^3].

## 발견한 스펙/제약

- 동일 지급월에 귀속월이 다른 CLOSED 신고서가 복수 존재할 수 있다 (예: 1월귀속 2월지급 + 2월귀속 2월지급)[^4]
- 전월미환급세액은 **가장 최신 귀속월**의 신고서에서 가져와야 하지만, 현재 코드는 ID 순서에만 의존한다[^3]
- 고객은 수동 키인으로 우회 가능하지만, 자동 반영이 되지 않는 것은 버그다
