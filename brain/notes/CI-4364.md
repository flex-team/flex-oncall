## 원인 분석

**수동 trigger 경로에서 조직 구성원 조회 시 재직 상태 필터링이 누락되어, 퇴사자에게도 할일(Todo)이 생성되고 알림이 발송됨.**

> 💡 **판단 근거**: 자동 trigger 경로(`ChecklistTaskTriggerAutoActionService`)에는 `isTriggerableStatus()` 필터가 존재[^5]하나, 수동 trigger 경로(`ChecklistTaskTriggerServiceImpl.triggerFromTargets()`)에는 동일한 필터가 없음[^6] → `triggerBasicTodo()`가 퇴사자 포함 전체 userIds를 받아 할일을 생성[^7]

### 가설 목록

| # | 가설 | 확인 방법 | 상태 |
|---|------|----------|------|
| 1 | 수동 trigger 시 `isTriggerableStatus()` 필터 누락 | 자동/수동 trigger 코드 경로 비교 | ✅ 확정 |

### 조사 과정

> 💡 `ChecklistStakeholderResolverImpl.resolveUsers()`에서 조직(DEPT/DEPT_WITH_CHILDREN) 조회 시 `SearchUserFilterableDto(departmentIds=...)` 만 설정[^8]
> → `userStatuses` 필터 미적용 → 퇴사자 포함 전체 구성원 반환
> → `triggerFromTargets()`가 모든 userId를 `trigger()`에 전달[^6]
> → `triggerBasicTodo()`가 전달받은 모든 userId에 대해 Todo 생성 + 알림 발송[^7]
> → `triggeredResult`가 모든 사용자에 대해 non-null → 전원 `TODO` 상태[^6]
> → FAILED 분기(라인 188-195)는 사실상 dead code

### 근본 원인 (5 Whys)

1. 왜 퇴사자에게 할일이 전송되나? → `triggerBasicTodo()`가 퇴사자 userId를 받아 Todo를 생성[^7]
2. 왜 퇴사자 userId가 전달되나? → `triggerFromTargets()`가 필터링 없이 모든 userId를 전달[^6]
3. 왜 필터링이 없나? → 자동 trigger에만 `isTriggerableStatus()` 필터가 적용되고, 수동 trigger에는 누락[^5]
4. 왜 FAILED 처리 로직이 있는데 동작하지 않나? → `triggerBasicTodo()`가 모든 사용자에게 Todo를 생성하므로 `triggeredResult`가 항상 non-null → FAILED 분기가 실행되지 않음[^6]

### 스펙 vs 버그 판별

**버그** — 자동 trigger에는 `isTriggerableStatus()` 필터가 존재하므로 퇴사자 제외가 의도된 스펙이다. 수동 trigger에 동일한 필터가 누락된 것은 구현 누락.

## 현재 상태 확인

- 체크리스트 할일 담당자 조회 시 조직 구성원 필터링 로직: **상태 필터 미적용 확인됨**[^8]
- 수동 trigger 경로에서 `isTriggerableStatus()` 필터 누락 확인됨[^6]

## 차단 범위

- 할일을 받은 퇴사자는 실제로 할일을 수행할 수 없으므로, 체크리스트 완료 처리에 영향 가능
- 우회: 조직 대신 개별 구성원을 직접 지정하면 회피 가능

## 해결
- EPDIV팀에 quick-pick 레이블로 수정 이슈 등록: EPDIV-119[^9]
- 수정 방향: `triggerFromTargets()`에서 `trigger()` 호출 전 `isTriggerableStatus()` 필터 추가

## 해결안 / 조사 방향

### 즉시 대응 (표면 원인 해결)
- 고객에게 조직 대신 개별 구성원 직접 지정으로 우회 안내

### 근본 해결 (코드 수정)

**Option A**: `triggerFromTargets()`에서 `trigger()` 호출 전 `isTriggerableStatus()` 필터 추가 + 비-triggerable 사용자에 대해 FAILED 레코드 별도 생성
- 장점: 자동 trigger와 일관된 동작, FAILED 레코드로 퇴사자 추적 가능
- 단점: 코드 변경량이 약간 더 많음

**Option B**: `triggerFromTargets()`에서 `trigger()` 호출 전 `isTriggerableStatus()` 필터만 추가 (자동 trigger와 동일하게 퇴사자 제외)
- 장점: 변경 최소화, 자동 trigger와 동일한 패턴
- 단점: FAILED 레코드 없이 퇴사자가 아예 target에서 누락 → 관리자가 "누가 제외됐는지" 확인 불가

> Option A 권장 — FAILED 상태로 기록하되 할일/알림은 발송하지 않는 것이 관리자 UX와 데이터 추적 관점에서 더 적합
