## 원인 분석

### 한 줄 요약

휴일대체 수정(CANCEL+재등록) 시 OpenSearch 문서에 새 이벤트 ID가 즉시 반영되지 않아, FE가 구 이벤트 ID로 취소를 요청하여 실패함. 약 4.5시간 후 자동 갱신됨.

### 발생 메커니즘

```
16:09:22  CANCEL(43783) → work-schedule-sync-to-es 발행
            → sync consumer 처리 시점: 44186이 아직 DB에 없음
            → originalHoliday.alteredHoliday = null
            → Jackson NON_NULL 설정 → JSON에서 alteredHoliday 필드 제외
            → doc() partial update → 기존 alteredHoliday(eventId=43783) 유지 ⚠️

16:09:31  POST(44186) → work-schedule-sync-to-es 발행
            → sync consumer 처리 시 44186이 DB에 존재
            → alteredHoliday = {eventId: 44186}
            → 그러나 이 sync가 문서를 갱신하지 못함 (원인 미확정)

16:09~18:18  search/by-departments API → OpenSearch에서 구 eventId=43783 반환
             → FE가 43783으로 DELETE 요청 → 이미 CANCEL된 ID → 400 오류

20:45        다른 트리거(추정: cron 또는 연쇄 이벤트)로 sync 재실행
             → OpenSearch 문서가 eventId=44186으로 정상 갱신됨
```

### 확인된 사실

**OpenSearch 문서 (3/22 날짜):**
- 16:09~20:45 KST: `alteredHoliday.eventId = 43783` (구 ID)[^15]
- 20:45 KST 이후: `alteredHoliday.eventId = 44186` (정상)[^18]

**DB 상태 (변동 없음):**
- 43783: CANCEL됨 (by 44185)
- 44186: 유효
- `toCategorizedEvents()` 필터 정상 작동 확인[^4]

**OpenSearch sync:**
- `work-schedule-sync-to-es.v1` 이벤트 정상 소비, 에러 없음[^17]
- `NON_NULL` + `doc()` partial update 조합이 null 필드를 기존값으로 유지시키는 구조[^16]

## 해결

### 즉시 대응 (완료)
- operation API(`/sync-os-work-schedule-advanced`)로 OpenSearch 재동기화 실행 → 문서 정상 갱신 확인[^18]
- 현재 OpenSearch 문서: `alteredHoliday.eventId = 44186` ✅

### 고객 대응
- 관리자에게 **페이지 새로고침 후 휴일대체 취소 재시도** 안내
- 현재 정상 데이터가 반영된 상태이므로 취소 가능할 것으로 예상

### 근본 원인 — 추가 조사 필요

**OpenSearch sync 타이밍 문제**: 휴일대체 수정(CANCEL+재등록)이 9초 간격으로 발생할 때, CANCEL sync와 POST sync 사이의 처리 순서에 따라 OpenSearch 문서가 잘못된 상태로 남을 수 있음.

원인 후보:
1. **`NON_NULL` + `doc()` partial update**: CANCEL sync에서 `alteredHoliday=null` → JSON 직렬화 시 필드 누락 → 기존 값(43783) 유지[^16]
2. **POST sync 미반영**: POST sync가 실행되었으나 결과가 문서에 반영되지 않은 이유는 미확정. 순서 역전, 동일 문서 동시 upsert 충돌 등 가능성
3. **자연 회복**: 20:45에 다른 트리거(cron, 연쇄 이벤트)로 sync가 재실행되어 자동 갱신

**수정 방향** (별도 티켓으로 분리 권장):
- `OpensearchClient.makeUpdateRequest()`의 `NON_NULL` 설정이 partial update에서 null 필드를 제거하지 못하는 문제 검토
- 또는 휴일대체 CANCEL+재등록을 하나의 트랜잭션으로 묶어 sync 이벤트를 1건만 발행
