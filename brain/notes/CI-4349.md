## 증상

4월 0.5일 사용 후 연차 잔여일이 13.002일로 표시됨. 0.002일의 출처가 불명확하다는 문의.

- 2월 0.333일 사용, 조정 0.833일로 잔여 13.5일이었음
- 4월 0.5일 사용 이후 잔여 **13.002일**로 표현
- 연차 부여 정책: 반차 이상 올림

## 원인 분석

**결론: 버그 아님. 소정근로시간 변경으로 인한 버킷 간 계산 단위 혼재가 원인.**

### 버킷 구성 (4/30, timestamp=1777561200000 기준)

| 버킷 | assignedTime | agreedDayWorkingMinutes | remainingDays |
|------|-------------|------------------------|---------------|
| REGULAR 2026 (2026-01-01) | 5841 min | 480 min/day | 12.16875 |
| ADJUST 2026-02-28 | 300 min | 360 min/day | 0.83333... |

```
5841 ÷ 480 = 12.16875
300 ÷ 360 = 0.83333...
합산 = 13.00208... → setScale(3, HALF_UP) = 13.002
```

### 소정근로시간 변경 이력

이 유저는 소정근로시간이 중간에 변경되었다:

- 2026-01-01 기준: **480 min/day** → REGULAR 버킷 생성 시점
- 2026-02-28 기준: **360 min/day** → ADJUST 버킷 생성 시점

각 버킷은 **생성 시점(assignedAt)의 소정근로시간**을 `agreedDayWorkingMinutes`로 저장하며, 잔여일 계산 시 이 값을 사용한다.

```kotlin
// AnnualTimeOffAssignForCalculationV1
override fun getAgreedDayWorkingMinute(): Int {
    return userWorkRule.getAverageDailyTimeOffMinutesPerWeekAtDate(assign.assignedAt)
}
```

```kotlin
// UserAnnualTimeOffBucketModel.kt
val remainingDays: TimeOffDays
    get() = remainingTimeOffMinutes.toBigDecimal()
        .divide(agreedDayWorkingMinutes.toBigDecimal(), MathContext.DECIMAL64)
```

```kotlin
// UserAnnualTimeOffBuckets.kt
fun getRemainingDays(): TimeOffDays {
    return this.buckets.sumOf { it.remainingDays }.setScale(3, RoundingMode.HALF_UP)
}
```

버킷마다 다른 소정근로시간으로 나누고 합산하기 때문에, 소정근로시간 변경 이력이 있는 유저는 합산 결과에 소수점이 발생할 수 있다.

#### "이전에는 안 보이다가 사용 후에만 보이는" 이유

소수점은 ADJUST 버킷 생성 시점(2026-02-28)부터 잠재적으로 존재했으나, 사용 전 두 버킷의 분수가 우연히 합산 시 정확히 상쇄되어 13.5로 보였다.

```
사용 전: 6080분 ÷ 480 + 300분 ÷ 360 = 38/3 + 5/6 = 81/6 = 13.5 (분수 상쇄)
사용 후: 5841분 ÷ 480 + 300분 ÷ 360 = 13.00208... → 13.002 (상쇄 관계 깨짐)
```

0.5일 사용으로 REGULAR 버킷 분자가 바뀌면서 분수 상쇄 관계가 깨져 처음으로 0.002가 드러났다.

### ADJUST 버킷 생성 경위

DB `v2_user_annual_time_off_adjust_assign` 확인:

- 2026-04-07 당일 같은 `adjust_group_id`로 3건 생성됨 (2건 삭제, 1건 생존)
- 최종 생존 조정: `assigned_at=2026-02-28`, `adjusted_hours=5`
  - `getMinutes() = 0 * 360 + 5 * 60 + 0 = 300 min`
  - `agreedDayWorkingMinutes = 360` (2026-02-28 기준 소정근로시간)

## 연차 조정 스펙 (이슈 조사 중 파악)

### 조정 목적 타입 (`adjust_type`)

`v2_user_annual_time_off_adjust_assign.adjust_type`. 버킷 타입(`AnnualTimeOffAssignType`)과 별개.

- `NORMAL` — 관리자 수동 +/- 조정
- `EMPLOYED` — 재직자 미사용 수당 정산
- `RESIGNED` — 퇴직자 정산

### 조정량 저장 구조

`adjusted_time` 컬럼은 JSON으로 저장: `{"days": 0, "hours": 5, "minutes": 0}`

실제 분(minutes)으로 변환:
```
assignedTime = days × sojeong_at_assignedAt + hours × 60 + minutes
```

`adjusted_days` 컬럼은 일 단위 환산 합계로 **표시 목적**으로만 저장. 실제 분 계산은 `adjusted_time` JSON에서 `getMinutes()`로 수행.

관리자 입력 방식은 두 가지:
1. 일시분 복합 입력(`{days, hours, minutes}`) → 그대로 저장
2. 소수점 일 단위(0.833일) → 내부에서 `{days, hours, minutes}`로 변환 후 저장

### 양수 vs 음수 조정

| 구분 | 처리 |
|------|------|
| `adjustedDays > 0` | **ADJUST 버킷** 신규 생성 → 잔여일 합산 포함 |
| `adjustedDays ≤ 0` | 기존 버킷에서 차감 → 소진 초과분은 **FOR_MINUS 버킷** 자동 생성 |

FOR_MINUS 버킷은 `getDetailRemainingDays`에서 `type !== FOR_MINUS` 필터로 **완전 제외**.

### 이 이슈와의 관련

이 케이스의 ADJUST 버킷(id=452335): `adjusted_time={hours:5}`, assignedAt=2026-02-28
- 양수 조정이므로 ADJUST 버킷으로 생성됨
- assignedAt 기준 소정근로시간 360분 → `assignedTime = 0×360 + 5×60 + 0 = 300분`
- `agreedDayWorkingMinutes = 360` (REGULAR 버킷의 480과 달라 합산 시 소수점 발생)

`-1분 조정`(id=452454): `adjusted_time={minutes:-1}` → `minutes` 직접 사용이므로 소정근로시간과 무관하게 -1분 적용

## 해결

코드 수정 불필요. **스펙상 정상 동작**이므로 고객에게 계산 근거를 안내.

- 13.002일 = 12.16875일(REGULAR 버킷) + 0.83333...일(ADJUST 버킷), 소수점 3자리 반올림
- 소정근로시간이 변경되면 각 버킷이 서로 다른 계산 단위를 사용하여 합산 시 소수점이 발생할 수 있음
- 실제 연차 잔여량(분 단위)은 정확하게 계산되어 있음
