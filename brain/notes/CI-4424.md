<summary>📋 04-07 변경 — 승인 정책 경로 확정 근거</summary>

**증거 체인:**

1. **Envers audit**: 286066의 end_date가 2026-04-07 18:48:58에 UPDATE됨 (실제 데이터 변경 발생)[^7]
2. **access log**: 해당 시간대에 `PUT /api/v2/core/work-experiences` 외부 API 호출 없음 → 직접 수정 아님[^13]
3. **consumer 로그**: traceId `7f02a4ff6fff5e8fe14a38f38c4cd3c5`가 `flex-prod-prod-review-consumer`에서 시작[^14]
4. **Kafka 토픽**: `prod.event.flex.approval.process.v1`, recordKey: `77952:USER_DATA:8dab458b9f234ea09dae1b2b67208f9b`[^14]
5. **코드 확인**: `UserDataApprovalApprovedServiceImpl.approve()` (237~262줄)에서 `UserDataPolicyType.WORK_EDU_EXPERIENCE` → `workExperienceApplicationService.update()` 호출[^15]

**확정 논리 — 왜 "승인 정책에 의한 경력 변경"이라고 할 수 있는가:**

Kafka 메시지나 토픽만으로는 확정할 수 없다. 아래 3가지 소거를 조합하여 확정한다.

**(a) 승인 vs 반려/취소 — approve() 경로 확정:**
- `approval.process` 토픽은 승인·반려·취소 모든 이벤트를 전달할 수 있음
- `reject()`/`cancel()`은 프로세스 상태만 변경하고 `workExperienceApplicationService`를 호출하지 않음[^15]
- Envers audit에 **실제 데이터 UPDATE가 기록됨** → `approve()` 경로만 가능

**(b) 경력 vs 학력 — WORK_EXPERIENCE subType 확정:**
- `WORK_EDU_EXPERIENCE` 타입은 subType으로 `WORK_EXPERIENCE`(경력)와 `EDUCATION_EXPERIENCE`(학력)로 분기[^15]
- Kafka recordKey(`77952:USER_DATA:...`)에는 subType 정보가 없음
- 그러나 `WORK_EXPERIENCE` 분기만 `workExperienceApplicationService.update()`를 호출 → `user_work_experience` 테이블에 write
- `EDUCATION_EXPERIENCE` 분기는 `educationExperienceApplicationService` → 다른 테이블에 write
- Envers audit에서 변경된 테이블이 **`user_work_experience`** → `WORK_EXPERIENCE` subType만 가능

**(c) 종합:**
- 토픽 `approval.process` + 실제 `user_work_experience` 데이터 변경 → `approve()` + `WORK_EXPERIENCE` subType 경로 확정
- 즉 **개인정보 변경신청(경력) 승인 후 자동 반영**

## 다음에 같은 문의가 오면

1. **먼저 확인**: `user_work_experience_audit` + `core_revinfo` JOIN 조회로 INSERT 시점 확인
   ```sql
   SELECT aud.id, FROM_UNIXTIME(ri.revtstmp/1000) AS created_at, aud.revtype, aud.begin_date, aud.end_date
   FROM flex.user_work_experience_audit aud
   JOIN flex.core_revinfo ri ON aud.rev = ri.rev
   WHERE aud.customer_id = ? AND aud.user_id = ?
   ORDER BY aud.id, ri.revtstmp;
   ```
2. **입력 경로 판별**: INSERT 시점의 access log에서 `POST /api/v2/core/work-experiences` 검색
   - 직접 API → 관리자 또는 본인 직접 입력
   - `/action/v2/core/work-experiences/bulk` → 대량 입력 (엑셀 등)
   - 워크플로우 경유 → 개인정보 변경신청
3. **요청자 확인**: audit log의 `actor.userIdHash` ≠ target `userIdHash`이면 관리자 입력

- `user_work_experience` 테이블에 `db_created_at`/`db_updated_at` 컬럼 존재 (DB 트리거, JPA Entity에는 미선언)[^11]
