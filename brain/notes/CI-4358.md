### Phase 5: DB 쿼리 구성 — Entity 기반 검증 (5분)

> 여기서 중요한 교훈이 있다.

**첫 번째 시도 (잘못된 접근)**: CI-4126 노트의 SQL을 기반으로 쿼리를 바로 작성했다.
```sql
SELECT ... FROM objective_v3 root_o WHERE root_o.customer_id = 193848 ...
```
→ **문제**: Entity 출처 없이 테이블/컬럼명을 "알고 있다고 생각"하고 작성. CLAUDE.md 규칙 위반.

**두 번째 시도 (올바른 접근)**: `ops-db-query-builder` 스킬을 통해 체계적으로 진행:

1. **Entity 탐색**: `ObjectiveV3JpaEntity.kt` 에서 `@Table(name = "objective_v3")` 확인
2. **컬럼 매핑 확인**: `@Column` 이 모두 생략 → Spring naming strategy(camelCase → snake_case) 추론
3. **DB 스키마 교차 검증**: `db_describe`로 실제 컬럼명 확인 — `customer_id`, `cycle_year`(int), `formatted_cycle`, `root_objective_id`, `deleted_at` 등 존재 확인
4. **database 특정**: `flex.objective_v3` → 존재하지 않음. `db_show_config` 캐시에서 `flex_goal.objective_v3`임을 확인
5. **허용 목록 추가**: `prod-allowed-tables.json`에 `objective_v3` 추가

**세 번째 시도 (코드 로직 검증)**: 사용자 피드백으로 추가.
- `ObjectiveV3JpaRepository.kt`의 `findRootObjectives` SQL 구조를 직접 읽어 쿼리 접근이 비즈니스 로직과 일치하는지 확인
- `ObjectiveSearchServiceImpl.kt`의 `filterRootObjective` 메서드에서 hitMap 생성 로직과 필터링 조건 확인
- 결과: 내 쿼리가 실제 SQL과 동일한 경로를 역추적하고 있음을 확인

**교훈**:
- 과거 노트의 SQL을 그대로 쓰면 테이블명/컬럼명/database가 변경됐을 때 실패한다
- Entity → DB 스키마 교차 검증 → 인덱스 확인 → 코드 로직 검증 순서를 지켜야 한다
- "코드에서 확인했는가?"라는 질문은 항상 유효하다. DB 쿼리가 맞더라도 비즈니스 로직과의 정합성을 확인해야 한다

### Phase 6: DB 검증 실행 (10분)

-- 2026 자식을 가진 2025 root 목표 찾기
SELECT HEX(root_o.id), root_o.title, root_o.formatted_cycle, root_o.cycle_year
FROM flex_goal.objective_v3 root_o
WHERE root_o.customer_id = 193848
  AND root_o.deleted_at IS NULL
  AND root_o.cycle_year = 2025
  AND root_o.id IN (
    SELECT DISTINCT o.root_objective_id
    FROM flex_goal.objective_v3 o

      AND o.cycle_year = 2026

| `019D6A71...` | 안전 교육 강화로 무사고 작업 환경 확립... | 2026_YEARLY |

### Phase 9: 트리 뎁스 검증 — 스크린샷과의 불일치 해소 (20분)

> 이 Phase는 사용자의 관찰("스크린샷에서 생산부문 하위에 25년 하위 목표만 보이는데 왜 26년은 안 보이지?")에서 시작됐다. **검증이 끝났다고 생각한 시점에서 나온 질문**이었고, 이것이 고객 안내의 정확성을 크게 높였다.

**문제 인식**: Phase 6에서 `root_objective_id`로 2026 자식을 찾았지만, `root_objective_id`는 트리의 최상위 root를 가리키는 필드이지 직접 부모(`parent_id`)가 아니다. 따라서 2026 목표가 생산부문의 직접 자식인지, 아니면 깊은 하위에 있는지 확인하지 않았다.

**추가 쿼리 1**: 생산부문 root 하위 전체 트리를 `parent_id`와 함께 조회
- 결과: 50건 조회, **모두 2025년** — 생산부문 직접 자식은 전부 2025

**추가 쿼리 2**: 2026 자식 2건의 `parent_id` JOIN으로 직접 부모 확인
```sql
SELECT child.title, child.formatted_cycle, parent.title AS parent_title, parent.formatted_cycle AS parent_cycle
FROM flex_goal.objective_v3 child
JOIN flex_goal.objective_v3 parent ON child.parent_id = parent.id
WHERE child.id IN (UNHEX('019D674E...'), UNHEX('019D6A71...'))
```
- 결과:
  - 2026 "무재해 0건/년" → 직접 부모: 2025 "위험요인 발굴 활성화를 통한 안전사고 예방"
  - 2026 "안전 교육 강화..." → 직접 부모: 2025 "안전 교육 강화로...생산2팀"

**실제 트리 구조**:
```
2025 "생산부문" (root, 회색)
  └── ... (여러 뎁스의 2025 목표들)
        ├── 2025 "위험요인 발굴 활성화를 통한 안전사고 예방"
        │     └── 2026 "무재해 0건/년"  ← 깊은 뎁스
        └── 2025 "안전 교육 강화로...생산2팀"
              └── 2026 "안전 교육 강화로..."  ← 깊은 뎁스
```

**영향**: 고객 안내가 "회색 목표 바로 하위를 펼치면 26년 목표가 있다"에서 "깊은 하위 어딘가에 26년 목표가 있어서 최상위까지 회색으로 올라온 것"으로 보정됨. 정확한 위치(목표명)까지 안내 가능해져 고객이 직접 찾을 수 있게 됐다.

**교훈**: `root_objective_id`와 `parent_id`는 다르다. cross-year 트리 검증 시 **반드시 `parent_id`로 직접 부모-자식 관계를 확인**해야 고객 안내가 정확해진다. "트리에 존재한다"와 "어디에 존재한다"는 다른 질문이다.

## 원인 분석

**CI-4126과 동일한 cross-year 트리 스펙.**

고객(종근당바이오)이 2025년 root 목표 "생산부문" 하위에 2026년 자식 목표("무재해 0건/년", "안전 교육 강화...")를 배치한 구조. `root-objectives` API는 2026 cycle 필터링 시 이 2025 root를 `hit=false`로 포함하여 관계성을 표시. FE에서 회색으로 구분.

## 해결

**스펙 확정 → CS 안내.** 코드 수정 불필요.

고객 안내 포인트:
1. 생산부문 하위 목표를 깊이 펼쳐보면 26년 목표가 있음 — 정확한 위치는 "위험요인 발굴 활성화를 통한 안전사고 예방" 하위의 "무재해 0건/년", "안전 교육 강화로...생산2팀" 하위의 "안전 교육 강화로..."
2. 이 26년 목표를 25년 트리 밖으로 분리(상위 목표 변경)하면 생산부문이 더 이상 표시되지 않음
3. 회색 = "직접 필터에 해당하지 않지만, 하위 어딘가에 해당하는 목표가 있어서 관계성 표시"

⚠️ **주의**: 26년 목표는 생산부문 바로 아래가 아니라 여러 뎁스 아래에 있다. "바로 하위를 펼치면 보인다"는 안내는 부정확 — 고객이 첫 뎁스만 펼쳐보고 "없는데요?"라고 할 수 있다.

### 개선 포인트

1. **과거 노트의 SQL을 그대로 쓰지 말 것** — 항상 Entity 출처부터 확인. 과거 노트는 "어떤 테이블을 봐야 하는지"의 힌트일 뿐, 그대로 실행 가능한 SQL이 아니다.
2. **DB 쿼리 구성 시 코드 로직 검증을 기본 단계로 포함할 것** — "이 쿼리가 비즈니스 로직과 같은 경로를 따르는가?"를 항상 확인.
3. **access log 응답 파싱 시 필드명을 가정하지 말 것** — `list` vs `objectives` 같은 차이는 실제 데이터를 먼저 원본 확인 후 파싱해야 한다.
4. **검증은 항상 이중으로** — DB로 구조를 확인하고, access log로 실제 동작을 확인하는 것이 가장 확실하다. 한쪽만으로 "확정"하지 말 것.

### 1. 목표 트리 시각화 운영 도구

**동기**: 이번 조사에서 가장 시간이 걸린 부분은 "2026 목표가 트리 어디에 있는지" 찾는 것이었다. `root_objective_id`로 존재를 확인하고, `parent_id` JOIN으로 직접 부모를 찾고, 전체 트리 구조를 머릿속으로 조립해야 했다. 고객사의 트리가 깊고 복잡할수록 이 과정이 느려진다.

**제안**: customer_id를 입력하면 해당 고객의 목표 트리를 시각화하는 운영 도구.

- 입력: customer_id + (선택) cycle 필터
- 출력: 트리 구조 시각화 (뎁스 들여쓰기 or 그래프)
  - 각 노드에 `title`, `cycle_year`, `hit` 여부 표시
  - cross-year 노드(자신과 다른 cycle의 자식/부모를 가진 노드) 하이라이트
- 구현 방식 후보:
  - **A. SQL + 텍스트 트리**: `ancestor_id_json` (JSON 배열)을 활용하면 뎁스를 재귀 없이 계산 가능. 가장 가벼움
  - **B. 온콜 스킬**: `ops-goal-tree` 같은 스킬로 DB 조회 → 텍스트 트리 출력. 현재 인프라 내에서 바로 가능
  - **C. 웹 대시보드**: Metabase 커스텀 쿼리 or 별도 내부 도구. 팀 공유 용이하나 구축 비용 있음

**기대 효과**:
- cross-year 이슈 조사 시간: ~20분 → ~2분 (트리 구조 파악)
- CS 안내 정확도 향상: 정확한 위치를 바로 안내 가능
- 반복 VOC 대응 시 고객 데이터를 매번 SQL로 파편적으로 보지 않아도 됨

### 2. cross-year 트리 탐지 SQL 템플릿

COOKBOOK에 재사용 가능한 SQL 템플릿 등록:

```sql
-- cross-year 트리 탐지: customer_id를 바꿔서 사용
-- Step 1: cross-year root 찾기
SELECT HEX(root_o.id), root_o.title, root_o.formatted_cycle
FROM flex_goal.objective_v3 root_o
WHERE root_o.customer_id = {고객ID}
  AND root_o.deleted_at IS NULL
  AND root_o.cycle_year = {이전연도}
  AND root_o.id IN (
    SELECT DISTINCT o.root_objective_id
    FROM flex_goal.objective_v3 o
    WHERE o.customer_id = {고객ID}
      AND o.deleted_at IS NULL
      AND o.cycle_year = {올해연도}
      AND o.root_objective_id IS NOT NULL
  );

-- Step 2: 올해 자식의 정확한 위치 (직접 부모) 확인
SELECT child.title AS child_title, child.formatted_cycle,
       parent.title AS parent_title, parent.formatted_cycle AS parent_cycle
FROM flex_goal.objective_v3 child
JOIN flex_goal.objective_v3 parent ON child.parent_id = parent.id
WHERE child.customer_id = {고객ID}
  AND child.deleted_at IS NULL
  AND child.cycle_year = {올해연도}
  AND child.root_objective_id = UNHEX('{Step1에서 찾은 root ID}');
```

→ 항상 Step 1 + Step 2를 세트로 실행. Step 1만 하면 "어디에 있는지" 모름.
