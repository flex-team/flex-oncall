### 성격
퇴사자가 승인 라인에 잔존하여 워크플로우 문서 3건의 승인이 차단된 운영 처리 요청. 고객이 제품의 "퇴직한 승인자 교체" 기능으로 자체 해결을 시도했으나 해당 구성원이 조회되지 않아 CS를 통해 강제 승인을 요청한 케이스[^1]. 문서번호 형식(2024-xxxx)과 라벨(워크플로우)로 보아 APPROVAL_DOCUMENT 카테고리의 workflow 문서[^10].

### 영향 범위
- 보고된 범위: 문서 3건, 1개 회사(브레인커머스)[^1]
- 진행형 여부: 아님 — 퇴사자의 기존 미승인 건이 잔존하는 일회성 상태. 신규 문서는 퇴사자를 거치지 않으므로 추가 피해 없음
- 이 패턴은 반복 인입 중 (CI-3951, CI-4228, CI-4266, CI-4286, CI-4244 — 5건 이상 누적)[^2][^3]

### 긴급도
일반 — 금전적/법적 영향 없음. 데이터 오염 진행 중 아님. 업무 차단이나 3건의 문서 처리가 지연되고 있으므로 당일 내 처리가 적절.

### 조사 방향
COOKBOOK 승인 F1 플로우(비활성 사용자 승인건 강제 승인) 적용[^11]. 절차:
1. 대상 사용자(퇴사자) 특정 — `view_user` 상태 확인
2. `approval_process`에서 문서 3건의 미처리 상태(ONGOING) 확인
3. 고객 동의 확인 후 `bulk-approve-for-user` API 호출 (categories: APPROVAL_DOCUMENT)
4. 워크플로우 문서이므로 `sync-with-approval`로 workflow_task 동기화 필요[^9]

## 원인 분석

**verdict**: `operational`

`yangyang0613@braincommerce.com` (user_id: 584593)이 시스템에서 hard-delete되어 승인 라인 step 0에서 PENDING으로 잔존, 문서 3건의 승인 흐름을 차단[^12]. "퇴직한 승인자 교체" 기능은 퇴사 이벤트 기반이므로 삭제된 구성원은 조회 불가[^8]. CI-4228과 동일한 패턴.

## 해결 대기

고객 응답에 따라 2가지 옵션:

**옵션 1: step 0만 강제 승인**
```
POST /api/operation/v2/approval/process/customers/135020/users/584593/bulk-approve-for-user
Body: { "categories": ["APPROVAL_DOCUMENT"] }
```
→ step 1은 `syoon@braincommerce.com`(557355)이 직접 승인

**옵션 2: 전체 강제 승인**
- 위 API 실행 후, 추가로 userId=557355에 대해서도 동일 API 실행
- 이후 3건 각각 `sync-with-approval` 호출하여 workflow_task 동기화[^9]
