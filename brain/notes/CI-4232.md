### 이슈 B: dev 재현 — ValuesContinuation null 직렬화 버그

> 💡 **판단 근거**: dev trace log에서 null employeeNumber 문서 구간 진입 시 커서 고정 확인[^8] → `ValuesContinuation` 직렬화 코드에서 null→"null" 변환 확인[^9] → OpenSearch가 문자열 "null"과 실제 null을 다르게 처리하여 커서가 동일 구간을 반복. **이 고객(211343)에서는 사번 null 구성원이 없어 해당하지 않음.**

<details>
<summary>📋 dev 버그 메커니즘 상세 (5 Whys)</summary>

1. **왜 continuation token이 고정되는가?** → search_after가 같은 문서 구간을 반복 반환하기 때문
2. **왜 같은 구간을 반복하는가?** → search_after에 전달된 값이 실제 sort position을 가리키지 못하기 때문
3. **왜 position을 가리키지 못하는가?** → null sort value가 문자열 "null"로 변환되어 전달되기 때문
4. **왜 "null" 문자열로 변환되는가?** → `ValuesContinuation.print()`에서 `joinToString()`이 null을 "null"로 변환하기 때문[^9]
5. **왜 이것이 문제인가?** → OpenSearch에서 keyword 필드의 null(missing value)은 정렬 순서 끝에 위치하지만, 문자열 "null"은 알파벳 순서에서 다른 위치 → **커서가 null 구간 앞으로 되돌아감**

</details>

## 해결

PR: https://github.com/flex-team/flex-core-backend/pull/5094 (draft)

1. **옵션 A — null sentinel 직렬화** (커밋 1-3)
   - `EsContinuation.print()`: null → `__NULL__` sentinel로 직렬화
   - `ValuesContinuation.parse()`: `__NULL__` → null로 역직렬화
   - 하위 호환성: 기존 토큰의 `"null"` 문자열은 그대로 유지 (세션 만료 시 자연 해소)

2. **옵션 B — keyword 필드 missing=_last 추가** (커밋 4, 김희찬님 리뷰 반영)
   - `EMPLOYEE_NUMBER`, `DISPLAY_NAME` 정렬에 `missing="_last"` 추가
   - 양방향(ASC/DESC) 모두 null-last ordering 보장
   - 이성환님 지적: String 필드라 Min을 정의하기 애매했던 것이 과거 해결 못한 이유[^18]

임준규님 확인: Relation 스쿼드 시절 known issue였으나 해법을 찾지 못했던 건[^17]

## 발견한 스펙/제약

| 발견 | 설명 |
|------|------|
| OpenSearch `search_after` null 처리 | keyword 필드의 null(missing)은 정렬 끝에 위치하며, 문자열 "null"과 다른 위치에 정렬됨 |
| `ValuesContinuation` 직렬화 한계 | `joinToString()`으로 tokenKeys를 직렬화하면 null→"null", 타입 정보 소실 발생 |
| 구성원 검색 정렬 구조 | sort = [EMPLOYEE_NUMBER_RAW, (script sort), USER_ID]. script sort는 퇴사 상태 필터에 따라 포함/미포함 |
| 겸직 인원 데이터 구조 | OpenSearch 문서는 userId 1건. position이 복수(겸직)여도 문서는 1건. 하지만 FE에서 position 기반 key 사용 시 중복 가능 |
