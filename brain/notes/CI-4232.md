**OpenSearch `search_after`[^13] 페이지네이션에서 null sort value가 문자열 "null"로 변환되어 커서가 진행하지 못하는 버그.**

| 3 | null sort value의 직렬화/역직렬화 오류 | trace log의 sort values 분석 | ✅ 확정 |

<details>
<summary>📋 버그 메커니즘 상세 (5 Whys)</summary>

1. **왜 continuation token이 고정되는가?** → search_after가 같은 문서 구간을 반복 반환하기 때문
2. **왜 같은 구간을 반복하는가?** → search_after에 전달된 값이 실제 sort position을 가리키지 못하기 때문
3. **왜 position을 가리키지 못하는가?** → null sort value가 문자열 "null"로 변환되어 전달되기 때문
4. **왜 "null" 문자열로 변환되는가?** → `ValuesContinuation.print()`에서 `joinToString()`이 null을 "null"로 변환하기 때문[^9]
5. **왜 이것이 문제인가?** → OpenSearch에서 keyword 필드의 null(missing value)은 정렬 순서 끝에 위치하지만, 문자열 "null"은 알파벳 순서에서 다른 위치 → **커서가 null 구간 앞으로 되돌아감**

</details>

## 해결안 / 조사 방향

### 즉시 대응 (표면 원인 해결)

없음 — 코드 수정 필요

### 근본 해결 (근본 원인 해결)

**옵션 A: `ValuesContinuation`에서 null을 특수 sentinel로 직렬화**
- `null` → `"__NULL__"` 등 sentinel 값으로 직렬화, 역직렬화 시 다시 null로 복원
- 장점: 기존 구조 변경 최소
- 단점: sentinel 값이 실제 데이터와 충돌할 가능성 (매우 낮지만)

**옵션 B: sort 구성 시 null 방지 — `missing` 파라미터 추가**
- `EMPLOYEE_NUMBER_RAW` sort에 `missing` 파라미터 설정 (예: `_last` 또는 `""`)
- OpenSearch가 null 대신 지정된 값을 sort value로 사용 → null이 tokenKeys에 들어가지 않음
- 장점: 근본적 해결, sort value 타입 일관성 보장
- 단점: `missing` 값에 따라 정렬 순서가 변경될 수 있음

**옵션 C: A + B 동시 적용** (권장)
- `missing` 파라미터로 null 발생을 방지하면서, `ValuesContinuation`에서도 null safety 추가
- 방어적 프로그래밍으로 동일 패턴의 재발 방지

## 발견한 스펙/제약

| 발견 | 설명 |
|------|------|
| OpenSearch `search_after` null 처리 | keyword 필드의 null(missing)은 정렬 끝에 위치하며, 문자열 "null"과 다른 위치에 정렬됨. `missing` 파라미터로 null 대체값 지정 가능 |
| `ValuesContinuation` 직렬화 한계 | `joinToString()`으로 tokenKeys를 직렬화하면 null→"null", 타입 정보 소실 발생. 현재 null을 처리하는 로직 없음 |
| 구성원 검색 정렬 구조 | sort = [EMPLOYEE_NUMBER_RAW, (script sort), USER_ID]. script sort는 퇴사 상태 필터 조건에 따라 포함/미포함 |
