# CI-4353: 캡스 외부 전송 후 퇴근 기록 미반영

## 증상

- 와이트라이브(Customer ID: 220398) je.lee@ytribe.co.kr
- 4/6(월) 캡스 퇴근 타각(18:15 KST) 후, 4/7 수동 전송했으나 퇴근 기록 미반영
- 출근 기록만 남아있는 상태

## 원인 분석

### 빠른 확인: Metabase

이 유형의 이슈는 DB 직접 조회 전에 Metabase로 먼저 확인 가능하다:

1. **[Metabase #3565](https://metabase.dp.grapeisfruit.com/question/3565)** — 세콤/캡스 이벤트 조회
   - customer_id + user_id + 날짜로 필터 → 이벤트 수신 순서·시각 즉시 확인
2. **[Metabase #4716](https://metabase.dp.grapeisfruit.com/question/4716)** — 위젯 draft 이벤트
   - fail_message, registered_id 확인 → 처리 성공/실패 여부 즉시 판별

→ 두 화면만으로 "이벤트가 왔는지 / 처리됐는지 / 왜 실패했는지"를 빠르게 파악할 수 있다.

### 이벤트 수신·처리 타임라인

> 핵심: **퇴근이 실시간으로 먼저 도착했으나, 출근이 아직 없는 상태여서 validation 실패.**
> 이후 수동 전송으로 출근이 뒤늦게 등록됐지만, 이미 실패한 퇴근은 재처리되지 않았다.

```

                                    ✗ flex에 실시간 전송 안 됨              ← 출근 미수신
                                      (transmitter polling 누락 추정)

                                       재처리 대상이 아님                  ← 퇴근은 여전히 누락

최종 상태:  4/6 출근 ✅  |  4/6 퇴근 ❌ (draft fail_message 남은 채 방치)
```

## 영향 범위

- 이 건은 개별 사용자 1명의 특정 날짜(4/6) 퇴근 기록 누락
- 캡스 연동 시 출근 이벤트가 실시간 전송되지 않고 퇴근만 먼저 도착하는 경우 동일 증상 발생 가능

## 해결

사용자(또는 관리자)가 4/6 퇴근 시간(18:15)을 수동 입력하면 해결.
- "내 근무" 화면에서 4/6 퇴근 기록 직접 입력
- 또는 관리자가 근무기록 관리에서 해당 구성원의 퇴근 기록 추가

## 비고

- 쿡북 체크리스트 #3 / F2(수동 전송 후 미반영 — 역순 수신) 패턴과 유사하나, 세콤이 아닌 캡스이고 "실시간 + 수동 전송 혼합"이라는 점이 다름
- 근본적으로 draft 실패 건에 대해 후속 START 등록 시 재처리하는 메커니즘이 없음 (시스템 한계)

### 개선 논의 (이성환)

- 이벤트 전달 순서가 역전되어 들어올 때, 고객사에서 세콤/캡스 데이터 기반으로 리커버리할 수 있는 방법이 현재 없음 → 개선 필요
- 역순 수신된 이벤트(퇴근 이후 뒤늦게 도착한 출근)를 삭제할 수 있는 운영 도구를 개발하면, 삭제 후 고객에게 수동 재전송을 제안하는 워크플로우가 가능
