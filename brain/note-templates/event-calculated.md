# 이벤트 기반 계산형 추가 섹션

> **해당 도메인**: 근태/휴가, 스케줄링, 교대근무, 연차촉진
>
> **특성**: 행동을 row로 적재 → 적재된 데이터 해석 → 해석 결과 계산.
> DB 레코드 ≠ 최종 결과. 기존 이벤트를 UPDATE하지 않고 새 이벤트를 추가하여 상태를 변경한다.
>
> **핵심**: "값이 이상하다" 신고 시 DB만 봐선 안 된다.
> 입력 이벤트(raw data) + 설정(근무유형/휴일/정책) + Calculator 로직의 조합으로 결과가 산출된다.

## 추가 섹션 (영향 범위 ~ 해결 사이에 삽입)

```markdown
## 소급 영향 범위

- 변경/수정이 적용되는 기간 (언제부터 언제까지 재계산되는가)
- 영향 대상자 수/범위 (특정 회사, 특정 근무유형 등)
- 설정 변경인 경우: 결과를 저장하지 않으므로 변경 즉시 과거 조회 결과에 반영됨
  - 근무유형 변경 → 자동근무기록 초기화 + 재등록 연쇄 발생
  - 연차 정책 변경 → 과거 부여 내역 재산정 가능
  - 포괄계약 시간 변경 → 해당 산정주기의 모든 초과근로에 영향
- `applyStartDateForDistributePeriodOver` 소급 적용 기준일 확인 (null이면 HARD_OPEN_DATE 적용)

## 배포 영향도

- 계산 로직 변경의 경우 배포 즉시 과거 데이터가 변경됨 (결과가 저장되지 않으므로)
- 페이롤 정산 기간에 걸치는가 → 페이롤팀 사전 공유 필요 여부
  - `WorkMinutesCalculator`가 페이롤 라이브러리로 사용되므로 양쪽 모두 영향
- ES와 DB 결과 불일치 가능성 — consumer lag 확인
- 롤백 시: 계산 로직이 되돌아가므로 결과도 되돌아감 (별도 데이터 롤백 불필요)

## 법적/컴플라이언스 리스크

- 연차촉진 cron 실패 시 법적 의무 불이행 (근로기준법 제61조) — 비즈니스 크리티컬
- 초과근로 계산 순서 의존성 — 포괄계약 공제 시 차감 순서가 법적으로 정해져 있음
- 불이익 변경 여부 (근로자에게 불리한 방향인가)
```
