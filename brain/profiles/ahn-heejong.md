## 인물 개요

> **직군**: manager (PM) | **한줄 특성**: 복잡한 근태 스펙을 데이터 기반으로 추적하고, 압박에 흔들리지 않고 맥락을 먼저 파악하는 현실주의 PM. 모르면 모른다고 하고, 잊으면 잊었다고 한다. 활용도 낮은 기능은 빼고, 반복 운영은 기능화한다.

### 전문 영역
근태(Time Tracking) 도메인의 핵심 의사결정자. 근무유형, 초과근무(주기연장·포괄임금·3종), 교대근무, 연차/월차/맞춤휴가, 보상휴가, 휴게시간, 세콤/CAPS 연동, 연차촉진 등 근태 전 영역에 걸쳐 스펙 결정 권한과 역사적 맥락을 보유. 근로기준법 해석(법개정 시점 2020-03-31까지 구분할 수준)까지 일정 수준 파악. AI 도구(Claude Code)를 PM 업무에 실전 적용하고 팀에 전파하는 얼리어답터.

### 초과근무 / 주기연장

#### 주기연장 일귀속 스펙

**스펙/규칙**
- 교대근무(근무표 기반이 아닌)에서 주기의 소정근로시간은 다음과 같이 계산:
  - 주기에 포함된 일자 중 근무유형 기준 소정근로일인 날짜의 일 소정근로시간 합 (배치를 보지 않음)
- 소정근로일의 주기 법정근로를 초과한 시간이 주기 연장으로 잡힘
- 월요일이 주휴일 → 일요일이 휴무일인 회사에서, 적용일에 따라 주기가 잘리면 소정근로일이 하루뿐이 되어 8시간 초과분이 전부 주기연장으로 잡히는 현상 발생
- 기간별(1개월 단위) 조회에서 주기연장은 일귀속 적용 전까지 미포함. 이유: 일 귀속이 없던 버전에서는 어떤 날에 포함시킬지가 애매했기 때문
- 일귀속 적용 후에는 기간별 조회에서도 주기연장이 보임
- 근태 대시보드 쪽은 (구현 상의 이슈로) 적용일과 무관하게 전체 기간에 대해서 주기연장을 일귀속 시키는 결정을 했음

**자주 오는 케이스**
- "월 기준 초과근무 리포트와 급여정산에서 숫자가 다릅니다"
  → 기간별 조회에서 주기연장이 빠져있어서. 초과근무 리포트나 급여에서는 잡히지만 1개월 단위 조회에서는 안 나옴. 일귀속 마이그레이션이 되면 해결됨.
  → 일귀속 지정일자를 확인하고, 그 이후 기간이면 동일하게 나와야 함
- "근무유형 바꿨더니 재현이 됐다가 안 됐다가 해요"
  → 적용일을 특정 날짜로 지정하면서 주기가 잘리는 경우에만 발생. 적용일을 바꾸거나 주기가 안 잘리게 하면 재발하지 않음

**의사결정 배경**
- "이런 결정을 했었다는 걸 아예 잊고있었네" — 본인도 과거 결정을 잊을 정도로 복잡한 히스토리
- 기간별 조회에서 안 보여주기로 한 것은 "보여줬을 때 이해하기 어려운 숫자가 나올 수 있어서". 예: 지난 달 마지막 주기가 2개월에 걸쳐있으면, 근무가 없는데 연장이 4시간 보이는 상황

#### 휴무일 연장 판단 기준

**스펙/규칙**
- 휴무일에 근무했을 때 연장근무로 인정하는 기준이 두 가지 존재:
  - 구현 A: (주 소정근로시간 / 5) — "일별 다른" 구현할 때 가이드라인이 없어서 이렇게 했음
  - 구현 B: 8시간 기준 — 이후에 나온 가이드
- 현재 두 방식이 혼재해서 이슈가 반복 발생

**자주 오는 케이스**
- "휴무일 근무 시 연장이 2시간 48분으로 잡혀요. 왜 8시간 기준이 아닌가요?"
  → 해당 근무유형이 (주 소정근로시간/5) 기준을 쓰고 있을 수 있음. 주 소정 14시간이면 (14*60/5) = 168분 = 2시간 48분 초과 시 연장.

**의사결정 배경**
- 일별 다른 구현 당시 가이드가 없어서 (주소정/5)로 했고, 한참 후에 8시간 기준 가이드가 나옴. 해당 가이드가 나왔을 때 소급 수정은 안 함.

#### 포괄 계약 주/월 단위 해석

**스펙/규칙**
- 기존: 주 단위 포괄 설정해도 내부적으로 월평균주수(4.345)를 곱해서 월 단위로 환산해서 동작 (사실상 주 단위 포괄 불가)
- 변경: 주 단위 해석이 실제로 가능하게 변경. 초과근무 보상정책에 포괄계약 해석 주기 관련 설정 추가
- 포괄 계약 단위(a)와 초과근무 보상정책 주기(b)의 조합이 4가지:
  1. a: 주 / b: 주 — 이번에 추가
  2. a: 월 / b: 월 — 기존
  3. a: 주 / b: 월 — 기존 (월평균주수 곱해서 환산)
  4. a: 월 / b: 주 — 기존에 없던 케이스. (월 계약분 / 월평균주수)를 주 단위 공제분으로, 항목별 내림처리

**자주 오는 케이스**

#### 교대근무 연장근무 산정 (주 소정근로 40시간 초과)

**스펙/규칙**
- 주 소정근로 = 일하는시간/일 × 일하는요일 (예: 8.5h × 6일 = 51시간)
- 40시간 초과분 = 11시간 = 660분
- 일수로 나누면 하루당 110분 = 1시간 50분
- 하루 소정근로는 8시간30분 - 1시간50분 = 6시간40분으로 취급
- 8시간30분 근무하면 1시간50분이 연장 (30분은 연장, 1시간20분은 법내연장)

**자주 오는 케이스**
- "왜 4시간07분부터 연장근무로 잡히나"
  → 위 소정근로 계산 로직에 의한 것. Notion 문서 참조: `flexnotion/40-13b0592a4a92802686b1d628d90b7afd`

### 연차 / 휴가

#### 연차·월차 미리쓰기 잔여 반영 차이

**스펙/규칙**
- 연차 미리쓰기: 현재 잔여에 **미반영** (내년 부여받을 연차에서 쓰는 거라 현재 잔여를 깎지 않음)
- 월차 미리쓰기: 현재 잔여에 **반영** (당겨쓰기 포함)
- 이 차이가 고객 혼란의 주 원인
- 안희종 입장: "이 이슈 몇 번 맞으면서 하나로 맞춰야한다는 쪽으로 생각이 기우는 중. 맞춘다면 연차 쪽으로 맞춰야하지 않을까 (연차, 월차 모두 미리쓰기는 잔여를 깎지 않도록)"

**자주 오는 케이스**
- "휴가개요 연차 카드에 -4일로 나오는데 -5일이어야 하지 않나요?"
  → 내년 1월 사용분(연차 미리쓰기)은 카드에 미반영. 월차 미리쓰기분만 반영되어 차이 발생.
  → 연차 상세에서 '조정 적용일'을 미래로 밀면 같은 숫자가 나오는지 확인시켜보면 이해에 도움
- "왜 연차는 안 깎이는데 월차는 깎이나요?"
  → 현재 스펙 차이. 미래에 부여받을 예정인 연차에서 사용될 내역이라 현재 잔여에 반영하지 않는 것이 연차 쪽 스펙. 월차는 당겨쓰기를 포함해서 보여줌.

**의사결정 배경**
- 당겨쓰기/미리쓰기 개발 당시 "현재 화면으로는 슬릭한 경험은 불가능하다는 결론"에 다다랐음
- 사용분에서 연차/월차 구분을 표시하면 정책 변경이나 과거 사용분 밀어넣을 때마다 바뀌어서 오히려 혼란 유발 → 부여/잔여/소멸에서만 구분하기로 결정

### 보상휴가

#### 보상휴가 부여건 매핑 로직

**스펙/규칙**
- 휴가 사용 시, 가용 부여건이 여러 건이면:
  1. 소멸일이 빠른 부여건부터 꺼내씀
  2. 소멸일 동일 시 부여일이 빠른 것 우선
- 한번 매핑된 부여건은 취소 전까지 고정됨. "앞에 걸 취소했더니 다음 부여건으로 넘어갔다" 같은 동작이 **아님**
- 취소 후 재사용하지 않으면 매핑된 부여건은 그때 정해진 것으로 고정

**자주 오는 케이스**
- "1/2 휴가 취소했더니 2/13으로 잠금이 넘어갔어요"
  → 1/2 사용 시점에 가용 부여건이 2/25 부여건 하나뿐이어서 그걸 썼음. 1/2을 취소해도 2/13, 3/6 등 다른 날짜의 사용도 같은 2/25 부여건에서 꺼냈으므로 잠금이 연쇄적으로 걸림. 해당 부여건과 매핑된 사용을 전부 취소해야 잠금 해제 가능.
- "보상휴가 회수했는데 잠금이 안 풀려요"
  → 회수만으로는 Lock이 안 풀림. 해당 부여건에 연결된 사용 내역을 모두 취소한 후 회수해야 함. 부분 사용 상태에서는 회수 불가.

**의사결정 배경**
- calibration을 내 근무에 맞추려고 적용 → 기존 부여건 매핑 이슈 발생 → calibration 롤백 → 주기연장 일귀속이 되면서 내 근무와 초과근무 쪽이 자연스럽게 일치하게 되어 calibration 없이도 해결됨
- 보상휴가 계산기에서 정책이 2개 이상 혼용되는 케이스(부여필요시간은 주기연장일귀속 on, 기부여시간은 off)는 고려하지 않음. 이유: 주기연장일귀속 전환이 미래로만 가능했기 때문에 거의 발생하지 않을 것으로 예상. 발생 시 운영 대응.

### 출퇴근 / 타각

#### 시작·종료 시각 vs 출근·퇴근 시각

**스펙/규칙**
- 시작/종료 시각: 현시점 기준 타임블록의 from-to (인정근무 기준)
- 출근/퇴근 시각: 실제 타각 기록 (실시간 기록)
- 실시간 위젯에서 등록하는 시점에는 둘이 동일
- 타임라인 편집하면 출퇴근 시각은 그대로이고 시작/종료만 달라짐
- 연차가 깔려있으면 그 위를 덮으면서 보여줌. 연차 취소 시 아래 깔려있던 실시간 기록이 드러남

**자주 오는 케이스**
- "툴팁에 오후 7시 50분이 뜨는데 왜 19시로 잘려서 나오나요?"
  → 인정 근무에 반영되는 시점에는 기존처럼 19시로 잘려서 남고, 실시간 기록 레이어에서 봐야 19시 50분 확인 가능 (인정 근무 레이어에선 19시 50분 확인 불가)

### 맞춤휴가

#### '사용 가능' vs '부여' 상태 구분

**스펙/규칙**
- 근속 시, 반복, 신청 시, 연차 소진 시 부여되는 휴가는 사실 '부여'가 아닌 '사용 가능' 상태

- "내 휴가 내역에 부여가 안 보인다"
  → 신청 시 지급 등은 사용 전까지 내역에 미노출이 정상
- "부여자가 구성원 이름으로 1월 1일 오전 12:00에 찍힌다"
  → 매년 N일 지급의 경우 사용 시점에 부여 내역이 생기며, 해당 연도 1/1 기준으로 찍힘
