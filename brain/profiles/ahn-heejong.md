## 인물 개요

### 전문 영역
타임트래킹(근태관리) 도메인의 핵심 PM. 출퇴근, 근무유형(고정/시차/선택적/교대), 초과근무, 휴가/연차, 승인, 리포트 전 영역에 걸쳐 스펙을 정의하고 의사결정을 내린다. 2021년 초에는 PM과 FE 개발자를 겸임하여 직접 React/TypeScript 코드를 작성했으며, 이후 PM 전업으로 전환했으나 프론트엔드 코드 수준의 기술적 이해를 바탕으로 의사소통한다.

카드 밀도 기준 상위 도메인:
1. **승인** — 요승설, 승인라인 resolve, 대결, 자동승인, 승인 개선 로드맵
2. **초과근무/연장** — 주기연장 일귀속, 포괄임금, 3종/7종, 연장근무 단위(올림/내림)
3. **휴가/연차** — 부여/소멸/촉진, 미리쓰기/당겨쓰기, 보상휴가, 미사용연차수당
4. **교대근무** — 스케줄 게시, 근무표, 근무지/근무조, 주휴일 밸리데이션
5. **출퇴근/위젯** — 실시간, 정시, 세콤/캡스 연동, 위젯 동작

### 의사결정 원칙 (3건 이상 반복 확인된 패턴)

1. **"반복 운영 대응하지 말고 기능화하라"**
   - 승인권자 퇴사 시 수동 데이터 수정 요청을 거절하고, 오프보딩 체크리스트/대결 기능으로 해결하는 방향을 주장 ([출처1](https://flex-cv82520.slack.com/archives/CRU35U9FC/p1683187890799249), [출처2](https://flex-cv82520.slack.com/archives/CRU35U9FC/p1684124954950899), [출처3](https://flex-cv82520.slack.com/archives/CRU35U9FC/p1668411683086529))
   - "30분, 1시간 걸리는 단순 처리들도 모이면 제품 개발 사이클에 꽤 심각하게 지장을 줌"

2. **"고객 데이터 직접 조작은 지양한다"**
   - 승인 내역/결재 데이터를 직접 수정하면 알림 발생 등 이상한 경험 + 법적 리스크 존재 ([출처1](https://flex-cv82520.slack.com/archives/CRU35U9FC/p1711355064260009), [출처2](https://flex-cv82520.slack.com/archives/CRU35U9FC/p1682560736944189))
   - 승인 대행은 사문서 위조에 해당한다는 원칙적 입장

3. **"의도를 크게 벗어나는 사용례의 편의성은 시급도 높지 않다"**
   - 재택 공유를 공지사항으로 쓰려는 고객 요구에 "한 번 해주면 앞으로 더 요구할 것이 자명" ([출처](https://flex-cv82520.slack.com/archives/CRU35U9FC/p1669966029504149))
   - 비공식 API(크롬 확장) 사용에 대해 "별도 고지 없이 차단 가능" 공식 입장

4. **"버그는 의도하지 않은 이슈만이 아니라, 잘못된 가정으로 경험이 이상한 경우도 버그다"**
   - 같은 책임감으로 접근해야 한다는 Product Philosophy ([출처](https://flex-cv82520.slack.com/archives/CRU35U9FC/p1670568501652799))

5. **"완벽보다 빠른 출시 후 개선"**
   - 리프레이밍 프로젝트에서 "일단 이 정도로도 내보낼 수 있고 이어서 고도화 가능하지 않을까" ([출처](https://flex-cv82520.slack.com/archives/C05PQ22NQS1/p1735795813159109))
   - 겹침 해석 규칙에서 validation 표기만 먼저 적용 후 느낌을 보자는 판단

6. **"법적 리스크가 있는 영역은 신중하게"**
   - 소정근로시간 표기 변경에 대해 "가볍게 대응했다가 법적 리스크를 만들 수 있으므로 노무 검토와 함께 신중하게 대응" ([출처](https://flex-cv82520.slack.com/archives/CRU35U9FC/p1736231539516789))
   - 연차촉진 휴직자 버그에서 보상 범위를 CS팀이 결정하도록 옵션만 제시

### 응대 패턴

1. **스펙 판단 요청**: 시스템 동작 원리를 구조적으로 설명(API 이름, 필드명, 로직 흐름 포함) → 의도된 동작인지 버그인지 명확히 판정 → 버그이면 우선순위와 영향 범위를 동시에 제시
2. **기능 요청**: "인지하고 있고 우선순위 고려 중, 구체적 시점 약속은 어렵다" → 내부적으로 로드맵 Bet/Cycle에 배치
3. **긴급 이슈**: "온도감 중인 이슈에 대해서는 다음날 오전 중으로 확인하고 예상 해결 일정을 말씀드리는 것이 프랙티스" ([출처](https://flex-cv82520.slack.com/archives/CRU35U9FC/p1669906055018289))
4. **고객 온도가 높은 케이스**: 고객의 온도와 실제 버그의 영향도를 분리해서 판단. 공감 표현은 하되 원칙은 유지

### 2. 초과근무/연장

#### 2-1. 주기 단위 연장근무 — 리포트 귀속일
- 주기 단위 연장근무 발생 시 리포트에서는 주기 말일에 발생했다고 처리
- 일 단위 연장은 귀속일이 명확하지만 주기 단위는 특정 불가
- 고정 근무제 통상근로자: 일 8시간 초과(일 단위 연장) + 주 40시간 초과(주기 단위 연장)
- **출처**: [스레드](https://flex-cv82520.slack.com/archives/CRU35U9FC/p1725266388467489)

#### 2-2. 연장근무 승인 — 일 단위 변경 구조의 한계
- 근무 등록이 무조건 일 단위로 변경되는 구조
- "일부만 받아들인다"는 개념이 없어 전체 승인/반려만 가능
- 반려 시 마지막으로 기록된 상태로 되돌림 (출퇴근 시각 정보는 유실됨)

#### 2-3. 포괄 임금 계약에서 목표 근무시간 / 초과근무 표시

- 변경 이력: 주 단위 포괄계약은 내부적으로 월 단위로 변환하여 처리
- **출처**: [스레드](https://flex-cv82520.slack.com/archives/CRU35U9FC/p1712569860691069)

### 1. 리프레이밍 뷰 전환 UX — 헤비유저 피드백 vs PM 판단
- **상황**: 세트렉아이 헤비유저가 "주/주기 뷰의 가로축-세로축이 바뀌는 것이 어이없다"고 세 차례 피드백
- **안희종 입장**: "보편적인 유저 목소리는 아닌 것 같다. 뷰타입과 조회기간을 커플링시킨 건 의도적이고 opinionated friction이라 이 의견 하나 때문에 바꾸는 것은 반대"
- **결과**: segmented control로 한 번 클릭 전환은 수용, 근본 구조는 유지
- **출처**: [스레드](https://flex-cv82520.slack.com/archives/C05PQ22NQS1/p1736412576164199)

### 2. 시몬스 휴일대체 — 배포 2주 전 기능 변경 요청
- **상황**: 기능 배포 2주 전 업데이트 노트도 사전/사후 두 번 발송했으나, 고객이 수령 못한 채 "오늘 테스트해보니 바꿔달라" 요청
- **안희종 입장**: "오늘 테스트해보고 기능을 바꿔달라는 걸 받아들이긴 어렵다"
- **결과**: 전체 기능 변경 대신 해당 고객 대상 긴급 제거 조치

### 3. 버그 대응 우선순위 vs CC팀 기대

### 4. 선택적 근무 반차 시간 범위 — 진재언 vs 안희종
