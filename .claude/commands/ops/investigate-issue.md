# Investigate Issue

## Purpose
Linear 티켓을 조회하고, 실제 조사를 수행하여 원인을 파악하고 해결안을 도출한다.
operation-note가 없으면 자동으로 생성하고, 있으면 최신 Linear 데이터로 보강한 뒤 조사를 진행한다.
조사 결과는 operation-note에 기록하여 **경험을 축적**한다.

## 문서 구조 원칙 — Progressive Disclosure

조사 결과 문서는 **점진적 공개(Progressive Disclosure)** 구조로 작성한다.
제3자가 최상위 레벨만 읽어도 핵심을 파악하고, 필요한 경우에만 세부 사항으로 들어갈 수 있어야 한다.

### 레벨 구조

| 레벨 | 내용 | 목표 |
|------|------|------|
| **L1 — 한 줄 요약** | 각 섹션의 핵심을 1문장으로 | 10초 안에 파악 |
| **L2 — 핵심 근거** | 결론을 지지하는 핵심 증거 2-3개 | 1분 안에 이해 |
| **L3 — 상세 (`<details>`)** | 코드 스니펫, SQL 쿼리/결과, 전체 추론 체인, 데이터 테이블 | 필요할 때만 열어봄 |

### `<details>` 사용 규칙

**반드시 접는 대상:**
- 3줄 이상의 코드 블록 (핵심 1-2줄은 L2에 인라인 가능)
- SQL 쿼리 + 결과 데이터 테이블
- 5 Whys 체인이 5단계 이상일 때의 중간 단계
- 소거된 가설의 상세 근거 (가설 테이블의 간략 상태는 L2에 유지)
- 코드 위치 테이블이 3행 초과일 때
- 과거 잘못된 분석/판단 이력

**접지 않는 대상 (항상 L1-L2에 노출):**
- 원인 확정 1문장
- 가설 목록 테이블 (# / 가설 / 상태)
- 스펙/버그 판별 결과
- "다음에 같은 문의가 오면" 전체
- 해결안 옵션 요약

### `<details>` 형식

```
<details>
<summary>📋 {접힌 내용 요약 — 예: "DB 조회 결과 (3건)"}</summary>

{상세 내용}

</details>
```

### 중복 제거 원칙

- **"현재까지 파악된 내용"과 "원인 분석"에 같은 내용을 쓰지 않는다.** 전자는 조사 전 알려진 사실, 후자는 조사 후 확정된 분석.
- **가설 테이블은 1개만 유지한다.** "가설 목록"과 "배제된 가설"을 별도 테이블로 만들지 않고, 가설 목록 테이블의 상태 컬럼으로 관리한다.
- **코드 위치는 핵심 2-3개만 L2에 노출하고, 나머지는 `<details>`에 넣는다.**
- **조사 과정의 잘못된 판단 이력은 `<details>`로 접는다.** 최종 확정 판단만 L2에 노출.

### 분량 가이드라인

| 섹션 | L1+L2 목표 분량 | 비고 |
|------|----------------|------|
| 증상 | 5-10줄 | 현행 유지 |
| 원인 분석 (조사 과정) | 5-15줄 (L2) | 상세 추론은 `<details>` |
| 원인 (확정) | 3-5줄 | 메커니즘은 `<details>` |
| 가설 목록 | 가설 수 x 1행 | 테이블 1개 |
| 해결안 | 옵션당 2-3줄 | 상세 장단점은 `<details>` |
| 코드 위치 | 핵심 2-3개 | 전체 목록은 `<details>` |
| **전체 문서 (L1+L2)** | **80-120줄** | `<details>` 제외 |

## Input
$ARGUMENTS

### Argument Resolution
- `$ARGUMENTS`가 비어있거나 티켓 ID 패턴(`[A-Z]+-\d+`, 예: CI-3861)이 없으면:
  1. 현재 git 브랜치명에서 티켓 ID 패턴을 추출 시도 (예: `fix/CI-3861-some-desc` → `CI-3861`)
  2. 현재 대화 세션에서 언급된 티켓 ID를 추출 시도 (직전 사용자 메시지, 이전에 실행한 커맨드 인자 등)
  3. 브랜치/대화에서 못 찾으면 현재 디렉토리명에서 추출 시도
  4. 찾은 경우: 사용자에게 `"티켓 ID를 {추출한ID}로 인식했습니다. 맞습니까?"` 확인 후 진행
  5. 못 찾은 경우: `"티켓 ID를 특정할 수 없습니다. 티켓 ID를 인자로 전달해주세요. (예: /ops:investigate-issue CI-3861)"` 출력 후 **즉시 종료**

### Branch Naming
worktree나 브랜치를 생성할 때 **`ops/{ticket-id}`** 를 브랜치명으로 사용한다. (예: `ops/CI-3861`)
이슈 번호가 브랜치명에 포함되어야 나중에 worktree 정리 시 어떤 이슈의 작업인지 식별할 수 있다.

## Operation Notes Directory Resolution

operation-notes 파일을 저장할 디렉토리를 아래 우선순위로 결정한다.
**스킬 실행 시작 시 한 번만 해석하고, 이후 모든 경로에 동일하게 적용한다.**

| 우선순위 | 경로 | 설명 |
|---------|------|------|
| 1 | `{repo-root}/operation-notes/` | repo 루트에 operation-notes 디렉토리가 존재 |
| 2 | `{repo-root}/.claude/operation-notes/` | .claude 하위에 존재 |
| 3 | `~/.claude/operation-notes/` | 글로벌 홈 디렉토리에 존재 |

- 디렉토리 **존재 여부**로 판단한다 (파일이 아닌 디렉토리).
- 셋 다 존재하지 않으면 사용자에게 `"operation-notes 디렉토리를 찾을 수 없습니다. 어디에 저장할까요?"` 로 물어본다.
- 해석된 경로를 이하 `{notes-dir}`로 표기한다.

### Note File Resolution

`{ticket-id}.md` 파일을 찾을 때 아래 순서로 탐색한다:
1. `{notes-dir}/{ticket-id}.md` (active — 진행 중)
2. `{notes-dir}/archive/{ticket-id}.md` (archive — 해결 완료)

- 파일을 **새로 생성**할 때는 항상 `{notes-dir}/` (루트)에 생성한다.
- 이슈가 **해결 완료**되면 `{notes-dir}/archive/`로 이동한다.
- 상대 링크 규칙:
  - 루트 → archive: `./archive/{ticket-id}.md`
  - archive → 루트: `../{ticket-id}.md`
  - archive → archive: `./{ticket-id}.md`

## Procedure

### Step 1: 문제 정의 및 이해관계자 파악

조사를 시작하기 **전에**, 문의의 핵심 문제가 무엇인지, 누구에 관한 이슈인지를 먼저 파악한다.
이 단계를 건너뛰고 바로 조사에 들어가면 잘못된 방향으로 시간을 낭비할 수 있다.

#### 1-1: Linear 이슈 읽기 + 핵심 문제 추출

Linear 이슈(제목, 설명, 코멘트)를 읽고, **최소한의 정보로 진짜 문제를 정의**한다.

**추출할 정보:**
| 항목 | 설명 | 예시 |
|------|------|------|
| **문제 한 줄 정의** | 문의의 핵심을 1문장으로 | "특정 구성원의 3월 근무기록이 누락됨" |
| **문의자(요청자)** | 이슈를 접수한 사람 (CS팀, 고객사 관리자 등) | CS팀 @김철수, 고객사 HR 담당자 |
| **대상자** | 실제로 문제를 겪는 사용자 | 구성원 홍길동 (employee_id: 12345) |
| **영향 범위** | 1명인지 다수인지, 특정 조건의 전체인지 | 해당 회사 세콤 연동 사용자 전원 |
| **회사 정보** | 회사명, Customer ID, company_id | (주)테스트 (Customer ID: 1234) |
| **시점/기간** | 문제가 발생한 시점이나 대상 기간 | 2026-03-10 이후, 3월 1일~15일 |

**핵심 원칙: "문의 내용"과 "실제 문제"를 구분한다.**
- 문의: "근무기록이 이상해요" → 실제 문제: "3/10 출근 기록이 누락됨" 또는 "근무시간 계산이 다름"
- 문의가 모호하면 **가능한 해석을 나열**하고 사용자에게 확인한다

#### 1-2: 불명확한 경우 — 사용자에게 질문

아래 항목 중 하나라도 불명확하면 **즉시 사용자에게 질문**한다. 추측으로 조사를 시작하지 않는다.

**질문이 필요한 상황:**
- 문제가 뭔지 모호함 (증상은 있지만 무엇이 잘못된 건지 명확하지 않음)
- 대상자가 누구인지 특정되지 않음 (이름/ID 없음)
- 영향 범위를 판단할 수 없음 (1명? 전체?)
- 문제 시점/기간이 불분명함

**질문 형식:**
```
❓ **문제 파악을 위한 확인**

Linear 이슈를 확인했습니다. 조사 전에 아래를 확인하고 싶습니다:

1. {구체적 질문 — 예: "근무기록이 누락된 건가요, 아니면 시간이 다르게 표시된 건가요?"}
2. {구체적 질문 — 예: "영향받는 구성원이 홍길동 1명인가요, 같은 조건의 다른 구성원도 해당되나요?"}
```

**금지사항:**
- 한 번에 3개 이상의 질문을 하지 않는다 (최소한의 질문으로 핵심 파악)
- "혹시 ~도 해당되나요?" 같은 탐색적 질문은 하지 않는다 — 조사 과정에서 확인할 수 있는 건 조사에서 한다
- 사용자 응답 없이 다음 단계로 넘어가지 않는다

#### 1-3: 문제 정의 확정 및 문서화

문제가 명확해지면, operation-note의 **"## 증상"** 섹션에 아래 형식으로 기록한다.
이 정보는 이후 조사 과정과 결과 보고에서 반복적으로 참조되므로, 여기에 한 번만 정확히 기록한다.

```markdown
## 증상
- **문제 정의**: {문제 한 줄 정의}
- **회사**: {회사명} (Customer ID: {id})
- **요청자**: {문의를 접수한 사람 — CS팀 담당자, 고객사 관리자 등}
- **대상자**: {실제 문제를 겪는 사용자 — 이름, ID 포함}
- **영향 범위**: {1명 / 특정 조건의 N명 / 전사}
- **문제 시점**: {발생 시점 또는 대상 기간}
- 문의 내용:
  1. {핵심 문의 사항 나열}
```

> 이 정보가 정리되어야 이후 DB 조회, 로그 검색 등에서 WHERE 조건을 정확하게 설정할 수 있다.

#### 1-4: 사실 기반 문제 검증

1-3에서 정의한 문제가 **텍스트 기반 추정**일 수 있다. 문제를 정확히 특정해야 이후 어떤 코드를 볼지, 어떤 로그를 더 볼지, DB를 볼지 판단할 수 있다.

**이 단계의 목적**: 문의 내용으로 추정한 문제를 **실제 데이터로 검증**하여, 정확한 문제 정의를 확정한다.

**절차:**

1. **관련 API/코드 식별**: 문제 정의에서 어떤 API가 관련되는지 코드베이스에서 확인한다.
2. **스펙 초기 판단**: 코드를 읽으면서 현재 동작이 **의도된 스펙인지, 버그인지 초기 판단**한다.
   - 코드에 의도적 분기/조건 처리가 있는가?
   - 주석이나 커밋 메시지에 해당 동작의 근거가 있는가?
   - 이 판단은 확정이 아닌 **초기 방향 설정**이다. 이후 조사에서 뒤집힐 수 있다.
   - **스펙으로 보이면** → 이후 조사 방향이 "고객 안내 근거 확보"로 전환
   - **버그로 보이면** → 이후 조사 방향이 "원인 추적 + 수정"으로 전환
   - **판단 불가면** → access log/DB 등 데이터로 판단을 보류하고 계속 진행
3. **access log 조회 (API 요청/응답 확인이 필요한 경우)**:
   - `os-query-log` 로 대상자의 access log를 조회하여 **실제 요청과 응답**을 파악한다.
   - 검색 조건: 대상자 ID, 문제 시점, 관련 API 엔드포인트
   - 요청 파라미터와 응답 상태/내용을 확인하여, 문의 내용과 실제 동작이 일치하는지 대조한다.
4. **사용자 확인**: 파악한 코드 동작과 요청/응답을 사용자에게 보여주고, **"이 문제가 맞는지, 스펙/버그 초기 판단이 맞는지"** 확인받는다.
5. **문제 정의 보정**: 실제 데이터와 문의 내용이 다르면 문제 정의를 수정한다. 1-3의 "증상" 섹션도 함께 업데이트한다.

**이 단계를 수행하는 조건:**
- API 요청/응답이 문제의 핵심인 경우 (예: "조회가 안 된다", "저장이 실패한다", "데이터가 다르게 보인다")
- 문의 내용만으로는 실제 어떤 동작이 발생했는지 특정하기 어려운 경우

**건너뛰는 조건:**
- 문제가 이미 명확한 경우 (예: "설정값이 잘못되어 있다", 스크린샷으로 증상이 확인된 경우)
- API 요청/응답이 아닌 배치/스케줄러 등 백그라운드 처리 이슈

---

### Step 2-3: 데이터 수집 (🔀 병렬 처리)

아래 두 작업을 **subagent로 병렬 실행**한다:

| Agent | 작업 | 상세 |
|-------|------|------|
| 🤖 Agent A | **note 준비** | `note-issue.md` Step 1~3 수행 → `operation-notes/{ticket-id}.md` 최신화 |
| 🤖 Agent B | **연관 이슈 스캔** | `operation-notes/` 전체 노트 읽기 → 관련 경험 수집 |

**Agent A: operation-note 준비**
`{notes-dir}/{ticket-id}.md` 를 최신 상태로 준비한다.

1. **문서 존재 확인**: `{notes-dir}/{ticket-id}.md` 또는 `{notes-dir}/archive/{ticket-id}.md` 존재 여부 확인
2. **문서 없음** → `note-issue.md` 를 읽고 **Step 1~5를 정식 실행**하여 operation-note를 생성한다 (Step 6 보고는 생략).
   이때 **Step 1에서 정의한 문제 정의 및 이해관계자 정보를 "증상" 섹션에 반드시 반영**한다.
   ```
   Read: .claude/commands/ops/note-issue.md
   ```
3. **문서 있음** → Linear 최신 정보와 동기화:
   - MCP CLI로 이슈 + 코멘트 조회 (`mcp-cli info` 로 스키마 확인 후 호출)
   - 새 코멘트/상태 변경을 반영
   - Step 1의 문제 정의/이해관계자 정보가 "증상" 섹션에 누락되어 있으면 추가
   - `investigate-issue` 가 작성한 섹션(원인 분석, 해결안 등)은 건드리지 않음

**문서 작성 시 `note-writing-guide.md` 의 "문서 작성 원칙"과 "문서 포맷팅 규칙"을 반드시 읽고 준수한다.**
```
Read: .claude/commands/ops/note-writing-guide.md
```

이 단계가 끝나면 `{notes-dir}/{ticket-id}.md` 가 최신 상태로 존재한다.

**Agent B: 쿡북 참조 + 연관 이슈 탐색 및 경험 수집**

**쿡북 참조 (최우선):**
1. `{notes-dir}/COOKBOOK.md`가 존재하면 **먼저 읽는다**
2. 현재 이슈의 도메인에 해당하는 섹션의 진단 체크리스트/과거 사례를 수집
3. 이 정보를 Step 4 가설 수립 시 참고 자료로 전달
   - 쿡북의 진단 체크리스트 항목은 가설 우선순위 결정에 활용
   - 쿡북의 과거 사례 중 유사한 건이 있으면 가설에 반영
4. **쿡북 히트/미스 판정 결과를 반환한다**:
   - 참조한 플로우 ID 목록 (예: `F1`, `F3`)
   - 해당 도메인에 플로우가 없으면 `미스` 로 판정
   - 최종 히트/미스 판정은 Step 10에서 조사 결과를 바탕으로 확정

**연관 이슈 탐색:**
`{notes-dir}/` 루트의 active 노트만 전체 스캔한다. archive 노트는 `{notes-dir}/INDEX.md`에서 키워드로 연관 문서를 찾아 필요한 것만 `{notes-dir}/archive/`에서 읽는다.

**연관 판단 기준:**
- **동일 회사/Customer ID**: 같은 고객의 이전 문의
- **동일 기능 영역**: 같은 기능(예: 세콤연동, 근태, 급여 등)에 대한 이슈
- **유사 증상/키워드**: 제목이나 증상 설명에서 공통 키워드
- **동일 라벨**: Linear 이슈 라벨이 일치

연관 이슈를 찾으면:
- 해당 이슈의 **해결 방법**이 현재 이슈에 적용 가능한지 판단
- 과거 해결 사례, 원인 패턴, 조사 방법을 참고

### Step 4: 초기 가설 수립 (1-2개만)

Step 1~3에서 수집한 데이터를 바탕으로 **가장 유력한 가설 1-2개만** 수립한다.
이 단계에서 모든 가능성을 나열하지 않는다. 첫 번째 가설을 소거한 뒤 다음 가설을 추가하는 방식으로 진행한다.

#### 4-1: 쿡북/과거 사례 기반 초기 방향 결정

Agent B가 수집한 쿡북 진단 체크리스트와 과거 사례를 **먼저** 확인한다.

- 쿡북에 해당 도메인의 진단 체크리스트가 있으면, 체크리스트 항목을 순서대로 대조하여 **가장 먼저 해당하는 항목**을 초기 가설로 채택한다.
- 과거 사례에 유사 증상이 있으면, 그 사례의 원인을 초기 가설로 채택한다.
- 쿡북/과거 사례에 해당하는 것이 없으면, 증상에서 가장 직접적으로 추론되는 원인 1개를 초기 가설로 설정한다.

#### 4-2: 가설 테이블 초기화

가설 테이블을 **1-2행으로 시작**한다. 나머지 가설은 첫 번째 가설이 소거된 후에 추가한다.

**초기 가설 테이블:**
```
| # | 가설 | 확인 방법 | 상태 |
|---|------|----------|------|
| 1 | {초기 가설} | {구체적 확인 방법} | ⏳ 조사 중 |
```

**금지사항:**
- 이 단계에서 3개 이상의 가설을 나열하지 않는다.
- 이 단계에서 모든 확인 지점을 도출하지 않는다.
- 이 단계에서 관련 코드를 전부 읽지 않는다. **초기 가설 검증에 필요한 코드만 읽는다.**

> 최종적으로 가설 테이블에는 소거된 것 포함 3개 이상이 있어야 한다. 그러나 그것은 Step 5 루프를 통해 점진적으로 쌓인다.

### Step 5: 소거법 기반 조사 (반복 루프)

**한 번에 하나의 가설만 조사한다.** 아래 루프를 가설이 확정되거나 모든 가설이 소거될 때까지 반복한다.

#### 루프: 가설 검증 사이클

각 사이클은 다음 순서로 진행한다:

**① 현재 가설에 필요한 코드/데이터만 조사**
- 이 가설을 검증/반증하는 데 **필요한 코드만** 읽는다.
- 필요한 데이터가 있으면 SQL 쿼리를 작성하여 사용자에게 요청한다.
- 다른 가설을 위한 코드는 이 단계에서 읽지 않는다.

**② 가설 판정 + 테이블 즉시 업데이트**
- 조사 결과를 바탕으로 가설을 판정한다:
  - `⏳ 조사 중` → `❌ 소거` (근거 1줄)
  - `⏳ 조사 중` → `✅ 확정` (근거 1줄)
  - `⏳ 조사 중` → `🔍 추가 확인 필요` (필요한 데이터 명시)
- operation-note의 가설 테이블을 **즉시 업데이트**한다.

**③ 중간 결과 보고**
- 사용자에게 현재 가설의 판정 결과를 1-2문장으로 보고한다.
- 데이터 요청이 필요하면 이 시점에 요청한다 (하단 "조사 방법 가이드" 참조).
- 사용자 응답을 기다리는 동안 다음 가설로 넘어가지 않는다 (데이터 요청 시).

**④ 다음 가설 결정**
- 현재 가설이 소거되면:
  - 조사 과정에서 **새로 발견된 단서**를 바탕으로 다음 가설을 수립한다.
  - 단서가 없으면, 증상에서 두 번째로 유력한 원인을 가설로 추가한다.
  - 가설 테이블에 새 행을 추가하고 `⏳ 조사 중`으로 설정한다.
- 현재 가설이 확정되면: **루프를 종료**하고 Step 6으로 진행한다.
- `🔍 추가 확인 필요`이면: 사용자 데이터를 기다리며, 그 사이 다른 독립적 가설이 있으면 병렬 조사 가능.

**루프 종료 조건:**
- 가설 1개가 `✅ 확정`
- 또는 모든 가설이 `❌ 소거` (→ 근본 원인 미확정, 추가 데이터 필요 명시)
- 또는 가설 조사가 5회 반복 (→ 현재까지의 결론 정리 후 사용자와 방향 협의)

**핵심: 소거 과정 자체가 문서의 가치다.** 제3자가 유사 이슈를 만났을 때:
- 어떤 가능성들이 있는지 빠르게 파악하고
- 이미 배제된 가설을 반복 조사하지 않으며
- 어떤 것을 먼저 확인하면 효율적인지 알 수 있다

#### 조사 방법 가이드

**사용 가능한 조사 방법:**

| 방법 | 도구 | 적합한 상황 |
|------|------|-------------|
| 코드 검색 | Grep, Glob, Read | 관련 로직/서비스 코드 추적 |
| Linear 검색 | `linear/list_issues` | 유사 이슈, 동일 고객/기능 이슈 검색 |
| Slack 스레드 조사 | `claude_ai_Slack` MCP | 코멘트에 포함된 Slack 링크의 스레드 내용 조회. **Slack MCP가 연결되어 있을 때만 사용** |
| 연관 이슈 참고 | operation-notes 읽기 | 동일 영역의 해결 사례 활용 |

**데이터 확인이 필요한 경우 — SQL/API를 통한 데이터 요청:**

근본 원인을 특정하려면 코드만으로는 부족하고 **실제 데이터 상태**를 확인해야 하는 경우가 많다. 이때 사용자에게 데이터를 요청한다.

**데이터 요청 원칙:**
1. **SQL 쿼리를 직접 작성**하여 제공한다. "DB를 확인해주세요"가 아니라 실행 가능한 쿼리를 준다.
2. 쿼리에는 **확인 목적, 기대값, 해석 가이드**를 함께 제공한다.
3. 한 번에 필요한 데이터를 **최대한 모아서 요청**한다 (여러 번 왕복하지 않도록).

**SQL 쿼리 작성 가이드:**

> ⚠️ **필수**: 테이블명과 컬럼명을 추측하지 않는다. 반드시 코드베이스의 **실제 JPA Entity 클래스**(`@Table`, `@Column` 어노테이션)를 확인한 뒤 쿼리를 작성한다.

1. **Entity 확인 의무**: SQL 쿼리를 작성하기 전에 반드시 관련 Entity 클래스를 Grep/Read로 찾아서 `@Table(name = "...")`, `@Column(name = "...")` 값을 확인한다. 추측으로 테이블/컬럼명을 사용하면 안 된다.
2. **Entity 검색 방법**:
   - 도메인 키워드로 Entity 클래스를 검색 (예: `Grep "UserWorkClockStopPreference"` → 해당 필드를 가진 Entity 발견)
   - `@Table` 어노테이션으로 실제 테이블명 확인
   - `@Column(name = "...")` 으로 실제 컬럼명 확인 (필드명과 컬럼명이 다를 수 있음)
   - key-value 패턴 테이블인 경우 key 값도 확인
3. 이슈 재현에 필요한 **회사ID, 사용자ID 등의 조건**을 Linear 이슈 내용에서 추출하여 WHERE 절에 반영한다.
4. 단순 조회뿐 아니라, 상태 변경 이력을 추적하기 위한 **audit/history 테이블**이 있는지도 확인한다.
5. 쿼리에 사용한 테이블/컬럼의 **출처(Entity 파일 경로:라인)**를 각주로 명시한다.

**요청 형식 예시:**
```
📊 **데이터 확인 요청**

목적: {가설 검증 목적}

**쿼리 1**: {조회 대상 설명}
\`\`\`sql
SELECT column_a, column_b, column_c
FROM actual_table_name
WHERE condition = 'value';
\`\`\`
- 출처: `{repo}` > `{Entity파일경로}:{라인}` — `@Table(name = "actual_table_name")`
- 기대값: {정상이면 어떤 값이어야 하는지}
- 이상 시: {가설과의 대조 해석}
\`\`\`
- 목적: 언제부터 갱신이 실패했는지 시점 특정
```

**Operation API 활용:**
- 어드민 API가 존재하는 경우, 사용자에게 어드민 화면에서 확인할 경로도 함께 안내한다.
- API 엔드포인트를 알고 있다면 `curl` 명령어 형태로 제공한다.

**로그 확인 요청:**
- 로그 검색이 필요한 경우, **검색 키워드, 시간 범위, 로그 레벨**을 구체적으로 명시한다.
- 예: `"2024-01-15 09:00~10:00 사이, company_id={id} 포함, ERROR 레벨 로그를 Datadog/CloudWatch에서 검색"`

**OpenSearch 로그 분석 결과 테이블 포맷:**

OpenSearch에서 traceId 기반으로 요청 흐름을 조회한 경우, 아래 포맷으로 정리하여 보고한다.
테이블은 box-drawing 문자(`┌─┬─┐`)를 사용하며, 한국어 문자 폭(width 2)을 고려하여 정렬한다.
**시간은 반드시 KST(UTC+9) 기준으로 표시한다.**

```
🔍 요청 흐름 — traceId: [{traceId 앞 16자리}]({dashboard URL with traceId filter})
   요청자: {이름}({userId}) / {회사명}({customerId})

┌────┬───────────────────────────┬──────────────────────────────────────────┬────────────────────────────────────┐
│ #  │ 시간 (KST)                │ API                                      │ 핵심 내용                          │
├────┼───────────────────────────┼──────────────────────────────────────────┼────────────────────────────────────┤
│  1 │ 2026-03-11 18:49:05.767   │ POST /action/v2/shift-schedule/query     │ 교대근무 조회 요청                 │
│  2 │ 2026-03-11 18:49:05.862   │ POST /api/internal/v2/work-rule/...      │ 근무유형 조회                      │
│  3 │ 2026-03-11 18:49:05.911   │ POST /api/internal/access-check          │ work_schedule READ → 100명         │
└────┴───────────────────────────┴──────────────────────────────────────────┴────────────────────────────────────┘

#1 [교대근무 조회 요청]({dashboard URL with spanId filter})
#2 [근무유형 조회]({dashboard URL with spanId filter})
#3 [work_schedule READ 권한 체크]({dashboard URL with spanId filter})
```

**대시보드 링크 생성 규칙:**

- 기본 URL: `https://log-dashboard.grapeisfruit.com/_dashboards/app/data-explorer/discover#?`
- 인덱스 패턴 ID: `f16cda60-f2fb-11ee-9a9d-4b897330ccb0` (prod)
- 시간 범위:
  - `from`: 첫 번째 access log 시각에서 elapsedTime을 뺀 시각(= 요청 시작 시각) -1분
  - `to`: 마지막 access log 시각 +1분
  - 형식: ISO 8601 UTC (예: `2026-03-11T09:48:06.789Z`)
- **헤더 traceId 링크**: traceId 필터만 포함
  ```
  _q=(filters:!(('$state':(store:appState),meta:(alias:!n,disabled:!f,index:{indexPatternId},key:json.traceId,negate:!f,params:(query:{traceId}),type:phrase),query:(match_phrase:(json.traceId:{traceId})))),query:(language:kuery,query:''))
  ```
- **#N spanId 링크**: traceId + spanId 필터 모두 포함, 시간 범위는 헤더 traceId 링크와 동일
  ```
  _q=(filters:!(('$state':(store:appState),meta:(alias:!n,disabled:!f,index:{indexPatternId},key:json.traceId,negate:!f,params:(query:{traceId}),type:phrase),query:(match_phrase:(json.traceId:{traceId}))),('$state':(store:appState),meta:(alias:!n,disabled:!f,index:{indexPatternId},key:json.spanId,negate:!f,params:(query:'{spanId}'),type:phrase),query:(match_phrase:(json.spanId:'{spanId}')))),query:(language:kuery,query:''))
  ```
- spanId 값은 작은따옴표로 감싸야 함 (`'{spanId}'`)
- 공통 파라미터:
  ```
  _a=(discover:(columns:!(json.ipath,json.message,json.traceId,'tempo%20link'),isDirty:!f,sort:!()),metadata:(indexPattern:{indexPatternId},view:discover))
  _g=(filters:!(),refreshInterval:(pause:!t,value:15000),time:(from:'{timeFrom}',to:'{timeTo}'))
  ```

**테이블 생성 시 한국어 정렬 처리:**

한국어 문자는 터미널에서 폭 2를 차지하므로, 정렬이 맞지 않을 수 있다. Python 스크립트로 `unicodedata.east_asian_width()`를 사용하여 display width를 계산하고 패딩을 맞춘다.

**받은 데이터 해석 및 가공:**
- 데이터를 받으면 **가설과 대조**하여 검증 결과를 즉시 분석한다.
- 예상과 다른 데이터가 나오면 **가설을 수정**하고 추가 확인 지점을 도출한다.
- 데이터 해석 결과를 사용자에게 설명하고, 필요시 후속 쿼리를 요청한다.
- **제3자 가독성을 위한 데이터 가공** (중요):
  사용자가 제공한 원시 데이터(SQL 결과, 로그, API 응답 등)를 operation-note에 기록할 때, **문서만 읽는 제3자도 맥락을 이해할 수 있도록 가공**해야 한다. 원시 데이터를 그대로 붙여넣지 않는다.
  - **컬럼 설명 추가**: 테이블 결과에 각 컬럼이 무엇을 의미하는지 주석 또는 설명 행을 붙인다.
  - **핵심 행 하이라이트**: 전체 결과 중 원인 파악에 결정적인 행을 강조하고, 왜 그 행이 중요한지 설명한다.
  - **정상값과 비교**: 문제가 되는 값 옆에 정상이라면 어떤 값이어야 하는지를 병기한다.
  - **시간순 정리**: 여러 이벤트/레코드가 있으면 시간순으로 나열하여 사건의 흐름을 재구성한다.
  - **요약 문장 선행**: 데이터 블록 앞에 1-2줄로 "이 데이터가 보여주는 것"을 먼저 서술한다.

  가공 형식 예시:
  ```
  > 📊 **연동 설정 상태** (조회 시점: 2024-01-15)
  >
  > 아래 결과에서 `status=INACTIVE`, `token_expires_at`이 과거 시점인 것으로 보아,
  > 토큰 만료 후 갱신에 실패하여 연동이 비활성화된 것으로 판단된다.
  >
  > | company_id | status | token_expires_at | updated_at |
  > |---|---|---|---|
  > | 1234 | **INACTIVE** ⚠️ (정상: ACTIVE) | 2024-01-10 (만료됨) | 2024-01-10 15:30 |
  ```

조사 중 발견한 중요 링크(Slack 스레드, metabase 쿼리, 관련 코드 위치 등)는 모두 수집한다.

### Step 6: 근본 원인 정리 + 스펙 vs 버그 판별

가설이 확정된 후에 이 단계를 수행한다.

#### 6-1: 근본 원인 추적 (5 Whys)

**확정된 원인에 대해서만** 5 Whys 체인을 작성한다. 조사 중에는 미리 작성하지 않는다.

단순히 "무엇이 잘못되었나"가 아니라, **"왜 잘못되었나"를 반복**하여 근본 원인까지 추적한다.

**원인 추적 체인:**
```
현상(Symptom) → 표면 원인(Direct Cause) → 근본 원인(Root Cause)
```

각 단계에서 **"왜?"를 반복**한다 (확인된 사실 기반으로, 추측이 아닌 출처가 있는 답변만):
1. 왜 출퇴근 기록이 누락되었나? → 동기화가 실행되지 않았음
2. 왜 동기화가 실행되지 않았나? → 스케줄러가 해당 회사를 건너뜀
3. 왜 건너뛰었나? → 연동 설정의 상태값이 INACTIVE
4. 왜 INACTIVE가 되었나? → 토큰 갱신 실패 시 상태를 INACTIVE로 변경하는 로직 존재
5. 왜 토큰 갱신이 실패했나? → (근본 원인) 외부 시스템 API 변경으로 인한 인증 방식 불일치

**분기가 있는 원인 추적 시 mermaid flowchart 활용:**
원인 추적에서 **2개 이상 분기**가 발생하여 텍스트 체인으로 전체 구조를 파악하기 어려울 때, operation-note에 mermaid flowchart를 포함한다. 단순 선형 체인(분기 없음)이면 텍스트 5 Whys가 더 간결하므로 mermaid를 사용하지 않는다.

#### 6-2: 스펙 vs 버그 판별

수집한 정보를 바탕으로 현상이 **의도된 스펙인지, 실제 버그인지** 판단한다.

**판별 기준:**
- 코드상 의도적 분기 처리가 있는가?
- 관련 문서/스펙에 해당 동작이 명시되어 있는가?
- 과거 유사 이슈에서 "스펙"으로 결론 난 적이 있는가?
- 데이터/상태 조건에 따른 정상 동작인가?

**판별 결과:**
- **스펙이면**: 고객에게 안내할 설명을 정리하고, 왜 이렇게 동작하는지 근거 제시
- **버그이면**: 원인과 영향 범위를 정리하고, 수정 방향 제안. **→ Step 6-3 사이드이펙트 분석 진행**
- **판단 불가이면**: 추가로 확인해야 할 항목을 명시

#### 6-3: 사이드이펙트 분석 (버그 판정 시)
버그로 판단된 경우, 수정 시 **의도치 않은 영향이 발생할 수 있는 지점**을 사전에 파악한다.
수정 코드 자체의 정확성뿐 아니라, 해당 로직에 의존하는 다른 기능들까지 고려해야 한다.

**분석 관점:**

| 관점 | 확인 사항 |
|------|----------|
| **호출자 추적** | 버그가 있는 메서드/함수를 호출하는 다른 코드가 있는가? 그 호출자들은 현재 동작(버그 포함)에 의존하고 있지 않은가? |
| **데이터 정합성** | 버그로 인해 이미 잘못 저장된 데이터가 있는가? 수정 후 기존 데이터와의 불일치가 발생하는가? |
| **다른 고객 영향** | 이 버그가 문의한 고객만의 문제인가, 같은 조건의 다른 고객에게도 발생하는가? |
| **연동/이벤트** | 버그 로직이 이벤트를 발행하거나 외부 시스템에 데이터를 전달하는가? 수정 시 이벤트 발행 패턴이 변경되는가? |
| **UI/API 계약** | 수정으로 API 응답 구조나 값이 변경되는가? FE가 현재 응답에 의존하고 있는가? |
| **성능** | 수정이 쿼리 범위 확대나 추가 조회를 유발하는가? |

**기록 형식** (operation-note에 추가):
```
### 수정 시 사이드이펙트 포인트

| # | 영향 지점 | 위험도 | 설명 |
|---|----------|--------|------|
| 1 | {호출자/기능명} | 🔴/🟡/🟢 | {왜 영향받을 수 있는지, 확인 방법} |
```

모든 항목을 깊이 조사할 필요는 없다. **대략적인 위험 지점을 식별**하여 수정 담당자가 주의할 포인트를 제공하는 것이 목적이다.

### Step 7: 해결안 도출
판별 결과에 따라:
1. **원인 분석**: 표면 원인과 근본 원인을 분리하여 기술한다.
   - **표면 원인**: 현상을 직접 유발한 조건. 이것만 해결하면 당장 증상은 사라지지만 재발 가능.
   - **근본 원인**: 표면 원인이 발생하게 된 구조적/로직적 이유. 이것을 해결해야 재발을 방지.
   - **근본 원인을 특정하지 못한 경우**: 현재까지의 조사 결과와, 근본 원인을 확인하기 위해 **추가로 필요한 데이터(테이블, 로그, API 응답)**를 명시한다.
   - 확인된 사실과 추정을 명확히 구분
2. **근로기준법 검토** (해당 시): 이슈가 근태/근무/휴가 관련이면 관련 법령을 조사하고, 각 해결안이 근로자에게 유리/불리한지 판단한다.
   - 근로기준법은 **근로자 보호법**이므로, 근로자에게 유리한 방향의 변경은 법적 리스크가 낮다
   - 근로자에게 불리한 방향의 변경은 노사 합의 필요 여부를 확인한다
   - 해결안 옵션별로 근로자 유불리를 명시한다 (예: "근로자에게 유리 ✅" / "근로자에게 불리 ⚠️")
3. **해결안 제시**: 표면 원인 해결과 근본 원인 해결을 구분하여 옵션을 나열한다.
   - **즉시 대응(표면 원인 해결)**: 당장 증상을 해소하는 방법 (데이터 패치, 설정 변경 등)
   - **근본 해결(근본 원인 해결)**: 재발 방지를 위한 코드/구조 수정
   - 각 옵션에 장단점 포함
   - 연관 이슈에서 참고할 해결 방법이 있으면 "참고: [ticket-id]에서는 ~방식으로 해결" 형태로 기재
   - **근로자에게 유리하면서 구현이 간단한 옵션을 우선 권장**한다
4. **다음 단계 제안**: 추가 조사가 필요하면 구체적 방향 제시

### Step 8: operation-note 업데이트
조사 결과를 `{notes-dir}/{ticket-id}.md`에 반영한다.
**`note-writing-guide.md`의 "문서 작성 원칙"을 반드시 준수한다** (출처 표기, 과정 기록, 용어 설명, **문서 포맷팅 규칙**).

**추가/업데이트하는 섹션 (Progressive Disclosure 구조 적용):**

- **"현재까지 파악된 내용"**: 조사 **이전에** 수집된 사실만 기록. investigate-issue의 조사 결과는 이 섹션이 아닌 "원인 분석"에 기록한다. **이미 "원인 분석"에 기록된 내용을 이 섹션에 중복 기재하지 않는다.**
- **"원인 분석"**: Progressive Disclosure 구조로 작성.
  - **L1 (최상단)**: 원인 확정 1문장
  - **L2**: 핵심 증거 2-3개 (코드 위치 인라인, 데이터 결과 요약 1줄씩)
  - **가설 목록** 하위 섹션: **테이블 1개만 유지**. 상태 컬럼(`✅ 확정` / `❌ 소거` / `🔍 추가 확인`)으로 관리. "배제된 가설" 별도 테이블을 만들지 않는다.
  - **L3 (`<details>`)**: 상세 조사 과정, 5 Whys 전체 체인, 코드 스니펫, SQL 쿼리/결과, 소거된 가설의 상세 분석
  - **"조사 과정"** 하위 섹션:
    - L2에는 **최종 확정 판단 경로만** 기록 (3-5 화살표 이내)
    - 중간에 잘못된 판단이 있었으면 `<details>`로 접어서 보존
    - 형식: `> 💡 **판단 근거**: {단서} → {확인}[^n] → {결론}` (출처는 각주로 분리)
  - 코드를 분석한 경우 **repo명과 파일 경로:라인 번호** 포함 (형식: `` `{repo}` > {경로}:{라인} ``)
  - **사용자 제공 데이터는 가공하여 `<details>` 안에 기록**: Step 5의 "제3자 가독성을 위한 데이터 가공" 규칙을 따른다.
  - **"근본 원인 미확정 시"**: L2에 현재까지의 가설 요약, `<details>` 안에 추가 확인 필요 데이터를 명시
- **"코드 위치"**: **핵심 파일 2-3개만** 본문(L2)에 노출. 5개 이상이면 나머지는 `<details>` 안에 전체 테이블로 기록.
- **"수정 시 사이드이펙트 포인트"**: 버그로 판정된 경우에만 작성. Step 6-3에서 분석한 영향 지점을 테이블로 기록.
- **"다음에 같은 문의가 오면"**: 이슈가 해결 완료된 경우에만 작성. 조사·해결 과정을 모르는 제3자가 **이 섹션만 읽고 빠르게 대응**할 수 있도록, 확인 → 판별 → 조치 순서로 간결하게 정리한다.
  - 구체적 확인 방법(SQL, 로그 검색 등)을 포함할 것
  - 확인 결과에 따른 분기("X이면 A, Y이면 B")를 명시할 것
  - 이 섹션은 3~5 단계 이내로 짧게 유지
- **"해결안 / 조사 방향"**: 도출한 해결안과 다음 조사 방향 기록. 각 옵션의 **장단점 근거**도 포함.
- **"참고 자료"**: 발견한 Slack 스레드, metabase 쿼리 링크, 관련 코드 위치 등 추가. **조사에 쓰이지 않았더라도 이후 참고할 수 있는 자료는 모두 기록.**
- **"미결 사항"**: 확인된 항목 체크, 새 항목 추가

**연관 이슈 링크:**
- Step 2-3에서 찾은 연관 이슈를 "## 연관 이슈" 섹션에 추가 (없으면 섹션 생성)
- 연관 이슈 문서에도 역방향 링크 추가 (중복 방지)
- 링크는 Note File Resolution의 상대 링크 규칙에 따라 대상 파일 위치에 맞는 경로를 사용
- 연관 이슈를 참고한 경우, **어떤 부분을 참고했는지** 구체적으로 기재 (예: "CI-3800의 세콤 동기화 해결 방법 참고")

### Step 8-1: 문서 검토 (🤖 검토 에이전트)
note 업데이트 후 **별도 subagent로 문서 검토**를 수행한다. `note-writing-guide.md`의 **"문서 검토 체크리스트" (A~E)** 를 전체 수행하고, 아래 조사 전용 항목을 추가로 검증한다. 위반 사항은 자동으로 수정한다.

**`note-writing-guide.md` 체크리스트 A~E 전체 수행** + 아래 추가 항목:

#### F. 조사 과정 품질 (investigate-issue 전용)
- [ ] 가설 목록 테이블에 `#, 가설, 확인 방법, 상태` 컬럼이 모두 있는가?
- [ ] 소거된 가설에 배제 근거가 명시되어 있는가?
- [ ] 5 Whys 체인이 `왜? → {답변}` 형태로 근본 원인까지 기록되어 있는가?
- [ ] SQL 쿼리 요청/결과가 목적-쿼리-기대값-해석 형태로 구조화되어 있는가?
- [ ] 사용자 제공 원시 데이터가 가공되어 있는가? (컬럼 설명, 정상값 비교, 핵심 하이라이트, 요약 문장)
- [ ] 원인 추적에 분기가 2개 이상이면 mermaid flowchart가 포함되어 있는가?
- [ ] 버그 판정 시 "수정 시 사이드이펙트 포인트" 테이블이 존재하는가?

#### G. 간결성 검증 (Progressive Disclosure)
- [ ] 전체 문서의 L1+L2 분량이 120줄 이내인가? (초과 시 `<details>`로 이동할 내용 식별)
- [ ] 3줄 이상의 코드 블록이 `<details>` 안에 있는가?
- [ ] SQL 쿼리/결과 데이터가 `<details>` 안에 있는가?
- [ ] 가설 테이블이 1개인가? (가설 목록 + 배제 테이블이 분리되어 있으면 통합)
- [ ] 코드 위치 테이블이 3행 이내로 노출되고, 나머지는 `<details>` 안에 있는가?
- [ ] 동일 내용이 "현재까지 파악된 내용"과 "원인 분석"에 중복 기재되어 있지 않은가?
- [ ] 잘못된 과거 분석/판단이 `<details>`로 접혀 있는가?
- [ ] 5 Whys 체인이 5단계 이상이면 중간 단계가 `<details>`에 있는가?

### Step 9: 결과 보고
- 조사 결과 요약 (3-5줄)
- 스펙/버그 판별 결과
- 추천 다음 액션:
  - 코드 수정 필요 → `ops:fix-issue {ticket-id}` 안내
  - 추가 조사 필요 → 구체적 방향 제시
  - 고객 응대 필요 → 답변 초안 제안
  - 해결 완료 → `ops:note-issue {ticket-id}`로 최종 기록 정리 및 Gherkin 시나리오 작성 안내

## Rules
- **외부 시스템 변경/조회 시 사전 확인 필수**: 아래 작업을 수행하기 전에 **반드시 사용자에게 먼저 확인**을 받는다. 자동으로 실행하지 않는다.
  - **Linear 코멘트 추가**: 조사 결과나 중간 보고를 Linear 이슈에 코멘트로 남기려는 경우
  - **DB 조회 (db:db-query)**: SQL 쿼리를 실제 DB에 실행하려는 경우 (쿼리 작성·제시는 자유, 실행만 확인 필요)
  - **OpenSearch 조회 (opensearch:os-query-log, opensearch:os-query-service)**: 로그나 서비스 문서를 OpenSearch에서 검색하려는 경우
- MCP CLI 호출 전 반드시 `mcp-cli info`로 스키마 확인.
- bot이 생성한 동기화 코멘트(예: "This comment thread is synced to...")는 무시.
- **추론/예측과 사실을 명확히 구분한다.** 중간 보고, 결과 보고, operation-note 기록 모두에 적용:
  - 사실(코드 확인, DB 조회 결과, 로그 등)에는 **근거를 함께 제시**한다.
  - 추론이나 예측에는 `(추정)` 태그를 붙이고, 왜 그렇게 추론했는지 간략히 설명한다.
  - 사용자에게 보고할 때 "~인 것으로 보입니다(추정)"와 "~입니다(확인됨)" 표현을 구분하여 사용한다.
- 발견한 링크(Slack 스레드, metabase 등)는 빠짐없이 참고 자료에 기록.
- 문서는 **한국어**로 작성한다.
- 기존 operation-note 내용을 덮어쓰지 않고, **추가/보강**하는 방식으로 업데이트.
- 과거 해결 경험을 적극 참고하되, 현재 이슈의 컨텍스트에 맞게 적용한다.
- 유저 데이터 요청 시 **왜 필요한지, 어디서 확인하는지** 반드시 함께 안내한다.
- 연관 이슈 판단 시 억지로 연결하지 않는다. 실제로 유의미한 연관이 있을 때만 링크한다.
- **`note-writing-guide.md`의 "문서 작성 원칙"을 준수한다**: 모든 정보에 **각주(`[^n]`)로 출처 표기**, 결론까지의 과정 기록, 도메인 용어는 각주로 설명 분리, **문서 포맷팅 규칙**(테이블 작성, 데이터 가공, 시각 요소 사용 기준 등). 인라인 출처(`*(출처)*`)는 사용하지 않는다.
- 코드를 분석했을 때 **repo명과 파일 경로:라인 번호** 형태로 위치를 명시한다 (형식: `` `{repo}` > {경로}:{라인} ``). 핵심 로직은 코드 스니펫도 포함한다.
- **사용자 제공 데이터(SQL 결과, 로그, API 응답 등)는 원시 형태로 붙여넣지 않는다.** 제3자가 문서만 읽어도 맥락을 이해할 수 있도록 컬럼 설명, 정상값 비교, 핵심 하이라이트, 요약 문장을 포함하여 가공한다.
- **각주 섹션**: operation-note 하단에 `## 각주` 섹션을 유지하고, 모든 출처·보충 설명 각주를 이 섹션에 모은다.
