운영/CS 이슈를 체계적으로 처리하기 위한 Claude Code 커맨드 모음.

Linear 이슈 조회부터 조사, 코드 수정, PR 생성, 지식 축적까지 일관된 파이프라인을 제공한다.

== 커맨드 아키텍처

커맨드는 **Primitive**(단일 책임 빌딩블록)과 **Orchestrator**(조합), **Specialized**(특수 목적)의 3계층으로 구성된다.

== 각 커맨드 상세

=== note-issue

**목적**: Linear 티켓 내용을 `{notes-dir}/{ticket-id}.md` 에 기록한다. 문서 작성/동기화에 집중하며, 조사 자체는 수행하지 않는다.

**주요 기능**:

* 신규 이슈: 이슈 상태(해결 완료 / 진행 중)에 따른 템플릿으로 operation-note 생성
* 기존 문서: Linear 최신 정보(새 코멘트, 상태 변경)와 동기화
* 해결 완료 건: Gherkin 시나리오 작성 (스펙 확인/변경 건)
* 연관 이슈 자동 탐색 및 양방향 링크

**산출물**: `{notes-dir}/{ticket-id}.md`

**사용 예시**:
[source]
----
/ops:note-issue CI-3861
----

=== investigate-issue

**목적**: 이슈를 실제로 조사하여 원인을 파악하고 해결안을 도출한다. 소거법 기반의 체계적 조사 프로세스를 따른다.

**주요 기능**:

* 5 Whys 기반 근본 원인 추적
* 가설 목록 수립 → 우선순위 순 소거
* COOKBOOK.md 참조로 과거 경험 활용
* SQL 쿼리 / 로그 검색 조건 작성하여 데이터 확인 요청
* 스펙 vs 버그 판별
* 사이드이펙트 분석 (버그 판정 시)

**산출물**: `{notes-dir}/{ticket-id}.md` 에 조사 결과 추가 (원인 분석, 배제된 가설, 해결안 등)

**사용 예시**:
[source]
----
/ops:investigate-issue CI-3861
----

=== fix-issue

**목적**: Linear 이슈를 기반으로 코드 조사 → 설계 승인 → 구현 → PR 생성까지 수행한다.

**주요 기능**:

* Linear 이슈 조회 + 상태를 In Progress로 변경
* gitflow 브랜치 자동 생성 (fix/, feature/, chore/)
* 코드 조사 및 설계 옵션 제시 → 사용자 승인 후 구현
* 빌드/테스트 검증
* Draft PR 생성 + Linear 이슈에 PR 링크 코멘트

**산출물**: Draft PR + `/tmp/` 아래 조사 문서

**사용 예시**:
[source]
----
/ops:fix-issue CI-3861
----

=== capture-issue

**목적**: `note-issue` + `update-cookbook` 증분을 티켓 ID 하나로 한번에 처리한다. 개별 이슈 원스톱 처리용.

**실행 흐름**:

* Phase 1: `note-issue` Step 1~6 수행 (operation-note 생성/업데이트)
* Phase 2: 쿡북 반영 대상 자동 판단 → `update-cookbook` 증분 모드 수행
* Phase 3: 통합 결과 보고

**산출물**: `{notes-dir}/{ticket-id}.md` + `COOKBOOK.md` 업데이트 (해당 시)

**사용 예시**:
[source]
----
/ops:capture-issue CI-3861
----

=== update-cookbook

**목적**: operation-notes에 쌓인 이슈 기록에서 반복 가능한 지식(진단 체크리스트, SQL 쿼리 템플릿, 과거 사례)을 추출하여 `COOKBOOK.md` 에 정리한다.

**두 가지 모드**:

* **전체 스캔 모드**: 인자 없이 호출. 모든 노트를 읽고 COOKBOOK.md 전체 재구성.
* **증분 모드**: 이슈 ID 전달. 해당 노트만 읽고 COOKBOOK.md에 증분 반영.

**산출물**: `{notes-dir}/COOKBOOK.md`

**사용 예시**:
[source]
----
# 전체 재구성
/ops:update-cookbook

# 특정 이슈만 반영
/ops:update-cookbook CI-3914
----

=== update-index

**목적**: operation-notes의 검색 인덱스(`INDEX.md`)를 갱신한다. 키워드 → 문서 매핑으로 전체 노트를 읽지 않고 관련 문서를 빠르게 찾을 수 있게 한다.

**주요 기능**:

* 증분 모드: 대상 노트에서 요약·도메인·키워드 추출 → INDEX.md 행 추가/갱신
* 전체 재구성: 모든 노트 스캔 → 도메인별 그룹핑 → INDEX.md 재작성

NOTE: `note-issue` Step 6에서 자동 호출되므로 직접 호출할 일은 드물다.

**산출물**: `{notes-dir}/INDEX.md`

**사용 예시**:
[source]
----
# 특정 노트만 인덱싱
/ops:update-index CI-3914

# 전체 재구성
/ops:update-index
----

=== maintain-notes

**목적**: 아카이브 안 된 활성 노트를 일괄 유지보수한다. Linear 이슈 상태 확인 → 노트 갱신 → 쿡북 반영 → 완료 노트 아카이브까지 한번에 처리.

**주요 기능**:

* 활성 노트 전체의 Linear 이슈 상태를 병렬 조회
* 상태 변경된 노트의 상태 표기 자동 업데이트
* 완료된 이슈 중 쿡북 반영 대상을 `update-cookbook` 증분 모드로 처리
* 아카이브 조건(이슈 완료 + 쿡북 반영) 충족 시 `archive/` 로 이동
* 관련 링크(COOKBOOK.md, INDEX.md, 연관 노트) 일괄 업데이트

**산출물**: 노트 갱신 + COOKBOOK.md 업데이트 + archive 이동 + 최종 리포트

**사용 예시**:
[source]
----
/ops:maintain-notes
----

=== sync-scenario-agents-md

**목적**: operation-notes에서 유저 대면 API를 찾아 `scenario-test/AGENTS.md` 에 누락된 도메인/API를 추가한다.

**제약**: `flex-team/flex-timetracking-backend` 레포지토리에서만 실행 가능.

**주요 기능**:

* Operation notes에서 API 경로 / Controller / MappingService 참조 추출
* AGENTS.md와 갭 분석
* 코드베이스에서 실제 존재 여부 검증
* 변경 사항 미리보기 → 사용자 승인 → PR 생성

**산출물**: AGENTS.md 업데이트 PR

**사용 예시**:
[source]
----
/ops:sync-scenario-agents-md
----

== 워크플로우

=== 기본 워크플로우: 이슈 접수 → 해결

일반적인 운영 이슈 처리 흐름.

[mermaid]
....
graph LR
    A["이슈 접수"] --> B["note-issue"]
    B --> C["investigate-issue"]
    C --> D["fix-issue"]
    D --> E["capture-issue"]
....

[cols="1,4,2"]
|===
| 단계 | 설명 | 커맨드

| 1. 기록
| 이슈 내용을 operation-note로 문서화
| `/ops:note-issue CI-XXXX`

| 2. 조사
| 원인 가설 수립 → 소거법 조사 → 해결안 도출
| `/ops:investigate-issue CI-XXXX`

| 3. 수정
| 승인된 설계로 코드 수정 → PR 생성
| `/ops:fix-issue CI-XXXX`

| 4. 데이터화
| 완료된 이슈를 note + 쿡북 증분으로 데이터화
| `/ops:capture-issue CI-XXXX`
|===

NOTE: `capture-issue` 는 **완료된 이슈**를 대상으로 note 동기화 + 쿡북 증분 반영을 한번에 처리한다. 이슈 접수 시점에는 쿡북에 반영할 내용이 없으므로 `note-issue` 를 사용한다.

=== 조사만 필요한 경우 (코드 수정 불필요)

스펙 확인, 데이터 패치, 설정 변경 등으로 해결되는 이슈.

[mermaid]
....
graph LR
    A["note-issue"] --> B["investigate-issue"]
    B --> C["해결"]
    C --> D["capture-issue"]
....

* `investigate-issue` 로 원인 파악 후, 코드 수정이 아닌 다른 방법으로 해결
* 해결 후 `capture-issue` 로 완료 상태 동기화 + 쿡북 반영
* 스펙 확인 건이면 Gherkin 시나리오도 자동 생성됨

=== 간단한 이슈 (원인이 명확한 경우)

과거 사례가 있거나 원인이 바로 파악되는 경우.

[mermaid]
....
graph LR
    A["fix-issue"] --> B["capture-issue"]
....

* `fix-issue` 가 내부적으로 Linear 이슈를 조회하고 코드를 조사하므로, 별도 note/investigate 없이 바로 진행 가능
* 완료 후 `capture-issue` 로 note + 쿡북 반영

=== 정기 유지보수

이슈가 여러 건 쌓인 후 일괄 정리.

[source]
----
/ops:maintain-notes          # 활성 노트 상태 갱신 + 쿡북 반영 + 아카이브
/ops:update-cookbook          # COOKBOOK.md 전체 재구성 (필요 시)
/ops:sync-scenario-agents-md # AGENTS.md 갭 보완
----

NOTE: `maintain-notes` 는 완료된 이슈의 쿡북 반영 + 아카이브까지 자동 처리한다. 쿡북 전체 재구성이 필요한 경우에만 `update-cookbook` 을 인자 없이 실행.

== 공통 규칙

=== 티켓 ID 자동 인식

모든 커맨드에서 티켓 ID를 생략하면:

1. 현재 git 브랜치명에서 패턴 추출 시도 (예: `fix/CI-3861-desc` → `CI-3861`)
2. 현재 대화 세션에서 언급된 티켓 ID를 추출 시도 (직전 사용자 메시지, 이전에 실행한 커맨드 인자 등)
3. 브랜치/대화에서 못 찾으면 디렉토리명에서 추출 시도
4. 찾으면 사용자 확인 후 진행, 못 찾으면 종료

=== 문서 작성 원칙

`note-issue` 에 정의된 원칙이 모든 커맨드에 적용된다:

* **출처 표기**: 모든 사실 정보에 출처 필수 (Linear 코멘트, 코드 분석, DB 데이터 등)
* **과정 기록**: 결론뿐 아니라 판단에 도달한 과정도 기록
* **용어 설명**: 도메인 특화 용어를 처음 사용할 때 설명 추가
* **포맷팅 규칙**: 테이블, mermaid, blockquote 등 상황별 시각 요소 사용 기준 준수

=== investigate-issue가 작성한 섹션 보호

`note-issue` 로 문서를 업데이트할 때, `investigate-issue` 가 추가한 섹션(원인 분석, 해결안, 참고 자료 등)은 건드리지 않는다. 역방향으로도 마찬가지.

== 활용 팁

=== COOKBOOK.md를 먼저 참조

`investigate-issue` 는 조사 시작 시 자동으로 COOKBOOK.md를 참조하여 과거 경험을 활용한다. COOKBOOK.md가 잘 정리되어 있을수록 조사 시간이 단축된다.

=== 연관 이슈 활용

`note-issue` 와 `investigate-issue` 는 operation-notes 전체를 스캔하여 동일 회사/기능 영역/증상의 이슈를 자동으로 찾는다. 과거에 같은 유형의 문의가 있었다면 해결 방법을 참고할 수 있다.

=== 이슈 완료 후 데이터화

이슈가 해결되면 `/ops:capture-issue CI-XXXX` 로 note 동기화 + 쿡북 증분 반영을 한번에 처리한다. 이슈가 여러 건 쌓였으면 `/ops:maintain-notes` 로 일괄 정리.

=== fix-issue 전에 investigate-issue

복잡한 이슈는 `fix-issue` 로 바로 가기보다 `investigate-issue` 로 원인을 충분히 파악한 뒤 진행하는 것이 효율적이다. `investigate-issue` 의 해결안이 `fix-issue` 의 설계 옵션 선택에 도움이 된다.

=== 근로기준법 관련 이슈

근태/근무/휴가 관련 이슈에서는 `note-issue` 와 `investigate-issue` 모두 관련 법령을 자동 조사한다. 해결안 선택 시 **근로자에게 유리한 방향**이 우선 권장된다.
