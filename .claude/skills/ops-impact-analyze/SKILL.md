# Impact Analyze

## Purpose
버그 확정 후, 수정이 미치는 영향 범위를 분석하고 해결안을 도출한다.
"이 유저만 고쳐주면 끝"이 아니라 "고치면 어디까지 영향이 가는지"를 먼저 판단.

## 전제 조건
- operation-note에 `## 원인 분석` 섹션이 존재해야 함 (investigate 완료)
- 원인 분석 섹션에 `**verdict**: \`bug\`` 마커가 있어야 함 (버그 판정)
- 전제 미충족 시 `ops-investigate-issue` 먼저 실행하라고 안내 후 즉시 종료

## Input
$ARGUMENTS

### Argument Resolution
- `$ARGUMENTS`가 비어있거나 티켓 ID 패턴(`[A-Z]+-\d+`, 예: CI-3861)이 없으면:
  1. 현재 git 브랜치명에서 티켓 ID 패턴을 추출 시도 (예: `fix/CI-3861-some-desc` → `CI-3861`)
  2. 현재 대화 세션에서 언급된 티켓 ID를 추출 시도 (직전 사용자 메시지, 이전에 실행한 커맨드 인자 등)
  3. 브랜치/대화에서 못 찾으면 현재 디렉토리명에서 추출 시도
  4. 찾은 경우: 사용자에게 `"티켓 ID를 {추출한ID}로 인식했습니다. 맞습니까?"` 확인 후 진행
  5. 못 찾은 경우: `"티켓 ID를 특정할 수 없습니다. 티켓 ID를 인자로 전달해주세요. (예: /ops:impact-analyze CI-3861)"` 출력 후 **즉시 종료**

### Operation Notes Directory Resolution

operation-notes 파일을 저장할 디렉토리를 아래 우선순위로 결정한다.
**스킬 실행 시작 시 한 번만 해석하고, 이후 모든 경로에 동일하게 적용한다.**

| 우선순위 | 경로 | 설명 |
|---------|------|------|
| 1 | `{repo-root}/brain/notes/` | repo 루트 brain/notes 디렉토리가 존재 |
| 2 | `{repo-root}/.claude/operation-notes/` | .claude 하위에 존재 (서브모듈용) |
| 3 | `~/.claude/operation-notes/` | 글로벌 홈 디렉토리에 존재 |

- 디렉토리 **존재 여부**로 판단한다 (파일이 아닌 디렉토리).
- 셋 다 존재하지 않으면 사용자에게 `"operation-notes 디렉토리를 찾을 수 없습니다. 어디에 저장할까요?"` 로 물어본다.
- 해석된 경로를 이하 `{notes-dir}`로 표기한다.

### Note File Resolution

`{ticket-id}.md` 파일을 찾을 때 아래 순서로 탐색한다:
1. `{notes-dir}/{ticket-id}.md` (active — 진행 중)
2. `{notes-dir}/archive/{ticket-id}.md` (archive — 완료)

## Procedure

### Step 1: 원인 + 코드 위치 로드

- operation-note에서 확정 원인, 5 Whys, 코드 위치 읽기
- 버그 코드의 정확한 위치 특정
- 수정이 필요한 메서드/클래스 식별
- 전제 조건 불충족 시 (원인 분석 섹션 없음 또는 버그 미판정):
  `"원인 분석이 완료되지 않았습니다. ops-investigate-issue {ticket-id} 를 먼저 실행해주세요."` 안내 후 종료

### Step 2: 사이드이펙트 분석

아래 6개 관점을 검토한다. 모든 항목을 깊이 조사할 필요 없음.
대략적인 위험 지점을 식별하여 수정 담당자가 주의할 포인트를 제공하는 것이 목적이다.

| 관점 | 확인 사항 |
|------|----------|
| 호출자 추적 | 버그가 있는 메서드/함수를 호출하는 다른 코드가 있는가? 그 호출자들은 현재 동작(버그 포함)에 의존하고 있지 않은가? |
| 데이터 정합성 | 버그로 인해 이미 잘못 저장된 데이터가 있는가? 수정 후 기존 데이터와의 불일치가 발생하는가? |
| 다른 고객 영향 | 이 버그가 문의한 고객만의 문제인가, 같은 조건의 다른 고객에게도 발생하는가? |
| 연동/이벤트 | 버그 로직이 이벤트를 발행하거나 외부 시스템에 데이터를 전달하는가? 수정 시 이벤트 발행 패턴이 변경되는가? |
| UI/API 계약 | 수정으로 API 응답 구조나 값이 변경되는가? FE가 현재 응답에 의존하고 있는가? |
| 성능 | 수정이 쿼리 범위 확대나 추가 조회를 유발하는가? |

기록 형식:

```markdown
### 수정 시 사이드이펙트 포인트

| # | 영향 지점 | 위험도 | 설명 |
|---|----------|--------|------|
| 1 | {호출자/기능명} | 🔴/🟡/🟢 | {왜 영향받을 수 있는지, 확인 방법} |
```

위험도 기준:
- 🔴 높음: 수정 전 반드시 검증 필요 (타 기능 깨질 가능성, 데이터 보정 필요)
- 🟡 중간: 수정 후 모니터링 필요 (성능 변화, 이벤트 발행 변경 등)
- 🟢 낮음: 참고 수준 (호출자 있지만 버그에 의존하지 않음)

### Step 3: 해결안 도출

1. **표면 원인 vs 근본 원인 분리**
   - 표면 원인: 증상으로 드러난 직접 원인
   - 근본 원인: 표면 원인을 유발한 구조적/설계적 원인

2. **근로기준법 검토** (해당 시)
   - 근태/급여/휴가 도메인이면 관련 법령 조항 확인
   - 법령 해석에 따라 어느 방향이 근로자에게 유리한지 판단

3. **해결안 제시**

   각 옵션을 아래 형식으로 작성:

   **옵션 A: 즉시 대응** (표면 원인 해결)
   - 내용: 데이터 패치, 설정 변경 등 코드 수정 없이 적용 가능한 조치
   - 장점: {빠른 적용, 리스크 낮음 등}
   - 단점: {근본 원인 미해결, 재발 가능 등}
   - 데이터 보정 필요 시: 보정 방법 포함

   **옵션 B: 근본 해결** (근본 원인 해결)
   - 내용: 코드/구조 수정
   - 장점: {재발 방지, 구조적 개선 등}
   - 단점: {개발·검증 시간 필요, 영향 범위 등}
   - 데이터 보정 필요 시: 보정 방법 포함

   **권장**: 근로자에게 유리하면서 구현이 간단한 옵션을 명시하고 이유를 한 줄로 작성
   - 연관 이슈나 과거 사례가 있으면 참고로 기재

### Step 4: operation-note 업데이트

operation-note에 아래 섹션을 추가한다. 기존 내용은 수정하지 않는다.

```markdown
## 영향 분석

{사이드이펙트 테이블 (Step 2 결과)}

## 해결안

{즉시 대응 vs 근본 해결 옵션 (Step 3 결과)}
```

### Step 5: 보고 + 다음 단계 안내

- 영향 분석 + 해결안 요약 보고 (5-10줄)
- 사실과 추론을 구분하여 보고
- 다음 단계 안내:
  ```
  코드 수정을 진행하려면: ops-fix-issue {ticket-id}
  ```

## Rules
- 외부 시스템(DB, OpenSearch 등) 조회 전 사용자 확인 필수
- 문서는 한국어로 작성
- 추론/예측과 사실을 구분 — 확인되지 않은 영향 지점은 `[추론]` 태그 부착
- DB 쿼리 실행은 메인 세션에서 직접 수행 (subagent 위임 금지)
- 코드 분석 시 repo명과 `파일 경로:라인 번호` 명시

## 🔬 v1 이후 진화 포인트

- 사이드이펙트 6개 관점이 실제로 다 쓰이는지, 빠지는 관점이 있는지
- "수정 범위 결정"을 별도 단계로 분리할 필요가 있는지
- impact-analyze가 필요 없었던 케이스가 있는지
