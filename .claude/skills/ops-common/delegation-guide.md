# Delegation Guide

ops 스킬 실행 중 DB 조회, 로그 검색, 메트릭 수집 등이 필요하지만 실행할 수 없을 때의 fallback 원칙과 위임 포맷을 정의한다.

## Fallback 원칙

1. **코드 분석 우선**: 코드에서 파악 가능한 것은 반드시 코드로 먼저 수행한다 (Controller → Service → Repository 추적, Entity 분석, 에러 핸들링 확인)
2. **쿼리 구성은 항상 수행**: 실행할 수 없어도 Entity 기반 SQL 구성(도메인 라우팅 → Entity 탐색 → SQL 작성)은 동일하게 수행한다. 실행만 위임한다.
3. **위임은 모아서 한 번에**: 여러 쿼리/검색이 필요하면 가능한 한 모아서 위임 요청한다 (왕복 최소화)
4. **결과 수신 후 조사 계속**: 위임 결과를 받으면 즉시 해석하여 이하 절차를 이어간다

## 위임 포맷

실행할 수 없는 쿼리/명령을 다른 담당자에게 전달할 수 있는 형태로 출력한다.

### DB 쿼리 위임

```markdown
📊 **DB 조회 위임 요청**

**목적**: {이 쿼리로 확인하려는 것}
**환경**: {dev/qa/prod} (database prefix: `{flex./flexdev./flexqa.}`)
**긴급도**: {즉시/일반}

\`\`\`sql
-- {질문 요약}
{SQL 쿼리 — LIMIT 포함, database prefix 포함}
\`\`\`

[근거]
- 테이블: `{테이블명}` ← `{repo}` > `{Entity 파일 경로}` `@Table(name = "{테이블명}")`
- 컬럼: `{컬럼명}` ← `{Entity 파일}` `@Column(name = "{컬럼명}")`
- 인덱스: {확인됨 / JPA 추론 / 미확인}

**기대 결과**: {정상이면 어떤 값이어야 하는지}
**이상 시 해석**: {어떤 값이면 어떤 의미인지}

**실행 방법**:
- DB 접근 가능한 담당자에게 이 블록을 전달
- 또는 Metabase/DBeaver에서 직접 실행
```

### OpenSearch 로그 검색 위임

```markdown
🔍 **로그 검색 위임 요청**

**목적**: {이 검색으로 확인하려는 것}
**환경**: {prod/dev}
**Kibana URL**: {base URL 또는 시간 범위·필터 파라미터 포함 URL}

**검색 조건**:
- 인덱스: `{인덱스 패턴}` (예: `flex-app.be-access-2026.04.10`)
- 시간 범위: {KST 기준 시작} ~ {KST 기준 끝}
- 필터:
  - `{필드명}`: `{값}`
- 정렬: `@timestamp` DESC

**기대 결과**: {어떤 로그가 있어야 하는지}
**이상 시 해석**: {로그가 없거나 특정 패턴이면 어떤 의미인지}

**실행 방법**:
- Kibana 대시보드에서 위 조건으로 검색
- 또는 OpenSearch 접근 가능한 담당자에게 전달
```

### AWS CLI 명령 위임

```markdown
☁️ **AWS 메트릭 확인 위임 요청**

**목적**: {확인하려는 메트릭}
**환경**: {dev/prod}

**콘솔 URL** (직접 접속):
{AWS 콘솔 URL — RDS PI, CloudWatch 등}

**CLI 명령** (AWS 접근 가능한 담당자용):
\`\`\`bash
{전체 CLI 명령어}
\`\`\`

**확인 포인트**: {어떤 수치를 봐야 하는지, baseline 대비 비교 방법}
```

## 위임 가이드

| 위임 대상 | 채널 | 적합한 상황 |
|----------|------|-------------|
| 같은 팀 BE 온콜 | Slack DM 또는 팀 채널 | DB 쿼리, OpenSearch 로그 |
| 인프라 담당 | Slack 인프라 채널 | AWS PI/CloudWatch, Grafana |

**위임 시 원칙:**
1. 위임 요청에는 **목적, 쿼리/명령, 기대 결과, 해석 가이드**를 모두 포함한다
2. 한 번의 위임에 필요한 쿼리를 **최대한 모아서** 요청한다
3. 결과를 받으면 **즉시 해석하여 조사를 계속**한다
4. 위임한 쿼리와 결과는 operation-note에 기록한다
