ops-learn의 Phase 4 ② COOKBOOK.md 갱신 시 참조하는 규칙.

## COOKBOOK.md 구조

```markdown

> investigate-issue 실행 전에 이 문서를 먼저 참조하면 조사 시간을 단축할 수 있다.
> 각 항목의 상세는 출처 이슈 노트(`[CI-XXXX](./CI-XXXX.md)`)를 참조.

## 쿡북 추가 대상 판별

operation-note 또는 Linear 티켓이 소스일 때, 먼저 이슈의 해결 방식을 판별한다:

| 해결 방식 | COOKBOOK 반영 | 이유 |
|-----------|-------------|------|
| 코드 수정 (버그 픽스, 기능 추가) | ❌ 스킵 | 코드가 바뀌면 기존 진단 패턴이 무의미 |
| 운영 대응 (DB 수동 수정, config 변경, 고객 안내) | ✅ 반영 | 같은 문의가 반복될 수 있음 |
| 스펙 확인 | ✅ 반영 | 스펙은 운영 지식 — 알아두면 조사 시간 단축 |
| 외부 문서 (Notion, Slack) | ✅ 항상 반영 | 외부 지식은 항상 학습 대상 |

## 조사 플로우 추출 규칙

### 소스 섹션

다음에서 조사 순서를 재구성한다:
- `## 원인 분석` > `### 조사 과정` — 실제 수행한 단계별 기록
- `## 원인 분석` > `### 5 Whys` — 인과관계 체인
- `## 원인 분석` > `### 가설 목록` — 확정/소거된 가설에서 분기 조건 추출
- `## 다음에 같은 문의가 오면` — 이미 정리된 진단 절차 (있으면 우선 참조)

### 작성 원칙

1. **재현 가능한 레시피**로 작성 — 누가 따라해도 같은 결론에 도달
2. 각 단계는 **도구 + 쿼리/조건 + 기대 결과** 구체적 명시
3. **분기점** 명확히 — `├─` / `└─` 로 표현
4. 한 플로우 **5~8단계 이내** — 너무 길면 분리
5. **플로우 ID**: `F{N}` 형식, 도메인 내 순번
6. **트리거**: 이 플로우를 시도해볼 만한 증상/조건 한 줄

### 히트 추적

- 새 플로우 추가 시 `히트: 1`로 시작
- 동일 플로우로 다른 이슈 해결 시 히트 +1, 출처 이슈 추가
- 히트율 높은 플로우를 위로 배치

### 기존 플로우와의 관계

- 기존 플로우와 일치 → 히트 +1, 출처 이슈 추가
- 기존 플로우의 변형 → 분기 추가 또는 별도 플로우 신설
- 완전히 새로운 패턴 → 새 플로우 추가

## COOKBOOK 작성 규칙

- 각 항목은 **1-2줄 요약** — 상세는 원본 소스 참조 링크로 대체
- SQL에서 **고객/유저 ID 제거**, 파라미터(`?`)로 일반화
- **코드 진입점은 포함하지 않는다** — 코드 변경 시 싱크 불일치 위험
- 도메인 분류는 강제하지 않는다 — 해당 도메인이 없으면 신설
- **변경 이력은 항상 기록**
- 스펙 확인 시 시나리오 테스트 연계: 시나리오 테스트가 없으면 `<!-- TODO: 시나리오 테스트 추가 권장 -->` 코멘트

## 외부 문서 소스 (Notion/Slack)에서의 추출

외부 문서는 operation-note와 구조가 다르다. 다음을 중심으로 추출:

- **API 매핑**: 화면/탭 → API 엔드포인트 → 요청/응답 구조
- **성능 제약**: 건수 제한, 타임아웃, 서버 내부 제약
- **설계 의도**: 왜 이렇게 만들었는지 (맥락 섹션)
- **필터/파라미터 조합**: 어떤 조건에서 어떤 API를 호출하는지
- **응답 필드 의미**: 각 필드의 의미와 null/false 조건

이들을 COOKBOOK의 해당 도메인 섹션에 다음 형식으로 추가:
- 진단 체크리스트: 문의 유형별 확인 순서
- API 매핑 테이블: 탭/조건 → API → 비고
- 스펙 설명: 응답 필드 의미, 제약사항
- Swagger/문서 링크: 참조용
