## 증상

- 세콤으로 출퇴근 기록을 받아 flex에 반영하는 구조
- **3/12(목)부터** 특정 구성원(alsrlfkdl@bos-semi.com)만 퇴근이 16:00으로 고정 기록됨[^1]

- DB 확인: `config_value = 'ON_TIME'`, `updated_at = 2026-03-10 13:38:23 KST`[^8]

### 5 Whys

1. 왜 퇴근이 16:00으로 기록되나? → preference가 `ON_TIME`이면 정시 이후 퇴근은 모두 정시로 기록하는 설계[^3]
2. 왜 preference가 `ON_TIME`인가? → `v2_time_tracking_user_config` 에 `ON_TIME` 값이 저장되어 있음[^8]
3. 왜 3/12부터 발생했나? → 2026-03-10 13:38 KST에 설정이 변경됨[^8]
4. 왜 설정이 변경되었나? → **유저 본인이 iOS flex 앱(v2.120.0)에서 2026-03-10 13:38 KST에 직접 변경**[^10]
5. 왜 안희종은 "회사 설정 따름"이라고 판단했나? → (추정) admin 화면에서 **회사 수준의 기본 설정**(REAL_TIME)을 확인한 것으로 보임. 유저 개인 설정(`ON_TIME`)이 회사 설정보다 우선 적용되는 구조를 인지하지 못했을 가능성

### 코드 분석

`WorkClockTimeAdjuster.adjustWorkClockStopTime()` 의 핵심 분기[^3]:

1. preference가 `REAL_TIME` → eventTime(실제 타각 시각) 그대로 기록
2. preference가 `ON_TIME` → 정시 이후 퇴근은 **모두 정시로 기록** (의도된 설계)

### Preference 해석 흐름

1. 유저 설정 조회: `v2_time_tracking_user_config` (key: `work-clock.stop.preference`)[^4]
2. 값이 `CUSTOMER_DEFAULT` → 회사(근무유형) 설정 참조[^5]
3. 회사 설정: `v2_customer_work_record_rule.work_clock_stop_preference`[^6]
4. `rawWorkClockStopPreference`가 null이면 기본값 `REAL_TIME`[^7]

> 💡 **확정**: DB 조회 결과 유저 개인 설정이 `ON_TIME`으로 확인됨[^8]. `CUSTOMER_DEFAULT` → 회사 설정 해석 경로를 타지 않고, 직접 `ON_TIME`이 적용됨.

## 스펙 vs 버그 판별

**스펙에 의한 동작.**

- `ON_TIME` preference가 설정되면 정시 이후 퇴근은 모두 정시로 기록하는 것이 의도된 설계[^3]
- 코드에 주석으로 명시: "캡콤 타각 시각 > 퇴근 정시 시각인 경우도 정시 퇴근으로 간주한다"[^3]
- 문제는 **설정값 자체**이지 시스템 동작이 아님

> ⚠️ 안희종이 "본인 설정은 회사 설정 따름이네"라고 판단한 것[^2]과 DB의 `ON_TIME` 값[^8]이 불일치. OpenSearch 로그 확인 결과, 유저 본인이 iOS 앱에서 직접 `ON_TIME`으로 변경한 것이 확인됨[^10]. 안희종은 admin 화면에서 회사 수준 기본 설정(REAL_TIME)을 확인한 것으로 추정.

## 다음에 같은 문의가 오면

> 세콤/캡스/텔레캅 퇴근 시 퇴근 시간이 정시로 기록되거나, 특정 시간으로 고정되는 문의

1. **퇴근 기록 데이터 수집**: 문의 대상 구성원의 최근 퇴근 기록을 날짜별로 정리 — **이벤트 시간(세콤 타각)** vs **기록된 시간(flex)** 비교
2. **변곡점 특정**: 정상 기록 → 비정상 기록으로 전환된 날짜를 찾는다
3. **설정 확인** (변곡점 전후):
   - `v2_time_tracking_user_config` WHERE `config_key='work-clock.stop.preference'` → `ON_TIME`이면 원인 확정
   - `v2_customer_work_record_rule` WHERE `customerId=?` → 회사 설정이 `ON_TIME`이면 전체 영향
   - `updated_at`이 변곡점과 일치하는지 확인
4. **설정 변경 주체 확인**: OpenSearch `flex-app.be-access-{날짜}` 에서 `PATCH /api/v2/time-tracking/user-config` 검색 → 본인/관리자/시스템 구분
5. **안내**: 본인 변경이면 → flex 앱에서 "현재 시각으로 퇴근" 또는 "회사 설정 따름"으로 재변경 안내

> ⚠️ 퇴근 시간이 **자정(00:00)으로 잘리는** 경우는 다른 원인 — [CI-3979](./archive/CI-3979.md) (다음날 휴가 겹침) 참조
