## 원인 분석

**교대근무에서 휴무일에 연장 발생 시, 퇴근 시각 자동 조정 로직이 `baseAgreedDayWorkingMinutes=0`을 참조하여 일연장 조건을 산출하지 못하는 버그.**

핵심 증거:
- `flex-timetracking-backend` > `UserWorkClockOverWorkInputRuleAdjustedTimeLookUpServiceImpl.kt:229-233` — `baseAgreedDayWorkingMinutes`가 휴무일에 0이므로 `.takeIf { it > 0 }` 결과 null[^5]
- `flex-timetracking-backend` > `WeeklyUnpaidHolidayWorkMinutesCalculationDayBuilder.kt:18` — 휴무일 빌더가 `baseWorkingMinutesPerDay=0`으로 고정[^6]
- 유급휴일(WEEKLY_PAID_HOLIDAY)은 L199-207에서 별도 분기로 `MAX_LEGAL_BASE_WORK_MINUTES_IN_DAY`(480분) 기준 정상 처리[^7]

### 가설 목록

| # | 가설 | 확인 방법 | 상태 |
|---|------|----------|------|
| 1 | 퇴근 자동 조정 로직에서 휴무일의 일연장 기준시간이 0 → 조정 불가 | `UserWorkClockOverWorkInputRuleAdjustedTimeLookUpServiceImpl.kt` L229-233 확인 | ✅ 확정 — `baseAgreedDayWorkingMinutes=0` → null |
| 2 | 스케줄 2개 등록(오전/오후)으로 우선순위 데이터 충돌 | 진재언이 확인: 공통점은 근무유형상 휴무일이며 스케줄 중복이 아닌 휴무일 자체가 원인[^8] | ❌ 소거 |
| 3 | 쉬는 날 설정으로 3/2가 휴일로 잡혀 별도 처리됨 | 김현욱이 "쉬는 날도 별도 설정되어 있어서 3/2가 휴일로 잡히지 않는다"고 확인[^9] | ❌ 소거 |

### 조사 과정

> 💡 **판단 근거**: 진재언이 QnA 스레드에서 코드 위치를 특정[^5] → 코드 확인으로 `baseAgreedDayWorkingMinutes=0` 확정[^6] → 유급휴일과의 비대칭 처리 확인[^7]

<details>
<summary>📋 5 Whys 분석</summary>

1. **왜 퇴근 자동 조정이 안 되나?** → `overWorkMinutesBasis` 계산에서 일연장 조건이 빠져 기준시간이 잘못 산출됨
2. **왜 일연장 조건이 빠지나?** → `일연장조건`이 null이 됨 (`takeIf { it > 0 }` 실패)
3. **왜 null이 되나?** → `baseAgreedDayWorkingMinutes`가 0
4. **왜 0인가?** → `WeeklyUnpaidHolidayWorkMinutesCalculationDayBuilder`가 휴무일의 baseWorkingMinutesPerDay를 0으로 빌드 (설계상 휴무일은 소정근로일이 아니므로)
5. **왜 0이 문제가 되나?** → 교대근무에서는 휴무일에 스케줄을 등록하여 실제 근무할 수 있으나, 퇴근 자동 조정 로직이 이 케이스를 고려하지 않음

</details>

<details>
<summary>📋 코드 상세 분석</summary>

**L229-233: 일연장조건 산출**
```kotlin
val 일연장조건 = context.calculationResult
    .days
    .first { it.date == targetDayWorkSchedule.date }
    .baseAgreedDayWorkingMinutes  // 휴무일: 0
    .takeIf { it > 0 }            // → null
```

**L235-246: 일법내연장조건도 동일하게 0 기반 → null**
```kotlin
val 일법내연장조건 = context.calculationResult
    .days
    .first { it.date == targetDayWorkSchedule.date }
    .let {
        maxOf(0, it.baseAgreedDayWorkingMinutes - ...)  // 0 - ... = 0
    }
    .takeIf { it > 0 }  // → null
```

**L269: 최종 계산에서 null 제외**
```kotlin
listOfNotNull(일연장조건, 일법내연장조건, 주기연장조건, 주기법내연장조건).min()
// 일연장조건=null, 일법내연장조건=null → 주기연장 조건에만 의존
```

**비교: 유급휴일(L199-207)은 별도 분기**
```kotlin
return if (paidHolidayAtTargetDate) {
    getWorkBlocks(
        workMinutesToAdd = maxOf(0, MAX_LEGAL_BASE_WORK_MINUTES_IN_DAY - ...)  // 480분(8시간) 기준
    )
}
```

### 스펙 vs 버그

**버그** — 진재언이 "교대근무상에서 휴무일에 연장이 발생한 경우에 세콤캡스 타각시 자동 조정이 안되는 버그"로 확인[^10]. 유급휴일은 정상 처리되나 휴무일만 누락된 것은 의도된 분기가 아닌 미고려 케이스.

## 해결안 / 조사 방향

### 즉시 대응 (표면 원인 해결)
- 해당 고객의 휴무일 퇴근 기록을 수동으로 조정 (어드민에서 퇴근 시각 직접 수정)

### 근본 해결 (코드 수정)

진재언 제안[^11]: `baseAgreedDayWorkingMinutes`가 아닌 **연장 판단의 기준시간이 되는 값을 `calculationResult`로부터 넘겨받아야** 함.

**옵션 A**: 휴무일도 유급휴일처럼 `MAX_LEGAL_BASE_WORK_MINUTES_IN_DAY`(480분) 기준으로 처리
- 장점: 단순, 유급휴일과 일관성
- 단점: 비즈니스 요구사항 재검토 필요 (휴무일에 정말 일연장이 8시간 기준이어야 하는지)
- 근로자에게 유리 ✅ (기준시간이 높아지면 연장이 덜 발생 → 자동 조정 범위가 넓어짐)

**옵션 B**: `calculationResult`에서 실제 연장 판단 기준시간을 별도 필드로 전달
- 장점: 정확한 기준시간 사용 가능
- 단점: 구현 복잡도 증가, `calculationResult` 인터페이스 변경 필요

> 진재언이 이미 원인을 파악하고 담당하기로 함[^12]. PR [#12008](https://github.com/flex-team/flex-timetracking-backend/pull/12008) 올림, 2026-03-18 배포 예정[^13].
