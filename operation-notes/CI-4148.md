### 스펙 vs 버그

**버그**. 이유:
- 이 검증 로직의 의도는 "취소 후 주기연장일귀속 매핑이 0개가 되는 것을 방지"[^10]
- 그러나 이 케이스에서 취소 후 복귀 대상(181404)은 원래 주기연장일귀속이 아니므로, 0개가 되어도 문제 없음
- 검증이 "취소 후 상태"가 아닌 "취소 대상"만 보고 판단하고 있어, **새로 생성된 근무유형에 distributePeriodOverToDay=true가 붙으면 해당 유저의 모든 예약 취소가 불가능해지는 의도하지 않은 사이드이펙트** 발생

### 근본 원인 (5 Whys)

1. 왜 예약 취소가 안 되나? → `UserWorkRuleCannotRemoveException` 발생[^7]
2. 왜 예외가 발생하나? → `allowCancelMapping = false`[^10]
3. 왜 false인가? → 조건 3(`isAppliedNotUniqueNewWorkRule`)이 false[^9]
4. 왜 조건 3이 false인가? → 자동 생성된 근무유형(267146)에 `distributePeriodOverToDay=true`가 설정되어, 해당 유저에게 매핑된 유일한 주기연장일귀속 근무유형이 됨
5. 왜 자동 생성 근무유형에 이 플래그가 true로 설정되었나? → (추정) 근무유형 복제/자동생성 시 원본의 설정을 그대로 복사했거나, 기본값이 true로 설정된 것으로 보임

## 해결안 / 조사 방향

### 즉시 대응 (표면 원인 해결)

**옵션 A: Operation API로 강제 취소**
- `UserWorkRuleUpdateServiceImpl.removeForMini`는 `skipFutureAppliedWorkRuleValidation=true`, `skipLastAppliedWorkRuleValidation=true`로 호출하여 조건 1, 2를 우회함
- Operation API에 이 메서드를 호출하는 엔드포인트가 있는지 확인 필요
- 42명 전원에 대해 실행 필요

**옵션 B: DB에서 distributePeriodOverToDay를 false로 패치**
```sql
UPDATE v2_customer_work_rule SET distribute_period_over_to_day = 0 WHERE id = 267146;
```
- 패치 후 UI에서 정상 취소 가능
- 단, 해당 근무유형이 실제로 주기연장일귀속이 필요한지 확인 필요

### 근본 해결 (근본 원인 해결)

**옵션 C: 조건 3 로직 수정**
- 현재: "취소 대상이 유일한 주기연장일귀속이면 차단"
- 수정안: "취소 후 남은 활성 매핑 중 주기연장일귀속이 0개가 되면 차단" (취소 후 상태 기준 판단)
- 또는: 미래 예약(아직 적용 전)인 경우 조건 3을 면제

**옵션 D: 근무유형 자동 생성 시 distributePeriodOverToDay 설정 로직 검토**
- 왜 자동 생성된 "10-7시_260317"에 이 플래그가 true로 설정되었는지 확인
- 원본 근무유형(181404)은 NULL/false인데 새 근무유형에 true가 붙은 경위 조사

### 권장 순서
1. **옵션 A 또는 B**로 42명 즉시 해결
2. **옵션 D**로 자동 생성 로직의 distributePeriodOverToDay 설정 경위 조사
3. **옵션 C** 검토 — 조건 3 로직이 "취소 후 상태"를 기준으로 판단하도록 개선
