## 원인 분석

**정산 자물쇠 해제 후 재처리 시, 구성원의 기본 공제 대상(부양가족 수)이 최신 상태로 갱신되어 소득세가 변경됨.**

고객이 "일부 근로자들의 특이사항이 있어 추가 반영"한 것이 **부양가족 추가**였으며, 정산 재처리(PAYEES 단계 재실행) 시 전체 구성원의 payee 스냅샷이 최신화되면서 joy/yaaung의 `dependent_families_count`도 갱신됨.[^14]

- 간이세액표는 가족 수에 따라 다른 세액을 조회하므로(`tax1Person`, `tax2People`, ...), 부양가족 수가 변경되면 소득세가 달라지는 것은 정상 동작[^12]
- 고객은 joy/yaaung을 직접 수정하지 않았지만, 정산 재처리는 **전체 대상자**의 payee 정보를 최신화함[^14]

**가설 3 (징수세액 비율)**:
- 두 구성원의 모든 원천세 공제 대상자 레코드에서 `income_tax_charge_type=100` 확인[^11]

- 건강보험 관련 변경(76번 코드 키인, 2026-03-05)은 소득세 무관

**가설 5 (과세표준 변경)**:

### 조사 과정

> 소득세 계산 공식: `최종 소득세 = 간이세액표 세액(과세표준, 가족수) × chargePercent - 누적감면액`[^12]

세액 결정 입력값을 순서대로 소거한 뒤, 고객 추가 정보에서 원인 확정:
1. 간이세액표 버전 → CSV 동일 (소거)
2. 미성년자녀 수 → under_age=0, 변경 없음 (소거)
3. 징수세액 비율 → 100 고정 (소거)
4. 코드 변경 → 관련 변경 없음 (소거)
5. 과세표준 → 고객 진술상 변동 없음 (소거)
6. **부양가족 수(dependent_families_count)** → 정산 재처리 시 최신화되어 변경됨 (**확정**)[^14]

### 스펙 vs 버그: **스펙**

정산 자물쇠 해제 후 재처리 시 PAYEES 단계가 전체 대상자의 payee 스냅샷을 최신화하는 것은 **의도된 동작**이다. 부양가족 수가 변경된 구성원의 소득세가 재계산되는 것은 간이세액표의 정상적인 가족 수 기반 세액 조회 결과이다.

고객이 혼동한 지점: "직접 수정하지 않은 구성원"이라도 정산 재처리 시 최신 구성원 정보가 반영되므로, 1차 정산과 수정 정산 사이에 구성원 정보(부양가족 등)가 변경되었다면 소득세가 달라질 수 있다.

## 다음에 같은 문의가 오면

1. **증상 확인**: 정산 수정(자물쇠 해제 후 재처리) 시 수정하지 않은 구성원의 소득세가 변경됨
2. **원인 확인**: 1차 정산 완료 ~ 정산 수정 사이에 해당 구성원(또는 다른 구성원)의 **기본 공제 대상(부양가족 수)** 변경이 있었는지 확인
   <details>
   <summary>payee 스냅샷 확인 쿼리</summary>

   ```sql
   SELECT user_id, dependent_families_count, under_age_dependent_families_count
   FROM flex_payroll.work_income_settlement_payee
   WHERE settlement_id = {settlement_id}
     AND user_id = {user_id};
   ```

   </details>
3. **안내**: 정산 재처리 시 PAYEES 단계에서 전체 대상자의 구성원 정보가 최신화되므로, 1차 정산 이후 변경된 부양가족 정보가 반영되어 소득세가 달라질 수 있음 → **정상 동작(스펙)**

## 해결

- 코드 수정 불필요 (스펙)
- 고객에게 안내: 정산 수정 시 전체 구성원의 최신 정보가 반영되며, 부양가족 수 변경으로 인한 소득세 재계산은 정상 동작임[^15]

    그러면 해당 구성원의 소득세는 이전 정산과 동일하다
