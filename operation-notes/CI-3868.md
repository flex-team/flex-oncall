## 증상

- 1월 26일자로 연장 포괄임금계약을 30시간 → 52시간으로 변경 (1월 26일보다 뒤늦은 날짜에 적용)
- 1월 1일~1월 31일 기간 내 52시간 이내의 연장근무가 발생
- 포괄계약시간 이내임에도 불구하고 공제되지 않는 현상

## 원인 분석 (확정)

### 판정: 스펙에 의한 정상 동작

API 데이터(`/api/operation/v2/exceeded-work/customers/46230/users/861174/exceeded-works`)를 통해 확인한 결과, 시스템이 설계대로 동작하고 있음을 확인.

### 데이터 기반 분석

유저: Yunseok.Choi@teamsparta.co (userId: 861174)
근무유형: 선택근무제 (FULL_FLEXIBLE), 산정주기: MONTH/1

시스템이 1월을 다음 3개 범위로 분할:

#### Range 1: 1/1~1/25 (구계약 30h)

| 항목 | 값 |
|------|-----|
| 포괄계약시간(연장) | 1451min (= 1800min × 25/31) |
| 잔여 포괄계약시간 | **1451min (전량 미사용)** |
| 해당 범위의 OVER 초과근무 | **0min** |
| 일별 초과근무 유형 | NIGHT, HOLIDAY만 존재 (OVER 없음) |

#### Range 2: 1/26~1/31 (신계약 52h)

| 항목 | 값 |
|------|-----|
| 포괄계약시간(연장) | 603min (= 3120min × 6/31) |
| 잔여 포괄계약시간 | **0min (소진)** |
| 주기연장(REGARDED_OVER) | 2010min |
| 실제 연장(OVER) origin | 1044min |
| 공제 후 인정(OVER) recognized | **441min** (= 1044 - 603) |

#### Range 3: 2/1~2/28 (52h 전체 적용)

| 항목 | 값 |
|------|-----|
| 포괄계약시간(연장) | 3120min (= 52h, 온전한 월) |
| 잔여 포괄계약시간 | 3120min |

### 핵심 메커니즘

1. **주기연장(REGARDED_OVER)은 주기 종료일이 속한 Range에 귀속됨**
   - 1월 산정주기 종료일 = 1/31 → Range 2에 귀속
   - Range 1에는 주기연장이 배분되지 않음 → 포괄계약시간 1451min이 **통째로 미사용**

2. **Range 2의 포괄계약시간(603min)만으로 OVER 1044min을 공제**
   - 603min만 공제 → 나머지 441min이 초과근무 수당 대상으로 인정

3. **결과적 괴리**
   - 총 포괄계약시간: 1451 + 603 = **2054min (≈34.2h)**
   - 실제 OVER: **1044min (≈17.4h)** → 포괄계약시간 내
   - 실제 공제: **603min** → 나머지 441min이 수당 발생
   - Range 1의 1451min이 사용되지 않아 발생하는 불일치

### 왜 이렇게 동작하는가

`ExceededWorkMinuteCalculator`의 설계:
- 범위별로 독립적으로 포괄계약시간 잔여량을 관리
- 주기초과근무(periodResult)는 주기 종료일이 포함된 범위에만 귀속
- 범위 간 잔여량 이월 없음

이 설계는 계약이 변경되지 않는 일반적인 경우에는 문제가 없으나, **월 중간에 계약 변경 + 주기연장이 주기 종료일 범위에 집중**되는 경우 직관과 다른 결과를 만든다.

### 처리 흐름

```
getUserExceededWorkResultsByBasis()
  ├─ 1. 초과근무 산정주기 조회 및 변환 (L273-298)
  ├─ 2. 근무 스케줄 조회 (L315-403)
  ├─ 3. 임금 계약 병합 (L405-431)
  ├─ 4. UserExceededWorkCalculationBasisGenerator.generate() 호출 (L434-448)
  │     └─ 포괄계약 + 산정주기의 경계점 기반 날짜 범위 분할
  └─ 5. 분할된 범위별 초과근무 분 계산 (L451-463)
```

### 경계점 기반 분할 알고리즘 (`UserExceededWorkCalculationBasisGenerator.generate()`)

1. 임금계약(포괄계약)과 초과근무 산정주기의 **모든 시작일/종료일+1일**을 경계점으로 수집
2. 경계점들을 정렬 후 중복 제거
3. 인접한 경계점 쌍으로 날짜 범위 생성 (`zipWithNext`)
4. 각 범위에 해당하는 산정주기와 포괄계약을 `find()`로 매칭
5. 산정주기가 없는 범위는 제외

이 이슈의 경우:
- 경계점: `1/1`, `1/26`, `2/1`, `3/1`
- 분할 결과: `1/1~1/25`(구계약 30h), `1/26~1/31`(신계약 52h), `2/1~2/28`(52h)

## 해결 방향

### 고객 안내 (즉시)

1. 포괄계약 변경 시 비율 적용 방식 설명:
   - "1월 중간(26일)에 계약을 변경하면, 1~25일은 구계약(30h) 기준, 26~31일은 신계약(52h) 기준으로 각각 일수 비율 적용됩니다"
   - "월 전체에 52h를 적용하려면 계약 시작일을 1월 1일로 설정해야 합니다"

2. 현재 상태에서의 해결 방법:
   - 포괄계약의 beginDate를 1월 1일로 변경 → 1월 전체에 52h 적용
   - 단, 이 경우 1/1~1/25에도 52h 기준이 적용되므로 기존 30h 계약과의 정합성 확인 필요

### 장기적 개선 검토 (선택)

- 주기연장을 범위 간 분산 배분하는 옵션 검토 가능
- 단, 기존 계약 유지 고객에게 영향이 없도록 설계 필요
- 현재로서는 스펙 변경보다 운영 가이드 강화가 적절
