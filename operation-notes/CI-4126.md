## 원인 분석

**사용자가 2025년 root 목표 하위에 2026년 자식 목표를 생성한 cross-year 트리 구조가 원인.** BE는 이를 `hit=false`로 구분하여 의도대로 동작하나, FE가 `hit` 필드를 무시하고 동일하게 렌더링하여 사용자 혼란 유발.

### 데이터 구조 (access log 응답에서 확인)[^12][^24]

2025 root 3건은 모두 `depth=0`, `parent=null`인 진짜 root 목표. 하위에 2026 자식이 존재:

```
2025 "해긴 전사 목표" (root, hit=false, owner=경영기획실)
  └── ... → 2026년 자식 목표 (root_objective_id가 이 2025 목표를 가리킴)

2025 "[현지화 파트]" (root, hit=false, owner=현지화파트)
  └── ... → 2026년 자식 목표

2025 "[운영파트]" (root, hit=false, owner=운영파트)
  └── ... → 2026년 자식 목표
```

### 근본 원인 (5 Whys)

1. **왜 2025 목표가 보이나?** → BE 응답에 `hit=false`로 2025 목표 포함[^12]
2. **왜 BE가 2025를 포함하나?** → `findRootObjectives` SQL이 root에 cycle 필터 미적용[^14]
3. **왜 root에 cycle 필터가 없나?** → 설계 의도: cross-year 트리의 root를 보여주되 `hit` 필드로 구분[^13]
4. **왜 사용자가 혼동하나?** → **FE가 `hit=false`를 시각적으로 구분하지 않음** ← 진짜 원인
5. **왜 cross-year 트리가 존재하나?** → 사용자가 2026 목표를 2025 root 하위에 생성 (정상 데이터)[^24]

### 근본 원인 (5 Whys)

1. **왜 2025 목표가 보이나?** → BE 응답에 2025년 cycle 목표가 포함됨[^12]
2. **왜 BE가 2025를 반환하나?** → `findRootObjectives` SQL이 root objective에 cycle 필터를 적용하지 않음[^14]
3. **왜 root에 cycle 필터가 없나?** → 설계 의도: "2026년에 해당하는 목표의 root를 보여준다"는 트리 탐색 목적[^13]
4. **왜 사용자에게 혼동을 주나?** → FE가 `hit: false`인 목표를 시각적으로 구분하지 않고 동일하게 표시

<details>
<summary>📋 원인 SQL 분석</summary>

`ObjectiveV3JpaRepository.findRootObjectivesOrderByDisplayOrderAsc()`[^14]:

```sql
SELECT root_o.*
FROM (
    SELECT DISTINCT o.root_objective_id
    FROM objective_v3 o
    WHERE o.customer_id = :customerId
      AND o.deleted_at IS NULL
      AND o.formatted_cycle IN (:formattedCycles)  -- ✅ 2026년 사이클로 필터
) AS distinct_roots
INNER JOIN objective_v3 root_o ON distinct_roots.root_objective_id = root_o.id
WHERE root_o.customer_id = :customerId AND root_o.deleted_at IS NULL
-- ⚠️ root_o에는 cycle 필터가 없음!
```

예시 데이터 흐름:
- 2026년 목표 A → `root_objective_id`가 2025년 목표 B
- 서브쿼리: 2026년 목표 A의 `root_objective_id` = B를 수집
- JOIN: B를 조회 → B는 2025년이지만 cycle 필터 없이 반환

서비스 레이어에서 `hitMap`으로 구분[^13]:
```kotlin
// cycle 조건에 맞지 않을 수 있다. Hit 여부를 어플리케이션 레벨에서 처리
val cycleIds = cycles.toSet()
val hitMap = objectives.associate { it.identity.id.toString() to (it.cycle in cycleIds) }
```
→ 2025 root는 `hit: false`로 설정되어 응답에 포함됨

### 스펙 vs 버그 판별

**BE는 의도된 스펙, FE는 UX 버그.**

- **BE 스펙 근거**: 코드 주석에 명시적으로 "root는 cycle 조건에 맞지 않을 수 있다"고 설계[^13]. `hit` 필드로 구분하도록 설계됨.
- **FE 버그 근거**: `hit=false`인 목표를 시각적으로 구분하지 않아 사용자가 "2025년 목표가 보인다"고 인식.

## 해결안

### 옵션 A: FE 수정 (권장)
- `hit=false` 목표를 흐리게 표시 / 접기 / 숨기기
- 가장 설계 의도에 부합하는 수정

### 옵션 B: BE 서비스 레이어 수정
- `filterRootObjective`에서 `hit=false`인 목표를 응답에서 제외
- FE 수정 없이 즉시 효과, 단 트리 탐색 UX가 변경됨

### 옵션 C: BE SQL 수정
- `findRootObjectives` SQL의 root JOIN에 cycle 필터 추가
- cross-year 트리 탐색이 불가능해지므로 목표 팀 협의 필수

### 옵션 D: Product 결정
- cross-year 트리 생성 자체를 제한 (2026 목표는 2026 root 아래에만 생성 가능)
- 기존 cross-year 데이터 마이그레이션 필요
