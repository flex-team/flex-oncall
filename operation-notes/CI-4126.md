## 원인 분석

**사용자가 2025년 root 목표 하위에 2026년 자식 목표를 생성한 cross-year 트리 구조가 원인.** BE는 이를 `hit=false`로 구분하여 의도대로 동작하고, FE도 `hit=false` 목표를 회색으로 구분 표시하여 정상 동작. 사용자가 회색 표시의 의미를 인지하지 못해 혼란이 발생한 것.[^25][^27]

### 데이터 구조 (access log 응답에서 확인)[^12][^24]

2025 root 3건은 모두 `depth=0`, `parent=null`인 진짜 root 목표. 하위에 2026 자식이 존재:

```
2025 "해긴 전사 목표" (root, hit=false, owner=경영기획실)
  └── ... → 2026년 자식 목표 (root_objective_id가 이 2025 목표를 가리킴)

2025 "[현지화 파트]" (root, hit=false, owner=현지화파트)
  └── ... → 2026년 자식 목표

2025 "[운영파트]" (root, hit=false, owner=운영파트)
  └── ... → 2026년 자식 목표
```

### 각 레이어별 판정

| 레이어 | 판정 | 근거 |
|--------|------|------|
| **데이터** | ✅ 정상 | 사용자가 의도적으로 2026 목표를 2025 root 하위에 생성[^24] |
| **BE SQL** | ✅ 의도된 설계 | 주석에 "cycle 조건에 맞지 않을 수 있다" 명시[^13]. `hit` 필드로 구분 |
| **Matrix** | ✅ 정상 | 2025 사이클로 정상 저장. goal-api가 ID를 보냈을 뿐[^17] |
| **FE** | ✅ 정상 | `hit=false` 목표를 **회색으로 구분 표시**하고 있음[^27] |

### 흐름 요약

```
FE: cycles=[2026_YEARLY, ...] 요청
  → goal-api: findRootObjectives SQL
    → 서브쿼리: 2026 목표 찾기 → root_objective_id 수집 (2025 root 포함)
    → root JOIN: cycle 필터 없이 root 조회 → 2025 root 3건 포함
  → goal-api → Matrix: id IN (...) 호출 (spanId: 959080cad667104e)[^17]
    → Matrix: 정상 반환 (2025 데이터 포함)
  → goal-api: hit=false 표시하여 응답 (2026 14건 hit=true + 2025 3건 hit=false)[^12]
FE: hit=false → 회색 표시, hit=true → 정상 표시 (정상 동작)
→ 사용자가 회색 목표의 의미를 인지하지 못해 혼란
```

### 근본 원인 (5 Whys)

1. **왜 2025 목표가 보이나?** → BE 응답에 `hit=false`로 2025 목표 포함[^12]
2. **왜 BE가 2025를 포함하나?** → `findRootObjectives` SQL이 root에 cycle 필터 미적용[^14]
3. **왜 root에 cycle 필터가 없나?** → 설계 의도: cross-year 트리의 root를 보여주되 `hit` 필드로 구분[^13]
4. **왜 사용자가 혼동하나?** → FE는 회색으로 구분 표시하지만[^27], 사용자가 회색 목표의 의미를 인지하지 못함
5. **왜 cross-year 트리가 존재하나?** → 고객이 최근(지난주 금요일 이후) 2025 root 하위에 2026 목표를 배치[^26] ← **진짜 원인**

<details>
<summary>📋 원인 SQL 분석</summary>

`ObjectiveV3JpaRepository.findRootObjectivesOrderByDisplayOrderAsc()`[^14]:

```sql
SELECT root_o.*
FROM (
    SELECT DISTINCT o.root_objective_id
    FROM objective_v3 o
    WHERE o.customer_id = :customerId
      AND o.deleted_at IS NULL
      AND o.formatted_cycle IN (:formattedCycles)  -- ✅ 2026년 사이클로 필터
) AS distinct_roots
INNER JOIN objective_v3 root_o ON distinct_roots.root_objective_id = root_o.id
WHERE root_o.customer_id = :customerId AND root_o.deleted_at IS NULL
-- ⚠️ root_o에는 cycle 필터가 없음!
```

예시 데이터 흐름:
- 2026년 목표 A → `root_objective_id`가 2025년 목표 B
- 서브쿼리: 2026년 목표 A의 `root_objective_id` = B를 수집
- JOIN: B를 조회 → B는 2025년이지만 cycle 필터 없이 반환

서비스 레이어에서 `hitMap`으로 구분[^13]:
```kotlin
// cycle 조건에 맞지 않을 수 있다. Hit 여부를 어플리케이션 레벨에서 처리
val cycleIds = cycles.toSet()
val hitMap = objectives.associate { it.identity.id.toString() to (it.cycle in cycleIds) }
```
→ 2025 root는 `hit: false`로 설정되어 응답에 포함됨

### 스펙 vs 버그 판별

**스펙 확정.** BE·FE 모두 의도된 동작.[^25][^27]

- **BE**: cross-year 트리의 root를 `hit=false`로 표시하여 응답에 포함 — 관계성 표시 목적[^13][^25]
- **FE**: `hit=false` 목표를 **회색으로 구분 표시** — 정상 동작[^27]
- **원인**: 고객이 최근 2025 root 하위에 2026 목표를 배치하여 cross-year 구조가 생성됨[^26]

## 해결

**스펙 확정 → 고객 안내로 해결.** 임준규(목표 팀)가 데이터를 직접 확인하여 2025 목표 하위에 2026 목표가 실제로 존재함을 확인[^33][^34]. 코드 수정 불필요.

안내 포인트:
1. 2025년 목표(회색) 하위를 펼치면 2026년 목표가 있음 — 이것이 회색 표시의 원인[^33]
2. 2025년 목표 하위에서 2026년 목표를 분리하면 2025년 목표가 더 이상 표시되지 않음[^25]
3. 회색 목표는 "직접 필터에 해당하지는 않지만, 하위에 해당하는 목표가 있어서 관계성을 보여주기 위해 표시"되는 것[^25]

## 다음에 같은 문의가 오면

1. **먼저 확인**: `root-objectives` API 응답에서 `hit=false` 목표가 있는지 확인 (access log의 responseBody)
2. **원인 판별**: `hit=false` + `hasChild=true`이면 → cross-year 트리. 해당 목표 하위에 요청 사이클에 해당하는 자식 목표가 존재
3. **조치**: 고객에게 안내 — "회색 목표 하위를 펼쳐보시면 올해 목표가 있습니다. 이전 연도 목표 하위에서 올해 목표를 분리하시면 해결됩니다"
4. **참고**: `root-objectives-by-aside` API는 Matrix 기반 검색이라 이 현상 없음. `root-objectives` API에서만 발생
