## 원인 분석

일별 리포트의 "스케줄 시작/종료" 컬럼에 **잘못된 시간이 표시되는 버그**. Converter에서 교대 그룹의 시간 범위를 추출하는 로직에 2가지 결함이 존재한다.

### 조사 과정

> 💡 **판단 근거**: 엑셀 컬럼 정의[^2] → Converter 변환 로직[^3] → `toGroupDtos()` 정렬 없음[^4] → 정렬 미보장 확정

<details>
<summary>📋 상세 코드 분석</summary>

**1. 엑셀 컬럼 정의** (`UserDailyWorkScheduleResultByShiftWorkRuleExcelWriter.kt:35-37`):

```kotlin
column("근무 스케줄") { it.shiftScheduleNames.oneOrAllAsString() }
column("스케줄 시작") { it.shiftSchedules.firstOrNull()?.from ?: "" }
column("스케줄 종료") { it.shiftSchedules.lastOrNull()?.to ?: "" }
```

- `firstOrNull()?.from` → 컬렉션의 **첫 번째** TimeRange의 시작
- `lastOrNull()?.to` → **마지막** TimeRange의 종료
- 컬렉션 순서가 시간순이 아니면 잘못된 값 표시

**2. Converter의 TimeRange 생성** (`DailyShiftUserWorkScheduleExportDataNightType5Converter.kt:67-79`):

```kotlin
shiftSchedules = workShiftScheduleMap[model.date]
    ?.mapNotNull {
        // FIXME: 첫 블록이 근무 블록이라는 가정이 들어감
        val from = it.shiftBlocks.firstOrNull()?.timeBlockFrom
        val to = it.shiftBlocks.firstOrNull()?.timeBlockTo
        if (from == null || to == null) null
        else TimeRange(from, to)
    }
    ?: emptyList()
```

- 각 `timeBlockGroup`의 `shiftBlocks.firstOrNull()`에서 from/to **모두** 추출
- 교대 그룹에 여러 블록(근무-휴게-근무)이 있으면 **첫 블록의 시간만 반영**
- 예: 17-20시 교대 = [17:00-18:00, 18:00-18:30(휴게), 18:30-20:00] → TimeRange(17:00, 18:00)만 생성

**3. `toGroupDtos()` 정렬 없음** (`UserDayWorkShiftTimeBlockGroupDto.kt`):

```kotlin
fun Collection<TemplateWorkPlanTimeBlockModel>.toGroupDtos() =
    this.groupBy { it.timeBlockGroupId }
        .map { UserDayWorkShiftTimeBlockGroupDto(...) }
```

- `groupBy().map()` → LinkedHashMap 순서 = 입력 순서. **시간순 정렬 없음.**
- 17-20시 그룹이 8-9시 그룹보다 먼저 오면 `firstOrNull()?.from` = 17:00

**4. `associateBy` 문제** (`DailyShiftUserWorkScheduleExportDataNightType5Converter.kt:45-51`):

```kotlin
val workShiftScheduleMap = userShiftSchedule
    ?.shiftSchedules
    ?.associateBy(
        keySelector = { it.date },
        valueTransform = { it.timeBlockGroups }
    )
```

- `shiftSchedules`는 `workPlanLookUpModels.flatMap { it.toResultDtos() }.sortedBy { it.date }`로 생성[^5]
- 사용자에게 2개의 **별도 교대 배치**(workPlan)가 있으면 동일 날짜 항목 2개 → `associateBy`가 마지막 것만 보존
- 1개 배치 안에 2개 timeBlockGroup이면 영향 없음

**5. NightType3 Converter도 동일 로직**: `DailyShiftUserWorkScheduleExportDataNightType3Converter.kt`에 같은 코드 존재

</details>

### 근본 원인 (5 Whys)

1. **왜** 리포트의 스케줄 시작시간이 실제와 다른가? → `firstOrNull()?.from`이 시간상 가장 이른 스케줄을 가져오지 못함
2. **왜** 가장 이른 스케줄을 못 가져오는가? → `timeBlockGroups`가 시간순으로 정렬되지 않음[^4]
3. **왜** 정렬되지 않는가? → `toGroupDtos()`가 `groupBy(timeBlockGroupId).map()`으로 생성하여 **입력 순서**(LinkedHashMap)를 따름
4. **왜** 정렬 처리를 하지 않았는가? → (추정) 하루에 1개 교대 그룹만 있는 것이 일반적이어서, 복수 그룹 시나리오가 고려되지 않음

### 스펙 vs 버그 판별

**버그**. 스케줄 화면에서는 두 스케줄이 정확히 표시되므로, 리포트도 동일한 시간을 반영해야 한다. FIXME 주석의 존재가 개발자도 이 가정의 위험성을 인지하고 있었음을 보여준다.

## 해결안 / 조사 방향

### 즉시 대응 (표면 원인 해결)

**Option A**: Converter에서 `shiftSchedules` 생성 후 **시간순 정렬** 추가

<details>
<summary>Option A 코드 예시</summary>

```kotlin
shiftSchedules = workShiftScheduleMap[model.date]
    ?.mapNotNull { ... }
    ?.sortedBy { it.from }  // 시간순 정렬 추가
    ?: emptyList()
```

</details>

- 장점: 최소 변경으로 `firstOrNull()`/`lastOrNull()` 정합성 확보
- 단점: `shiftBlocks.firstOrNull()` 문제(to 값 부정확)는 미해결

### 근본 해결 (근본 원인 해결)

**Option B**: TimeRange 생성 시 그룹 전체의 min(from)/max(to) 사용 + 정렬

<details>
<summary>Option B 코드 예시</summary>

```kotlin
shiftSchedules = workShiftScheduleMap[model.date]
    ?.mapNotNull { group ->
        val from = group.shiftBlocks.minOfOrNull { it.timeBlockFrom }
        val to = group.shiftBlocks.maxOfOrNull { it.timeBlockTo }
        if (from == null || to == null) null
        else TimeRange(from, to)
    }
    ?.sortedBy { it.from }
    ?: emptyList()
```

</details>

- 장점: 다중 블록 교대의 전체 시간 범위 정확히 반영 + 정렬
- 단점: 변경 범위가 Option A보다 넓음 (NightType3, NightType5 모두 수정)

**Option C** (가설 2 해당 시): `associateBy` → `groupBy` + `flatMap`

<details>
<summary>Option C 코드 예시</summary>

```kotlin
val workShiftScheduleMap = userShiftSchedule
    ?.shiftSchedules
    ?.groupBy { it.date }
    ?.mapValues { (_, v) -> v.flatMap { it.timeBlockGroups } }
    ?: emptyMap()
```

</details>

- 이 수정은 Option B와 **함께 적용** 권장

### 추천

**Option B + C** 를 함께 적용. 정렬 + 전체 범위 반영 + 중복 날짜 보존을 모두 해결.
