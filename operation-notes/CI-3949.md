## 원인 분석

이 티켓에는 **두 가지 별개의 이슈**가 혼재:

| # | 이슈 | 증상 | 상태 |
|---|------|------|------|
| A | OS 문서 미생성 | 3/2가 휴일대체 탭에 안 보임 | 수동 sync로 즉시 해결. 근본 수정 미완료 |
| B | 2/14 `holidayProps` null | OS 문서는 있으나 조회 안 됨 | **스펙대로 동작** |

### 이슈 A: OS 문서 미생성

**자동근무 미사용 + 근무 미수정 유저는 sync 이벤트 미발생 → OS 문서 없음 → 휴일대체 탭 미표기. 알려진 아키텍처 이슈.**[^4][^5]

- 휴일대체 API는 `ORIGINAL_HOLIDAY` exists 필터로 조회[^6] — OS 문서 없으면 결과 없음
- OS 문서 생성은 이벤트 기반 sync에 의존[^7] — 근무 미수정 유저는 이벤트 미발생
- **즉시 대응**: 수동 sync 실행 완료, 이주화 확인 "네엡 잘 보입니다!"[^8]

### 이슈 B: 2/14 `holidayProps` null — 스펙

**2/14 시점 활성 근무유형 "기본근무"(223469)의 토요일 = `WEEKLY_UNPAID_HOLIDAY`(휴무일)[^9] → 제외 대상. "생산 근로자" 적용은 2/23부터.[^10]**

- 코드에서 `WEEKLY_PAID_HOLIDAY`만 originalHolidays에 추가[^11] — 휴무일은 제외
- 근로기준법 제55조: 휴일대체는 **유급휴일** 대상 → 무급 휴무일 제외는 법적으로 정합[^17]

### 조사 과정

> 💡 **이슈 A**: OS에 데이터 없음[^4] → `ORIGINAL_HOLIDAY` exists 필터[^6] → sync 이벤트 의존[^7] → Slack 동일 패턴 보고[^5] → **확정**

> 💡 **이슈 B**: re-sync 후에도 `holidayProps` null[^3] → 고정근무유형 확인[^12] → "생산 근로자" 토요일 = `WEEKLY_PAID_HOLIDAY`[^13]... 이상한데? → `v2_user_work_rule` 확인: 적용 2/23부터[^10] → 2/14 시점 "기본근무" 토요일 = 휴무일[^14] → 코드에서 제외[^11] → **스펙 확정**

<details>
<summary>5 Whys (이슈 A)</summary>

1. 왜 휴일대체 탭에 3/2가 안 보이나? → `ORIGINAL_HOLIDAY` exists 필터를 통과하지 못함
2. 왜 `ORIGINAL_HOLIDAY`가 없나? → 유저+날짜의 OS 문서 자체가 없음
3. 왜 OS 문서가 없나? → sync 이벤트가 미발생
4. 왜 sync 이벤트가 없나? → 자동근무 미사용 + 근무 미수정으로 트리거 조건 미충족
5. 왜 월간 배치가 커버하지 못했나? → enrichment에서 근무 데이터 없는 유저는 빈 결과 반환[^16]

</details>

## 관련 법령

<details>
<summary>근로기준법 제55조 (휴일대체 법적 근거)</summary>

3월 2일은 2026년 삼일절 대체공휴일. 근로기준법 제55조 제2항 단서가 휴일대체의 법적 근거.

> **제1항**: 사용자는 근로자에게 1주에 평균 1회 이상의 유급휴일을 보장하여야 한다.
> **제2항**: 사용자는 근로자에게 대통령령으로 정하는 휴일을 유급으로 보장하여야 한다. 다만, 근로자대표와 서면으로 합의한 경우 특정한 근로일로 대체할 수 있다.
