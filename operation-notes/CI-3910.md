## 현재까지 파악된 내용

### 참조자 알림 아키텍처 (코드 분석)

참조자(referrer) 알림은 **독립적인 알림 시스템이 아니라 승인(Approval) 프로세스의 일부**로 동작한다. *(코드: tracking-approval/service/.../TrackingApprovalServiceImpl.kt:29-129, time-tracking/consumer/.../TimeOffApprovalProcessStartedMassageHandler.kt:91-151)*

**알림 발송 플로우:**
```
1. 휴가 등록
   ↓
2. 승인 프로세스 생성 (TrackingApprovalServiceImpl)
   - 승인 정책에서 참조자(referrer) 목록 조회
   - 각 참조자의 resolveTarget을 사용자 ID로 resolve
   - notificationSetting (OFF / END_ONLY / START_AND_END) 설정 확인
   ↓
3. 승인 이벤트 발행 (외부 approval-starter 라이브러리)
   - notificationSetting 기반으로 수신자 필터링
   ↓
4. 알림 팩토리 호출 (ApprovalProcessStartedMessageContextFactory)
   - onStartToReferrers(receiverUserIds) — 이미 필터링된 사용자 ID 전달
   ↓
5. 핸들러 실행 (TimeOffApprovalProcessStartedMassageHandler.handleReferrers())
   - FlexTimeTrackingTimeOffRequestApproveReferMessageContext 생성
   ↓
6. 메시지 큐 발송 → 외부 notification 서비스 → 이메일 발송
```

### 참조자 resolve 구조

참조자의 `resolveTarget`은 **참조자를 어떻게 구성원으로 변환할지를 정의하는 필드**이다. 예를 들어, "대표이사" 조직을 참조자로 등록했다면, 실제 사용자를 구하기 위해 그 조직의 구성원을 조회(resolve)하는 방식을 `resolveTarget`에 정의한다. 그 타입은 5가지가 있다: *(코드: tracking-approval/model/.../ResolvedApprovalProcessModel.kt:104-110)*

| 타입 | 설명 | resolve 방식 |
|------|------|-------------|
| `USER` | 특정 사용자 | 직접 매핑 |
| `DEPT` | 조직(부서) | 조직 구성원 조회 |
| `DEPT_WITH_CHILDREN` | 조직 + 하위 조직 | 조직 + 하위 구성원 조회 |
| `RELATIVE_DEPT_HEAD` | 상대적 부서장 | 대상자 기준 부서장 조회 |
| `CUSTOMER_SUPER_ADMIN` | 최고 관리자 | 회사 관리자 조회 |

**resolve 실패 시 에러 처리 — resolve 과정에서 지정된 대상(조직, 사용자 등)을 찾을 수 없으면 해당 참조자는 승인 프로세스에서 자동 제외됨:** *(코드: tracking-approval/service/.../NewApprovalResolverOutboundPort.kt:112-122)*
- `EMPTY_DEPARTMENT` — 조직에 구성원 없음
- `DELETED_DEPARTMENT` — 삭제된 조직
- `NOT_FOUND_DEPARTMENT` — 존재하지 않는 조직
- `DELETED_USER` — 삭제된 사용자
- `RESIGNED_USER` — 퇴사한 사용자
- `LEAVE_OF_ABSENCE_USER` — 휴직 중인 사용자
- `COMPANY_JOIN_SCHEDULED_USER` — 입사 예정 사용자

**핵심:** resolve가 실패한 참조자는 승인 프로세스에서 **자동으로 제외**된다. *(코드: tracking-approval/service/.../TrackingApprovalServiceImpl.kt:123-128)*

```kotlin
// 매칭된 스텝의 참조자 중에서 resolve 실패한 것들을 필터링
referrers = matchSteps.matchedStep.referrers
    .filter { referrer ->
        // errorTargets에 포함된 referrer는 제외 (resolve 실패한 대상)
        errorTargets.none { referrer.resolveTarget.type == it.type && referrer.resolveTarget.value == it.value }
    }
    .map { referrer -> convertToModel(referrer) },
```

### 알림 설정 (notificationSetting)

각 참조자에는 `TrackingApprovalNotificationSetting`이 설정되어 있다. 이는 **관리자가 승인 정책 템플릿에서 설정하는 값으로, 해당 참조자가 언제 알림을 받을지를 제어**한다: *(코드: tracking-approval/model/.../ResolvedApprovalProcessModel.kt:97-101)*

| 설정 | 동작 |
|------|------|
| `OFF` | 알림 미발송 |
| `END_ONLY` | 승인 완료 시에만 발송 |
| `START_AND_END` | 승인 시작 + 완료 시 발송 |

> ⚠️ 이 설정은 **승인 정책(Approval Policy) 템플릿 레벨**의 설정이며, 사용자가 UI에서 개인 계정에서 보는 "참조 알림 활성화" 설정과는 다른 개념이다. 전자는 관리자가 통사적으로 참조자가 언제 알림을 받을지 정하는 것이고, 후자는 사용자 개인이 모든 알림 수신 여부를 제어하는 것이다.

### 이메일 발송 후 처리

메시지 큐 이후의 실제 이메일 발송은 **외부 notification 라이브러리** (`team.flex.notification` v3.78.0)에서 처리한다. *(코드: tracking-adapter/service/.../DefaultTrackingNotificationService.kt:26-51)*
이 단계에서 사용자별 "이메일 알림 안받기" 설정이 체크될 수 있으나, 해당 로직은 이 코드베이스 외부에 있음.

### notification 시스템 경계 분석 (추가 코드 분석)

**`notification_deliver` 테이블은 이 코드베이스에 존재하지 않음** — 외부 `team.flex.notification` 서비스에서 관리한다. *(코드베이스 전체 검색 — entity 클래스, repository, 테이블 매핑: 0건 발견, 2026-02-13)*

**이 코드베이스와 외부 서비스의 경계:**

```
[이 코드베이스 — timetracking-backend]
  ↓ SendNotificationCommandV1 → Kafka
[외부 — team.flex.notification 서비스]
  → Kafka 컨슘 → notification_deliver DB 기록 → SES 발송 → delivery 상태 업데이트
```

| 구분 | 이 코드베이스 | 외부 notification 서비스 |
|------|-------------|----------------------|
| 역할 | Kafka 메시지 생성·발행 | Kafka 메시지 소비·이메일 발송·기록 |
| `notification_deliver` | 접근 불가 | 관리 주체 |
| SES 발송 | 관여 안함 | 실제 SES API 호출 |
| 채널 선택 (email/push/slack) | 관여 안함 | 결정 주체 |

> 💡 **핵심 시사점**: notification_deliver 쿼리 결과(0건)와 SES 로그(발송 성공)의 불일치는 **외부 notification 서비스 내부의 데이터 적재 로직** 문제일 수 있다. 이 코드베이스에서는 Kafka 메시지를 발행하는 것까지만 책임지며, 이후는 외부 서비스의 영역이다.

**notification_deliver 불일치 가능 시나리오:**
1. notification_deliver는 특정 채널(push 등)만 기록하고, 이메일은 별도 경로로 발송되어 기록되지 않을 수 있음
2. 원래 쿼리의 `message_data_map LIKE` 조건이 부정확하여 레코드를 누락했을 수 있음
3. notification_deliver에 기록되기 전에 SES 발송이 먼저 완료되는 비동기 처리 구조일 수 있음

### 조사 과정 — 5 Whys 추론 체인

> 💡 **추론 과정 (단계별)** — 현상(알림 미수신)에서 출발하여 각 단계마다 "왜?"를 반복하며 원인을 추적:
>
> **[왜?] 744767이 알림을 받지 못했는가?**
> → 알림 자체가 생성되지 않았거나, 생성되었으나 전달되지 않음
>
> **1단계: 알림 아키텍처 파악**
> → 참조자 알림의 출발점은 어디인가? 승인 프로세스 기반임을 확인 *(코드: tracking-approval/service/.../ApprovalProcessStartedMessageContextFactory.kt:66-99)*
>
> **2단계: 참조자 resolve 흐름 추적**
> → 744767이 참조자 목록에 포함되어 있는가? 아니면 초기 단계에서 제외되었는가?
> - DEPT 타입 참조자(조직)는 `approvalPolicyResolveTargetInternalService.resolveAll()`을 통해 실제 조직 구성원을 조회 *(코드: tracking-approval/service/.../NewApprovalResolverOutboundPort.kt:34)*
> - resolve 실패 시(조직 삭제, 구성원 없음 등) 해당 참조자가 자동 제외되는 로직 *(코드: tracking-approval/service/.../TrackingApprovalServiceImpl.kt:123-128)*
>
> **3단계: notificationSetting 원천 확인**
> → 744767이 참조자 목록에 있어도 알림 설정이 OFF라면 알림이 발송되지 않을 수 있음
> - `notificationSetting`은 **관리자가 승인 정책(Approval Policy) 템플릿에서 설정**하는 값 *(코드: tracking-approval/dto/.../TimeTrackingApprovalPolicyCommandModel.kt:31-34)*
> - 승인 정책 조회 시 외부 approval 라이브러리의 `ApprovalPolicyMatchedStepQueryInternalService.matchStep()`에서 반환 *(코드: tracking-approval/service/.../step/NewApprovalStepOutboundPort.kt:35-50)*
> - 반환된 `NotificationSetting` → `TrackingApprovalNotificationSetting`으로 변환 *(코드: tracking-approval/service/.../ApprovalExtensions.kt:208-213)*
>
> **4단계: 전체 알림 발송 체인 추적 (완료)**
> → 744767이 최종 수신자 목록에 포함되는가? 참조자가 제외될 수 있는 **2개 지점**을 확인:
>
> | 지점 | 위치 | 제외 조건 | 결과 |
> |------|------|----------|------|
> | ① resolve 실패 필터링 | `TrackingApprovalServiceImpl.kt:123-128` | 참조자의 resolveTarget resolve가 실패하면 승인 프로세스 모델에서 제외 | FE에게 반환되는 승인 프로세스에 참조자가 아예 없음 → 등록 시에도 포함 안 됨 |
> | ② notificationSetting 필터링 | 외부 `approval-starter` 라이브러리 내부 | `notificationSetting`이 `OFF` 또는 `END_ONLY`이면 승인 시작(START) 알림에서 제외 | `onStartToReferrers(receiverUserIds)`에 해당 사용자 ID 미포함 |
>
> **5단계: 조건부 스텝(conditionalStep) 구조 발견 — 새로운 유력 가설 (가설 6)**
> 승인 정책에는 `defaultStep`과 `conditionalSteps`가 있으며, **각 스텝마다 별도의 referrers 목록**이 존재함을 확인 *(코드: tracking-approval/model/.../template/ApprovalPolicyDetailModel.kt)*

> - `conditionalSteps`: 특정 조건(예: 휴가 유형, 부서, 기간 등)에 따라 선택적으로 적용되는 스텝들. 조건이 충족되면 **그 스텝의 referrers만 사용**되고, defaultStep의 referrers는 무시됨
> - 외부 `ApprovalPolicyMatchedStepQueryInternalService.matchStep()`이 매칭된 단일 스텝을 반환 *(코드: tracking-approval/service/.../step/NewApprovalStepOutboundPort.kt:35-50)*

>   - 동일 건이면: 가설 7 확정 방향, flex 측 문제 아님
