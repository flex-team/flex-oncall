## 현재까지 파악된 내용

### 클라이언트 측 사후신청 버튼 노출 로직

이지선님이 분석한 클라이언트 측 로직[^3]:

1. `/api/v2/time-tracking/users/{userIdHash}/alternative-holidays/original-holiday-info` API로 휴일 정보 및 휴일대체 정보 조회
2. 응답에서 휴일대체 가능한 날 판별 — 휴일 정보 있음 && 휴일대체 안 된 휴일
3. **휴일대체 가능한 날 && 근무 편집 중 아님 && 이미 등록된 인정근무 존재** → 사후신청 버튼 노출

### 현재 대응

김한나님이 요승설[^4] 또는 근태관리 > 휴일대체에서 관리자가 직접 대체일을 지정하는 방안으로 고객에게 안내 완료[^5]

### 해결 방향 논의

- 이지선님: 사후신청 버튼 노출 시점에서 승인 발생 여부를 체크할 수 없으므로, 모든 사용자에게 안내를 해야 하는데 그러면 노이즈가 심함[^6]
- 이성환님: actor에게 안내하는 수준으로 끊을 수 없는지 질문[^7]
- 이지선님: 승인라인 리졸브[^8]는 `templateKey`[^9] 기반으로 수행되는데, 사후신청 버튼 노출 시점에서는 근무 편집 중이 아니라 리졸브 불가[^10]
- 이성환님: @yj.kim BE가 나서야 할 부분인지 확인 요청[^11]

## 원인 분석

**`original-holiday-info` API가 휴일근무 승인 발생 여부를 체크하지 않고 데이터를 반환하여, 승인이 불가능한 구성원에게도 사후신청 버튼이 노출되는 버그.**

### 핵심 근거

1. `CustomerUserAlternativeHolidayStatusLookUpService`[^12]는 휴일 목록과 대체 상태만 조회하며, 승인라인 리졸브를 수행하지 않음
2. `needAlternativeHolidayInfo()`[^13]는 휴일 조건(근로자의 날, 이미 대체됨, 스케줄근무 주휴일)만 체크하며, 승인자 존재 여부는 미검증
3. 1차 조직장 미배치 시 승인라인 리졸브가 `NOT_FOUND_RELATIVE_DEPT_HEAD` 에러를 반환[^14]하지만, 이 체크가 사후신청 버튼 노출 경로에는 포함되지 않음

### 가설 목록

| # | 가설 | 확인 방법 | 상태 |
|---|------|----------|------|
| 1 | `original-holiday-info` API가 승인 여부 미확인 → 사후신청 버튼 무조건 노출 | API 호출 흐름 전체 추적 | ✅ 확정 |
| 2 | FE 자체적으로 승인라인 체크 로직이 있으나 누락 | FE 코멘트 분석 | ❌ 소거 — FE에서는 `original-holiday-info` 응답만으로 버튼 노출 판단[^3] |
| 3 | 1차 조직장 미배치가 아닌 다른 조건에서도 동일 문제 발생 가능 | 승인라인 에러 케이스 분석 | ✅ 확정 — 퇴사자, 휴직자, 삭제된 부서 등 8가지 에러 사유 모두 동일하게 영향[^14] |

<summary>📋 조사 과정 상세</summary>

**API 호출 흐름 추적:**

```
UserAlternativeHolidayLookUpController.getOriginalHolidayForAlternativeHolidayInfoByUser()
  → UserAlternativeHolidayLookUpMappingService.getAlternativeHolidayInfoByUser()
    → CustomerUserAlternativeHolidayStatusLookUpService.getUserAlternativeHolidayStatus()
      → 휴일 목록 조회 (승인 체크 없음)
      → 대체 상태 조회
      → 응답 반환
```

이 체인 어디에도 `TrackingApprovalServiceImpl` 이나 승인라인 리졸브 호출이 없음.

**승인라인 리졸브는 별도 경로에서 수행:**

```
근무 등록 (WorkScheduleRegister)
  → WorkScheduleHolidayWorkApprovalCheckerLogic.check()
    → needAlternativeHolidayInfo() — 휴일 조건만 체크
    → 승인 생성 시 TrackingApprovalServiceImpl.createApprovals() 호출
      → approvalStepOutboundPort.matchStepsForProcess()
      → approvalResolverOutboundPort.resolver() — 여기서 1차 조직장 리졸브
```

즉, 승인라인 리졸브는 **근무 등록 플로우** 안에서만 수행되고, `original-holiday-info` 조회 시에는 수행되지 않음.

</details>

### 5 Whys

1. **왜 사후신청 버튼이 표시되는가?** → `original-holiday-info` API가 휴일대체 가능한 날짜를 반환하기 때문[^12]
2. **왜 승인 불가 구성원에게도 반환하는가?** → API가 승인라인 유효성을 체크하지 않기 때문[^12]
3. **왜 체크하지 않는가?** → 사후신청 버튼 노출 시점에서는 `templateKey` 기반 승인라인 리졸브가 설계상 분리되어 있기 때문[^10]
4. **왜 분리되어 있는가?** → 승인라인 리졸브는 근무 등록 플로우에서만 수행되도록 설계됨. `original-holiday-info` 는 휴일 정보 조회용 API로, 승인 모듈과 독립적으로 동작[^15]
5. **근본 원인** → 휴일대체 사후신청 기능 설계 시, "승인이 실제 발생하는지"에 대한 사전 검증 단계가 누락됨

### 스펙 vs 버그 판별

**버그** — 의도된 동작이 아님.

- 사후신청 버튼은 "휴일에 근무한 후 대체일을 신청"하는 기능인데, 승인라인이 유효하지 않으면 신청 자체가 불가
- 사용자 입장에서 신청이 된 것처럼 보이지만 실제로는 처리되지 않는 상황 → UX 결함
- 근로자가 휴일근무 보상(대체휴일)을 받지 못하는 결과로 이어질 수 있어 **근로자에게 불리** ⚠️

## 해결안

### 즉시 대응 (표면 원인 해결)

현재 CS 대응 유지 — 관리자가 근태관리 > 휴일대체에서 직접 대체일 지정[^5]

### 근본 해결 (재발 방지)

| # | 옵션 | 장점 | 단점 | 근로자 유불리 |
|---|------|------|------|--------------|
| A | **BE: `original-holiday-info` 응답에 승인 가능 여부 필드 추가** | FE가 필드 기반으로 버튼 숨김 처리 가능. 다른 승인 에러 케이스도 일괄 해결 | 승인라인 리졸브 호출 추가로 API 응답 시간 증가 가능 | 유리 ✅ |
| B | **BE: 사후신청 submit 시점에 승인라인 검증 + 적절한 에러 반환** | API 변경 최소화. submit 시 정확한 에러 메시지 제공 | 사후신청 버튼은 여전히 노출 → UX 혼란 잔존 | 중립 |
| C | **A + B 조합** | 버튼 숨김 + submit 시 방어 로직 이중 안전 | 구현 범위 가장 넓음 | 유리 ✅ |

> **권장: 옵션 A** — 근로자에게 유리하고, `original-holiday-info` API 응답에 `approvalAvailable: Boolean` 필드를 추가하는 것으로 FE에서 깔끔하게 처리 가능. 리졸브 성능이 우려되면 캐싱 고려.
