## 스펙/버그 판별

**판정: 버그 (시스템 설계 누락)**

- 연차 정책 변경 시 기존 촉진 데이터를 정리하는 것은 의도적으로 설계에서 빠진 것이 아니라, **고려되지 않은 엣지케이스**
- 촉진 대상 **진입** 시점의 필터링은 엄격하지만, 대상에서 **이탈**할 때의 정리가 누락됨
- 근로자(대표이사)에게 불필요한 법정 절차(연차촉진 문서 작성)를 요구하는 것이므로 수정이 필요

## 해결안

### 즉시 대응 (표면 원인 해결) — 이 고객 한정

Operation API 또는 DB 직접 패치로 4건의 PENDING_WRITE 이력을 CANCELED 처리:

```sql
-- 대상 확인 (DRY RUN)
SELECT id, boost_type, status, boosted_at, task_key
FROM annual_time_off_boost_history
WHERE customer_id = 56765 AND user_id = 229449 AND status = 'PENDING_WRITE';

-- CANCELED 처리
UPDATE annual_time_off_boost_history
SET status = 'CANCELED',
    canceled_at = UNIX_TIMESTAMP() * 1000,
    canceled_user_id = 0,
    last_modified_date = NOW(),
    last_modified_by = 'system'
WHERE customer_id = 56765 AND user_id = 229449 AND status = 'PENDING_WRITE';
```

> ⚠️ 이것만으로 홈피드 카드가 사라지는지는 추가 확인 필요. 촉진 취소 코드(`cancelAnnualTimeOffPromotionDocuments`)에서 TODO도 CANCELED 처리하므로, **Operation API를 통한 촉진 취소가 DB 직접 수정보다 안전**함.

### 근본 해결 (근본 원인 해결) — 재발 방지

**옵션 A**: 정책 매핑 변경 시 미완료 촉진 이력 자동 정리
- 장점: 원인을 직접 해결. 정책 변경과 동시에 정리되어 사용자 경험 일관성 보장
- 단점: 정책 변경 로직에 촉진 모듈 의존성 추가. 영향 범위 넓음

**옵션 B**: 스케줄러에서 PENDING_WRITE 이력 중 정책 변경된 건 자동 CANCELED 처리
- 장점: 기존 정책 변경 로직 수정 불필요. 배치로 일괄 정리
- 단점: 정책 변경 ~ 다음 스케줄러 실행 사이에 짧은 기간 홈피드에 표시될 수 있음

**옵션 C**: 홈피드 조회 시 현재 정책 기준 필터링 추가
- 장점: 데이터 정합성 유지하면서 표시만 제어
- 단점: 홈피드 조회 성능 저하 가능. 근본적 해결이 아닌 화면 수준 우회

### 부수 이슈: 델리에서 촉진 이력 미표시 (원인 확인됨)

```kotlin
}
```

미지급 연차 정책(`enabledAnnualTimeOffPolicy = false`)이면 연차 버킷이 존재하지 않으므로, **모든 촉진 이력이 숨김 처리됨**. 이것은 의도된 동작이나, 홈피드의 notification/TODO는 별도로 정리되지 않아 불일치 발생.
